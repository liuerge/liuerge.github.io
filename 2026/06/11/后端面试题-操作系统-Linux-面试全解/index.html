0x00 进程与线程

0. 进程有哪几种状态？Linux 内核中如何表示？

标准答案：
三态模型：就绪（Ready）→ 运行（Running）→ 阻塞（Blocked/Waiting）。Linux 内核中进程状态更加细分：

状态 内核符号 说明
TASK_RUNNING R 正在运行或就绪队列中等待调度
TASK_INTERRUPTIBLE S 可中断睡眠，等待资源（如 socket 数据），收到信号可唤醒
TASK_UNINTERRUPTIBLE D 不可中断睡眠，通常等待 IO 完成，信号无法唤醒（经典 D 状态问题）
TASK_STOPPED T 被信号暂停（SIGSTOP）
TASK_TRACED t 被 ptrace 跟踪（如 gdb 断点）
EXIT_ZOMBIE Z 僵尸进程，子进程已结束但父进程未调用 wait() 回收
EXIT_DEAD X 死亡状态，正在被回收

深入思考：

  • D 状态（Uninterruptible Sleep）：这是生产环境最常见的「假死」原因。D 状态进程不可被 kill -9 杀死，因为内核认为它在执行不可被打断的操作（如 NFS 挂起、磁盘 IO 阻塞）。排查方法：cat /proc/进程ID/stack 看内核调用栈。
  • Z 状态（Zombie）：少量僵尸进程无害，但大量堆积会占满进程表（pid_max 默认 32768），导致 Cannot fork 错误。根治方法是修复父进程逻辑，或使用 SIGCHLD 信号处理。临时清理只能杀父进程（kill -9 PPID），让 init（systemd）收养子进程并回收。
  • 生产踩坑：某次线上 NFS 挂载点不可达，导致所有读写该目录的进程卡在 D 状态，umount -f 都解不了，最终重启机器。监控系统应该加上 D 状态进程数量告警。

参考来源：https://man7.org/linux/man-pages/man7/signal.7.html

1. 上下文切换(Context Switch)的开销到底有多大？如何优化？

标准答案：
上下文切换分为两类：

1. 进程上下文切换：

  • 切换页表（TLB 失效）
  • 保存/恢复 CPU 寄存器（通用寄存器、PC、栈指针）
  • 切换内核栈
  • 调用调度器算法选择下一个进程
  • 开销：微秒级（2-10μs），但 TLB 失效后的缓存缺失影响可能持续上百微秒

2. 线程上下文切换：

  • 同一进程的线程共享虚拟地址空间，不切换页表
  • 只需保存线程私有数据（栈指针、寄存器），TLB 不掉
  • 开销比进程切换小约 50%-80%

深入思考：

  • 量化测试：Intel Xeon 2.4GHz 上，进程切换约 3-5μs，线程切换约 1-2μs。关键是切换后的缓存暖机（cache warm-up）时间：L1 cache miss 成本 ~10ns，L2 ~50ns，LLC ~100ns。
  • 如何检查上下文切换频率：
  • 优化手段（生产经验）：
    • 减少线程数：不是线程越多越好，推荐 2 * CPU核数（CPU 密集型）或稍多（IO 密集型）
    • 使用协程替代线程：协程是用户态切换，0.1-1μs 量级
    • 绑定 CPU taskset：减少进程在核间漂移造成的 cache miss
    • 使用 epoll/io_uring 替代每个连接一个线程的模型
  • 生产踩坑：某 Java 应用 32 核机器上开 200 个线程池，commit 峰值的 vmstat 显示 cs 列 500k+/s，sys 列 CPU 使用率飙到 40%（正常应 <10%），优化线程数到 64 后恢复正常。

参考来源：https://www.brendangregg.com/perf.html

2. Linux 线程模型（NPTL）详解 —— 用户级 vs 内核级实现

标准答案：
Linux 目前使用的线程实现是 NPTL（Native POSIX Thread Library），2003 年引入，替代了早期的 LinuxThreads。

核心模型：1:1 模型：

  • 每个用户态线程（pthread）对应一个内核调度实体（task_struct）
  • clone() 系统调用创建线程，共享虚拟地址空间、文件描述符等
  • 线程调度由内核完成，可以利用多核

对比三种模型：

模型 用户态:内核态 代表 优点 缺点
多对一（M:1） 多用户线程 → 1 内核线程 早期 Green Threads 切换快，用户态完成 无法利用多核，一个阻塞全组挂
一对一（1:1） 1 用户线程 → 1 内核线程 NPTL / Windows 真并行，一个阻塞不影响其他 切换需要陷入内核，开销大
多对多（M:N） 多用户线程 ⇔ 多内核线程 某些 Solaris 实现 两全其美 实现复杂，调度难，已边缘化

深入思考：

  • clone() 系统调用：创建线程实际调用 clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | ...)，与 fork() 的区别就在这些 flag：
    • CLONE_VM：共享地址空间（不共享则 = 进程）
    • CLONE_FILES：共享文件描述符表
    • 不加 CLONE_VM 且设新 PID = fork() 创建进程
  • 线程与进程的 top 区别：top -H 可以看到线程级视图，ps -T 也行。默认 top 在 Linux 下显示的是进程（不一定是线程 ID）。
  • 生产踩坑：pthread_create 失败通常不是内存不足，而是 pid_max 限制（32768）或 ulimit -u（用户最大进程数）。检查 cat /proc/sys/kernel/pid_max 和 ulimit -u。

参考来源：https://man7.org/linux/man-pages/man7/pthreads.7.html

3. 协程比线程好在哪里？goroutine / 虚拟线程 / 纤程的底层原理

标准答案：
协程（Coroutine）是用户态轻量级线程，由运行时而非内核调度。

核心优势对比：

维度 内核线程（OS Thread） 协程（Goroutine / Virtual Thread）
创建成本 ~1MB 栈空间预分配 ~2KB-4KB 初始栈，可动态扩缩
切换成本 ~1-2μs（内核态） ~0.1-0.5μs（用户态）
数量上限 几千 ~ 几万（受进程表限制） 百万级别（仅受内存限制）
调度器 内核 CFS 调度器 用户态 M:N 调度
blocking 操作 阻塞整个内核线程 可以挂起当前协程，复用线程

三种主流实现：

  1. Goroutine（Go）：

    • M:N 模型：M 个 goroutine 运行在 N 个 OS 线程上
    • GMP 调度器：G（goroutine）、M（machine=OS 线程）、P（processor=本地队列）
    • 栈初始 2KB，按需增长（分段栈方案，后改为栈复制）
    • go func() 编译后变成 runtime.newproc() 调用

  2. 虚拟线程（Java Project Loom，JDK 21+）：

    • M:N 模型，携程由 JVM 管理
    • 当虚拟线程执行阻塞操作（socket IO、锁）时，yield 回载体线程
    • 不能使用 synchronized 阻塞虚拟线程（会 pin 住载体线程）
    • 栈存储在堆上，可以 GC

  3. 纤程/协程（C++ Boost.Fiber / libco）：

    • 纯用户态切换，不依赖运行时
    • 使用 swapcontext() 或 makecontext() 或汇编手动保存寄存器
    • 腾讯 libco 被大量用于 C++ 后端服务

深入思考：

  • 协程不是银弹：CPU 密集型任务中协程没优势（内核线程分时已经够了），协程主要解决 IO 密集型场景下的线程资源浪费
  • 协程的「伪阻塞」陷阱：如果协程内部调用了一个真实阻塞的系统调用（如文件 IO），整个 OS 线程都被阻塞，该线程上的所有协程都会暂停。Go 的 runtime 用 netpoller 处理网络 IO，但对文件 IO 仍会阻塞线程，所以 Go 用独立的 os/exec 协程池处理文件操作
  • Java 虚拟线程的坑：synchronized 会 pin 载体线程，ThreadLocal 的内存泄露风险远超平台线程。生产建议：搭配 ReentrantLock 和结构化并发使用
  • 生产踩坑：某 Go 服务 goroutine 数飙到 200 万，OOM 了。排查发现一个无缓冲 channel 的发送者没有接收者，goroutine 永久阻塞。协程虽轻，但无限创建一样会死。加一个 goroutine 泄漏检测工具（pprof goroutine）很重要。

参考来源：https://go.dev/src/runtime/proc.go

4. 死锁的四个必要条件是什么？如何检测和预防？

标准答案：
四个必要条件（Coffman 条件），缺一不可：

  1. 互斥（Mutual Exclusion）：资源一次只能被一个进程使用
  2. 持有并等待（Hold and Wait）：进程持有资源时，还在等待其他资源
  3. 不可剥夺（No Preemption）：资源不能被强制剥夺，只能由持有者主动释放
  4. 循环等待（Circular Wait）：存在一个进程-资源的循环等待链

预防策略（破坏上述任意一个条件）：

条件 预防方法 实际代价
互斥 使用无锁编程（Lock-Free CAS） 不一定适用所有场景
持有并等待 一次性申请所有资源 资源利用率低，可能饥饿
不可剥夺 申请失败时释放已持有的资源 回滚成本高
循环等待 对资源编号，按序申请 需要全局编号，难以维护

  • 银行家算法：Dijkstra 提出的避免死锁的经典算法，实际项目中用的极少（需要预先知道最大资源需求量，太理想化）
  • Linux 内核的死锁检测：lockdep（Lock Dependency Validator），内核编译时开启 CONFIG_PROVE_LOCKING，运行时自动检测潜在死锁。驱动开发必备。
  • 数据库中的死锁：InnoDB 用等待图（wait-for graph）检测死锁，超时阈值 innodb_lock_wait_timeout（默认 50s）。死锁发生时 MySQL 选择回滚持锁最少的事务。
  • 生产踩坑：某服务调用了 A → B → C → A 的回环 RPC，3 个服务各自有一个数据库连接池，A 占着连接等 B 的 RPC 响应，B 占着等 C，C 占着等 A。这不是数据库死锁而是应用层的「分布式死锁」。解法：RPC 调用设置全局超时（timeout < 连接池获取超时），并且不要在一个事务中发 RPC 请求。

0x01 内存管理

5. 虚拟内存、分页与分段——现代 OS 如何管理内存？

标准答案：

虚拟内存（Virtual Memory）：

  • 每个进程看到独立的线性地址空间（Linux 64 位下 128TB 用户空间）
  • 通过 MMU（内存管理单元）将虚拟地址映射到物理地址
  • 好处：隔离保护、简化编程、允许 > 物理内存的虚拟内存（通过 swap）

分页（Paging）：

  • 虚拟内存和物理内存切成固定大小的页（Linux 默认 4KB，Huge Page 2MB/1GB）
  • 页表（Page Table）维护 VA→PA 映射，TLB（Translation Lookaside Buffer）是页表的硬件缓存
  • 多级页表（x86-64 下 4 级：PML4 → PDPT → PD → PT）节省内存

分段（Segmentation）：

  • 按逻辑单位（代码段、数据段、栈段）划分，每段大小可变
  • x86 实模式依赖分段，保护模式下已退化（段寄存器仅用于权限检查）
  • Linux 采用平坦内存模型：所有段的基地址为 0，段限长为整个地址空间，分段实际失效

深入思考：

  • 页表开销：4KB 页面下，一个进程映射 4GB 虚拟内存，4 级页表需要约 4KB（PML4）+ 512*8B（PDPT）+ …，总约 8-10MB。如果是大页（2MB），只需 3 级页表，总约 8KB，TLB 覆盖范围扩大 512 倍。
  • 透明大页（THP）：Linux 内核自动将 4KB 小页合并为 2MB 大页，对 Redis/MongoDB 等数据库反而有害（大页导致内存分配延迟不稳定），数据库生产环境建议关闭：echo never > /sys/kernel/mm/transparent_hugepage/enabled
  • ASLR（地址空间布局随机化）：内核随机化 mmap 基地址、栈基地址等，增加漏洞利用难度。编译 gcc -no-pie 可以关闭。
  • 生产踩坑：某次 Java 应用在容器中频繁 OOM，监控显示物理内存远没到 limit，但进程被 OOM Killer 杀了。原因是容器限制的是 cgroup 内存，而 Java 用 cat /proc/meminfo（宿主机视图）计算可用内存，设置 -Xmx 过大。解法：用 -XX:+UseCGroupMemoryLimitForHeap（JDK 8u131+）或 -XX:MaxRAMPercentage=75.0（JDK 10+）。

参考来源：https://www.kernel.org/doc/html/latest/admin-guide/mm/index.html

6. 页面置换算法有哪些？LRU 在生产中的缺陷和改进

标准答案：
当发生缺页中断（Page Fault）且物理内存满时，OS 需要选择一页换出。

经典算法对比：

算法 策略 实现复杂度 性能 实际使用
FIFO 最早进来的先换出 低 差（Belady 异常：增加页面数命中率反而下降） 不用
Optimal 未来最久不用的先换出 不可能实现 理论最优 基准对照
LRU（最近最少使用） 长时间未访问的先换出 中 好 近似实现
Clock（二次机会） 环形扫描，给第二次机会 低 接近 LRU Linux 内核使用
LFU（最不经常使用） 访问次数最少的先换出 中 较好，但热点数据难适应 特殊场景
ARC 自适应缓冲区替换 高 好，动态平衡 ZFS、文件系统

Linux 内核实际使用：

  • LRU 近似算法：Linux 维护 active_list 和 inactive_list，每个页面有 PG_referenced 和 PG_active 标志位
  • 双链策略：页面先在 inactive list 中观察，如果被访问一次则移到 active list；active list 中的页面如果长时间未被访问，降级回 inactive list
  • Kswapd：后台守护进程，当 free 内存低于 watermark[low] 时开始回收，watermark[min] 触发直接回收

深入思考：

  • LRU 的真实缺陷：
    1. 扫描开销：全链表扫描 O(n)，生产环境中百万级页面，LRU 成本太高
    2. 顺序扫描污染：一次大文件顺序读（如 cp、grep 大文件），会把缓存全刷成自己的数据，挤掉热点数据。Linux 用 inactive list + 临时标记 缓解
    3. Cache 抖动（Thrashing）：工作集 > 物理内存时，页面不断换入换出，系统吞吐量骤降
  • 改进方案：
    • Clock 算法：环形缓冲区，每个页维护一个 access bit，指针遍历时若 access bit=1 则清 0 并跳过，若 access bit=0 则换出。时间复杂度 O(1)。
    • MySQL Buffer Pool 的 LRU：InnoDB 使用改进 LRU，将链表分为 young（5/8）和 old（3/8），新页面插入 old 区头部，第二次访问才移入 young 区。防止一次性扫描污染缓存。
    • 2Q 算法：维护两个队列（A1in/A1out/Am），被淘汰页面保留其 key 在 A1out 中，若再次访问则直接入 Am，对周期性访问模式友好。
  • 生产踩坑：某 Redis 实例启用 allkeys-lru 淘汰策略，业务批量写入 1000 万 key 时热 key 被大量淘汰。原因是 Redis 的 LRU 是采样近似 LRU（maxmemory-samples, 默认 5），批量写入时采样的全部是新 key，导致热 key 被误淘汰。增大 samples 到 10-15 缓解，或改用 allkeys-lfu。

参考来源：https://www.kernel.org/doc/gorman/html/understand/understand013.html

    3. 进程「超量申请」+ 同时使用（overcommit_memory=1）

    • 设置 vm.overcommit_memory = 2（禁止超额分配），但可能影响 fork()
    • 关键服务设置 oom_score_adj = -500 ～ -1000
    • 设置 vm.panic_on_oom = 1（直接 panic 重启，比随机杀进程更可控）
    • 使用 cgroup 内存限制（让 OOM Killer 只杀容器内的进程）
    • 监控 cat /proc/meminfo | grep 'Committed_AS' 观察超额分配总量
  • Java 应用特别：JVM 的堆外内存（DirectBuffer、Metaspace、Native Stack）容易被忽略，OOM Killer 看到的 RSS 可能远大于 Xmx。排查：pmap -x PID | grep -i anon 看匿名映射大小。
  • 生产踩坑：某次线上 Kafka Broker 频繁 OOM 被杀，oom_score 显示 Java 进程 900+ 最高分。调试发现 Kafka 依赖的 GlobalKTable（RocksDB）在堆外使用了大量内存 + Kafka 的 socket buffer 默认 100MB，加上 Java heap 后 RSS 超过 cgroup limit。解法：调小 server.socket.buffer，设置 RocksDB 的 block cache 大小。

参考来源：https://man7.org/linux/man-pages/man5/proc.5.html

8. mmap 原理详解——零拷贝和共享内存的基石

标准答案：
mmap（Memory Map）将文件或设备映射到进程的虚拟地址空间，对内存的读写等于对文件的读写。

工作流程：

四种使用模式：

模式 flag 效果 典型应用
共享映射 MAP_SHARED 修改回写文件，多进程共享 文件 IO、IPC
私有映射 MAP_PRIVATE 写时复制（COW），修改不写回文件 动态库加载、fork()
文件映射 fd != -1 映射文件到内存 大文件读取、数据库
匿名映射 fd = -1, MAP_ANONYMOUS 分配大块内存，不关联文件 heap 扩展、跨进程共享内存

深入思考：

  • mmap vs read/write：
    • 传统 read/write：文件 → Page Cache（DMA）→ 用户缓冲区（CPU copy），至少一次 DMA + 一次 CPU copy
    • mmap：文件 → Page Cache（DMA），省掉了一次 CPU copy
    • 小文件（<64KB）：read/write 更快（mmap 有缺页中断开销）
    • 大文件（>64KB）：mmap 更快，且访问模式更灵活（随机读写优势显著）
  • MAP_SHARED 的注意事项：
    • 多进程共享内存时需加锁（futex、mutex 放共享内存里）
    • 文件截断（ftruncate）后继续访问映射区域会收到 SIGBUS 信号
    • mmap 文件大小不能为 0（否则 SIGBUS）
    • mremap() 可以动态扩展映射区域
  • 生产踩坑：
    1. mmap 与 OOM：mmap 映射大文件 + MAP_PRIVATE 时，写操作触发 COW，如果进程 fork() 了，子进程修改映射页会复制，可能消耗大量物理内存。某些 Java 应用使用 mmap 做文件缓存时踩到这个坑。
    2. tcmalloc/jemalloc 与 mmap：这两个内存分配器在分配大块内存（> 256KB）时会直接用 mmap 匿名映射，而不是用 brk()。如果频繁分配释放，会导致大量 VMA（虚拟内存区域）碎片。
    3. MongoDB WT 引擎：WiredTiger 引擎使用 mmap 管理数据文件，如果内存压力大触发大量 page fault，性能会急剧下降。MongoDB 4.0+ 开始减少对 mmap 的依赖，引入 wiredTigerCacheSize 参数控制缓存。
  • mmap 的限制：
    • vm.max_map_count（默认 65530）限制了 mmap 映射段数量。Elasticsearch 大量使用 mmap 映射索引文件，生产环境要调高：sysctl -w vm.max_map_count=262144
    • 32 位系统下地址空间有限（3GB），mmap 大文件困难
    • mmap 的页对齐要求（必须按 PAGE_SIZE 偏移量映射）

参考来源：https://man7.org/linux/man-pages/man2/mmap.2.html

9. BIO/NIO/AIO/IO 多路复用——五种 IO 模型详解

标准答案：
Unix 下有五种 IO 模型（参考 Stevens《UNP》）：

模型 阻塞点 并发能力 回调/通知方式 代表
阻塞 IO（BIO） 整个 read/write 过程阻塞 低（1 连接 1 线程） 无 传统 Socket
非阻塞 IO（NIO） 轮询时占用 CPU 中（需要 busy loop） 无 O_NONBLOCK
IO 多路复用 select/poll/epoll_wait 处阻塞 高（单线程管理千级连接） 内核通知就绪 epoll/select/kqueue
信号驱动 IO（SIGIO） 信号到达前不阻塞 中 SIGIO 信号 SIGIO
异步 IO（AIO） 完全不阻塞 高 完成通知/callback io_uring/IOCP

BIO（Blocking IO）：

  • 缺点：连接数上千时线程过多，上下文切换开销巨大
  • 适用：连接数少（< 100）、低并发场景

NIO（Non-blocking IO）：

  • 优点：单线程可处理多个连接
  • 缺点：轮询消耗 CPU，数据就绪时如何感知？不能空闲时阻塞等待
  • 适用：几乎不用原生的 NIO，而是用 NIO + 多路复用（如 Java NIO Selector）

AIO（Asynchronous IO）：

  • 真实异步：read/write 调用立即返回，由内核完成操作后通知
  • Linux 上：AIO 实现（libaio）需要 O_DIRECT（绕过 Page Cache），不符合大部分场景
  • io_uring（Linux 5.1+）是真正的高效异步 IO

深入思考：

  • BIO 到 NIO 的演进本质：从「每个连接一个线程」到「事件驱动模型」的转变，核心是意识到 线程是昂贵的资源
  • select/poll 的局限性：
    • select：FD_SET 上限 1024，每次调用要把 fdset 从用户态拷贝到内核态
    • poll：无上限但仍是 O(n) 扫描，大连接数下性能差
  • epoll 的突破：O(1) 就绪通知 + 零拷贝 fd 到内核（epoll_create 时建立 epoll 实例，后续只需拷贝变化的 fd）
  • 生产踩坑：Java 的 NIO.2（AIO）在 Linux 上用的是 epoll 模拟而非真实 AIO，性能不如 Windows 的 IOCP。直到 JDK 17 的 io_uring 实验性支持才接近真实 AIO。

参考来源：https://man7.org/linux/man-pages/man7/epoll.7.html

10. epoll/poll/select 对比——百万并发背后的功臣

标准答案：

指标 select poll epoll
时间复杂度 O(n) O(n) O(1)（就绪列表）
最大 fd 数 1024（FD_SETSIZE） 无上限（受系统限制） 无上限
fd 传递方式 全量拷贝到内核 全量拷贝到内核 内核维护事件表，epoll_ctl 增量更新
工作模式 LT（水平触发） LT（水平触发） LT + ET（边缘触发）
就绪通知 全部返回，遍历所有 fd 全部返回，遍历所有 fd 只返回就绪的 fd，内核通过回调机制将就绪 fd 加入链表
系统调用 每次都需要 每次都需要 只需 epoll_wait
适用连接数 < 1000 < 10000 > 10000（百万级）

epoll 核心数据结构：

边缘触发（ET）vs 水平触发（LT）：

  • LT（默认）：只要 fd 还有数据可读，epoll_wait 会一直返回
    • 编程简单，不易出错
    • 但效率稍低（数据没读完会重复通知）
  • ET：fd 从不可读变成可读时才通知一次
    • 必须循环 read 直到 EAGAIN，否则数据读不完
    • 效率更高（减少 epoll_wait 调用）
    • 需要搭配非阻塞 IO 使用（防止最后一次 read 阻塞）

深入思考：

  • epoll 的瓶颈在哪？
    • 单个 epoll 实例可以管理数十万 fd，但 epoll_wait 返回后应用程序需要遍历就绪列表执行回调
    • 惊群（Thundering Herd）：多线程/多进程同时 epoll_wait 同一个 fd，accept 时多个进程被同时唤醒但只有一个能成功。解法：SO_REUSEPORT（Linux 3.9+）让内核做负载均衡，或 EPOLLEXCLUSIVE（Linux 4.5+）只唤醒一个等待者。
    • 对于 100 万连接场景，内存占用约：epoll 实例 ~ 每个 fd 约 160 字节（64 位系统），100 万约 160MB
  • 为什么 epoll 比 select/poll 快这么多？
    1. 省去了每次把全部 fd 从用户态拷贝到内核态
    2. 避免了 O(n) 全部扫描（epoll 只返回就绪 fd）
    3. 回调注册机制避免轮询
  • 生产踩坑：
    • epoll_wait 返回 EINTR 表示被信号中断，需要重新调用
    • ET 模式下漏读数据（没有循环到 EAGAIN）会导致 fd 永远不再通知
    • ngx_epoll_module（Nginx）就是用 ET 模式 + 事件驱动实现 100 万并发的经典范例
    • Redis 也用 epoll（Linux）实现单线程管理数万连接

参考来源：https://man7.org/linux/man-pages/man7/epoll.7.html

11. 零拷贝（Zero-Copy）技术原理——Kafka 每秒写百万条消息的秘密

标准答案：
零拷贝是指数据在从磁盘到网卡的传输过程中，避免在用户态和内核态之间多次拷贝。

传统 IO 路径（四次拷贝 + 四次上下文切换）：

合计：2 次 CPU 拷贝 + 2 次 DMA 拷贝 + 4 次上下文切换

零拷贝实现方式：

1. mmap + write（三次拷贝）：

  • 仍有一次 CPU 拷贝，但省去了 read() 的第一次用户态拷贝
  • Java 中使用：FileChannel.map() → 直接操作 ByteBuffer

2. sendfile（两次拷贝）：

  • 0 次 CPU 拷贝！ 仅在支持 SG-DMA 的硬件上可实现
  • 如果硬件不支持，退化为 1 次 CPU 拷贝

3. splice（两次拷贝，管道实现）：

  • 在 Page Cache 和 Socket Buffer 之间通过管道缓冲区传输
  • 无需硬件支持

4. io_uring（Linux 5.1+，真正的异步零拷贝）：

  • 支持 read/write/sendfile/splice/fsync 等几乎所有操作
  • 通过 IORING_OP_PROVIDE_BUFFERS 实现 buffer 共享
  • 是 IO 领域的未来方向

深入思考：

  • Kafka 如何利用零拷贝：
    • Kafka 的消息存储是连续追加的 Segment 文件
    • 消费者拉取消息时，Kafka 使用 FileChannel.transferTo()（底层 sendfile）直接将数据从磁盘发到网卡
    • 避免了 Page Cache → JVM Heap 的拷贝，也避免了 JVM GC 压力
    • Kafka 的 100 万 msg/s 很大程度上依赖零拷贝 + 页缓存 + 顺序 IO
  • Nginx 的零拷贝：
    • sendfile on; 配置启用零拷贝
    • 静态文件（图片、CSS、JS）都通过 sendfile 发送
    • 实测静态文件吞吐量提升 2-3 倍
  • Netty 的零拷贝：
    • 网络传输层面：FileRegion（transferTo 封装）
    • 内存层面：CompositeByteBuf 合并缓冲区避免拷贝
    • 序列化层面：直接操作 DirectBuffer，避免堆内拷贝堆外
  • 零拷贝的适用边界：
    • 数据不需要在用户态处理（如只是透传）时效果最好
    • 如果需要修改数据（加 Header、加密、压缩），零拷贝就无法用
    • 小文件（< 几 KB）零拷贝收益不明显，上下文切换开销摊不掉
  • 生产踩坑：
    1. sendfile 对 NFS/CIFS 等网络文件系统支持不好
    2. 使用 mmap 做零拷贝时，大文件的 page fault 可能引入延迟抖动
    3. Kafka 2.6+ 支持基于 fetch.max.bytes 的零拷贝，但如果开启 SSL 加密则退化为传统拷贝（加密过程需要用户态介入）

参考来源：https://www.kernel.org/doc/html/latest/filesystems/splice.html

12. IO 多路复用 + Reactor/Proactor 模式——高性能网络框架的核心

标准答案：

Reactor 模式（同步事件分发）：

  • 事件多路分离器（demultiplexer）等待事件（epoll_wait/select）
  • 事件就绪后，分发给对应的 Handler（读就绪 → callback 读数据）
  • 典型架构：
  • 代表：Redis (单 Reactor)、Netty (主从 Reactor)、Nginx (多 Reactor)

Proactor 模式（异步事件分发）：

  • 应用发起异步操作后，由 OS 完成实际操作后通知应用
  • 不需要应用手动 read/write（OS 已经完成）
  • 代表：Windows IOCP（真正的 Proactor）、Linux 的 Boost.Asio（模拟 Proactor）

Reactor 的三种变体：

模式 结构 优点 缺点 代表
单 Reactor 单线程 一个 Reactor 做所有事 简单，无并发问题 CPU 核利用率低，耗时操作阻塞所有 Redis
单 Reactor 多线程 Reactor 分发，Worker 线程池处理 CPU 利用率高 Reactor 单点瓶颈 Netty 默认配置
主从 Reactor 多线程 Main Reactor 负责 accept，Sub Reactor 负责 IO 分担 Reactor 压力 实现复杂 Netty (bossGroup/workerGroup)

深入思考：

  • 为什么 Redis 6.0 之前坚持单 Reactor 单线程？
    • 内存操作本身就是 CPU 密集的「快操作」
    • 多线程引入的锁竞争、缓存失效可能抵消并发收益
    • 性能瓶颈不在 CPU 而在网络 IO（epoll 已经高效）
    • 6.0+ 引入 IO 多线程处理网络读写（仍是 Reactor，只是 IO 阶段多线程）
  • Netty 的主从 Reactor 设计：
    • bossGroup 通常 1 个线程（accept 比较轻量）
    • workerGroup 线程数通常 = CPU 核数的 2 倍
    • 每个 EventLoop 绑定一个 Selector，负责 N 个 Channel
  • 生产踩坑：
    1. Netty 默认的 io.netty.leakDetectionLevel 是 SIMPLE，生产环境禁用或改为 ADVANCED 以便检测 DirectBuffer 泄漏
    2. Reactor 线程中绝不能做阻塞操作（如 JDBC 查询、RPC 调用），必须丢给业务线程池。Netty 的 ChannelHandler 中若误用阻塞操作，会导致该 EventLoop 上的所有 Channel 无响应
    3. epoll 空轮询 bug：某些 JDK 版本的 epoll 实现存在 bug，select 返回 0 但仍会继续循环，CPU 100%。Netty 有专门的 IntervalWakeUp 策略和 DeadConnectionWatcher 应对

参考来源：https://netty.io/wiki/user-guide-for-4.x.html

13. CPU 爆满如何排查——从 top 到 perf 的全链路指南

标准答案：
五步排查法：

第一步：确认 CPU 情况

  • 重点看 us（用户态）vs sy（内核态）比例
  • sy > 30% 通常是系统调用频繁（上下文切换/锁竞争）
  • wa > 30% 说明 IO 瓶颈导致 CPU 等待

第二步：找消耗 CPU 的进程/线程

第三步：分析热点代码路径

第四步：生成火焰图（Flame Graph）

  • 火焰图 X 轴是采样频率（宽度 = CPU 占用率），Y 轴是调用栈
  • 平顶（table top）：函数本身占用 CPU 高（需要优化）
  • 高层塔（tall tower）：调用栈深，可能包含锁等待或系统调用

第五步：针对不同原因的对策

CPU 模式 原因 解决
us 高 用户态计算密集 代码优化、增加并行度、编译优化（-O2/-O3）
sy 高 系统调用频繁/锁竞争 减少锁粒度、CAS 替代锁、减少系统调用
wa 高 IO 等待 换 SSD、调整 IO 调度器、异步 IO
hi 高 硬件中断密集 网卡多队列（RSS）、中断绑定（irqbalance）
st 高 宿主机超分配 云厂商 sidecar 或同宿竞争

深入思考：

  • CPU load average 误解：load average > CPU 核数不等于负载高。Linux 下 load 包含 D 状态进程，如果 IO 卡住（D 状态堆积），load 会飙升但 CPU 是空闲的（idle 高）。所以必须结合 %idle 和 %wait 判断。
  • perf 的采样原理：基于硬件 PMU（Performance Monitoring Unit）的采样中断，默认 1000Hz。采样并非连续，而是统计概率，采样数不够可能导致函数未被命中。
  • 生产踩坑：
    1. 自旋锁（Spinlock）消耗：top 显示 sy 高，perf top 看到 _raw_spin_lock 或 futex 在顶位。原因是多线程争用锁时大量时间花在等待上。解法：减少锁竞争或改用读写锁（rwlock）。
    2. st 模式（Steal Time）高：云服务器上被 hypervisor（宿主机）偷走了 CPU 时间片。检查是否有同宿大流量虚拟机竞争。KVM/Xen 环境下 st > 10% 需要提工单。
    3. Java 应用 CPU 高但看不出热点：JIT 编译后的 code 在 perf 中地址是随机化的，需要 Java 的 -XX:+PreserveFramePointer 配合 perf-map-agent 才能解析出 Java 方法名。
    4. perf 采样过密导致系统变慢：perf record -F 99（99Hz）一般安全，-F 1000 在某些内核版本会引入可感知的性能开销。

参考来源：https://www.brendangregg.com/perf.html

IO 高通常表现为 top 中的 wa（iowait）列大于 30%，系统吞吐下降，但 CPU 并不繁忙。

1. iostat 看整体

关键指标：

17. Cgroup（Control Group）详解——容器资源隔离的基石

标准答案：
Cgroup 是 Linux 内核提供的资源隔离和限制机制，是实现容器化的基石之一（另一个是 Namespace）。

Cgroup v1（传统版本）：

  • 每个子系统独立的层级结构（hierarchy）
  • 子系统包括：
    • cpu：CPU 时间片分配（cpu.shares、cpu.cfs_period_us、cpu.cfs_quota_us）
    • memory：内存限制（memory.limit_in_bytes、memory.oom_control）
    • blkio：块设备 IO 限制（blkio.throttle.read_bps_device）
    • cpuset：CPU 核绑定
    • devices：设备访问控制
    • freezer：进程冻结/恢复
    • net_cls/net_prio：网络分类和优先级
    • pids：进程数量限制
    • hugetlb：大页限制

Cgroup v2（统一版本，Linux 4.5+）：

  • 所有子系统统一在一个 hierarchy 中
  • 更安全的接口设计（写时复制）
  • 支持 cgroup.kill 原子杀死整个 cgroup 下的进程
  • 不可逆的升级（一旦启用 v2，无法部分回到 v1）
  • 主要改进：
特性 v1 v2
架构 多 hierarchy 单 hierarchy
内存记账 page counter LRU-based
IO 控制 独立 blkio 统一 IO controller（io.max）
CPU 控制 cpu + cpuset 分开 cpu 整合 cpuset 策略
BPF 集成 不支持 原生支持（cgroup-bpf）
cgroup.events 无 有（OOM 通知，populated 通知）

深入思考：

  • 容器的「资源视图」伪造：即使 cgroup 限制了容器只能使用 1GB 内存，/proc/meminfo 看到的仍是宿主机内存总量。这就是为什么 Java 在容器中容易踩 Xmx 的坑——JVM 读取 /proc/meminfo 判断可用内存。修复：JDK 8u131+ 的 -XX:+UseCGroupMemoryLimitForHeap（v1）或 -XX:MaxRAMPercentage=75.0（通用）。
  • Cgroup v2 在 Kubernetes 中的普及：
    • Kubernetes 1.24 开始 GA 支持 cgroup v2
    • containerd 1.5+ 默认使用 cgroup v2
    • 必须启用 systemd.unified_cgroup_hierarchy=1 内核参数
    • 主流发行版：Fedora 31+、Ubuntu 21.10+、Debian 11+ 默认 cgroup v2
  • 内存 cgroup 的 OOM 行为：
    • v1：memory.oom_control 设为 1 时 OOM 杀死 cgroup 内进程
    • v2：自动 OOM，可通过 cgroup.events 监听 OOM 事件
    • 生产防护：使用 memory.high（软限制）而不是 memory.max（硬限制），触发内存回收而非 OOM kill
  • 生产踩坑：
    1. v1 的 memory.move_charge_at_immigrate：进程迁移到新的 memory cgroup 时，旧的内存不会自动跟随。如果限制新 cgroup 内存小于已有内存，会立即 OOM。v2 修复了这个问题。
    2. IO 限制的统计不准：v1 的 blkio 只统计 direct IO（O_DIRECT 打开的文件），对 Page Cache（缓存的读写）不准确。v2 的 io 控制器基于 page cache 统计，更精确。
    3. Docker 默认使用的 cgroup 驱动：cgroupfs vs systemd。如果宿主机使用 systemd 作为 init 系统，建议 Docker daemon --exec-opt native.cgroupdriver=systemd，否则资源紧张时 systemd 和 cgroupfs 可能产生冲突。

参考来源：https://www.kernel.org/doc/Documentation/cgroup-v2.txt

21. 容器 CPU 限制的真相——CFS 配额 vs 核心绑定

标准答案：
Docker/Kubernetes 中的 CPU 限制有两种方式：CFS 配额（软限制） 和 cpuset（硬绑定）。

CFS（Completely Fair Scheduler）配额：

实现原理：cpu.cfs_period_us（默认 100000μs = 100ms）和 cpu.cfs_quota_us

cpuset 绑定：

两种方式的对比：

维度 CFS Quota cpuset
灵活度 高（可分配小数核） 低（必须整数核）
性能一致性 差（throttle 导致抖动） 好（独占核）
CPU 利用率 高（可超卖） 中（核被独占，不可超卖）
上下文切换 高频跨核切换，TLB miss 多 核内切换，cache 亲和性好
适合场景 一般业务、微服务 延迟敏感型、高频交易

深入思考：

  • CFS 的 throttle 问题：
    • cpu.stat 中的 nr_throttled 和 throttled_time 指示节流频率
    • 100ms period 在高性能场景下太粗：一个线程排队 99ms 才获得 1ms 运行时间，频繁 throttle
    • 解决方案：
    • Kubernetes 1.20+ 支持 cpu.cfs_period_us 配置（--cpu-cfs-period）
  • CPU Manager（Kubernetes）：
    • static 策略：将 Guaranteed QoS Pod 绑定到特定 CPU 核（cpuset）
    • 减少上下文切换抖动，适合数据库、DPDK 等服务
    • 代价：CPU 核被占用，不可用于其他 Pod
  • 生产踩坑：
    1. CPU Throttling 导致性能陡降：某 Java 应用设置 --cpus=0.5，CFS quota 每 100ms 只有 50ms CPU，但实际上应用需要 70ms 才能完成一次请求处理，结果每 100ms 里有 50ms 被 throttle，吞吐量直接减半。解法：调大 quota 或改用 --cpuset-cpus。
    2. 弹性伸缩的误区：HPA 基于 CPU 使用率扩缩容时，如果 Pod 被 throttle 了（CPU 受限），实际 CPU 使用率反而会被压低（因为用不了），导致 HPA 感知不到压力，不会扩容。此时应看 cpu_cfs_throttled_seconds_total（Prometheus metric）而不是 CPU 使用率。
    3. Java Runtime.availableProcessors() 的 Bug：旧版 JDK 中此 API 返回的是宿主机 CPU 核数而非容器限额（cgroup.cfs_quota_us/cfs_period_us），导致 ForkJoinPool 启动过多线程。JDK 10+ 通过 UseContainerSupport 修复。
