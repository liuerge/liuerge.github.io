Redis 高级面试全解

0x00 基础与数据结构

0. Redis 为什么这么快？

标准答案：

  1. 纯内存操作：数据在内存中，读写纳秒级
  2. 单线程模型（< 6.0）：避免上下文切换和锁竞争；6.0+ 网络 IO 多线程，命令执行仍单线程
  3. IO 多路复用：epoll/kqueue 事件驱动，单线程处理大量连接
  4. 高效数据结构：SDS（动态字符串）、ziplist、skiplist、quicklist 等定制化结构
  5. 避免内存碎片：jemalloc/tcmalloc 内存分配器优化

深入思考：

  • CPU 瓶颈：纯内存场景 CPU 通常在 70% 以下；若 CPU 爆满往往是慢查询、BigKey、或者 AOF rewrite 频繁 fork 导致的上下文切换
  • 单线程的代价：单个慢查询（如 KEYS *、HGETALL 大 hash）会阻塞所有请求，务必用 SCAN 替代
  • IO 多线程：Redis 6.0+ io-threads 仅处理网络读写，命令执行仍是单线程，实测在 10w+ QPS 场景提升约 30%-50%
  • 对比延伸：Memcached 多线程模型在纯 KV 场景有优势，但 Redis 因丰富的数据结构胜出

参考来源：https://redis.io/docs/reference/optimization/latency/

1. String 的底层实现 —— SDS

标准答案：
Redis 的 String 底层是 SDS（Simple Dynamic String），而非 C 字符串。SDS 结构如下：

优点：

  • O(1) 获取长度：C 字符串需要遍历，SDS 直接读 len
  • 杜绝缓冲区溢出：修改前检查空间，不够则扩容
  • 减少内存重分配：预分配策略（len < 1MB → 加倍；len >= 1MB → 多分配 1MB）
  • 二进制安全：不依赖 \0 判断结尾，可存图片/序列化对象

深入思考：

  • SDS v2 演进：Redis 7.0+ 使用 sdshdr8/16/32/64 按字符串长度选择合适的 header，节省内存——短字符串用 uint8_t len 而非 uint64_t
  • 内存碎片：APPEND 频繁操作导致预分配空间膨胀，可用 SETRANGE 替代场景下更省内存
  • 🔥 生产踩坑：某团队用 String 存 50MB 的用户画像 JSON，触发 active-defrag 导致 CPU 爆满，迁移到 Hash 分片后解决

参考来源：https://redis.io/docs/data-types/strings/

2. Hash 的底层实现 —— ziplist / hashtable

标准答案：
Redis Hash 有两种底层编码：

  • ziplist（压缩列表）：当 field-value 数 < 512 且单个元素 < 64 字节时使用，连续内存块存储，节省空间
  • hashtable：元素超过阈值时转换为哈希表，O(1) 读写

深入思考：

  • Listpack 替代 ziplist：Redis 7.0+ 引入 listpack 替代 ziplist，解决 ziplist 的连锁更新问题（中间插入导致后续所有节点的 prevlen 字段需重新计算）
  • Hash 分片 BigKey：当 Hash 有数万 field 时，即使是 hashtable 也可能成 BigKey。建议按业务前缀分片，如 user:info:{userId % 1000}，每片控制 field < 1000
  • 对比延伸：HINCRBY 做计数器聚合非常高效，但注意原子性——没有事务保护时多次 HINCRBY 不是事务性操作

参考来源：https://redis.io/docs/data-types/hashes/

3. List 的底层实现 —— quicklist

标准答案：
Redis List 底层是 quicklist（3.2+），由多个 ziplist 节点组成的双向链表：

  • 每个节点是一个 ziplist，存储连续元素
  • list-max-ziplist-size 控制每个 ziplist 大小（默认 8KB）
  • 两端操作 O(1)，中间操作 O(n)

深入思考：

  • quicklist vs linkedlist+ziplist：3.2 之前是 ziplist 或 linkedlist 二选一；3.2 之后 quicklist 统一两者
  • 🔥 生产踩坑：消息队列用 LPUSH + BRPOP 实现，某团队设置 list-max-ziplist-size -2（允许单节点 8KB），结果消息体平均 100KB → 每个 ziplist 只存 1-2 个元素，quicklist 退化为纯链表，内存和性能骤降
  • 替代方案：Redis Stream（5.0+）更适合消息队列场景，支持消费者组、ACK、持久化

参考来源：https://redis.io/docs/data-types/lists/

4. Set 的底层实现 —— intset / hashtable

标准答案：

  • intset（整数集合）：当所有元素是整数且数量 < 512 时，有序紧凑数组存储
  • hashtable：元素超出阈值或含有非整数元素时切换

深入思考：

  • intset 升级策略：当插入一个更大范围的整数（如 int16 → int32），所有元素会升级，但不支持降级
  • SADD 的批量操作：SADD key m1 m2 m3 比多次 SADD 减少网络往返，但注意单命令过大会阻塞

参考来源：https://redis.io/docs/data-types/sets/

5. ZSet 的底层实现 —— ziplist / skiplist + dict

标准答案：

  • ziplist：元素数 < 128 且单个元素 < 64 字节时使用
  • skiplist + dict（跳表+字典）：跳表维护 score 有序性（O(logN) 范围查询），字典维护 member→score 映射（O(1) 查分）

深入思考：

  • 为什么用 skiplist 而非平衡树：跳表实现简单、范围遍历友好（双向链表）、调整平衡时只需局部更新、平均 O(logN) 性能接近红黑树
  • ZI score 相同：ZSet 的 score 相同时按 member 字典序排列（二进制比较）
  • 🔥 生产踩坑：某排行榜场景，用 ZINCRBY 每 5 秒更新一次分数，跳表频繁调整节点位置导致 CPU 使用率 85%+。优化：把实时分存在 Hash，每 10 分钟批量重新计算 ZSet
  • 内存优化：ZRANK 内部通过 dict 查 member 再走 skiplist 算排名，多层 skiplist 指针（Redis 7.0+ 每层 50% 概率上升）内存占用约 8-12 字节/元素

参考来源：https://redis.io/docs/data-types/sorted-sets/

6. BigKey 问题 —— 发现与处理

标准答案：
BigKey 指单个 key 包含大量数据（如 10MB+ String 或数万成员的集合），危害：

  • 阻塞命令：DEL 大 key 会阻塞单线程
  • 网络带宽：GET 一个 50MB 的 key 传输耗时高
  • 内存不均：集群模式下数据倾斜

发现手段：

  1. redis-cli --bigkeys 扫描（采样）
  2. DEBUG OBJECT key 查看 serializedlength
  3. 慢查询日志发现 DEL/GET 耗时久
  4. RAM 分析工具：rdb-tools、redis-rdb-parser

处理方式：

  • 拆分：Hash 按 field 分片、List 分段、大 String 压缩
  • 异步删除：UNLINK（Redis 4.0+）后台线程回收内存
  • lazyfree-lazy-server-del yes 自动开启惰性删除

深入思考：

  • 🔥 生产踩坑：某团队误将一张 20 万条 SQL 查询结果序列化后存入 String（约 80MB），导致 Redis 内存 OOM。排查过程：redis-cli --bigkeys 未发现（String 超 10MB 即触发告警但未设置），后通过 MONITOR 发现 GET large_key_xxx 耗时 2.3 秒
  • UNLINK 并非完全无阻塞：后台线程回收大内存时，如果内存碎片严重，free() 可能短暂卡顿

参考来源：https://redis.io/docs/reference/optimization/latency/

7. Hot Key 发现与处理

标准答案：
Hot Key 指某 key 的 QPS 极高，导致单节点 CPU/带宽打满。

发现手段：

  1. redis-cli –hotkeys（Redis 4.0+）：基于 OBJECT freq 的 LFU 信息
  2. MONITOR 采样：抓取 10s 流量，统计访问频率（生产慎用，对性能有影响）
  3. 客户端统计：在 Jedis/Lettuce 调用层埋点
  4. 代理层：Codis/Twemproxy 可在代理层统计

处理方案：

  1. 本地缓存：客户端或代理层用 Caffeine/Guava Cache 缓存热点数据
  2. 读写分离：主节点写，从节点读，但要接受秒级不一致
  3. 打散扩容：在 key 后加随机后缀（如 hotkey:0 ~ hotkey:99），客户端轮询或 hash 访问
  4. 限流降级：保护下游数据库

深入思考：

  • 🔥 生产踩坑：某直播弹幕场景，room:9999:danmu List 被 10 万用户同时读取，单节点带宽打满。解决：本地缓存弹幕列表，每 500ms 刷新一次，带宽骤降 95%
  • Hot Key vs BigKey：Hot Key 是访问频率问题，BigKey 是数据体量问题，但常同时出现（大活动时热点商品既是 Hot Key 又是 BigKey）

参考来源：https://redis.io/topics/rediscli

0x01 持久化

8. RDB 与 AOF 的原理与对比

标准答案：

维度 RDB AOF
原理 快照全量持久化当前内存数据 追加写命令到日志文件
触发方式 save（阻塞）、bgsave（fork子进程）、自动策略 always/everysec/no（fsync 策略）
文件大小 小（二进制压缩） 大（写命令文本，即使开启 AOF rewrite）
恢复速度 快（直接加载快照） 慢（重放命令）
数据安全 丢失上次备份后的全部数据 everysec 最多丢 1s 数据
对性能影响 bgsave fork 子进程，COW 写时复制 always 严重影响写性能

混合持久化（Redis 4.0+）：aof-use-rdb-preamble yes

  • AOF rewrite 时生成的 AOF 文件前半段是 RDB 格式（全量内存快照），后半段是增量 AOF 日志
  • 结合 RDB 的恢复速度和 AOF 的数据安全性

深入思考：

  • fork 的代价：Redis 主进程 fork() 时需拷贝页表，内存越大 fork 越慢（12GB 内存 fork 约 300ms-1s）；期间 redis 无法响应请求
  • COW 写时复制：fork 后父子进程共享内存页；子进程写 RDB，父进程若有写请求则触发页复制。写量大的场景下 COW 会导致内存翻倍风险
  • 🔥 生产踩坑：某电商 Redis 实例 32GB 内存，开启 save 900 1 自动 RDB；大促期间写流量暴涨，bgsave fork 耗时 2.3s，期间请求全部堆积；更严重的是 COW 导致 RSS 从 32GB 飙升到 58GB，OOM killer 杀掉 Redis 进程
  • 优化方案：
    1. 关闭自动 RDB，仅保留 AOF
    2. 大实例（>20GB）考虑用 Redis Cluster 分片
    3. 替换物理机用更快的 CPU 降低 fork 时间

参考来源：https://redis.io/docs/management/persistence/

9. AOF Rewrite 阻塞分析

标准答案：
AOF rewrite 过程：

  1. BGREWRITEAOF 触发
  2. 主进程 fork 子进程
  3. 子进程将内存数据写为最小命令集到临时文件
  4. 父进程将增量写命令缓存到 AOF rewrite buffer
  5. 子进程完成，父进程将 buffer 追加到临时文件
  6. 原子替换旧 AOF 文件

阻塞点：

  1. fork() 阻塞：主进程需要拷贝页表
  2. 写入阻塞：当 aof-use-rdb-preamble yes 时，父进程写 RDB preamble 可能因内存置换卡顿
  3. fsync 阻塞：appendfsync always 每次写操作 fsync，在磁盘繁忙时阻塞

深入思考：

  • 🔥 生产踩坑：某 AOF rewrite 执行 15 分钟未完成，排查发现 IO 利用率 100%（同机有日志采集程序写同块磁盘）。隔离方案：Redis 使用单独 SSD 或云盘
  • 优化建议：
    1. appendfsync everysec 平衡数据安全与性能
    2. 大实例 auto-aof-rewrite-min-size 64GB 避免频繁 rewrite
    3. 使用 no-appendfsync-on-rewrite yes，Rewrite 期间不 fsync，防 IO 竞争

参考来源：https://redis.io/docs/management/config/

10. RDB vs AOF 生产选型

标准答案：

场景 推荐方案
缓存场景，可接受丢数据 RDB 或关闭持久化
数据重要，不能丢数据 AOF everysec
大实例（>20GB） AOF（RDB fork+COW 风险大）
快速恢复 混合持久化（RDB preamble + AOF 增量）
极致性能 关闭持久化，仅做主从复制

深入思考：

  • Redis 数据可靠性：多数生产环境 开启 AOF everysec + 混合持久化，同时至少 1 个从节点异步备份
  • 备份策略：每日将 RDB 备份到 OSS/S3，保留最近 7 天备份，用于灾难恢复

参考来源：https://redis.io/docs/management/persistence/

11. 🔥 生产踩坑：RDB 快照文件损坏恢复

标准答案：
某团队因磁盘坏道导致 RDB 文件损坏，Redis 重启失败：

恢复过程：

  1. 使用 redis-check-rdb 工具修复或提取可读数据
  2. 若损坏不可修复，使用从节点 RDB 恢复
  3. 若无从节点，启用 AOF 日志重放
  4. 最终方案：磁盘替换 + 从备份恢复 RDB + 增量同步

教训：

  • 不要只依赖单机持久化文件，务必有从节点或远程备份
  • 定期 redis-check-rdb 检查 RDB 文件完整性

参考来源：https://redis.io/topics/rediscli

0x02 集群与高可用

12. 主从复制原理 —— 全量同步与增量同步

标准答案：
全量同步（PSYNC 2）：

  1. 从节点发送 PSYNC ? -1
  2. 主节点执行 bgsave 生成 RDB
  3. 主节点将 RDB 发送给从节点，同时持续写入 replication buffer
  4. 从节点清空自身数据，加载 RDB
  5. 主节点将 replication buffer 增量发送给从节点

增量同步（PSYNC ）：

  1. 从节点发送自己的 replid 和 offset
  2. 若主节点 replication backlog 中仍有数据，则增量同步
  3. 若 offset 已超出 backlog，则触发全量同步

深入思考：

  • repl_backlog_size：默认 1MB，网络不稳定时容易触发全量同步。计算公式：repl_backlog_size = 主库写速率 × 最大断连时间 × 2
  • 🔥 生产踩坑：某主从间网络闪断 30s，因 repl_backlog_size 设置过小（1MB），写速率 10MB/s → 30s 产生 300MB 数据远超 backlog，从节点触发全量同步。大 RDB（8GB）传输过程中再次闪断 → 反复全量同步 3 次。解决方案：repl_backlog_size 512MB，并优化网络稳定性
  • PSYNC 2（Redis 4.0+）：主节点重启后 replid 不变，从节点可增量同步，避免全量

参考来源：https://redis.io/docs/management/replication/

13. Sentinel 故障转移 —— 原理与脑裂

标准答案：
Redis Sentinel 提供高可用保障：

  1. 监控：Sentinel 每秒向主从节点发送 PING
  2. 主观下线（sdown）：单个 Sentinel 节点超时未收到回复
  3. 客观下线（odown）：quorum 个 Sentinel 同意主观下线
  4. 领导者选举：Sentinel 之间用 Raft 类似协议选举 leader 执行故障转移
  5. 故障转移：选一个从节点升为主、修改配置、通知其他从节点

脑裂问题：
网络分区时，Sentinel 认为主节点挂了，提升从节点为新主；旧主恢复正常后继续接受写入，导致数据不一致。

解决方案：

  • min-replicas-to-write 1：至少有 1 个从节点确认时才可写
  • min-replicas-max-lag 10：从节点延迟超过 10s 时禁止主节点写入

深入思考：

  • 🔥 生产踩坑：某公司使用 Redis Sentinel 3 节点部署，因网络交换机故障导致主节点被隔离，Sentinel 提升新主；旧主交换机恢复后重新接入集群，但 min-replicas-to-write 0 导致旧主仍在接受写入，恢复后出现大量键冲突
  • Sentinel vs Cluster：Sentinel 适合主从架构高可用，Cluster 适合数据分片 + 高可用

参考来源：https://redis.io/docs/management/sentinel/

14. Redis Cluster 数据分片 —— 16384 slots

标准答案：

  • 16384 个哈希槽：对 key 用 CRC16 计算后模 16384 决定归属槽
  • 三主三从最小集群，主节点负责读写，从节点备份
  • 槽迁移：CLUSTER SETSLOT 系列命令实现平滑迁移
  • MOVED 重定向：客户端请求的槽不属于当前节点，返回 MOVED 错误 + 目标节点地址（直连模式）
  • ASK 重定向：槽迁移过程中，请求落在正在迁出的槽时，返回 ASK 错误 + 目标节点地址

为什么是 16384 不是更多的槽？

  • 心跳包用 2KB 位图表示所有槽（16384 bits = 2KB）
  • 若用 65536 个槽，心跳包 → 8KB，集群规模大时网络开销激增
  • 16384 在常见集群规模（<1000 节点）下够用

深入思考：

  • 🔥 生产踩坑：批量操作（MGET、MSET、管道）跨槽时，Redis Cluster 不支持跨槽原子操作。某团队用 MGET 批量查 500 个 key，发现 key 分布在 4 个槽上，MGET 返回 CROSSSLOT 错误。解决方案：用 Hash Tag {user:123} 确保相关 key 在同一槽
  • 重分片对性能的影响：槽迁移期间，源节点需 dump+restore 数据，CPU 和网络开销大；建议低峰期执行，每次最多迁移 100 个槽

参考来源：https://redis.io/docs/management/scaling/

15. Redis Cluster 重分片与 ASK/MOVED 处理

标准答案：
重分片（Resharding）：

  1. 确定目标节点和需要迁移的槽
  2. 在目标节点执行 CLUSTER SETSLOT <slot> IMPORTING <source-node-id>
  3. 在源节点执行 CLUSTER SETSLOT <slot> MIGRATING <target-node-id>
  4. 从源节点获取 slot 中的 key：CLUSTER GETKEYSINSLOT <slot> <count>
  5. 逐个迁移 key：MIGRATE <target-host> <target-port> <key> 0 5000
  6. 迁移完成后通知所有节点：CLUSTER SETSLOT <slot> NODE <target-node-id>

MOVED 与 ASK 的区别：

维度 MOVED ASK
场景 槽已永久迁移 槽正在迁移中
客户端缓存 更新本地槽映射 不更新
重试 直接连新节点 先发 ASKING 再操作

深入思考：

  • 🔥 生产踩坑：某团队上线自动重分片脚本，未检查集群负载，在 11:00 高峰期执行迁移。结果源节点 CPU 100%（MIGRATE 需要序列化 + 网络传输 + 反序列化），槽迁移耗时 2 小时，大量请求超时。教训：重分片必须在低峰期 + 控制迁移速率 + 监控 CPU 和网络
  • Redis Cluster 客户端要求：客户端必须实现 MOVED/ASK 重定向逻辑。常见客户端：Jedis Cluster（Java）、StackExchange.Redis（C#）、lettuce（Java+响应式）、redis-py-cluster（Python）

参考来源：https://redis.io/docs/management/scaling/

16. Redis Cluster 分片哈希标签（Hash Tag）

标准答案：
Hash Tag 是一种将多个 key 强制分配到同一槽的机制：

  • 语法：{tag}，如 user:123:profile、user:123:orders
  • Redis 只取 {} 内的 tag 计算 CRC16 决定槽位
  • 常用于需要事务/MULTI 操作的关联数据

深入思考：

  • Hot Key 问题加剧：所有带相同 tag 的 key 集中在同一槽，若 tag 设计不当（如所有 key 用 {global}），会导致单节点过热
  • Tag 长度限制：{} 内至少 1 字符，{} 外至少 1 字符，否则不作为 Hash Tag
  • 嵌套花括号：只有第一个左括号和第一个右括号之间的内容生效：{user:{123} → tag 为 user:{123

参考来源：https://redis.io/docs/reference/cluster-spec/

0x03 缓存设计

17. 缓存穿透 / 击穿 / 雪崩 —— 区分与解决方案

标准答案：

问题 描述 解决方案
穿透 请求不存在的数据，每次绕过缓存打到 DB 布隆过滤器 + 空值缓存（TTL 短）
击穿 热点 key 过期，大量请求同时打 DB 互斥锁（SETNX）、热点 key 不过期、本地缓存
雪崩 大量 key 同时过期，DB 被打垮 TTL 加随机值、多级缓存、限流降级

深入思考：

  • 布隆过滤器的局限性：有误判率（假阳性），不可删除元素。对于需要删除的场景不可用，考虑布谷鸟过滤器
  • 互斥锁的尺度：锁的粒度可按 key 维度（防击穿）而非全局锁；用 SETNX key value EX 5 实现，避免死锁
  • 🔥 生产踩坑：某电商双 11 活动，商品详情页缓存 TTL 统一设为 30 分钟，零点大促同时触发缓存刷新，DB 连接数瞬间打爆。修复方案：TTL = 1800 + random(0, 300)，错开过期时间

参考来源：https://redis.io/docs/manual/patterns/

18. 缓存一致性 —— Cache Aside / Read-Through / Write-Behind

标准答案：

Cache Aside（旁路缓存）：

  • 读：先查缓存 → 未命中查 DB → 写入缓存
  • 写：先更新 DB → 再删除缓存（延迟双删：先删缓存 → 更新 DB → 异步再删一次）

Read-Through：

  • 缓存层代理数据库查询，请求只与缓存交互
  • 缓存未命中时，缓存层自行加载 DB 数据

Write-Behind（Write-Back）：

  • 写请求只写缓存层，缓存异步批量刷回 DB
  • 写性能极高，但数据丢失风险大（缓存挂了数据丢失）

深入思考：

  • 为什么是「删缓存」而非「更新缓存」：

    1. 并发写场景下，更新缓存可能会导致旧值覆盖（A 写 DB=10，B 写 DB=20，B 先更新缓存=20，A 后更新缓存=10）
    2. 删缓存后，下次读一定会从 DB 拉最新值
    3. 懒加载策略，只在真正需要时才加载

  • 延迟双删的时序问题：

    • A 删缓存 → A 更新 DB → B 读缓存未命中 → B 读 DB（旧数据） → B 写缓存（旧数据） → A 第二次删缓存
    • 解决方案：第二次删除用延迟（500ms），确保 B 的写缓存操作在前

  • 🔥 生产踩坑：某支付订单场景，更新订单状态后删缓存，因删除失败（Redis 超时）导致缓存中一直是「已支付」状态。修复：用 binlog（Canal）监听 MySQL 变更，异步刷新缓存

参考来源：https://redis.io/docs/manual/patterns/

0x04 分布式锁

21. SETNX + Lua 实现分布式锁

标准答案：
加锁：

  • NX：key 不存在时才设置
  • EX 30：自动过期 30s，防死锁
  • value 用 uuid 标识锁的持有者

解锁（Lua 脚本，保证原子性）：

为什么要用 Lua：GET 值判断 + DEL 操作不是原子的，两个操作之间锁可能过期被其他线程获取

深入思考：

  • 锁续期（Watch Dog）：业务执行时间超过锁 TTL 怎么办？
    • Redisson 的实现：后台线程每 10s（锁 TTL 的 1/3）检查一次，如锁仍由当前线程持有，则 PEXPIRE key 30000
    • 客户端挂了？锁自动过期释放
  • 🔥 生产踩坑：未设置锁 TTL 导致死锁（服务器宕机、进程被 kill -9）；设置了 TTL 但业务执行超时导致并发执行
    • 优化：锁 TTL = 业务预估最大时间 × 2，配合 Watch Dog
  • 单点问题：SETNX 只在一个 Redis 节点上加锁，主从异步复制下主节点挂了，从节点未同步锁信息 → 其他线程可以加锁成功 → 锁失效

参考来源：https://redis.io/docs/manual/patterns/

22. Redlock 原理与争议

标准答案：
Redlock（Redis Distributed Lock）由 Redis 作者 Antirez 提出，解决单点锁的可靠性：

加锁步骤：

  1. 获取当前时间戳
  2. 依次向 N 个（通常 5 个）独立的 Redis 节点加锁，设置相同 key+value+TTL
  3. 计算加锁耗时，如果成功加锁节点数 ≥ N/2 + 1 且总耗时 < TTL，则加锁成功
  4. 加锁失败则向所有节点发送解锁

争议（Martin Kleppmann vs Antirez）：

  • Martin Kleppmann（《DDIA》作者） 认为 Redlock 不安全：

    1. 时钟漂移：依赖时间戳，时钟跳跃会导致锁过期判断错误
    2. GC/停顿：客户端加锁后发生 STW GC，锁自动过期，另一个客户端获得锁 → 第一个客户端 GC 恢复后以为自己仍持有锁
    3. Fencing Token 问题：Redlock 没有生成递增的 fencing token，无法在资源层做合法性校验

  • Antirez 反驳：

    1. 时钟漂移在生产环境中可管理（NTP 不会跳变太大）
    2. GC 停顿是通用问题，所有分布式锁都会遇到
    3. Redlock 适用于大多数业务场景，不需要 fencing token

深入思考：

  • 生产建议：
    • 大多数场景 SETNX 单节点锁就够了
    • 真的需要强一致锁？用 ZooKeeper（临时顺序节点 + Watch 机制）或 etcd（租约 + revision）
    • Redlock 在业务复杂度上不值得，除非你真的有 5 个独立 Redis 节点
  • 🔥 生产踩坑：某团队使用 Redlock 实现分布式锁，但因磁盘 IO 瓶颈导致 Redis 节点间时钟 skew 超过 2s，锁频繁失效。最终回退到 SETNX + 主从哨兵方案
  • Fencing Token：ZooKeeper 的 zxid 可充当 fencing token，资源层可校验 token 递增

参考来源：

  • https://redis.io/docs/manual/patterns/
  • https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html
  • http://antirez.com/news/101

23. 🔥 生产踩坑：分布式锁超时与续期

标准答案：
某定时任务场景，每 2 分钟执行一次，用 Redis 分布式锁确保单点执行：

  1. 线程 A 加锁成功（TTL=10s）
  2. 数据库查询 + 业务计算耗时 15s
  3. 第 10s 锁自动过期
  4. 线程 B 加锁成功（以为 A 已结束）
  5. 第 15s A 完成业务，执行 Lua 解锁
  6. A 解锁了谁的锁？—— A 的锁已过期，B 的锁被 A 误删！

解决方案：

  1. Watch Dog 自动续期（Redisson）：锁持有期间定期续期，直到业务结束释放
  2. Lua 解锁检查 value 是否是自己：每个线程用唯一 uuid 标识锁持有者
  3. 合理设置 TTL：基于业务预估最大耗时 × 2

参考来源：https://github.com/redisson/redisson/wiki/8.-distributed-locks-and-synchronizers

24. 分布式锁选型对比

标准答案：

方案 强一致性 可靠性 性能 复杂度
SETNX + Lua 低（主从异步复制丢锁） 中 极高 低
Redlock 中（有争议） 较高 高 高
ZooKeeper 高（ZAB 协议，顺序节点） 高 中 中
etcd 高（Raft 共识） 高 中 中

深入思考：

  • 性能优先级：SETNX 单节点 ≈ 5-10μs，ZooKeeper ≈ 10-20ms（需要磁盘 fsync）
  • ZooKeeper 的惊群效应：大量客户端监听同一节点时，节点变更会触发所有客户端同时加锁 → 大量网络风暴
  • etcd 租约（Lease）：与 TTL 类似，但 etcd 通过 gRPC keepalive 续期，比 Redis Watch Dog 更可靠

28. OOM 排查

标准答案：
现象：OOM command not allowed when used memory > 'maxmemory'

排查步骤：

  1. INFO memory 查看内存分配
  2. redis-cli --bigkeys 检查大 Key
  3. MEMORY STATS 查看内存细分（overhead、数据集、碎片）
  4. 检查 maxmemory-policy 配置是否合理
  5. MEMORY DOCTOR 诊断内存健康度

常见原因：

  1. BigKey 未拆分
  2. 内存碎片率高（used_memory_rss / used_memory > 1.5）
  3. 没有设置 maxmemory 或 maxmemory-policy noeviction
  4. 业务写入速率远大于过期/淘汰速率

解决方案：

  1. 设置合理的 maxmemory（物理内存 70%-80%）
  2. 选择合适的淘汰策略：allkeys-lru（缓存场景）、volatile-ttl（有时效数据）
  3. activedefrag yes 开启自动碎片整理
  4. 升级到 jemalloc 5.x 减少碎片

深入思考：

  • 🔥 生产踩坑：某实例 maxmemory=8GB，used_memory 显示 6GB，但 RSS 达到了 11GB → OOM killer 杀进程。原因：activedefrag no，碎片率 > 1.8。开启 activedefrag yes 后 RSS 降回 7.5GB
  • COW 导致的伪 OOM：bgsave 时 COW 翻倍内存，即使 used_memory < maxmemory，RSS 也可能超物理内存。关闭自动 bgsave 或调大物理内存

参考来源：https://redis.io/commands/info/

  • 复杂操作 → Lua script 或服务端聚合

深入思考：

  • 🔥 生产踩坑：某定时任务每 5 分钟执行 KEYS * 清理过期 session，导致 Redis CPU 持续 80%+。原因：KEYS 是 O(N) 操作，遍历 200 万 key 耗时约 500ms，阻塞所有其他请求
  • slowlog-max-len 设置过小：日志被冲掉，无法追溯问题。建议设置 10000 以上，低频消费到外部存储

参考来源：https://redis.io/commands/slowlog/

30. CPU 爆满排查

标准答案：
排查流程：

  1. redis-cli INFO cpu：检查 used_cpu_sys / used_cpu_user 比例
  2. redis-cli --hotkeys：找 Hot Key
  3. SLOWLOG GET 50：看慢查询
  4. INFO commandstats：各命令耗时占比
  5. 检查 AOF/RDB 是否正在执行
  6. INFO clients：检查客户端连接数

常见原因与解决：

原因 特征 解决
慢查询 KEYS * / BigKey 改用 SCAN，拆分 BigKey
Hot Key 单 key QPS 极高 本地缓存/打散
fork 负载 bgsave/bgrewriteaof 关闭自动持久化/调大实例
客户端风暴 连接数 > 10000 连接池/限流
内存碎片整理 activedefrag 导致 限制 defrag 速度
复杂 Lua 脚本 执行时间长 拆分/优化脚本

深入思考：

  • 🔥 生产踩坑：某 Redis 实例 CPU 100%，排查发现 INFO commandstats 中 ZREMRANGEBYSCORE 占比 70%。原因：某定时任务每 10 秒执行一次 ZREMRANGEBYSCORE 清空过期数据（ZSet 有 80 万成员），每次耗时 300ms。优化：改为每日一次，或用 Sorted Set 的 TTL 机制
  • Linux 层面的排查：top 看 redis-server 进程 CPU，perf top -p <pid> 看热点函数，/proc/<pid>/status 看上下文切换

参考来源：https://redis.io/docs/reference/optimization/latency/

31. 🔥 生产踩坑：Redis 内存无限增长

标准答案：
某日志采集系统使用 Redis List 作为消息缓冲，未设置 maxmemory 和淘汰策略。业务高峰期日志量暴涨，Redis 内存从 4GB 增长到 28GB，最终 OOM 宕机。

根因分析：

  1. maxmemory 0（无限制）
  2. maxmemory-policy noeviction（默认）
  3. 消费端消费速度 < 生产速度，List 持续堆积
  4. 没有监控告警

修复方案：

  1. maxmemory 12GB：限制 Redis 最大内存
  2. maxmemory-policy allkeys-lru：开启淘汰策略
  3. 改为 Redis Stream 替代 List，支持消费者组和消息过期
  4. 给 List 设置上限：LTRIM key 0 99999 保留最近 10 万条
  5. 增加消费端并发度

教训：

  • 所有 Redis 实例必须设置 maxmemory
  • 缓存场景使用 allkeys-lru，有时效数据使用 volatile-ttl
  • 监控 used_memory / maxmemory 比率，超过 80% 告警

参考来源：https://redis.io/docs/manual/configuration/

32. Redis 优化 Checklist

标准答案：

配置层面：

  • ✅ maxmemory 设置为物理内存的 70-80%
  • ✅ maxmemory-policy 按业务选择淘汰策略
  • ✅ activedefrag yes（Redis 4.0+）开启自动碎片整理
  • ✅ hz 10（默认）适当调高到 20 以加速过期 key 清理
  • ✅ lazyfree-lazy-eviction yes 惰性淘汰
  • ✅ 关闭 save 自动 RDB，仅用 AOF everysec

数据层面：

  • ✅ 拆分 BigKey（Hash 分片、List 分段、大 String 压缩）
  • ✅ Hot Key 本地缓存或打散
  • ✅ 合理的 TTL + 随机偏移防雪崩
  • ✅ 避免 KEYS *、SMEMBERS、HGETALL 全量操作

集群层面：

  • ✅ Cluster 每个节点内存 < 20GB（fork 风险）
  • ✅ repl_backlog_size 按写速率计算
  • ✅ min-replicas-to-write 和 min-replicas-max-lag 防脑裂
  • ✅ 客户端 Hash Tag 合理规划，防 Hot Slot

运维层面：

  • ✅ slowlog-log-slower-than 10000 监控慢查询
  • ✅ 内存利用率告警（>80%）
  • ✅ 定期 redis-cli --bigkeys 扫描
  • ✅ 备份 RDB 到异地存储
