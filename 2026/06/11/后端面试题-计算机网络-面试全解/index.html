后端面试题：计算机网络 面试全解

本文覆盖计算机网络核心知识点：TCP/IP 协议栈、HTTP 演进、HTTPS/TLS、DNS/CDN、WebSocket/gRPC、生产实战与踩坑。面向高级研发面试，每道题附带深入思考与源码/生产视角。

0x00 TCP/IP

0. TCP 三次挥手的详细过程？为什么是三次不是两次或四次？

标准答案：

  1. 第一次（SYN）：客户端发送 SYN 报文，seq=x，进入 SYN_SENT
  2. 第二次（SYN+ACK）：服务端回复 SYN+ACK，seq=y，ack=x+1，进入 SYN_RCVD
  3. 第三次（ACK）：客户端回复 ACK，seq=x+1，ack=y+1，双方进入 ESTABLISHED

为什么不是两次：

  • 两次握手无法防止历史连接（旧的 SYN 被服务端接受后建立连接，客户端已经关闭了）
  • 三次握手通过第三次 ACK 携带 seq 让服务端确认客户端是否 alive；两次握手服务端无法区分历史 SYN

为什么不是四次：

  • 三次已经能可靠交换双方初始序列号（ISN），第四次是冗余的
  • SYN + ACK 合并为一个报文就是优化过的三次

深入思考：

  • SYN Flood 攻击：攻击者伪造大量 SYN，服务端回复 SYN+ACK 后收不到 ACK，半连接队列被填满。防护：tcp_syncookies（1 表示开启）、增大 tcp_max_syn_backlog、tcp_synack_retries 调小
  • TCP Fast Open（TFO）：第二次握手时就可以携带数据，减少 RTT，适用于短连接场景（Chrome 默认开启）
  • 初始序列号（ISN）：RFC 793 基于时钟生成，RFC 6528 改为 HMAC 哈希防止序列号预测攻击

生产踩坑：

  • 高并发短连接场景（Nginx 反代、Redis 客户端），三次握手的 SYN+ACK 耗时导致 CPU 飙升，解决方案：启用 tcp_tw_reuse + tcp_fastopen，或使用连接池
  • SYN 超时默认 63 秒（tcp_syn_retries=5，指数退避 3/6/12/24/48+），用户在 5 秒内网络闪断会卡死 60 秒，建议 tcp_syn_retries=2 缩短到 9 秒

参考来源：RFC 793 / RFC 6528 / https://datatracker.ietf.org/doc/html/rfc793

1. TCP 四次挥手的详细过程？TIME_WAIT 的意义？

标准答案：

  1. 第一次（FIN）：主动关闭方发 FIN，进入 FIN_WAIT_1
  2. 第二次（ACK）：被动关闭方回复 ACK，进入 CLOSE_WAIT；主动方收到后进入 FIN_WAIT_2
  3. 第三次（FIN）：被动方发 FIN，进入 LAST_ACK
  4. 第四次（ACK）：主动方回复 ACK，进入 TIME_WAIT（等待 2MSL 后 CLOSED）

TIME_WAIT 的两个作用：

  1. 保证 ACK 到达：如果第四次 ACK 丢失，被动方重传 FIN，TIME_WAIT 状态下能重发 ACK 而不是直接 CLOSED 导致 RST
  2. 防止旧连接数据干扰新连接：2MSL 时间内旧报文（可能还在网络中游荡）全部消失，不会串到后续相同四元组的新连接

深入思考：

  • 2MSL 为什么是 2 倍 MSL 而不是 1 倍？因为 ACK 到对方需要 1 个 MSL，对方 FIN 重传也需要 1 个 MSL，加起来最多 2MSL
  • MSL 值：Linux 默认 30s（/proc/sys/net/ipv4/tcp_fin_timeout 实际上是 FIN_WAIT_2 超时），TIME_WAIT 实际 60s（内核硬编码 TCP_TIMEWAIT_LEN）
  • CLOSE_WAIT 过多：说明服务端代码没有正确 close socket，通常是被动关闭方的应用层忘了关连接
  • FIN_WAIT_2 过多：对端不发送 FIN 可能因为半关闭状态一直不关，tcp_fin_timeout 控制超时时间

生产踩坑：

  • TIME_WAIT 过多（短连接场景，4w+ 连接数时 TIME_WAIT 可能达到数万）：tcp_tw_reuse + tcp_tw_recycle（Linux 4.12+ 已移除 recycle，reuse 可用但仅用于客户端），或改用长连接/连接池
  • tcp_tw_reuse 只对客户端有效（发起连接时重用 TIME_WAIT 端口），对服务端无效
  • 容器环境（K8s）中 TIME_WAIT 问题更突出：每个 Pod 的 IP 有限，端口复用更频繁，建议 Pod 级别限制短连接

参考来源：RFC 793 / http://linux-ip.net/html/tcp-timewait.html

2. TCP 滑动窗口与流量控制

标准答案：

滑动窗口是 TCP 实现可靠传输和流量控制的核心机制：

  • 发送窗口 = min(拥塞窗口, 接收窗口)
  • 窗口内的数据可以连续发送，无需等待 ACK
  • 窗口滑动的条件：收到确认（ACK），窗口右移

流量控制：

  • 通过 TCP 首部的 窗口字段（Window Size） 实现
  • 接收方在 ACK 中告知自己的剩余缓冲区大小（rwnd）
  • 发送方不能超过 rwnd 的大小

零窗口探测（Zero Window Probe）：

  • 如果接收方窗口为 0，发送方停止发送，但定时发 1 字节的探测包
  • 防止接收方发送窗口更新通知丢失导致的死锁

深入思考：

  • 窗口缩放因子（Window Scaling）：TCP 首部窗口字段只有 16 位（最大 64KB），RFC 1323 引入缩放因子，实际可达 1GB
  • 糊涂窗口综合症：接收方每次只释放少量空间，发送方发送小包浪费带宽。Nagle 算法（合并小包）和 Clark 算法（接收方延迟更新窗口，直到有足够空间）共同解决
  • Nagle 算法 vs Delayed ACK：Nagle 发送方攒数据，Delayed ACK 接收方延后回复，两者冲突导致延迟 200ms（Nagle等 ACK，Delayed ACK 等数据）。解法：TCP_NODELAY 禁用 Nagle

生产踩坑：

  • 内网高带宽低延迟场景（Redis、Kafka），tcp_rmem / tcp_wmem 默认值偏小（内核 auto-tuning），需要手动调大提升吞吐
  • 外网跨境传输（中国到美国），大窗口 + BBR 拥塞控制比 CUBIC 效果更好，吞吐提升 2-3 倍
  • Nagle + Delayed ACK 互锁：典型表现是交互式 SSH 或 telnet 响应慢，setsockopt(TCP_NODELAY) 解决

参考来源：RFC 1323 / RFC 1122 / https://www.kernel.org/doc/Documentation/networking/ip-sysctl.rst

3. TCP 拥塞控制算法详解

标准答案：

拥塞控制防止网络内部（路由器/交换机）过载，与流量控制（端到端）不同。四个阶段：

  1. 慢启动（Slow Start）：cwnd 从 1 MSS 开始，每收到一个 ACK 翻倍（指数增长），直到 ssthresh 或发生丢包
  2. 拥塞避免（Congestion Avoidance）：cwnd 到达 ssthresh 后，每 RTT 增加 1 MSS（线性增长）
  3. 快重传（Fast Retransmit）：收到 3 个重复 ACK 立即重传，不等超时
  4. 快恢复（Fast Recovery）：不回到慢启动，cwnd = ssthresh + 3（TCP Reno），或 cwnd 折半（TCP NewReno）

主流算法对比：

算法 核心思想 适用场景 特点
Reno 丢包即拥塞，cwnd 折半 经典 丢包时降低 50%
NewReno 改进了快恢复阶段的部分 ACK 处理 有线网络 一个 RTT 内恢复多个丢包
CUBIC 三次函数窗口增长，独立于 RTT 高带宽长距离 Linux 默认算法
BBR 基于带宽和 RTT 模型，不依赖丢包 高延迟/丢包网络 Google 出品，吞吐更高
BBRv3 BBRv2 改进版 混合网络 兼顾公平性

深入思考：

  • BBR 为什么比 CUBIC 好？CUBIC 依赖丢包判断拥塞，在深缓冲区（bufferbloat）场景下丢包前已排队严重，吞吐上不去。BBR 通过探测最大带宽（BW）和最小 RTT 建模，不把丢包当拥塞信号，在高延迟和少量丢包场景表现优异
  • CUBIC vs BBR 公平性：BBR 在共享瓶颈链路上会挤占 CUBIC 流，需要 pacing 和 fair queuing
  • Linux 默认算法变更：5.19+ 默认 CUBIC，但 Google 生态（YouTube、GCP）广泛使用 BBR

生产踩坑：

  • BBR 在阿里云/腾讯云上的表现：国内云厂商内部网络 SNAT 场景下，BBR 的 RTT 探测被路由干扰，导致带宽不稳定。建议内网用 CUBIC，公网用 BBR
  • 快重传阈值：tcp_reordering 默认 3，对乱序敏感的网络（如 WiFi）频繁触发快重传，可调大到 5-7
  • 丢包率 1% 场景：CUBIC 吞吐下降 90%+，BBRv3 仍能维持 60%+，直播/实时音视频场景推荐 BBR

参考来源：https://github.com/google/bbr / https://datatracker.ietf.org/doc/rfc793/

4. TCP 粘包与拆包问题

标准答案：

粘包：多个小包合并为一个 TCP 数据段发送
拆包：一个完整包被拆分为多个 TCP 数据段到达

产生原因：

  • TCP 是流式协议（流式套接字），没有消息边界
  • Nagle 算法合并小包
  • MSS 限制（MTU 1500 - IP 20 - TCP 20 = 1460 字节）
  • 接收方缓冲区读取不及时

解决方案：

方案 原理 优缺点
固定长度 每个消息固定 N 字节 简单但浪费空间，不支持变长
分隔符 消息末尾加 \n 或 \r\n 灵活但内容中不能出现分隔符
长度前缀 前 4 字节存消息体长度 最主流，HTTP/2、gRPC 均使用
自定义协议 TL V 格式（Type-Length-Value） 适用于协议复杂的场景

Netty/Java 实践：

深入思考：

  • UDP 没有粘包问题：UDP 基于数据报（datagram），接收 recvfrom 一次对应一次 sendto，有明确边界
  • HTTP/2 多路复用：一个 TCP 连接上有多个 Stream，每个 Stream 有自己的 frame，依然需要长度前缀（frame header 中的 Length 字段）
  • 设计自己的消息协议时：建议用 Varint 编码（Protobuf 方式）代替固定 4 字节长度，压缩小消息的头部开销

生产踩坑：

  • 用 \n 做分隔符时，消息体里不可控地出现了换行符（用户输入），导致解析错乱。解决方案：转义（类似 HTTP 的 chunked transfer encoding），或改用长度前缀
  • 拆包后的半包数据需要积累缓冲区（Netty 的 Cumulator），缓冲区大小设置过小会频繁扩容，建议根据业务最大消息设置合理初始容量
  • 反向代理（Nginx/Envoy）的 body buffer 大小影响拆包组装，proxy_request_buffering off 关闭缓冲后用 chunked 流式传输

参考来源：https://netty.io/wiki/user-guide-for-4.x.html / RFC 7540

0x01 HTTP

5. HTTP/1.0 → 1.1 → 2 → 3 的核心演进

标准答案：

特性 HTTP/1.0 HTTP/1.1 HTTP/2 HTTP/3
连接 短连接（每个请求新建 TCP） 持久连接（Connection: keep-alive） 多路复用（单一 TCP） QUIC（基于 UDP）
管线化 不支持 支持但 HOL 阻塞 二进制分帧，无 HOL 无 HOL（UDP 天然）
头部压缩 无 无 HPACK（静态/动态表） QPACK（兼容乱序）
队头阻塞 有 有（HTTP 级 + TCP 级） TCP 级 HOL 仍然存在 完全消除
服务端推送 无 无 支持 支持
传输层 TCP TCP TCP QUIC (UDP)
加密 无/可选 无/可选 通常 h2 over TLS 强制 TLS 1.3

深入思考：

  • HTTP/1.1 的管线化为什么没人用？因为 HOL（队头阻塞）——第一个请求响应慢，后续所有请求等待；且服务端必须按序响应，代理/浏览器都不实现
  • HTTP/2 的 TCP 级 HOL：一个 TCP 连接上所有流共享一个拥塞控制，一个丢包导致所有流阻塞。这就是升级到 HTTP/3 的最大驱动力
  • QUIC 如何解决：基于 UDP，每个 Stream 独立丢包检测和重传，一个流丢包不影响其他；0-RTT 握手更是碾压 TCP+TLS 1.3 的 1-RTT

生产经验：

  • HTTP/2 在弱网环境（丢包率 > 2%）性能可能不如 HTTP/1.1 多连接，因为 TCP HOL 导致所有流一起降速。此时应降级到 HTTP/1.1 多连接或直接上 HTTP/3
  • Nginx 开启 HTTP/2：listen 443 ssl http2;，注意 gRPC 必须走 HTTP/2
  • CDN 边缘节点对 HTTP/3 支持参差不齐：Cloudflare 全量支持，阿里云仅部分区域，回源时自动降级

参考来源：RFC 7540 (HTTP/2) / RFC 7230 (HTTP/1.1) / RFC 9000 (HTTP/3/QUIC)

6. HTTPS 完整握手过程（TLS 1.2 vs 1.3）

标准答案：

TLS 1.2 完整握手（2-RTT）：

TLS 1.3 握手（1-RTT，可 0-RTT）：

关键区别：

  1. RTT 减少：TLS 1.3 1-RTT，0-RTT 模式复用前次会话
  2. 密码套件精简：移除 RSA 密钥交换（不支持前向安全），仅保留 AEAD（AES-GCM / ChaCha20-Poly1305）
  3. 握手消息加密：1.3 中 ServerHello 之后的全部消息加密，防中间人窥探证书内容
  4. 降级保护：1.3 的 ServerHello 包含 supported_versions 扩展，防止降级攻击

深入思考：

  • 前向安全（PFS, Perfect Forward Secrecy）：即使用私钥泄露，历史通信仍然安全。RSA 密钥交换（TLS 1.2 以前）不具备 PFS，因为 premaster secret 由客户端用服务器公钥加密传输——私钥泄露就能解密所有历史会话。ECDHE 每次会话生成临时密钥对，私钥泄露不影响历史会话
  • 0-RTT 的安全风险：重放攻击——攻击者截获 0-RTT 数据再发一次。服务器需要幂等性处理（GET 请求可以，POST 不行）。Cloudflare 方案：0-RTT 仅用于幂等请求
  • 证书链验证：客户端用操作系统/浏览器的 CA 根证书验证服务端证书链，同时检查域名（CN/SAN）、有效期、吊销状态（CRL/OCSP）

生产踩坑：

  • TLS 1.3 在部分旧客户端（Android 4.x、Windows 7 IE）不兼容，Nginx 需同时开启 TLS 1.2+1.3：ssl_protocols TLSv1.2 TLSv1.3;
  • 证书链不完整（缺少中间证书）是最常见的握手失败原因，用 openssl s_client -connect host:443 -showcerts 排查
  • OCSP Stapling 没有启用时，浏览器自行 OCSP 查询导致额外 500ms+ 延迟：ssl_stapling on;
  • 0-RTT 需要客户端和服务器都保存会话 ticket，服务端集群需要共享 ticket key（ssl_session_ticket_key），否则客户端每个请求都要完全握手

参考来源：RFC 8446 (TLS 1.3) / RFC 5246 (TLS 1.2) / https://www.cloudflare.com/learning/ssl/what-happens-in-a-tls-handshake/

7. HTTP 缓存策略全解

标准答案：

强缓存（不请求服务器）：

头部 优先级 说明
Expires 低 HTTP/1.0，绝对时间，依赖客户端时间准确
Cache-Control: max-age=3600 高 HTTP/1.1，相对秒数，优先级高于 Expires

协商缓存（向服务器验证是否可用）：

头部 生成方 原理
Last-Modified / If-Modified-Since 服务端 文件最后修改时间，秒级精度
ETag / If-None-Match 服务端 文件内容哈希，精确到字节

完整流程：

  1. 浏览器请求 → 检查强缓存（max-age 未过期）→ 直接使用本地缓存（200 from disk/memory cache）
  2. 强缓存过期 → 发起协商请求（带 If-Modified-Since / If-None-Match）
  3. 服务端比对 → 304 Not Modified → 使用缓存；否则 200 + 新资源

深入思考：

  • 强缓存命中 vs 协商缓存命中：强缓存完全没有网络请求（0ms），协商缓存有一个 304 请求（几十 ms）。静态资源应尽可能使用强缓存+文件名 hash（webpack contenthash）
  • Cache-Control 指令详解：no-cache 不是不缓存而是每次需协商；no-store 真正禁止缓存；private 仅浏览器可缓存；public 代理/CDN 也可缓存；must-revalidate 过期后必须向源站验证
  • Service Worker 缓存：更灵活的应用层缓存，可以缓存 HTML 页面（强缓存无法缓存 HTML）、实现离线访问

生产踩坑：

  • 版本更新后用户拿到旧资源：静态资源用 index.a1b2c3.js hash 命名，HTML 文件设 no-cache。CDN 刷新缓存时注意全网生效时间（分钟~小时级）
  • CDN 回源缓存不一致：CDN 节点使用的 cache key 包含 query string，?v=1 和 ?v=2 被视为不同资源——需要 CDN 配置忽略 query 或使用 rewrite
  • Cache-Control: immutable：提示浏览器永不变化，配合 hash 文件名，省掉 304 协商请求——Chrome 和 Firefox 支持
  • HTTP/2 Server Push 失败原因：推送的资源可能已经在缓存中，浪费带宽。Chrome 已移除 Server Push 支持，改用 103 Early Hints

参考来源：RFC 7234 / https://developer.mozilla.org/en-US/docs/Web/HTTP/Caching

8. Cookie vs Session vs Token

标准答案：

特性 Cookie Session Token (JWT)
存储位置 客户端浏览器 服务端内存/Redis 客户端（通常是 Header）
状态 有状态 有状态 无状态
跨域 受限（SameSite 策略） 需共享 Session 存储 天然跨域
安全 易 CSRF、XSS 需 Session 固定防护 防篡改（签名），需防 XSS
扩展性 需配合 Session 集群需共享存储 服务端无状态，水平扩展容易
性能 每次请求携带 服务端查存储 服务端仅验证签名
失效 设置 Expires/Max-Age 服务端设置 TTL JWT 签发后无法提前撤销

典型架构：

  • Session + Cookie：登录成功 → 服务端创建 Session（Redis）→ Set-Cookie: sessionId → 下次请求 Cookie 带入
  • JWT Token：登录成功 → 签发 JWT（签名防篡改）→ 返回给客户端 → 后续请求 Authorization: Bearer → 服务端验签

深入思考：

  • JWT 的缺点：无法主动失效（除非用黑名单，但那就变成有状态了）；payload 信息泄露（base64 可解码）；token 膨胀影响请求性能
  • 刷新令牌（Refresh Token）：Access Token 短期（15min）+ Refresh Token 长期（7天），减少 JWT 泄露风险。关键设计：Refresh Token 轮换（rotation）+ 重用检测（reuse detection）
  • SameSite Cookie：Strict（同站发送）、Lax（导航到目标站发送）、None（需 Secure，跨站发送）。Chrome 80+ 默认 Lax，有效防止 CSRF
  • Session 共享方案：Redis 集中存储（Spring Session Data Redis）；粘性 Session（不推荐，AZ 故障不可用）；JWT 无状态化

生产踩坑：

  • JWT 在 Nginx 日志中记录 Authorization header，导致 token 泄露：确保 access_log 不输出 Authorization 或用 HTTPS
  • Cookie 的 HttpOnly 和 Secure 标志必须设置，防止 XSS 窃取和中间人截获
  • 微服务架构下每个服务都要解析 JWT，重复解码浪费 CPU：在网关层统一切换为内部 Session token（短、无签名）
  • JWT 的 iat/exp 使用服务器时间，客户端和服务器时间偏差导致认证失败：设置合理的 clockSkew（通常 30 秒）

参考来源：RFC 6265 (Cookie) / RFC 7519 (JWT) / OWASP Cheat Sheet

9. DNS 完整解析过程

标准答案：

以访问 www.example.com 为例：

详细步骤：

  1. 浏览器缓存：Chrome chrome://net-internals/#dns，TTL 一般为 60s
  2. 操作系统缓存：Linux /etc/hosts、systemd-resolved；Windows C:\Windows\System32\drivers\etc\hosts
  3. 本地 DNS 解析器（LDNS，通常是 8.8.8.8 或 114.114.114.114 或 DHCP 分配）
  4. 根服务器：全球 13 组（A-M），返回 .com TLD 服务器的 NS 记录
  5. TLD 服务器：.com 服务器，返回 example.com 权威服务器的 NS 记录
  6. 权威服务器：返回 www.example.com 的 A/AAAA 记录

两种查询方式：

  • 递归查询：LDNS 替客户端完成全部分析（客户端只问一次）
  • 迭代查询：LDNS 依次询问各级服务器（各级返回下一级地址）

深入思考：

  • DNS 缓存优化：TTL 设置权衡——TTL 太短（60s）导致 LDNS 频繁查权威，加载慢；TTL 太长（86400s）域名切换 IP 后生效慢。建议：正常 TTL 600s，变更前 48 小时降到 60s
  • CNAME 记录 vs A 记录：CNAME 可以指向另一个域名（CDN 常用），但会多一次 DNS 查询；A 记录直达 IP。根域不能使用 CNAME
  • DNS 负载均衡：一个域名多 A 记录，DNS 轮询返回不同 IP。缺点：DNS 客户端会缓存、不会自动剔除故障节点

生产踩坑：

  • DNS 劫持/污染：国内运营商 DNS 劫持常见，将 NXDOMAIN 转向广告页。解决方案：DoH（DNS over HTTPS）、DoT（DNS over TLS）、国内用 114.114.114.114（不劫持）
  • SLB（Server Load Balancer）切换时：必须提前降 TTL（600→60），等 48 小时生效后再切换 IP，否则用户仍然访问旧 IP
  • K8s 中的 DNS 问题：Pod DNS 搜索域过长导致解析延迟 +N 次查询后缀尝试，CoreDNS 配置 ndots:1 避免

参考来源：RFC 1034 / RFC 1035 / https://www.cloudflare.com/learning/dns/what-is-dns/

10. CDN 加速原理及关键技术

标准答案：

CDN（Content Delivery Network） 通过在全球部署边缘节点，将内容缓存到离用户最近的位置。

核心流程：

  1. 用户请求 cdn.example.com/image.jpg
  2. 本地 DNS 解析 → CNAME 指向 CDN 全局负载均衡器（GSLB）
  3. GSLB 根据用户 IP 选择最近的边缘节点 IP（智能 DNS）
  4. 用户请求边缘节点
  5. 命中缓存：直接返回
  6. 未命中：边缘节点回源（Origin Pull）获取内容 → 缓存 → 返回

关键技术：

  • GSLB（全局负载均衡）：基于 Anycast（BGP 广播同一 IP）+ 地理位置/网络延迟选优
  • 缓存策略：遵循源站 Cache-Control，支持强制缓存、分层缓存（Edge → Mid-Tier → Origin）
  • 动态加速（DCDN）：动态内容不走缓存，通过最优路由（专用探测线路）到源站，减少跨国延迟
  • 边缘计算：在 CDN 节点运行计算逻辑（Serverless @ Edge），如图像压缩、A/B 测试

深入思考：

  • 回源策略：源站需要允许 CDN 回源 IP 段；HTTPS 回源需要 CDN 和源站之间信任证书；OSS 回源（对象存储）减少源站压力
  • 缓存穿透：大量请求未命中（热点文件过期、恶意请求动态 URL），直接压垮源站。解决方案：CDN 层限频、MISS 限速回源、设置较长 TTL
  • 预热（Preload）：大促前将热点资源推送到 CDN 边缘节点，避免大量 MISS 回源。云厂商 API：PreloadCache
  • Private CDN：金融/游戏行业自建 CDN 节点，用 BGP Anycast + 自研缓存调度，成本高于公有 CDN 但可控

生产踩坑：

  • CDN 缓存不更新：修改了源站文件但 CDN 还是旧内容。原因：CDN 没有收到 purge 通知，或 CDN 设置了 ignore-cache（忽略源站 Cache-Control）。处理：在 CDN 控制台手动刷新 URL/目录
  • HTTPS 证书回源问题：CDN 到源站走 HTTP（明文）虽然快但不安全，走 HTTPS 时 CDN 的 SNI 需要正确设置
  • 多 CDN 容灾：大厂用多厂商（阿里+腾讯+Cloudflare），通过 DNS 轮询/HTTPDNS 分发流量。问题：各 CDN 节点质量不均，GSLB 切换速度不同

参考来源：https://www.cloudflare.com/learning/cdn/what-is-a-cdn/ / https://developer.aliyun.com/cdn

11. DNS 劫持与防护

标准答案：

DNS 劫持类型：

类型 攻击方式 场景
本地劫持 篡改 hosts 文件、路由器 DNS 配置 局域网内劫持
运营商劫持 中间人修改 DNS 应答 国内 ISP 插入广告页
缓存投毒 向 DNS 递归服务器注入虚假记录 DNS 服务器漏洞
域名劫持 盗取域名管理账号，修改 NS 记录 即 DNS 被盗

防护方案：

  1. DoH（DNS over HTTPS）：DNS 查询走 HTTPS 443 端口，与普通流量混合，运营商难以区分和劫持
    • Chrome/Firefox 默认开启（使用 Cloudflare 1.1.1.1 或 8.8.8.8）
  2. DoT（DNS over TLS）：DNS 查询走 853 端口 TLS 加密
  3. DNSSEC（DNS Security Extensions）：对 DNS 记录加数字签名，验证应答真实性
    • 部署率低（全球 ~20% 顶级域已签名）
  4. HTTPDNS：绕过本地 DNS，直接通过 HTTPS API 获取 IP（阿里云、腾讯云提供）

深入思考：

  • DNSSEC 为什么没有广泛部署？配置复杂（需要 Zone Signing Key / Key Signing Key 双密钥管理）；签名增加了 DNS 响应大小（从 512B 到 2K+ 需要 EDNS0）；部分 CDN 不兼容
  • HTTPDNS 是更好的解决方案：移动端 SDK（阿里云 HTTPDNS、腾讯云 MSDK）直连调度服务器，彻底绕过运营商的 LocalDNS 劫持，同时实现更精准的 GSLB 调度
  • 域名防盗：注册商开启域名锁（Registry Lock）、两步验证；域名到期前 30 天续费；WHOIS 信息隐私保护

生产踩坑：

  • 运营商劫持导致用户 HTTPS 证书告警（中间人用自签名证书）：用户看到「不安全」提示，公司背锅。解决方案：DoH + HSTS Preload
  • DNSSEC 配置错误导致域名解析失败：签名过期、算法不匹配。测试工具：dig +dnssec example.com、https://dnssec-analyzer.verisignlabs.com
  • 移动端 App DNS 劫持严重：iOS/Android 上使用 okhttp/NSURLSession 的 DNS over HTTPS 配置，统一绕过运营商

参考来源：RFC 7858 (DoT) / RFC 8484 (DoH) / RFC 4033 (DNSSEC) / https://developer.apple.com/documentation/network/dns_over_https

13. WebSocket vs HTTP 长轮询 vs SSE

标准答案：

特性 长轮询 (Long Polling) SSE (Server-Sent Events) WebSocket
方向 服务端→客户端 服务端→客户端（单向） 全双工（双向）
协议 HTTP HTTP（EventStream） WebSocket 协议
连接数 每个请求新建 HTTP 连接 一个 HTTP 长连接 一个 TCP 长连接
浏览器兼容 全部 IE 不支持（polyfill 可用） IE 10+
自动重连 需手动实现 内置（EventSource 自动重连） 需手动实现
二进制 需 Base64 需 Base64 原生支持
HTTP 开销 每次请求都带 HTTP 头部 仅第一次握手 仅第一次握手
典型场景 老系统兼容 股票行情/通知推送 实时游戏/聊天/协作

深入思考：

  • 长轮询为什么不好：每次轮询 HTTP 请求头部 ~800 字节浪费带宽；请求到达时可能没有新数据（空响应）；高并发场景服务端需要维持大量挂起请求
  • SSE 的优点：浏览器原生支持自动重连（EventSource），自带事件 ID 断点续传，比 WebSocket 简单得多。如果只需要服务端推送，SSE 比 WebSocket 更合适
  • WebSocket 的缺点：维护复杂（心跳、重连、粘包处理），DDoS 防护困难（Connection: Upgrade 绕过 CDN 防护），浏览器有并发限制（每个域名 6 个 WebSocket）

生产踩坑：

  • 长轮询在 Nginx 后端（proxy_read_timeout 默认 60s）导致频繁超时重连：调大 proxy_read_timeout 300s
  • WebSocket 连接数过多导致服务端文件描述符（FD）耗尽：ulimit -n 65535 + worker_connections 65535
  • SSE 在 HTTP/2 下多路复用效率高，HTTP/1.1 有浏览器 6 连接限制：使用 HTTP/2
  • 混用 WebSocket 和 HTTP API 时，身份认证 token 需要在 WebSocket 握手时传递（query parameter 或自定义 header，WebSocket API 不支持自定义 header，只能用 URL query）

参考来源：WebSocket RFC 6455 / SSE W3C / https://hpbn.co/websocket/

14. gRPC 四种通信模式详解

标准答案：

gRPC 基于 HTTP/2 的 4 种通信模式：

1. 一元 RPC（Unary RPC）：

客户端发送一个请求，服务端返回一个响应。等价于传统 HTTP API。

2. 服务端流式 RPC（Server Streaming）：

客户端发一次请求，服务端通过 stream 返回多个消息。

3. 客户端流式 RPC（Client Streaming）：

客户端通过 stream 发送多个消息，服务端收到所有消息后返回一个响应。

4. 双向流式 RPC（Bidirectional Streaming）：

双方通过 stream 独立读写，读写顺序没有限制。真正的全双工。

深入思考：

  • gRPC 性能优势：Protobuf 二进制序列化（比 JSON 小 3-10x、快 20-100x）、HTTP/2 多路复用（减少连接数）、头部压缩（HPACK）
  • Deadline/Timeout：客户端设置超时，服务端通过 context.Context 感知。gRPC 默认没有超时，需要显式设置
  • 流控：HTTP/2 流级流控，默认 64KB，大流量场景需调大 grpc.initial_conn_window_size
  • 拦截器（Interceptor）：类似中间件，实现日志、认证、限流、监控，客户端和服务端各一套

生产踩坑：

  • gRPC 负载均衡：gRPC 使用 HTTP/2 长连接，L4（TCP）LB 无法感知 RPC 级别的负载。解决方案：L7 LB（Envoy/Linkerd）、客户端负载均衡（pick_first/round_robin）、K8s headless service + DNS 轮询
  • gRPC 在浏览器中不可用：浏览器无法直接使用 HTTP/2 的 trailer 和 gRPC 自定义 header。需要 gRPC-Web + Envoy 代理转换
  • 大消息问题：Protobuf 默认 4MB 限制，超过需要手动设置 MaxCallRecvMsgSize。超大消息（>100MB）建议用流式传输，避免内存暴涨
  • 连接管理：gRPC 客户端默认使用连接池（每个地址一个 HTTP/2 连接），频繁建连导致 TIME_WAIT 堆积：设置 grpc.keepalive_time 和 grpc.keepalive_timeout
  • 序列化性能：Protobuf 在 Java 中反射序列化比 Go 慢很多，需要预编译（protoc-gen-grpc-java 插件）或使用 Kotlin serialization

参考来源：https://grpc.io/docs/what-is-grpc/core-concepts/ / https://protobuf.dev/

15. gRPC 与 Thrift/HTTP API 的选型

标准答案：

特性 gRPC Thrift HTTP (JSON/REST)
序列化 Protobuf（二进制） Apache Thrift 二进制 JSON（文本）
传输层 HTTP/2 TCP（原始）/HTTP HTTP/1.1 / 2
流式 四模式原生支持 部分帧级别 不支持（SSE 单向）
代码生成 protoc 多语言 thrift compiler OpenAPI + 手工
浏览器支持 需 gRPC-Web 不直接支持 原生
社区生态 广泛（K8s/Envoy） 较小 最广泛
性能 高 高 中等
调试 需 grpcurl / gRPC UI 需 Thrift 工具 curl / Postman

选型建议：

  • 内部微服务通信：gRPC 或 Thrift（首选 gRPC，因为 HTTP/2 流控+多路复用优势）
  • 外部 API / 开放平台：REST/GraphQL（浏览器友好、调试方便）
  • 移动端 → 后端：gRPC-Web 或 HTTP API（gRPC-Web 有性能优势但调试受限）
  • 网关：使用 Envoy/gRPC-Gateway 将 gRPC 转为 REST，对外暴露 HTTP，内部走 gRPC

深入思考：

  • gRPC vs Thrift 性能对比：单次请求差异不大（微秒级），但在吞吐量上 gRPC 的 HTTP/2 多路复用胜过 Thrift 的多连接池管理
  • Schema 演进：Protobuf 的 field number 不可变，reserved 标记已删除字段；Thrift 类似。JSON 无 schema 约束，但 OpenAPI 可定义
  • 长连接健康检查：gRPC Health Check Protocol（grpc.health.v1.Health/Check）是标准做法，Envoy 侧支持主动健康检查

生产踩坑：

  • 混合协议架构（gRPC + HTTP）导致运维复杂度翻倍：监控（Prometheus 指标）、日志、调用链都需要兼容两套
  • gRPC 的 Deadlines 未正确传递导致级联故障：A→B 调用超时但 B→C 继续执行，浪费资源
  • 错误处理：gRPC 错误码（Status.Code）比 HTTP 状态码更丰富（UNAVAILABLE、DEADLINE_EXCEEDED、UNAUTHENTICATED），但客户端需要正确处理重试

参考来源：https://grpc.io/docs/guides/ / https://thrift.apache.org/

    • DV（Domain Validation）：仅验证域名所有权，免费，浏览器绿锁
    • OV（Organization Validation）：验证企业身份，地址栏显示公司名
    • EV（Extended Validation）：严格验证，浏览器显示公司名+绿条——Chrome 56+ 已取消绿条显示，EV 价值下降
  • 证书链优化：使用最少中间证书（通常是 2 级：根 CA + 中间 CA + 服务端证书），减少握手阶段证书传输量
  • TLS False Start：TLS 1.2 中允许在 ChangeCipherSpec 后立即发应用数据，节省 1 RTT。所有现代浏览器支持

生产踩坑：

  • 证书到期事故：没有自动续期，检测到的时候已经挂了。解决方案：acme.sh cron job + Prometheus 告警双重保障
  • 证书链不完整：Nginx 只配了 cert.pem 漏了 chain.pem，部分客户端（Android）会证书验证失败
  • 通配符证书（*.example.com）：不能匹配 example.com（裸域），需要额外申请一张裸域证书或用 SAN 包含
  • 多域名 SAN 证书：同一证书包含 api.example.com 和 www.example.com，优点是减少证书数量，缺点是如果某个子域私钥泄露全部受影响
  • OCSP 服务器不可达导致握手延迟：开启 ssl_stapling 由 Nginx 代理 OCSP 查询，同时配置 ssl_stapling_responder fallback

参考来源：https://letsencrypt.org/zh-cn/docs/ / https://certbot.eff.org/ / https://ssl-config.mozilla.org/

18. 全链路排查思路（用户慢 → 找到根因）

标准答案：

六步排查法：

Step 1：确认问题范围

  • 用户反馈「网站打不开」还是「某个接口慢」
  • 所有用户还是部分用户（地域、运营商、设备）
  • 持续还是偶发（有无规律、时段）

Step 2：客户端网络诊断

Step 3：服务端诊断

Step 4：中间件排查

  • 反向代理墙（Nginx/Envoy）：access_log 的 $upstream_response_time vs $request_time
    • $request_time 大但 $upstream_response_time 小 → Nginx 自身瓶颈（backlog、连接数）
    • 两者都大 → 后端服务慢
  • Redis 慢查询：SLOWLOG GET 10
  • MySQL 慢查询：慢查询日志，SHOW PROCESSLIST 看锁等待

Step 5：应用层

  • 全链路追踪（Jaeger/Zipkin/SkyWalking）：trace_id 跟踪一次请求经过的所有服务
  • APM（Pinpoint/Datadog/Arms）：机器级、方法级热点
  • 业务日志：grep trace_id app.log | tail -100

Step 6：深入定位

  • TCP 抓包：tcpdump -i eth0 -w trace.pcap port 443 → Wireshark 分析握手耗时、重传、零窗口
  • eBPF/BCC 工具：bcc/tcpconnect 看连接频率，tcplife 看连接生命周期

深入思考：

  • TTFB（Time To First Byte）大的常见原因：
    1. DNS 解析慢 → DoH 或预解析
    2. TCP 连接慢 → 跨国跨运营商延迟
    3. TLS 握手慢 → 会话复用、OCSP Stapling
    4. 后端业务逻辑慢 → SQL/微服务调用
    5. 网络丢包导致 TCP 重传 → tcpdump 确认
  • 502 和 504 的区别：
    • 502 Bad Gateway：上游返回非法响应（或连接被拒绝）
    • 504 Gateway Timeout：上游在代理超时内没有响应

生产踩坑：

  • DNS 解析耗时 2s：某次排查发现 pod 内 /etc/resolv.conf 的 search 域有 6 个后缀，每次 DNS 查询尝试 7 次拼接才成功。解决：ndots:1 + 减少 search 域
  • TCP 重传 30% 导致接口 5s 超时：云主机网卡协商速率异常（1000M 降到 100M），ethtool eth0 确认后找云厂商换实例
  • TIME_WAIT 积累到 6w 导致端口耗尽：gRPC 短连接未启用 keepalive，配置 keepalive_time=10s 改为长连接解决
  • Redis 连接超时但 ping 正常：某次 telnet 通但实际是 iptables 规则限制 tcp_max_syn_backlog 太小导致 SYN 被丢。ss -s 看 TCPBacklogDrop 确认
  • Nginx worker 连接数爆满：netstat -s | grep -i "listen queue" 看到 listen queue overflow，调大 worker_connections + net.core.somaxconn

场景 命令/工具

TCP 连接排查 ss -tan, netstat -s

参考来源：https://www.brendangregg.com/linuxperf.html / https://blog.cloudflare.com/tcp-tcpdump/

19. 网络性能评估与基准测试

标准答案：

吞吐测试（Bandwidth）：

  • iPerf3：最常用

延迟测试（Latency）：

  • ping：RTT 基础值，ping -c 100 -i 0.1 看方差
  • netperf TCP_RR：每秒请求-响应次数（TPS）
  • sockperf：更精细的延迟分布（P50/P99/P999）

连接数测试：

  • wrk：HTTP 压测
  • h2load：HTTP/2 压测

深入思考：

  • 测试环境注意事项：同机房延迟通常 < 0.5ms，跨 AZ 0.5-2ms，跨区域 20-100ms。测试结果标注网络拓扑
  • TCP_RR vs TCP_STREAM：TCP_STREAM 测大文件带宽，TCP_RR 测短连接 RPS，两者侧重点不同
  • PPS（Packets Per Second）瓶颈：小包场景（Redis、消息队列）吞吐受 CPU 中断数限制，CPU 软中断（softirq）成为瓶颈。解决方案：RPS/RFS、XDP、DPDK

生产踩坑：

  • iPerf3 测试结果比实际带宽低：-P 并行流不够，单流受限于 TCP 窗口大小，调大 tcp_rmem/tcp_wmem
  • wrk 压测本机 Nginx 的 Connection refused：somaxconn 和 nginx backlog 未同步调大
  • UDP iPerf 测试丢包率高但实际业务不丢：UDP 测试时默认缓冲区太小，调大 net.core.rmem_max 和 wmem_max
  • 测试结果波动大：云上同租户争抢（噪声邻居），跑多轮取 P50/P99，对比不同时段的基线和异常值

参考来源：https://iperf.fr/ / https://github.com/wg/wrk / https://www.brendangregg.com/usemethod.html

总结

计算机网络是后端工程师的基础功，本文从 TCP/IP 协议栈底层原理到 HTTP 协议演进，从 DNS/CDN 到 WebSocket/gRPC，再到生产实战调优与排障，覆盖了高级研发面试的核心知识点。

一句话记忆口诀：

  • TCP 三握四挥：SYN 握手，FIN 挥手，TIME_WAIT 防旧包
  • HTTP 演进：1.0 短连接，1.1 长连管线，2 多路 HPACK，3 QUIC 无 HOL
  • HTTPS：TLS 1.2 两 RTT，1.3 一 RTT，ECDHE 前向安全
  • DNS 解析：浏览器→缓存→LDNS→根→TLD→权威
  • CDN 加速：CNAME→GSLB→边缘节点→回源
  • WebSocket：HTTP Upgrade 101，全双工
  • gRPC：Protobuf + HTTP/2，四种流模式
