0x00 Git 基础：暂存区与对象模型

0. Git 的暂存区（Stage/Index）是什么？为什么设计它？

标准答案：
Git 有三大区域：工作目录（Working Directory）、暂存区（Stage/Index）、仓库（Repository/History）。暂存区是一个位于 .git/index 的二进制文件，记录了即将提交的文件快照清单。

提交过程：git add 将文件快照写入 .git/objects/ 并更新索引 → git commit 将索引中的快照树打包成一个 commit 对象。

深入思考：

  • 为什么设计暂存区？ 允许分批提交：你可以在一次开发中修改多个文件，只把相关的一组文件放入暂存区，分多次提交。这是 Git 对比 SVN 的关键优势之一。
  • 底层实现：.git/index 是一个排序的路径-元数据映射表，包含文件名、mtime、SHA1、模式位等。git ls-files --stage 可以看到它的内容。
  • 性能考量：大项目数千文件时，暂存区操作仍 O(1) —— 因为 hash 查找而非遍历。
  • 与 SVN 差异：SVN 只有工作副本和中央仓库，没有暂存区概念；Git 的暂存区是一层安全网，让你在提交前能精细控制。

生产踩坑：
某团队在 CI 脚本中执行 git add . && git commit，但 .gitignore 漏配了 node_modules/，导致 200MB 的 node_modules 被 add 到暂存区。git commit 产生了一个巨大的 blob，后续 git gc 每次都要处理这个庞然大物。修复需要用 git filter-branch 或 git filter-repo 从历史中清除，极为痛苦。

参考来源：https://git-scm.com/book/en/v2/Git-Basics-Recording-Changes-to-the-Repository

1. Git 对象模型：Blob、Tree、Commit 分别是什么？

标准答案：
Git 是内容寻址文件系统。四种对象类型：

对象 内容 标识 示例
Blob 文件内容（二进制/文本） SHA1 哈希 e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
Tree 目录结构：文件名→blob/tree + 权限 + SHA1 的映射 SHA1 类似文件系统的目录节点
Commit 指向一棵 tree + 父 commit + 作者/提交者信息 + 时间 + 消息 SHA1 一次提交的快照
Tag 指向一个 commit（注解标签还包含签名信息） SHA1 补丁发布标记

对象存储路径：.git/objects/XX/XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX（前两位为目录名，后 38 位为文件名）。

深入思考：

  • Blob 只存内容不存文件名：文件重命名后内容不变，SHA1 相同，Git 自动检测为 rename。
  • 内容寻址的威力：相同内容的文件在 Git 中只存一份 blob，无论它出现在多少目录中。
  • git cat-file -p <SHA1> 可以查看任意对象的内容。git cat-file -t <SHA1> 查看对象类型。
  • Tree 是完整快照，不是 diff：每次 commit 都保存一棵完整 tree，Git 通过对比两棵 tree 的 SHA1 来高效计算 diff。

生产踩坑：
某开发者误以为 .git/objects/ 下的对象可以随意删除以节省空间。删除后 git log 报 fatal: bad object，整个仓库损坏。正确做法是用 git gc 或 git prune 清理 dangling objects。不要手动操作 objects 目录。

参考来源：https://git-scm.com/book/en/v2/Git-Internals-Git-Objects

2. HEAD 指针和分支的本质是什么？

标准答案：

  • HEAD：一个指向当前所在位置的指针。通常指向一个分支引用（如 refs/heads/main），此时称为 attached HEAD。如果指向一个具体 commit，称为 detached HEAD。
  • 分支：本质上是一个可移动的指针，指向一个 commit。main 分支就是 .git/refs/heads/main 文件，内容是一个 40 位的 SHA1。
  • HEAD~N / HEAD^N：相对引用，~ 表示父提交，^ 表示第几个父提交（merge commit 有多个父提交时用）。

cat .git/HEAD 输出类似 ref: refs/heads/main。

深入思考：

  • 分支的轻量性：SVN 的分支是目录拷贝（O(n) 文件复制），Git 分支只是一个 40 字节的文件（O(1)）。这就是 Git 鼓励”多建分支、频繁建分支”的根本原因。
  • Detached HEAD 的危险：在 detached HEAD 下做的 commit 没有分支指向，切换分支后这些 commit 变成 dangling objects，最终被 GC 回收。除非你 git checkout -b new-branch 创建一个分支引用它。
  • HEAD@{0} / HEAD@{yesterday}：reflog（引用日志）记录了 HEAD 的移动历史，是 Git 的”后悔药”。

生产踩坑：
一个常见的 CI/CD 场景：在 Jenkins pipeline 中执行 git checkout <某个 SHA> 进入 detached HEAD 状态，然后执行构建。构建完成后没有切回分支就直接 git push，结果提交到了无分支的 dangling commit。修复方法：始终用 git checkout -B temp-branch SHA 创建临时分支，构建完成后强制推送到目标分支。

参考来源：https://git-scm.com/book/en/v2/Git-Internals-Git-References

3. git add 和 git commit 底层发生了什么？

标准答案：

  1. git add file.txt：Git 压缩 file.txt 为 blob，写入 .git/objects/，然后在 .git/index 中记录路径→SHA1 映射
  2. git commit -m "msg"：Git 将当前 index 中的 tree 快照写入 objects，创建一个 commit 对象（指向该 tree + 父 commit），然后将 HEAD 指向的 branch ref 更新为这个新 commit

深入思考：

  • git add 之后但未 commit 前，文件内容已经在 .git/objects/ 中了！只是没有被任何 commit 引用。git fsck --lost-found 可以找回这些 dangling blob。
  • git commit --amend 本质是创建一个新的 commit 替换当前分支的 tip，而不是修改原 commit。
  • git add -p（交互式暂存）允许只暂存文件的部分修改，底层是对同一个文件创建多个不同内容的 blob。

生产踩坑：
某开发者 git add -A 后才发现不小心加入了敏感信息（如数据库密码）。此时 commit 尚未发生，直接 git reset HEAD 即可清空暂存区。但如果是 git add 之后的文件在工作目录中被删除，blob 仍然在 objects 中。正确做法：git rm --cached <file> 从暂存区移除，且确保 .gitignore 已更新。

参考来源：https://git-scm.com/book/en/v2/Git-Internals-Plumbing-and-Porcelain

0x01 分支与合并

4. Merge 与 Rebase 的区别是什么？各自什么场景使用？

标准答案：

特性 Merge Rebase
提交历史 保留分支拓扑，产生 merge commit 线性化历史，无 merge commit
冲突范围 一次性解决所有冲突 逐个 commit 解决冲突
可追溯性 可以看到”合入来自 feature 分支” 无法区分哪些是 rebase 过来的
安全性 安全，不重写历史 危险，会重写 commit SHA
适合场景 公共分支（main/release） 私有/本地分支

核心原则：

  • 公共分支永远用 merge，不要 rebase：rebase 会生成全新的 commit（新 SHA），其他协作者拉取时会爆炸。
  • 私有分支在合入公共分支前 rebase 一次：git checkout feature && git rebase main，让 feature 的基底在 main 的最新 commit，然后 git checkout main && git merge feature（fast-forward）。

深入思考：

  • Merge 的三种模式：git merge --no-ff（强制创建 merge commit，保留分支信息）、--ff（默认 fast-forward，不创建 merge commit）、--squash（将分支所有改动压缩为一个工作树变更，不保留提交历史）。
  • Rebase 交互模式：git rebase -i HEAD~3 可以 squash/reword/reorder/delete commits，是非常强大的历史整理工具。
  • 黄金法则：不要在已经推送到远程的分支上执行 rebase。重写已公开的 SHA 会导致其他协作者 git pull 后出现大量冲突。

生产踩坑：
某团队的 feature 分支已经 push 到远程并开了 PR，开发者在本地 git rebase main 后 git push --force 推上去。另一位协作者已经基于旧 feature 分支做了二次开发，git pull 后出现数百个莫名其妙的冲突。最终花费 2 天手工合并。教训：rebase 未通知团队 + 强制推送 = 灾难。

修复方法：git rebase --abort 取消 rebase；如果已强制推送，协作者可以用 git fetch origin feature:tmp 获取旧分支，然后手动 cherry-pick。

参考来源：https://git-scm.com/book/en/v2/Git-Branching-Rebasing

5. Cherry-pick 的使用场景和最佳实践

标准答案：
git cherry-pick <commit> 将指定的 commit 的变更应用到当前分支，生成一个新的 commit（内容相同但 SHA 不同）。

典型场景：

  1. Hotfix 同步：在 release 分支修复了一个 bug，需要同步到 main 分支，但不想合并整个 release
  2. 选择性移植：功能 A 在 feature/1 分支上开发了一半，但部分代码对 feature/2 有用
  3. 回退恢复：某 commit 被 revert 后，确认需要恢复时 cherry-pick 原 commit
  4. 分支间协调：submodule 的依赖提交需要引入到父项目

深入思考：

  • 冲突处理：cherry-pick 也可能产生冲突，用 git cherry-pick --continue / --skip / --abort 控制。
  • 批量 cherry-pick：git cherry-pick A..B 拣选 A 到 B 之间的所有 commit（不含 A）。
  • Cherry-pick 的副作用：原 commit 的作者信息和时间被保留，但 committer 信息和时间是当前操作者的。git cherry-pick -x <commit> 会在 commit message 中附加一行 (cherry picked from commit <SHA>)，便于追踪来源。

生产踩坑：
某项目在一次 release 分支上 cherry-pick 了 main 分支的 20+ 个 hotfix commit。没有用 -x 标记，导致后期排查某个修复为何在 release 上没有效果时，完全找不到来源。后来强制要求所有 cherry-pick 加 -x 参数，并在 commit message 中注明 JIRA ticket 号。

参考来源：https://git-scm.com/docs/git-cherry-pick

6. 冲突解决的策略和最佳实践

标准答案：
冲突标记格式：

冲突解决五步法：

  1. git status 查看哪些文件冲突
  2. 手动编辑冲突文件，选择保留哪一侧（或组合）
  3. git add <file> 标记为已解决
  4. git merge --continue（或 git rebase --continue）完成操作
  5. 验证编译通过 + 测试通过

高级策略：

  • 使用 merge tool：git mergetool 调用 vimdiff/meld/kdiff3 等可视化工具
  • 直接择一侧：git checkout --ours <file>（保留当前分支）、git checkout --theirs <file>（保留外来分支）
  • 合并策略参数：git merge -X theirs（自动选择外来方，仅用于已知无关的变更）

深入思考：

  • 冲突的本质：Git 按三路合并（3-way merge）算法工作。base（公共祖先）、ours、theirs 三方对比，当 ours 和 theirs 在同一个区域都做了不同修改时才产生冲突。
  • 减少冲突的技巧：频繁合并上游、原子化 commit、模块化代码减少多人修改同一文件。
  • 重命名检测：git merge --find-renames 会检测文件重命名，但配置不当可能导致误判。

生产踩坑：
某次合并，A 同事删除了 src/old-utils.js（已废弃），B 同事在该文件末尾追加了新功能。合并时 Git 报告”deleted by us”冲突（A 删了文件，B 改了文件）。B 同事看到冲突以为是误删，git checkout --theirs 恢复了文件。结果 old-utils.js 中大量废弃代码进入 main，导致 lint 失败、安全扫描报警。

教训：处理”文件删除”冲突时，必须理解上下文，不要机械地用 –theirs 或 –ours。最佳做法：git diff HEAD...MERGE_HEAD -- <file> 看清双方变更后再决定。

参考来源：https://git-scm.com/book/en/v2/Git-Tools-Advanced-Merging

0x02 进阶操作

7. Reset、Checkout、Revert 三兄弟的区别

标准答案：

操作 作用范围 是否修改历史 典型用法
git reset 移动 HEAD 和分支引用 是（重写历史） 撤销 commit、取消暂存
git checkout 切换分支/恢复文件 否（除非 detached HEAD） 切换分支、恢复工作区文件
git revert 创建新的反提交 否（安全追加 commit） 撤销已推送的公开 commit

git reset 的三个模式：

  • --soft：只移动 HEAD，暂存区和工作区不变（可以重新 commit）
  • --mixed（默认）：移动 HEAD + 重置暂存区，工作区不变
  • --hard：移动 HEAD + 重置暂存区 + 重置工作区（不可逆！）

深入思考：

  • git reset 的可逆性：--hard 后文件被丢弃，但 git reflog 可以找回。只要对象还在 .git/objects/ 中，git reset --hard <原来 SHA> 就能恢复。
  • git checkout <file> vs git restore <file>：Git 2.23+ 引入了 restore 命令，职责更清晰：git restore --staged <file> 取消暂存（替代 reset HEAD <file>），git restore <file> 恢复工作区（替代 checkout -- <file>）。
  • git revert 的多 commit：git revert HEAD~3..HEAD revert 最近 3 个 commit（逆序逐个 revert）。

生产踩坑：
某开发者在本地 git reset --hard HEAD~5 后才发现这 5 个 commit 中有 3 个是今天辛苦写的内容。幸好 git reflog 找到了原来的 HEAD 位置，git reset --hard HEAD@{1} 恢复了所有内容。教训：--hard 之前一定确认，或者先用 --soft 看看文件列表。

另一个踩坑：git checkout <commit-SHA> 不是重置，而是进入 detached HEAD。某新人意图取消最近一次 commit，用了 git checkout HEAD~1，结果所有工作区文件变成了旧版本，他以为数据丢失了。正确做法是 git reset --soft HEAD~1 或 git revert HEAD。

参考来源：https://git-scm.com/docs/git-reset

8. Stash 的使用场景与进阶技巧

标准答案：
git stash 临时保存工作区中未提交的修改（tracked 文件），恢复干净的工作目录。

基础用法：

进阶技巧：

  • git stash push -m "wip: 数据库连接优化"：带信息的 stash
  • git stash -u / git stash --include-untracked：包括未跟踪的新文件
  • git stash --all：包括被 .gitignore 忽略的文件
  • git stash branch <new-branch>：从 stash 创建新分支并应用

深入思考：

  • Stash 的底层：stash 实际上是创建了两个特殊的 commit（index 和 worktree），存放在 refs/stash 引用上。git stash list 本质是遍历这个引用链。
  • Stash 的冲突：git stash pop 时如果当前工作区和 stash 内容冲突，会弹出冲突并保留 stash（类似冲突的 merge）。解决后需要手动 git stash drop。
  • 替代方案：Git 2.13+ 引入了 git worktree，可以创建多个工作目录，比 stash 更适合长期切换场景。

生产踩坑：
某开发者在 feature 分支上 git stash 后切换到 main 修复紧急 bug，修复后 git checkout feature && git stash pop，结果发现 stash 的内容和 feature 当前状态产生了大量冲突——因为 feature 分支被其他同事 push 了新 commit。更糟糕的是，他忘了 stash 里还有未跟踪的配置文件，导致 git stash pop 后配置文件没恢复，本地环境启动失败。

教训：

  1. 长时间用 stash 保存修改不如直接创建 WIP commit（git commit -m "WIP"）后用 git reset HEAD~1 恢复
  2. stash 尽量短命，不要跨天
  3. 未跟踪的文件用 -u 参数确保被保存

参考来源：https://git-scm.com/docs/git-stash

9. Git Bisect 二分查找：如何精准定位引入 Bug 的 commit？

标准答案：
git bisect 通过二分查找，在已知好（good）和坏（bad）的 commit 之间自动 checkout 中间 commit，帮助定位第一个引入 bug 的 commit。

标准流程：

进阶用法：

  • 自动化 bisect：git bisect run <script> 让脚本自动判断 good/bad，适合 CI 集成。脚本 exit 0 表示 good，非 0 表示 bad。
  • 跳过不可测试的 commit：git bisect skip 跳过编译失败的 commit，Git 会选另一个候选。
  • 可视化 bisect：git bisect visualize 用 gitk 或 git log --graph 查看 bisect 进度。

深入思考：

  • 效率：1000 个 commit 只需要 ~10 次测试，20000 个 commit 只需要 ~15 次。
  • 适用条件：bug 必须可稳定复现。偶发性 bug 不适合 bisect（会误判）。
  • 局限：如果 bug 是由多个 commit 的组合引入的（不是单一 commit），bisect 找到的 commit 可能只是”最后一根稻草”。
  • 性能 bisect：也可用于定位性能回归的 commit，用基准测试脚本作为判断标准。

生产踩坑：
某项目有一个偶发性的 OOM 问题，团队用 git bisect 多次得到不同的”罪魁祸首”，浪费了两天时间。后来发现是因为测试机器资源波动导致误判。修复方法：使用固定的、隔离的测试环境，并且用脚本多次运行取中位数。

另一个坑：bisect 过程中遇到编译失败的 commit（比如代码不全、依赖缺失），如果直接标记为 bad 会干扰二分。正确做法是 git bisect skip，Git 会智能挑选相邻的其它 commit 继续。

参考来源：https://git-scm.com/docs/git-bisect

0x03 团队协作

10. Fork + PR 协作流程详解

标准答案：
Fork + PR（Pull Request）是 GitHub/GitLab 上最流行的开源协作模式。

流程：

  1. Fork：在 GitHub 上 fork 上游仓库到自己的账号
  2. Clone：git clone git@github.com:<你的用户名>/project.git
  3. 添加上游远程：git remote add upstream git@github.com:<原作者>/project.git
  4. 创建 feature 分支：git checkout -b feat/my-feature
  5. 开发 + 提交：多次 commit
  6. 同步上游：git fetch upstream && git rebase upstream/main
  7. 推送：git push origin feat/my-feature
  8. 创建 PR：在 GitHub 上创建 Pull Request
  9. Code Review：维护者审查、讨论、修改
  10. 合并：维护者 merge/squash/rebase 方式合入

深入思考：

  • 保持 fork 同步：gh repo sync <你的用户名>/<repo>（GitHub CLI）或手动 git fetch upstream && git merge upstream/main。
  • PR 的合并选项：
    • Create a merge commit：保留所有分支历史
    • Squash and merge：压缩为一个 commit，适合小型修复
    • Rebase and merge：线性化，适合长期开发
  • 多 PR 协作：每个功能一个独立分支，不要把所有改动堆在一个分支上。

生产踩坑：
某开发者在 fork 的仓库上直接向 main 分支推送了修改（未建 feature 分支），然后通过 main 分支开了 PR。PR 审查期间，他又从上游同步了 main 分支，导致 PR 中包含了上游的无关 commit，Code Review 无法进行。最终不得不关闭 PR 重新开。

教训：永远不要在 fork 的 main 分支上直接开发。main 只用来同步上游。feature 分支才是开发主阵地。

参考来源：https://docs.github.com/en/get-started/quickstart/fork-a-repo

11. Git Flow vs GitHub Flow vs GitLab Flow 对比

标准答案：

特性 Git Flow GitHub Flow GitLab Flow
分支数量 多（main/develop/feature/release/hotfix） 少（main + feature） 中（main + feature + environment/pre-production）
复杂度 高，需要 branch manager 低，几乎零学习成本 中
发布频率 低（如每两周 release） 高（CI 通过即发布） 中到高
适用场景 有版本号的大型产品 SaaS/web 应用，持续部署 企业级项目，需要环境/合规管理
Hotfix 专用 hotfix 分支，修复后合入 main 和 develop 直接在 main 上修复 环境分支向前修复
历史清晰度 高（有 develop 区分进行中和已完成） 中（全靠 feature 分支命名） 高（环境分支描述部署状态）

深入思考：

  • Git Flow 的弊端：develop 分支的存在增加了复杂性，在持续部署场景下 develop 和 main 的意义模糊。实际上很多团队只用 main 分支 + release branches。
  • GitHub Flow 的前提：完善的 CI/CD pipeline + 完善的测试覆盖 + 小步频繁提交。没有这些，main 分支随时可能变红。
  • GitLab Flow 的特色：引入环境分支（staging/pre-production/production），支持”上游优先”原则——所有修复先合入 main，然后反向 cherry-pick 到环境分支。
  • Trunk-Based Development：另一种流派，比 GitHub Flow 更激进——所有开发直接在 main 上或非常短命的 feature 分支上完成，强调每日合并。

生产踩坑：
某团队采用 Git Flow 但每个 sprint 要发布 3-4 次，develop 和 main 之间频繁 merge，产生大量 merge commit，历史极其混乱。而且 hotfix 和 develop 的冲突每周都要花半天解决。最终切换到 GitHub Flow + CI 自动部署，发布效率提升了 3 倍。

教训：Git Flow 适合低频版本发布（如移动 App），不适合高频部署的 SaaS 产品。工作流的选择应匹配团队的发布频率和部署能力。

参考来源：https://nvie.com/posts/a-successful-git-branching-model/

12. Submodule 与 Subtree 的区别及选型

标准答案：

特性 Submodule Subtree
存储方式 指针（commit SHA 引用） 完整代码拷贝到父仓库
版本管理 子仓库独立 子仓库代码成为父仓库的一部分
克隆体验 git clone --recursive 或额外 git submodule init && git submodule update 普通 clone 即可
修改体验 需要进入子目录操作独立仓库 直接在父仓库中修改，像普通文件一样
冲突风险 低（各仓库独立） 中（修改可能被覆盖）
推送到子仓库 手动进入子目录 push 用 git subtree push 推送
学习成本 中（需要理解 submodule 的概念） 低（看起来像普通文件）

选型建议：

  • 用 Submodule 的场景：你的团队需要严格隔离代码（如第三方库）、子仓库有独立生命周期、多人分工在不同仓库
  • 用 Subtree 的场景：你需要锁定某个库的特定版本、不希望团队成员理解 submodule 的复杂度、你希望父仓库可以离线工作

深入思考：

  • Submodule 的常见问题：git pull 不会自动更新 submodule，需要 git submodule update --init --recursive。版本切换时 submodule 容易处于 detached HEAD。
  • Subtree 的限制：git subtree split 操作在大仓库上很慢（需要遍历所有历史）。git subtree merge 冲突比普通 merge 更难处理。
  • 现代替代：Git LFS（大文件存储）和包管理器（npm/Maven/pip）在很多场景比 submodule/subtree 更好。大多数项目不应该用 submodule 来管理自己的微服务——用 monorepo 或包管理器更好。

生产踩坑：
某项目的 lib/ 目录包含公共工具库，用 submodule 管理。某次紧急 hotfix 时，开发者 git clone 主项目忘了加 --recursive，导致 lib/ 目录为空，构建直接失败。修复花了 20 分钟才定位到问题。

更严重的坑：submodule 的版本被误更新了。A 同事更新了 submodule 的指针（在父仓库中）并 push，B 同事 git pull 后 submodule 自动指向新版本，但 B 的本地 submodule 没有对应的 blob，构建失败。解决方案：团队约定更新 submodule 指针时必须通知全员。

subtree 坑：某团队用 subtree 引入了第三方的 UI 组件库。某次他们直接手动修改了第三方代码来修复 bug（不在子仓库中修复），后续 git subtree pull 时产生了灾难性的冲突，花了 2 天才解决。

参考来源：https://git-scm.com/book/en/v2/Git-Tools-Submodules

13. 大仓库优化：Sparse Checkout 与 Shallow Clone

标准答案：

Shallow Clone（浅克隆）：

优点：大幅减少克隆时间和磁盘占用（10GB 仓库可能降到 500MB）。
缺点：无法查看完整历史、无法 git bisect、无法 git log --all。

Sparse Checkout（稀疏检出）：

优点：工作目录中只包含你关心的文件，适用于 monorepo。
缺点：克隆时仍下载全部 Git 对象（2025+ Git 已经支持 --filter=blob:none 改善）。

最佳组合：

  • --depth 1：只拉最新 commit
  • --filter=blob:none：不拉取任何 blob（按需懒加载）
  • --sparse：只检出默认目录
    后续需要某个文件时，Git 自动从远程获取。

深入思考：

  • --filter=tree:0：比 blob:none 更激进，连 tree 对象也不拉，只拉 commit。适合只需要元数据的场景。
  • git maintenance：Git 2.30+ 引入的后台维护功能，git maintenance start 自动执行 git gc/git prefetch，优化大仓库体验。
  • partial clone 的限制：一些命令（如 git log --follow）需要完整历史时仍然会自动补全，可能比普通 clone 还慢。

生产踩坑：
某公司的 monorepo 有 50GB+（包含所有历史），新入职的开发者 git clone 耗时 30+ 分钟，磁盘 50GB 直接报警。使用 --depth 1 --filter=blob:none 后，克隆时间降到 2 分钟，磁盘占用 800MB。

但问题来了：开发者 git log 只能看到 1 条 commit，git blame 无法追溯。CI 上也用 shallow clone，导致某些依赖版本验证（需要检查历史中的 package.json 变化）失败。

解决方案：CI 用 git fetch --unshallow 在需要时补全历史；本地开发用 --depth=100 兼顾速度和功能。

参考来源：https://git-scm.com/docs/git-sparse-checkout

14. Git 在 CI/CD 中的集成最佳实践

标准答案：
CI 中 Git 的典型使用场景：

  1. 检出代码：git clone --depth 50 避免拉取全部历史
  2. 标签和版本：git tag v1.2.3 标记发布版本
  3. 状态检查：git diff --name-only HEAD~1 获取变更文件列表，实现增量构建
  4. 自动回滚：git revert <commit> 配合 CI pipeline 自动回滚
  5. 版本号生成：git describe --tags --always 生成形如 v1.2.3-5-gabc1234 的版本
  6. 触发检查：git log --oneline HEAD...origin/main 检查是否有未合并的提交

CI 中的 Git 配置要点：

深入思考：

  • CI 中执行 git push 的风险：CI 环境中的 Git 操作可能被中断，导致仓库状态不一致。建议用 API（如 GitHub API）而非 git push。
  • 语义版本自动生成：基于 commit message（Conventional Commits）+ git tag 自动计算版本号，工具如 semantic-release。
  • CI 中的 merge vs rebase：CI 中的自动合并永远用 git merge --no-ff，绝对不用 rebase。rebase 在 CI 中重写历史会导致 pipeline 混乱。

生产踩坑：
某团队在 CI pipeline 中写了一段自动发布脚本，包含 git push --tags。但某次 CI 执行时，两台并行的 runner 同时推送了 v1.2.3 标签，导致 tag 冲突覆盖，生产环境的部署脚本拉到了错误的版本。

教训：

  1. CI 中任何写 Git 操作都需要加锁或串行化
  2. 标签推送到远程前先检查是否已存在：git tag -l v* | grep v1.2.3 && exit 1
  3. 用 Git 标签做部署标记时，最好用时间戳+commit SHA（如 deploy-prod-20260611-abc1234）避免冲突

参考来源：https://docs.github.com/en/actions/using-workflows/workflow-commands-for-github-actions

15. Git Hooks：自动化工作流的关键

标准答案：
Git Hooks 是在 Git 操作的关键节点触发的脚本，放置在 .git/hooks/ 目录下。

常用 Hook 类型：

Hook 触发时机 用途
pre-commit git commit 前 代码风格检查、lint、单元测试
prepare-commit-msg 提交信息编辑前 自动添加 JIRA ticket 号、模板
commit-msg 提交信息编辑后 验证 commit message 格式（如 Conventional Commits）
pre-push git push 前 运行集成测试、检查分支命名规范
post-receive git push 完成后（服务端） 触发 CI/CD、部署
post-merge git merge 完成后 自动更新 submodule、重新安装依赖

最佳实践：

  • 团队共享 Hooks：.git/hooks/ 不会被 push 到远程。用工具如 husky（npm）/ pre-commit（Python）将 hooks 配置纳入版本控制。
  • 性能考虑：hooks 脚本应轻量。pre-commit hook 运行超过 30 秒会严重影响开发者体验。
  • 跳过 Hooks：git commit --no-verify 或 git push --no-verify 临时跳过，但应在紧急修复时使用。

深入思考：

  • 服务端 vs 客户端 Hooks：
    • 客户端 hooks：每个开发者的本地验证（pre-commit、pre-push）
    • 服务端 hooks：GitHub Actions / GitLab CI / Jenkins 等远程触发（post-receive）
  • commit-msg hook 示例（检查 Conventional Commits）：

生产踩坑：
某团队在 pre-commit hook 中集成了格式化工具（Prettier）和 lint，但格式化工具自动修改了文件内容，而 hook 没有重新将修改后的文件 git add。结果 git commit 成功后，工作目录中还有未暂存的格式化改动。下一次 git status 看到一堆已提交+未暂存的 diff，非常困惑。

解决方案：在 hook 中格式化后显式 git add 修改过的文件，或者用 lint-staged（配合 husky）只处理暂存区的文件。

另一个坑：某团队的 pre-push hook 会在推送前运行全部 6000+ 测试用例（耗时 40 分钟）。本意是保证代码质量，但开发者为了绕过这个 hook，统一用 git push --no-verify。剩下的 hooks 形同虚设。

教训：Hook 的性能至关重要。pre-push 只应运行关键的集成测试（<3 分钟），全面测试留给 CI。

参考来源：https://git-scm.com/book/en/v2/Customizing-Git-Git-Hooks

16. 历史重写与灾难恢复

标准答案：
Git 提供多种历史重写工具，但必须谨慎使用。

常见历史重写操作：

操作 命令 风险
修改最近 commit git commit --amend 低（仅最后一个 commit）
交互式 rebase git rebase -i HEAD~N 中（重写 N 个 commit）
批量替换 git filter-branch（已废弃） 高（重写整个历史）
专业清理 git filter-repo（推荐） 高（整个历史）
删除文件 git rm --cached + .gitignore 低（仅工作区）

灾难恢复必杀技：

  • reflog 救场（lost 但 referenced）：git reflog 查看所有 HEAD 操作，找到被 reset/rebase 前的 SHA，git reset --hard <SHA> 或 git checkout <SHA> -b recovery-branch
  • git fsck --lost-found：找回 dangling commits/blobs，写入 .git/lost-found/
  • git cherry-pick 逐个捞回：如果知道每个 commit 的 SHA，可以逐个 cherry-pick 到新分支

预防措施：

  1. 推送到远程前，创建备份分支：git branch backup/feature-name
  2. 大操作前先用 --dry-run 预览
  3. 敏感数据泄漏时，假设数据已泄漏，不仅仅是 git filter-branch 清除，还要轮换所有密钥

深入思考：

  • filter-repo vs filter-branch：
    • filter-branch 已废弃（Git 2.24+ 提示用它替代），性能差，处理大仓库可能跑数小时
    • filter-repo（Python 工具，pip install git-filter-repo）速度快 10-100 倍，支持并行处理
  • BFG Repo-Cleaner：Java 工具，专注删除大文件/敏感数据，比 filter-repo 更简单
  • 重写历史后的团队协调：一旦重写了已推送的历史，需要 git push --force-with-lease（比 --force 更安全，会检查是否有人先推了），并通知所有协作者 git fetch origin && git reset --hard origin/main

生产踩坑（最严重的案例）：

案例 1：误提交 500MB 大文件
某团队不小心将 500MB 的测试数据文件 commit 到了仓库。虽然事后 git rm 并新增 .gitignore，但 blob 已经永久存在于历史中。git clone 每次都要下载 500MB，即使最新代码不包含该文件。使用 git filter-repo --path-glob '*.data' --invert-paths 从整个历史中删除，仓库从 1.2GB 降到了 200MB。

案例 2：rebase 后 push –force 导致同事数据丢失
开发者 A 在 feature 分支上 git rebase main 后 git push --force。开发者 B 之前从旧 feature 分支 pull 过，还在上面做了新 commit。B 的 git push 失败，被迫 git pull --rebase，结果 B 的 commit 被错误地应用到错误的位置，产生了 10+ 个重复 commit。最终 B 靠 git reflog 找回原始分支，但和 A 的冲突花了 1 天解决。

教训：强制推送前团队需要在 IM 群发公告，且用 --force-with-lease 替代 --force。

案例 3：误删 main 分支
某团队在清理远程仓库时，误执行了 git push origin :main（删除远程 main 分支）。发现后紧急补救：因为本地 main 还在，git push origin main 恢复。但之后 5 分钟内不断有同事跑来说 git pull 报错，因为他们的本地 main 追踪的远程 ref 已经丢失。解决方案：git branch --set-upstream-to=origin/main main。

应急清单（灾难发生时按顺序执行）：

  1. 不要惊慌，不要 git gc（对象被清理前可恢复）
  2. git reflog 查看本地操作历史
  3. git fsck --lost-found 找回 dangling 对象
  4. 如果分支被删除但远程还在，git fetch origin <branch>:<branch>
  5. 如果 push –force 覆盖了远程，找其他协作者要他们的 reflog
  6. 最后的希望：GitHub/GitLab 的 Ref Log 或仓库恢复功能（有些平台保留 90 天内被删除的引用）
  7. 如果你什么工具都用了还不行 — 去 .git/objects/ 看看有没有残留的 pack 文件

参考来源：https://git-scm.com/docs/git-filter-repo

17. 生产实战综合题：一次完整的 Git 灾难复盘

标准答案：
以下是一个真实案例的综合复盘。

背景：某电商平台团队，10 人开发，Git Flow 工作流，代码托管在 GitLab。

事件经过：

  1. 周三 14:00，开发者 A 在 feature/checkout-optimize 分支上开发了 3 天，共 12 个 commit
  2. 14:15，A 执行 git rebase -i origin/main 想整理 commit 历史，但操作中误选了 drop 删除了 2 个关键 commit
  3. 14:16，A 发现不对，git rebase --abort，但不知道 rebase 已经改写了一部分 reflog
  4. 14:20，A 在慌乱中执行了 git reset --hard HEAD~5 想回到 rebase 之前的状态，但 rebase 前后 SHA 已经不同了
  5. 14:25，A 发现修改的代码都不见了，git log 看不到了，用 git reflog 找到了之前的 SHA
  6. 14:30，A 恢复成功，但误以为需要强制推送到远程才能同步，执行了 git push --force origin feature/checkout-optimize
  7. 14:35，开发者 B 在同一个分支上做 code review 时，git pull 报奇怪的冲突
  8. 14:45，B 尝试 git fetch origin && git reset --hard origin/feature/checkout-optimize，丢失了自己的 review 笔记和 minor fixes
  9. 15:00，团队开始争吵，认为有人”破坏”了分支

根本原因分析：

  • A 在团队使用的分支上执行 rebase（黄金法则：shared branches never rebase）
  • rebase 后发现错误没有立刻求助，而是继续操作
  • git push --force 代替 --force-with-lease
  • B 也缺乏灾难恢复经验，不应该 reset --hard

正确做法：

  1. A 发现误 drop commit 后：git rebase --abort 立即取消
  2. 如果 --abort 不可用（部分已应用），用 git reflog 找到 rebase 前的 SHA，git reset --hard <SHA>
  3. 绝不对已经推送的共享分支做 force push
  4. 如果需要修正，开一个新的分支从头开始

事后改进措施：

  1. 将 main/develop 分支设为 Protected（GitLab/GitHub 设置），禁止 force push
  2. feature 分支在 merge 前由 CI 自动 squash，开发者不需要手工 rebase
  3. 团队培训：git push --force-with-lease 和 set -u origin/main 保护
  4. 建立事故处理 SOP：一旦发生历史重写，立刻在群里发公告，全员 git fetch 后确认
  5. Jenkins CI 增加 git push --force 的审计告警

参考来源：https://docs.gitlab.com/ee/topics/git/git_rebase.html

总结：Git 不仅是一个版本控制工具，更是团队协作的命脉。理解其对象模型和工作原理，才能在关键时刻做出正确决策。面试官关注的不只是你会用 git rebase -i，更关注你在生产事故面前能否冷静恢复。记住 Git 的黄金法则：公共分支永不 rebase，强制推送必先公告，reflog 是最后的救命稻草。
