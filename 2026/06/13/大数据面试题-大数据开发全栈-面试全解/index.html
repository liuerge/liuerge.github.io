0. CDC（Change Data Capture）工具对比：Canal / Debezium / Maxwell / Flink CDC

标准答案：

维度 Canal Debezium Maxwell Flink CDC
开源方 阿里巴巴 Red Hat Zendesk Apache Flink 社区
底层原理 模拟 MySQL Slave 解析 Binlog Kafka Connect 框架 + Binlog 解析 读取 Binlog 输出 JSON 封装 Debezium，通过 Flink Source 读取
输出格式 自定义 Message Avro / JSON / Protobuf JSON DynamicTable / DataStream
断点续传 支持（ZooKeeper 记录 Position） 支持（Kafka Connect Offset） 支持（bootstrapping） 支持（Flink Checkpoint）
Schema 变更 需重启 部分支持 DDL 需重启 支持 Schema Evolution
全量+增量 不支持一体化 1.x 需分开，2.x 支持 不支持 支持 Snapshot + Binlog 一体化
Flink 集成 无原生集成 需 Debezium Embedded 无 原生深度集成

深入思考：

  • Flink CDC 2.x 的改进：1.x 需要全局锁保证一致性导致 MySQL 写入 hang 住，2.x 采用增量快照（Incremental Snapshot）算法，分片读取不需要锁，支持并发读取多张表。
  • 选型建议：如果技术栈是 Flink，首选 Flink CDC；如果 Kafka 生态优先，选 Debezium；仅 MySQL 且不需要 Flink，Canal 轻量够用；Maxwell 适合简单 JSON 输出场景。
  • Debezium vs Flink CDC 的关系：Flink CDC 底层封装了 Debezium，但无需部署 Kafka Connect 集群，直接以 Flink Source Connector 形式运行，降低了运维复杂度。

生产踩坑案例：

某电商公司使用 Canal 采集 MySQL Binlog 到 Kafka，大促期间 Binlog 量激增导致 Canal 消费延迟 30 分钟以上。排查发现 Canal 默认的 batchSize 和 memoryBufferSize 参数太小，调整后恢复正常。另一个问题是 Canal 不支持 DDL 变更后的动态适配，一旦源表加列，Canal 任务必须重启才能感知新字段。

  • https://cloud.tencent.com/developer/article/1897501
  • https://zhuanlan.zhihu.com/p/426489574

1. 日志采集工具对比：Filebeat / Flume / Logstash / Fluent Bit

标准答案：

维度 Filebeat Flume Logstash Fluent Bit
开发方 Elastic Apache Elastic CNCF
语言 Go Java JRuby C
资源占用 极小（~5MB） 较大（JVM） 较大（JVM） 极低（~450KB）
吞吐量 高 中 中 高
插件生态 丰富（ELK 深度集成） 较丰富（Hadoop 生态） 丰富（Input/Filter/Output） 轻量，逐渐丰富
主要场景 轻量日志采集 Agent 大数据平台内部采集 日志解析与转换 边缘 / IoT / 容器
数据可靠性 内置背压，保证至少一次 Channel 事务机制 内置队列 + 持久化 背压机制

深入思考：

  • Filebeat 的优势：Go 编写无 JVM 依赖，自动处理多行日志（multiline）、支持自动发现（Autodiscover）、与 Elasticsearch 原生集成。
  • Fluent Bit 的崛起：在 Kubernetes 环境下，Fluent Bit 取代 Fluentd 成为 CNCF 毕业项目，资源消耗仅为 Fluentd 的 1/30，适合边车模式部署。
  • Flume 的现状：Flume 在大数据生态中逐渐被边缘化。新的项目很少用 Flume 做采集，更多地使用 Filebeat + Kafka 直接对接。

生产踩坑案例：

某公司用 Flume 采集业务日志，出现日志丢失问题。原因是 Flume Channel 使用了 Memory Channel，Agent 宕机后内存中的数据全部丢失。解决方案：换用 File Channel 或 Kafka Channel 换取可靠性。

2. 数据同步工具对比：DataX / Sqoop / SeaTunnel / ChunJun

标准答案：

维度 DataX Sqoop SeaTunnel ChunJun
开发方 阿里巴巴 Apache（已退役） Apache（原 Waterdrop） 袋鼠云
架构模式 单机多线程 MapReduce 作业 分布式 / Spark / Flink Flink 原生
支持数据源 MySQL/Oracle/HDFS/OTS 等 关系型 ↔ Hadoop 近百种，包括 Kafka/ES/HBase Oracle/MySQL/SQLServer/CDC
断点续传 支持 不支持 支持 支持
数据一致性 支持 有限 支持 Exactly-Once 支持 Exactly-Once
Flink 集成 无 无 原生 Flink Runner 原生 Flink
增量同步 支持 支持（–incremental） 支持 支持
社区现状 维护中，更新缓慢 已退役（2018 后无更新） 活跃 活跃

深入思考：

  • Sqoop 的退役：Sqoop 自 2018 年后几乎无更新，被 Apache 标记为退役项目。原因：Sqoop 强依赖 MapReduce 做并行化，但 MapReduce 本身的延迟和资源开销不适合数据同步场景。替代方案是 DataX（离线批量）和 SeaTunnel（批流一体）。
  • SeaTunnel 的定位：SeaTunnel（原 Waterdrop）定位为云原生数据集成平台，支持批量同步、实时同步、CDC 同步，并且可以基于 Spark/Flink 分布式执行。
  • ChunJun 的 CDC 能力：ChunJun 在数据库 CDC 采集方面做得较深，支持 Oracle LogMiner、MySQL Binlog、SQLServer CDC 等多种协议，且支持断点续传 + Exactly-Once。

生产踩坑案例：

某金融公司将 Sqoop 从 Oracle 每天同步约 5TB 数据到 Hive，频繁出现 OOM。原因是 Sqoop 的 --m 参数调的并行度过高，单个 Mapper 分配的数据量不均匀导致部分 Mapper OOM。同时 Sqoop 对 Oracle 的 Date 类型处理存在时区问题。迁移到 DataX 后，使用通道（channel）级别的限速和内存控制，问题解决。

参考链接：

  • https://github.com/alibaba/DataX
  • https://seatunnel.apache.org/
  • https://dtstack.github.io/chunjun/

3. 消息队列在数据管道中的角色（Kafka 作为数据总线）

标准答案：

Kafka 在大数据管道中的核心角色是 数据总线（Data Bus），承担以下职责：

  1. 解耦：数据生产者和消费者通过 Topic 解耦，生产者不需要知道下游消费方。
  2. 缓冲：应对上下游吞吐量不匹配的突发流量，削峰填谷。
  3. 持久化：数据可配置保留时间（如 7 天/30 天），支持回溯消费。
  4. 多订阅：同一份数据可以被多个消费者组独立消费（如数仓、实时、搜索等）。
  5. 异步能力：CDC 数据、日志数据、埋点数据通过 Kafka 异步传输，不阻塞业务系统。

典型数据管道架构：

深入思考：

  • Kafka 与 Pulsar 的对比：Pulsar 采用存算分离（BookKeeper 存储 + Broker 计算）架构，支持多租户、存储与计算独立扩缩容。Kafka 的优势在于生态成熟（99% 的公司都在用 Kafka）、运维简单、延迟更低。选型建议：需要多租户隔离和弹性伸缩的场景选 Pulsar，追求极低延迟和生态丰富选 Kafka。
  • Kafka 的 Exactly-Once 语义：Kafka 在 0.11 版本引入幂等生产者 + 事务 API 实现 EOS。幂等生产者保证单个 Partition 无重复，事务保证跨 Partition 的原子写入。

生产踩坑案例：

某在线教育平台 Kafka 集群凌晨峰值流量冲高到 800MB/s，Broker 频繁发生 Leader 重新选举。排查发现：replica.fetch.max.bytes 和 num.replica.fetchers 配置不合理，导致 ISR 频繁收缩。调整为 num.replica.fetchers=4 + replica.fetch.max.bytes=20971520 后恢复稳定。

  • https://pulsar.apache.org/

0x01 数据仓库建模

4. 维度建模：星型/雪花型/星座型

标准答案：

星型模型：一个事实表（Fact Table）居中，多个维度表（Dimension Table）围绕，维度表不进一步规范化。查询时事实表 JOIN 各维度表，路径短、性能高、业务直觉强。

雪花模型：维度表进一步规范化（第三范式），维度表分解为更细粒度的子表。数据冗余少，但查询需更多 JOIN，性能下降。

星座模型：多个事实表共享维度表，适用于有多业务过程的企业数仓（如订单事实表 + 库存事实表共用时间维度和产品维度）。

维度 星型 雪花型 星座型
查询性能 最高（1 层 JOIN） 较低（多层 JOIN） 高（共享维度）
数据冗余 高 低 中
设计复杂度 低 高 中
维度一致性 差（维度表独立） 好（规范化） 好（共享维度）
典型场景 BI 报表/OLAP 数据治理严格的环境 企业级数仓

深入思考：

  • 维度一致性（Conformed Dimension）：在星座模型中，多个事实表共享的维度表必须是”一致维度”——同一维度在不同事实表中的含义、粒度、属性定义完全相同。这是 Kimball 方法论的核心原则。
  • 退化维度（Degenerate Dimension）：事实表中只剩下键、没有其他属性的维度（如订单号）。通常保留在事实表中不做单独的维度表。
  • 杂项维度（Junk Dimension）：将多个低基数、不相关的属性（如订单状态、支付方式标志位）合并到一个杂项维度表中，减少维度表数量。

生产踩坑案例：

某电商团队设计了严格的雪花模型，维度表规范化到 5 层。结果 BI 报表查询需要 JOIN 7-8 张表，延迟 > 30 秒。最终回退到星型模型，将常用维度属性反规范化到维度表中，查询时间降到 2 秒以内。

参考链接：

  • https://www.selectdb.com/blog/896
  • https://zhuanlan.zhihu.com/p/618087399

5. 数仓分层（ODS / DWD / DWS / ADS / DIM）

标准答案：

分层 英文 职责 设计原则
ODS Operational Data Store 原始数据层，不经过任何业务逻辑加工 与源系统保持 1:1 映射，保留原始数据
DWD Data Warehouse Detail 明细数据层，清洗/去重/标准化/维度退化 最细粒度，面向业务过程的事实
DWS Data Warehouse Service 服务数据层，轻度汇总，按主题宽表化 公共指标汇总，减少重复计算
ADS Application Data Store 应用数据层，面向具体业务报表 高度定制化，按需产出
DIM Dimension 维度表，维度属性描述 一致性维度，缓慢变化维处理

深入思考：

  • 是否需要 DWD 层：很多团队直接 ODS → DWS，跳过 DWD。适合简单场景。但大规模数仓必须有 DWD，因为 ODS 的脏数据（字段类型不匹配、乱码、null 值）需要在 DWD 统一清洗。
  • DWS 宽表设计的陷阱：过度宽表化（一个 DWS 表几百个字段）会导致存储膨胀和计算资源浪费。建议按主题域拆分（订单域、流量域、用户域），每张宽表列数控制在 50-80 以内。

生产踩坑案例：

某互联网公司 ODS→DWD 做了全量清洗，但源系统其中一个字段从 int 改为 varchar，导致 DWD 层 ETL 任务全部失败。解决方案：在 ODS 层统一做一次 Schema 校验，不兼容字段先落地为 string，DWD 再通过 UDF 做类型转换。

6. 缓慢变化维（SCD Type 1/2/3/拉链表）

标准答案：

类型 处理方式 适用场景 局限性
Type 1 覆盖原值，不保留历史 修改错误数据、属性不重要 无法追溯历史
Type 2 新增记录，添加生效时间/版本号 需要历史追溯的业务维度 数据量膨胀快
Type 3 增加”原值”列，保留有限历史 只需保留上一次的值 只能看一步历史
拉链表 记录每条记录的生命周期（start_date / end_date / is_active） 需要全量历史轨迹 关联查询复杂

深入思考：

  • Type 2 的版本设计：常见的实现方式有（1）effective_date + expiration_date + is_active_flag，（2）version_number + 时间范围，（3）开窗函数 row_number() over(partition by natural_key order by effective_date)。
  • 拉链表 vs Type 2：拉链表本质上是一种 Type 2 的实现方式，但拉链表更强调用 start_date/end_date 标记生命周期，通常用于超大量历史数据（如用户地址变更），而 Type 2 可以配合代理键使用。
  • 代理键（Surrogate Key）：为解决自然键不稳定问题，用自增 ID 或 UUID 作为维度表的 PK。事实表引用代理键而非自然键，避免自然键变化导致事实表更新。

生产踩坑案例：

某金融公司用户维度表使用 Type 2 方式，三个月后维度表从 500 万行膨胀到 5 亿行。原因是用户每天都有十几个属性变更，每条变更都产生新记录。优化方案：归一化变更频繁的字段（如用户标签）到单独的事实表，低频变更字段（如性别、出生日期）留在维度表。

参考链接：

  • https://www.mianshiya.com/question/1837027894614880258

7. 指标体系：原子指标 / 派生指标 / 复合指标

标准答案：

类型 定义 示例 存储位置
原子指标 基础业务度量，不可拆分 订单金额、用户数、PV DWD 层直接产出
派生指标 原子指标 + 维度限定 + 聚合算子 近 7 天每个省份的订单总额 DWS 层汇总产出
复合指标 多个派生指标的运算结果 客单价 = 订单总额 / 订单数 ADS 层或应用端计算

指标体系设计原则：

  1. 一致性：同一指标在不同报表中定义必须一致（如 GMV 是否含退款）。
  2. 可拆解性：复合指标必须能拆解为原子指标，便于排查差异。
  3. 全量枚举：数据团队产出指标字典，业务方确认，避免口径争议。

深入思考：

  • 指标管理平台：大型公司需要统一的指标管理平台（如腾讯的 TData、阿里的 OneData），通过元数据系统管理指标的定义、口径、下游依赖、变更历史。
  • 指标一致性校验：不同来源（实时 vs 离线）的同一指标需要做交叉验证。如离线计算的昨日 GMV 与实时计算的前一天 GMV 差值应在 1% 以内，超过阈值触发告警。

生产踩坑案例：

某电商公司离线 GMV 和实时 GMV 差异达到 15%。排查发现：离线是按照”订单支付完成时间”统计，实时是按照”事件发生时间”统计，导致同一笔订单可能跨日期归属。统一口径为”支付完成时间”后差异降至 0.3%。

8. Kimball vs Inmon vs Data Vault 建模方法论对比

标准答案：

维度 Kimball（维度建模） Inmon（范式建模） Data Vault
核心理念 自底向上，先数据集市再企业数仓 自顶向下，先企业级模型再数据集市 审计/溯源驱动，可扩展建模
建模方式 星型/雪花型 3NF（第三范式） Hub / Link / Satellite
适用范围 部门级 BI 快速交付 企业级统一数据平台 大数据、复杂源、需要审计
开发周期 短（2-4 周可出第一个数据集市） 长（6-12 月建模完成再迭代） 中（模式灵活但学习曲线陡）
变更响应 快（新增维度/事实直接扩展） 慢（需修改模型全局影响） 快（新增 Hub/Sat 不影响已有）
历史追踪 通过 SCD 实现 通过时间戳实现 Satellites 天然支持

9. Lambda 架构与 Kappa 架构对比与选型

标准答案：

Lambda 架构：两套链路并行——批量层（Batch Layer）处理全量历史数据保证准确性，速度层（Speed Layer）处理实时增量数据保证低延迟，服务层（Serving Layer）合并两者结果。

Kappa 架构：仅保留实时处理层，所有数据以流的方式处理，通过重放历史数据来修正结果，无需单独的批量层。

维度 Lambda Kappa
处理模式 批 + 流两套代码 一套流引擎处理所有
准确性与一致性 批处理保证最终准确 重放数据修复（需消息队列持久化足够长）
开发成本 高（两套代码逻辑对齐） 低（一套代码）
运维成本 高（两条链路独立部署监控） 低（单一链路）
容错能力 强（任一条链路出问题另一条能兜底） 依赖于消息队列的持久化
适用场景 业务要求极端准确性（金融） 数据量大、追求敏捷开发

深入思考：

  • Kappa 架构的实际瓶颈：Kappa 架构假设消息队列（Kafka）可以保存全量历史数据用于重放，但实践中 Kafka 的磁盘成本过高。改进方案：Kappa + 数据湖（Iceberg/Hudi）的存储底座，实时写 Iceberg 增量，历史重放从 Iceberg 读取。
  • 混合架构：现在很多公司采用”Lambda 外壳 + Kappa 内核”——实时链路用 Flink SQL 产出 DWS，离线链路也用同一套 SQL 跑批，通过 Flink 的批流一体能力消除代码差异。

生产踩坑案例：

某短视频公司采用纯 Lambda 架构，实时指标和离线指标天天对不上。原因是实时链路的字典表（如城市映射）更新不及时。改为 Kappa 架构后，字典变更作为事件流处理，指标一致性从 92% 提升到 99.9%。

11. 实时指标计算（DWM → DWS → ADS 实时链路）

标准答案：

实时数仓在 DWD 之上构建多层实时计算：

DWM（Data Warehouse Middle）中间层：对 DWD 明细数据进行轻量聚合，预计算公共统计量（如 PV、UV、去重统计），减少重复计算。

DWS（Data Warehouse Service）服务层：按主题做宽表化，每个主题输出多个维度组合的汇总结果（如按小时 × 省份 × 品类的 GMV）。

ADS（Application Data Store）应用层：面向具体业务的指标，通常直接写 Redis/MySQL/ClickHouse 供应用查询。

深入思考：

  • 实时去重的工程实现：COUNT(DISTINCT) 在 Flink 中需要全量状态存储，数据量大时状态膨胀严重。优化方案：
    1. HyperLogLog：近似去重，误差 2%，状态极小。
    2. RoaringBitmap：精确去重，状态比原始值压缩 10-100 倍。
    3. 布隆过滤器：近似去重，适合黑名单场景。
  • 多层链路延迟：DWM → DWS → ADS 每层都引入处理延迟，总延迟 = 各层窗口大小 + 处理时间。建议统一窗口周期（如 1min DWM → 5min DWS → 1min ADS），避免窗口叠加导致延迟急剧增长。

生产踩坑案例：

某直播平台实时大屏接入层 ADS 用 Redis 缓存，Flink 每 1 秒写入一次，导致 Redis TPS 飙到 20 万，主从复制延迟超 10 秒。优化：Flink 端做 TumblingWindow 每 5 秒聚合一批写入，并使用 Pipeline 模式批量发送，Redis TPS 降至 2 万，大屏延迟反而降低。

14. OLAP 分类（ROLAP / MOLAP / HOLAP / MPP）

标准答案：

类型 原理 代表引擎 优势 劣势
ROLAP 直接查询关系表，实时计算 Presto/Trino/Impala/ClickHouse 灵活，支持明细查询 大查询性能依赖资源
MOLAP 预计算聚合结果，查询预聚合 Kylin/Druid/Doris（部分模式） 查询极快，亚秒级响应 灵活性差，需要预定义维度
HOLAP 混合模式，聚合走 MOLAP，明细走 ROLAP Kylin + Presto 组合 兼顾性能与灵活性 架构复杂

深入思考：

  • ROLAP vs MOLAP 的本质区别：ROLAP 不做预处理，查询时实时扫描数据，拼的是算力和资源；MOLAP 做预处理（Cubing），查询时返回已经算好的结果，拼的是预计算能力。

  • Doris 的分类：Apache Doris 属于 MPP 架构的 ROLAP，但其 Aggregate Key 模型支持部分预聚合（同维度相同 Key 自动合并）。

15. 预计算：Kylin / Druid / ClickHouse 物化视图

标准答案：

Apache Kylin：

  • 原理：预计算 Cube（多维立方体），将原始数据按所有维度组合预先聚合。
  • 核心概念：Cubes（多维数据集）、Cuboid（聚合维度子集）、Segment（数据分段）。
  • 优点：亚秒级查询海量数据（千亿级别）。
  • 缺点：Cube 构建耗时，维度组合爆炸（维度 N，Cuboid 数 ≈ 2^N），不支持实时更新。

Apache Druid：

  • 原理：时间分区 + 预聚合 + 列存，Segment 是核心存储单元。
  • 特点：支持实时摄入（Tranquility/Kafka Indexing Service），亚秒级时间序列分析。
  • 优点：实时写入、高并发、时间序列查询极快。
  • 缺点：不支持 Join（必须预关联）、SQL 支持有限。

ClickHouse 物化视图：

  • 原理：数据写入时同步触发物化视图的聚合，数据仍在原始表 + 物化视图副本。
  • 特点：数据实时写入，物化视图实时更新，不阻塞写入。
  • 优点：实时性好、灵活（可以建任意物化视图）、资源消耗可控。
  • 缺点：物化视图只对新写入数据生效，历史数据需手动处理。

深入思考：

  • Kylin 的”维度爆炸”解决方案：

    1. 必要维度裁剪：只选择查询频繁的维度组合构建 Cuboid。
    2. 层级维度：如 省 → 市 → 区 不构建所有组合，只构建 省、省+市、省+市+区。
    3. 聚合组：将互斥维度放在不同聚合组。

  • Druid vs Kylin 的选型：

    • 需要实时摄入 + 监控类查询 → 选 Druid。
    • 需要精确的 Join + 多维分析（BI 报表） → 选 Kylin。
    • 两者都做预计算，但 Druid 更偏时序，Kylin 更偏多维聚合。

生产踩坑案例：

某公司使用 Kylin，业务维度 20+ 个，开启自动构建 Cuboid，结果每天 Cube 构建耗时 8 小时以上，严重影响数据产出时效。优化：手动指定 mandatory_dimensions 和 hierarchy_dimensions，Cuboid 数从 100 万降到 2000，构建时间从 8 小时降到 30 分钟。

19. 数据血缘（Atlas / Spark Lineage / Flink Lineage）

标准答案：

数据血缘（Data Lineage）记录数据从产生到消亡的完整链路，包括数据从哪里来、经过什么转换、被哪些下游消费。

各引擎的血缘能力：

引擎 血缘粒度 实现方式 支持层级
Apache Atlas 字段级 Hook（Hive Hook/Spark Hook/Sqoop Hook）注入 Kafka → Atlas 消费 表级 + 列级
Spark Lineage 字段级 DataFrame/Dataset 的 QueryExecution 记录血缘 RDD 级/作业级
Flink Lineage 表级 Flink Table API 的 Catalog 自动记录 表级
DataHub 字段级 通过 Ingestion Recipe（Spark/Airflow/dbt/Great Expectations） 列级

深入思考：

  • 字段级血缘为何重要：表级血缘只能知道表 A → 表 B，但不知道是 A 的哪一列产生了 B 的哪一列。字段级血缘可以精准定位——例如报表中”订单金额”字段异常，溯源到源头发现是”优惠券抵扣”字段未纳入计算。
  • Spark Lineage 的实现：Spark 的 SQL 执行计划（LogicalPlan → PhysicalPlan）中记录了每个输出列由哪些输入列经过哪些表达式计算得来。通过 QueryExecution.analyzed 可以解析出字段间的映射关系。
  • Flink Lineage 的挑战：Flink 的 DataStream API 没有 SQL 执行计划，无法自动解析字段级血缘。Flink SQL 通过 QueryOperation 可以获取表级血缘。字段级血缘需要自研或通过 Calcite 解析 SQL 来实现。

生产踩坑案例：

某公司用 Atlas 做血缘，发现某些 Hive 表的血缘丢失。排查原因：ETL 任务使用了 INSERT OVERWRITE ... SELECT 但没有使用 Hive Hook，或者 Hook 版本不匹配。解决方案：统一 ETL 任务的提交方式（HiveServer2 或 Hive CLI 兼容的 Hook），并做定期的血缘校验——比对 Atlas 中的表和 Hive Metastore 中的表，找出缺失血缘的表。

24. DAG 调度依赖管理

标准答案：

DAG（有向无环图）是工作流调度的核心概念，定义任务间的先后执行关系。

常见的依赖类型：

类型 说明 示例
时间依赖 上游任务执行完成后触发 A → B（B 在 A 完成后启动）
数据依赖 上游数据就绪后触发 表 A 分区就绪 → 启动 B
条件依赖 满足某些条件才触发下游 A 成功 → B，A 失败 → C
循环依赖 不同 DAG 间的跨日依赖 DAG-1 今天产出 → DAG-2 今天开始

依赖管理最佳实践：

  1. 数据驱动调度（Data-driven Scheduling）：用”数据就绪事件”触发任务，而非固定时间。如 Hive 分区就绪后自动触发下游。
  2. DAG 拆分原则：一个 DAG 不要超过 100 个 Task，拆分为多个子 DAG 通过 Sensor 触发。
  3. 跨 DAG 依赖：通过 ExternalTaskSensor（Airflow）或 SubProcess 任务（DolphinScheduler）处理。
  4. 循环依赖检测：所有调度平台都应提供 DAG 检测功能，提交前校验是否有环。

深入思考：

  • 数据依赖的工程实现：

    1. Hive Partition Sensor：监控 PARTITION 是否存在，如果存在则认为数据就绪。
    2. HTTP Sensor：调用数据平台 API 确认数据状态。
    3. 文件传感器：HDFS/OSS 文件是否存在。

    传感器轮询策略：指数退避 + 最大超时时间，避免频繁轮询浪费资源。

  • 多租户隔离：调度系统需支持不同部门的 DAG 隔离，防止一个 DAG 耗尽所有 Worker。实现方式：

    • 标签匹配（Tag-based Scheduling）：DAG 打标签 → 特定 Worker 只处理对应标签的任务。
    • 资源池（Pool/Queue）：Airflow 的 Pool 机制限制指定 DAG 的并发 Slot。

生产踩坑案例：

某公司配置了跨 DAG 的时间依赖，但因为夏令时切换导致时间偏移，部分任务的触发时间错乱，导致联动错误。解决方案：所有调度使用 UTC 时间，业务显示时转换为本地时间。

25. 补数 / 重跑 / 告警机制

标准答案：

补数（Backfill）：

  • 定义：对历史某段时间的数据重新计算。
  • 场景：模型上线第一天跑历史数据、上游数据补录、发现历史数据错误。
  • 实现：
    • Airflow：airflow dags backfill -s 2024-01-01 -e 2024-01-10 my_dag
    • DolphinScheduler：选择补数日期范围 → 一键执行。

重跑（Rerun）：

  • 定义：仅重跑失败或错误的任务。
  • 场景：ETL 任务失败、数据质量问题需要修复。
  • 实现：
    • 仅重跑失败 Task，不重跑成功 Task。
    • 重跑时自动清空目标数据（DELETE 对应分区）再写入。

告警机制：

告警级别 触发条件 通知方式 响应
Info 任务正常完成 无/日志 无需处理
Warning 任务失败 < 3 次 钉钉/WeChat/邮件 值班人员关注
Error 任务失败 > 3 次 电话/SMS + 钉钉 立即响应
Critical 关键链路中断 > 30 分钟 电话 + 工单 拉群处理

核心告警指标：

  1. 延迟告警：任务产出时间超过 SLA 阈值。
  2. 失败告警：Task 重试 N 次后仍然失败。
  3. 数据波动告警：上下游表行数差异超过 10%（自动对账）。
  4. 链路中断告警：DAG 中关键路径断裂。

深入思考：

  • 幂等性设计：补数和重跑的基础是任务幂等——同一份数据跑多次得到相同结果。对于 SCD Type 2 的表，幂等性更难保证，需要额外的去重逻辑。
  • 并行补数策略：大范围补数（如补 3 个月数据），建议按分区并发执行，但控制并发度避免压垮集群。

生产踩坑案例：

某公司补数时并行度设置过高（30个分区同时计算），导致 HDFS NameNode 负载飙升，产线其他任务全部阻塞。引入补数限流机制：补数并发数 = min(分区数, 集群空闲资源比例 × 最大并发)。

28. 存算分离（HDFS 对象存储化 / 数据湖）

标准答案：

什么是存算分离：存储和计算解耦，存储层使用廉价的分布式存储（HDFS / 对象存储），计算层按需弹性伸缩，不需要时释放。

演进路径：

关键技术选择：

方案 存储后端 事务支持 Schema 演进 实时写入 适用场景
HDFS + Hive HDFS 不支持 有限 不支持 传统离线数仓
Iceberg HDFS/S3 ACID 支持 支持（Flink） 实时湖仓
Hudi HDFS/S3 ACID 支持 支持（Flink） 增量更新场景
Delta Lake HDFS/S3 ACID 支持 有限 批流一体

深入思考：

  • 为什么存算分离能省钱：传统 Hadoop 集群中，计算和存储资源绑定。即使只需要存储扩容，也得加一批带 CPU/内存的计算节点。存算分离后，存储用对象存储（价格约为 HDFS 的 1/3），计算用弹性集群（按需启动，用完释放）。
  • 存算分离的代价：
    1. 延迟增加：对象存储（S3）的读写延迟高于 HDFS，对延迟敏感的任务（如实时写入）有影响。
    2. 网络带宽：数据在网络中传输，需要大带宽支持。
    3. 数据本地性丧失：不再保证”计算靠近数据”，Shuffle 开销增大。

生产踩坑案例：

某公司将 HDFS 全部替换为 S3，跑 Spark 作业发现比原来慢 3-5 倍。原因是 Spark 的 Shuffle 依赖本地 SSD 的随机读写，S3 的随机读延迟太高。解决方案：

  1. 使用 S3 做冷存储（历史数据），HDFS 做热存储（最近 7 天数据）。
  2. Spark Shuffle 数据写本地 SSD，最终结果写 S3。

  • 劣势：不支持微服务、不支持 GPU 调度、扩展性有限

K8s 混部：

  • 调度模型：Pod-based
  • 调度平台：Kubernetes（Volcano / YunKai）
  • 适用场景：Flink on K8s、Spark on K8s、微服务 + 大数据混部
  • 优势：资源利用率高（在线 + 离线混部）、弹性伸缩

Volcano：

  • 华为贡献的 K8s 批处理调度器
  • 核心特性：Gang Scheduling（批量调度）、Fair Share、Queue 资源管理
  • 适用：Spark/Flink/MPI 等批处理作业

YunKai：

  • 阿里巴巴开源的 K8s 调度器
  • 核心特性：Binpacking 提高资源利用率、Cost-based Scheduling
  • 适用：在线 + 离线混合部署

深入思考：

  • YARN vs K8s 选型：
    • 已有 Hadoop 集群且稳定运行 → 继续用 YARN。
    • 新项目 / 云原生改造 / 需要混部 → 选 K8s。
    • Flink on K8s 已经是大势所趋（Flink 官方 1.12+ 原生支持 K8s）。
  • 混部的核心技术：
    • 资源隔离：cgroup v2 限制 CPU/内存，blkio 限制磁盘 IO。
    • 优先级抢占：在线应用优先，离线作业使用闲置资源，在线高峰时离线主动退让。
    • CPU 弹性：离线使用”绑核 + Burstable”模式，在线使用”独占 + Guaranteed”模式。

生产踩坑案例：

某公司 Flink on YARN 集群在高峰期出现资源不够，但集群总 CPU 利用率只有 30%。原因是 YARN 的 yarn.scheduler.capacity.maximum-am-resource-percent 默认值 0.1，限制了应用管理器资源。调整后资源利用率提升到 65%。后续迁移到 Flink on K8s 后进一步利用混部能力，资源利用率达到 85%。

31. 成本优化：冷热分层 / 压缩 / 生命周期管理

标准答案：

冷热分层存储：

类别 访问频率 存储方式 存储成本占比 查询性能
热数据 最近 7 天高频访问 SSD / HDFS（Replica=3） ~60% 亚秒级
温数据 7-90 天，偶尔查询 HDFS（Replica=2 + EC） ~25% 秒级
冷数据 90 天以上，极少查询 对象存储（归档） / 磁带 ~15% 分钟级

压缩策略：

压缩格式 压缩比 压缩速度 解压速度 是否可分割 推荐场景
gzip 4:1 慢 慢 否 单机小文件
snappy 2:1 极快 极快 是 实时计算中间数据
lz4 1.5:1 极快 极快 否 Flink 状态压缩
zstd 5:1 中 中 是 Hive 表压缩
zlib 6:1 慢 慢 是 历史归档

生命周期管理：

  • 表生命周期：设置保留周期（如 30 天），过期自动删除或转入冷存储。
  • 分区生命周期：按时间分区自动清理，如保留最近 90 天的分区。
  • 小文件合并：定时将小文件合并为 256MB~512MB 的大文件，减少 NameNode 压力。
  • 无效数据清理：定期扫描并清理中间表、临时表（超过 7 天无访问）。

深入思考：

  • HDFS 纠删码（Erasure Coding）：HDFS 3.x 支持 EC，将 3 副本的 300% 存储开销降低到 150%（如 RS-6-3：数据 6 块 + 校验 3 块，可靠性等价 3 副本）。适合冷数据。
  • 列存 + 压缩：ORC + zstd 是性价比最高的 Hive 表存储方案，压缩比可达 1:8，查询性能优于 Parquet + snappy。
  • Flink 状态后端优化：Flink 使用 RocksDB 状态后端时，启用 state.backend.rocksdb.block.blocksize=32KB + state.backend.rocksdb.compression.type=lz4，状态存储减少 50%。

生产踩坑案例：

某公司所有 Hive 表使用 gzip 压缩，虽然压缩比高（7:1），但查询时需要逐条解压，查询速度极慢。切换到 zstd + ORC 后，压缩比 6:1，但查询速度提升了 5 倍。进一步分析：gzip 不可分割，每个 Mapper 只能处理一个 gzip 文件，并行度受限，而 zstd 支持文件分割。

33. 实时链路延迟优化（从秒级到毫秒级）

标准答案：

实时链路端到端延迟优化是一个系统工程，需要从数据接入 → 计算 → 输出的每一个环节做优化。

延迟拆解：

环节 延迟来源 优化手段 延迟改善
T1：采集 Binlog 采集间隔 调整 Binlog 实时 Push / Canal 的 slaveId 避免长事务 500ms → 50ms

T4：输出 目标存储写入抖动 批量写入 + Buffer 500ms → 50ms
T5：应用 查询延迟 预计算 + Redis 缓存 100ms → 1ms

目标 批量大小 写入间隔 方式
Redis 100-500 条/批 100ms Pipeline 模式
ClickHouse 1000-3000 条/批 1s 本地表写入后分布式表查询

深入思考：

  • 端到端 Exactly-Once vs 延迟的取舍：
    • 开启 Checkpoint + 二阶段提交 → 保证 EOS，但延迟增加（每次 Checkpoint 有状态快照）。
    • 优化方案：增大 Checkpoint 间隔（建议 30-60s），使用增量 Checkpoint（RocksDB Incremental Checkpoint）。

    • 使用 MapState 替代 ValueState + ListState 的组合，减少序列化开销。

生产踩坑案例：

某金融风控系统要求实时延迟 < 50ms，但实际端到端延迟一直在 500ms 以上。逐层排查：

  1. 采集层：Canal 的 batchSize 太小（100），每条数据都触发一次 Kafka 发送 → 改 batchSize=1000 + linger.ms=10。
  2. 传输层：Kafka 的 acks=all 导致每次等待所有副本确认 → 改 acks=1（风控场景允许极少日志丢失容错）。
  3. 计算层：Flink 窗口使用 Event Time + 2s 容忍延迟 → 改 Processing Time + 直接计算。
  4. 输出层：写入 Redis 使用单条写入 → 改 Pipeline 模式 500 条/批。
    最终延迟降到 80ms，风控规则引擎可接受。
