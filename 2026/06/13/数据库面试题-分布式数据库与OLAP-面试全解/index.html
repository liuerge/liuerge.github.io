0x00 TiDB 深度

0x00-01 TiDB 整体架构是怎样的？各组件职责是什么？

标准答案：
TiDB 采用计算存储分离的三层架构：

  1. TiDB Server（无状态 SQL 层）：负责接收客户端连接，解析 SQL，生成分布式执行计划。自身不存储数据，通过 PD 获取元数据后下推计算任务到 TiKV/TiFlash。
  2. PD (Placement Driver)：集群的”大脑”。维护全局元信息（Region 路由表），分配全局事务 ID（TSO），执行 Region 调度（负载均衡、Leader 迁移、故障恢复）。由至少 3 节点通过 etcd/Raft 构成高可用组。
  3. TiKV（行存引擎）：分布式 Key-Value 存储。数据按 Key Range 切分为 Region（默认 96MB），每个 Region 通过 Multi-Raft 复制 3 副本保证强一致。支持 MVCC 和分布式事务。
  4. TiFlash（列存引擎）：TiKV 的列存 Learner 节点。通过 Raft Learner 实时同步 TiKV 数据，提供列式存储和向量化计算，为 OLAP 查询加速。与 TiKV 共享同一份数据的最新版本。

深入思考：

  • 边界情况：TiDB Server 无状态意味着可随意扩缩容，但大查询的中间结果集仍在 TiDB 内存中，需要配置 mem-quota-query 防止 OOM。
  • 源码层面：PD 使用 etcd 的 Raft 实现保证高可用，但 etcd 本身对磁盘 IO 敏感，PD 节点磁盘延迟升高会影响 TSO 分配，进而拖慢全集群写入。
  • 对比延伸：与 OceanBase 的 RootService 对比——OB 的 RootService 是主节点模式（Active-Standby），TiDB 的 PD 通过 Raft 选举，两者都是元数据管理核心但 HA 机制不同。

生产踩坑案例：
某金融客户 TiDB 集群 PD 节点磁盘使用率超过 80%，导致 etcd 写入延迟飙升，TSO 分配超时，全集群写入 P999 从 5ms 暴涨到 500ms。排查发现是 PD 的 gc-safe-point 历史记录未清理。解决：配置 pd-ctl scheduler config gc-emphasis 并清理历史 Region 信息。

0x00-02 TiKV 中的 Raft 是如何工作的？Region 是什么？Leader/Follower 如何调度？

标准答案：
TiKV 将整个 KV 空间按 Key 范围切分为若干 Region（默认 96MB）。每个 Region 是一个独立的 Raft Group，包含 3 个或更多副本（Peer）。每个 Peer 分布在不同的 TiKV 节点上。

Raft 工作流程：

  1. Leader 选举：每个 Region 有一个 Leader，负责处理所有读写请求。Leader 通过心跳维持任期，宕机后 Follower 触发选举。
  2. 日志复制：Leader 接收写请求后写入 Raft Log，并行复制到 Follower 节点，多数派（Quorum）确认后提交。
  3. 读写路径：读请求也由 Leader 处理（Lease Read / Read Index 保证线性一致读），Follower 可提供 stale read。

PD 调度机制：

  • Leader 均衡：将访问热点的 Region Leader 分散到不同 TiKV 节点
  • Region 均衡：根据 Store 的 Region 数量/大小，通过调度器移动 Region Peer
  • 故障恢复：检测到 Peer 不可用后，在健康节点上补建新 Peer
  • Merge/Split：Region 大小超过 96MB 时自动分裂，小于阈值时合并减少元数据开销

深入思考：

  • 边界情况：Region 分裂是 PD 触发 TiKV 在本地执行 split 操作，不涉及数据搬迁。但如果原始 Region 是热点，分裂后的两个 Region 可能都还在同一节点，需要调度器介入。
  • 源码层面：Raft 在 TiKV 中基于 etcd raft-rs 库实现，但 TiKV 使用 Multi-Raft（每个 Region 一个 Raft Group），相比 etcd 的单 Raft Group，元数据开销随 Region 数量线性增长。千万级 Region 场景下，Raft 心跳消息成为网络瓶颈。
  • 生产经验：大表创建后，一开始只有一个 Region，写入全部落在一个 TiKV 节点。需要开启 split-table + load-base-split 让 TiDB 预分裂 Region。

生产踩坑案例：
某 IoT 场景使用自增主键写入，所有数据落在最后一个 Region 上形成写入热点。虽然 PD 尝试迁移 Leader，但写流量集中在单一节点。解决：改用 Hash 分区键（如 user_id % 64）作为主键，或开启 load-base-split 让 PD 根据 QPS 自动分裂热点 Region。

  • https://pingcap.medium.com/tidb-internal-iii-scheduling-de1001ef207d

0x00-03 TiDB 分布式事务的工作原理？乐观锁与悲观锁的区别？

标准答案：
TiDB 的分布式事务基于 Google Percolator 模型，结合 MVCC（多版本并发控制）实现快照隔离（SI）/可重复读（RR）。

乐观事务（默认）：

  1. Prewrite 阶段：客户端从 PD 获取 start_ts，将所有写入的 Key 加锁并写入 TiKV。锁包含主锁（Primary Lock）和从锁（Secondary Lock）。
  2. Commit 阶段：获取 commit_ts，提交主锁释放主 Key，异步提交从锁。
  3. 冲突检测：Prewrite 时如果发现 Key 已被锁定，则事务冲突，需要回退重试。

悲观事务：

  1. 加锁阶段：事务执行 DML 时立即对涉及行加悲观锁（for_update_ts），防止其他事务修改。
  2. Prewrite/Commit：同乐观模式，但因为已持有锁，Prewrite 冲突概率极低。
特性 乐观锁 悲观锁
冲突检测时机 Commit 时 执行 DML 时
冲突场景性能 冲突高时大量回退重试 额外 RTT 加锁成本
适用场景 低冲突、短事务 高冲突、长事务
死锁检测 无（靠超时） TiDB 内置死锁检测器

深入思考：

  • 边界情况：乐观事务在冲突率高时（>5%），系统吞吐量断崖式下跌，因为大量事务在 Commit 时回退重试。这时必须切换到悲观锁。
  • 源码层面：Percolator 模型有个著名缺陷——Secondary Lock 提交失败会导致数据不一致（Primary 已提交，Secondary 未提交）。TiDB 通过异步提交 + 后台”锁清理”（Lock Resolver）巡检补偿来保证最终一致。
  • 对比延伸：与 Spanner 的 TrueTime 对比——Percolator 需要中心化的 TSO 分配器，受时钟漂移影响；TrueTime 用原子钟+GPS 消除了中心化瓶颈。

生产踩坑案例：
某电商库存系统使用乐观事务，高峰期库存扣减冲突率高达 30%，大量事务回退导致成功率不到 60%。改为悲观锁后，虽然单次延迟从 5ms 升到 15ms，但吞吐量提升 3 倍，成功率恢复到 99.9%+。

  • https://www.mydbops.com/blog/optimistic-and-pessimistic-locking-in-tidb

0x00-04 TiDB 的 HTAP 架构是如何实现的？TiFlash 如何保证与 TiKV 的数据一致性？

标准答案：
TiDB 的 HTAP 方案通过”行存 TiKV + 列存 TiFlash”双引擎实现，用户无需关心数据路由——优化器根据 SQL 类型自动选择引擎。

TiFlash 架构要点：

  1. Raft Learner：TiFlash 以 Raft Learner 角色加入 TiKV 的 Raft Group。Learner 不参与选举，只接收 Raft Log 实时同步数据。
  2. DeltaTree 引擎：列式存储引擎，数据按列压缩（LZ4/ZSTD），支持向量化执行（SIMD）。
  3. 一致性保证：TiFlash 提供 Snapshot Isolation 级别的读一致性，与 TiKV 一致。读请求检查 apply index，确保读到最新已提交数据。
  4. 智能路由：TiDB 优化器根据代价模型决定查询下推到 TiKV（行存）还是 TiFlash（列存），通过 tidb_isolation_read_engines 变量可手动控制。

深入思考：

  • 边界情况：如果 TiFlash 同步延迟较大（如网络抖动），TiDB 可能读取到旧版本数据？实际上 TiFlash 在查询时会等待 apply index 追上 read timestamp，所以不会读到过时数据，但会增加查询延迟。
  • 源码层面：TiFlash 通过 tiflash replica 命令建立 TiKV 表副本。每个 Region 在 TiFlash 中对应一个 Segment，Segment 大小默认 256MB 至 2GB，远大于 TiKV 的 Region。这是为了避免列存中过多小文件影响扫描性能。
  • 生产经验：HTAP 混合负载时注意资源隔离——通过 tidb_enforce_mpp 将 OLAP 查询强制走 TiFlash，避免消耗 TiKV 的 CPU/IO 资源影响 TP 业务。

生产踩坑案例：
某 BI 报表场景，运行大查询时 TiKV CPU 冲到 100%，TP 业务 P99 延迟从 10ms 飙到 2s。排查发现查询未走 TiFlash，而是直接在 TiKV 上全表扫描。根因：TiFlash 副本同步延迟过大，优化器判断走 TiKV 更快。解决：调整 tidb_allow_mpp 强制走 TiFlash，并优化 TiFlash 与 TiKV 之间的网络带宽。

参考链接：

  • https://docs.pingcap.com/tidb/stable/tiflash-overview
  • https://github.com/pingcap/tiflash/blob/master/docs/design/0000-00-00-architecture-of-distributed-storage-and-transaction.md

0x00-05 TiDB 7.x/8.x 有哪些重要新特性？

标准答案：

TiDB 7.1 LTS (2023)：

  • Resource Control GA：多租户资源隔离，通过 RESOURCE GROUP 限制 CPU/IO/内存，防止 OLAP 影响 OLTP
  • TiFlash 半持久化：减少 TiFlash 内存压力
  • Grafana 告警增强：内置 50+ 告警规则

TiDB 7.5 LTS (2024)：

  • Global Sort：减少 BR/Restore 和 DDL 过程中对缓存的依赖，降低大表 DDL 对 I/O 的冲击
  • 并行 ADD INDEX：支持对单表并发添加多个索引，DDL 时间大幅缩短
  • TiFlash 查询下推增强：更多算子下推（窗口函数、JSON 函数等）
  • AutoAnalyze 增强：根据查询模式智能选择统计信息收集时机

TiDB 8.x (2025)：

  • Global Memory Control：全局视角的内存管理，防止单条查询撑爆 TiDB 节点
  • TiFlash MPP 增强：支持跨节点 Shuffle Hash Join，大表 JOIN 性能飞跃
  • Placement Policy in SQL：通过 SQL 定义数据放置策略（跨 AZ/跨 Region）
  • TiKV 异步 IO：降低写延迟 P99
  • TTL 自动分区管理：时序数据自动过期清理

深入思考：

  • 生产经验：Resource Control 是集群级隔离，不是节点级——意味着即使配置了资源组，同节点上的不同租户仍共享 CPU 资源。
  • 选型意义：8.x 的 Global Memory Control 对于多租户场景是革命性特性——在此之前，一条大查询可以耗尽 TiDB Server 内存导致 OOM。现在可以精确限制每个 Query 的内存使用。

0x01 OceanBase

0x01-01 OceanBase 整体架构是怎样的？RootService 与 OBServer 的职责？

标准答案：
OceanBase 采用 Shared-Nothing 架构，所有节点对等，每个节点上运行 OBServer 进程。

核心组件：

  1. OBServer：每个节点运行一个 observer 进程，内含 SQL 引擎、事务引擎、存储引擎，对等提供服务。
  2. RootService（RS）：运行在某个 OBServer 上的主控进程（Active-Standby 模式），负责集群管理、分区调度、负载均衡、DDL 操作。通过选举产生主 RS，备 RS 随时接管。
  3. Zone：逻辑上的可用区概念，一个 Zone 包含一组 OBServer。数据在同一 Paxos Group 内的副本分布在不同的 Zone 中，实现跨 AZ/跨 Region 容灾。
  4. 租户（Tenant）：逻辑数据库实例，每个租户有独立的资源池（CPU/内存/存储），互不干扰。系统租户（sys）管理集群，普通租户提供业务服务。

数据分片：表按用户指定的分区规则（Hash/Range/List）水平切分为 Partition，每个 Partition 的多个副本组成 Paxos Group。

深入思考：

  • 对比延伸：TiDB 的 PD 通过 Raft 保证高可用（3 节点投票），OB 的 RootService 是 Active-Standby 选举，Leader 挂掉后在 OBServer 中重新选举。OB 的方式更轻量，但切 RS 期间（约 30s）新 DDL 无法执行。
  • 架构差异：OB 是”计算存储不分离”，每个 OBServer 同时承载计算和存储；TiDB 是真正的计算存储分离。OB 的好处是减少网络开销（本地 IO），坏处是扩缩容需要数据搬迁。

生产踩坑案例：
某银行 OB 集群 RS 节点硬件故障，RootService 切主后，新 RS 需要重新收集全集群 Partition 分布信息，耗时 5 分钟。期间 DDL 操作全部阻塞。解决：部署 RS 为独立资源组，确保 RS 所在节点配置和隔离性最佳。

参考链接：

  • https://oceanbase.github.io/oceanbase/architecture
  • https://en.oceanbase.com/docs/enterprise-oceanbase-database-en-10000000000384579

0x01-02 OceanBase 的分布式选举（Paxos/Paxos Group）是如何工作的？

标准答案：
OceanBase 使用自研的 Multi-Paxos 协议（基于 Basic Paxos 优化）实现数据副本的强一致。

Paxos Group：

  1. 每个 Partition 的多个副本组成一个 Paxos Group（通常 3 副本）。
  2. Redo Log 通过 Paxos 同步：写操作需要多数派（Quorum）确认才算提交。
  3. Leader 处理读写（Strong Read，强一致读），Follower 可提供弱一致读。

选举流程：

  • Leader 通过定期心跳维持领导地位
  • Leader 宕机后，Follower 触发选举（通过 Paxos Prepare/Accept 阶段）
  • 选出的新 Leader 通过日志追赶（Log Catchup）补全未同步的 Redo Log

与 Raft 的核心区别：

特性 Raft OceanBase Paxos
日志顺序 强顺序，日志连续 允许空洞，更加灵活
选举 随机超时触发 基于租约+优先级

深入思考：

  • 源码层面：OB 的 Paxos 实现中，日志副本数不是固定的 3——生产环境常用”5 副本跨 3 AZ 部署”：主 AZ 3 副本，备 AZ 2 副本，允许单 AZ 整体故障而不丢数据。
  • 生产经验：Paxos Group 太小（如 3 副本）时，多 Zone 场景下备 Zone 的磁盘延迟会影响主 Zone 写入——因为每次 Redo Log 都需要跨 AZ 确认。

0x01-03 OceanBase LSM-Tree 存储引擎的工作原理？

标准答案：
OB 使用基于 LSM-Tree 的自研存储引擎，将数据分为基线数据和增量数据：

分层结构：

  1. MemTable（内存表）：新写入/修改的数据先写入 MemTable（内存中的有序结构）。写入操作记录 Redo Log 后才算持久化。
  2. SSTable（静态基线）：MemTable 达到阈值（默认 64MB/256MB）后，冻结并刷入磁盘成为 Mini SSTable。多个 Mini SSTable 后台合并为 Major SSTable。
  3. Compaction（合并）：
    • Minor Compaction：多个 Mini SSTable 合并
    • Major Compaction：全量基线数据 + 增量数据合并为新的基线，清理删除标记和过期版本

读放大处理：

  • 读请求需要同时查 MemTable + 所有 SSTable 层级
  • OB 使用 Block Cache + Row Cache + Bloom Filter 三件套加速读路径
  • 每个 SSTable 带有 Bloom Filter，快速判断 Key 是否不存在

深入思考：

  • 对比 RocksDB：OB 的 LSM-Tree 与 RocksDB 最大的区别在于”每日合并”——OB 默认每天执行一次 Major Compaction（可配置），不像 RocksDB 用层级合并（L0→L1→L2…）。这使 OB 的写放大更可控，但空间放大略大。
  • 编码压缩：OB 支持行列混合编码（DICTIONARY / RLE / DELTA / PREFIX），压缩比可达 70-90%，远超 MySQL InnoDB 的页压缩。

生产踩坑案例：
某 OB 集群白天写入量大，MemTable 频繁冻结刷盘，Minor Compaction 跟不上产生速度，导致 SSTable 层级暴涨，读放大严重，查询延迟从 1ms 升到 50ms。解决：调大 freeze_trigger_percentage（从 20% 调到 50%），增加 MemTable 容量，减少冻结频率。同时将 Major Compaction 改到凌晨执行。

0x01-04 OceanBase 的多租户与资源隔离机制？

标准答案：
OB 通过”租户”实现多租户架构，每个租户是一个逻辑数据库实例，拥有独立的数据、用户、系统变量。

资源隔离层级：

  1. CPU 隔离：通过 cgroup（user-mode scheduling fallback）限制租户使用的 CPU 核数。OB 4.x 使用内置 SQL 引擎的优先级调度，以微秒粒度分配 CPU 时间片。
  2. 内存隔离：完全独立。每个租户的 MemTable、SQL 执行内存、Plan Cache 都分隔在不同内存池中。一个租户 OOM 不会影响其他租户。
  3. IO 隔离：4.x 引入了 IOPS 限制，通过 min_iops / max_iops 控制每个租户的磁盘读写带宽。
  4. 数据隔离：每个租户的元数据（系统表）完全独立，一个租户无法看到其他租户的数据。

资源池：

  • CREATE RESOURCE POOL 定义资源规格（CPU/内存/磁盘）
  • 一个租户绑定一个或多个资源池
  • 资源池分布在不同 Zone 上实现高可用

深入思考：

  • 对比延伸：TiDB 的 Resource Control 是集群维度限制（Group），OB 是租户维度隔离。OB 的隔离粒度更细（完全独立内存池），适合严苛的金融级多租户场景。
  • 生产经验：多租户场景下需要合理规划 UNIT 数量——OB 的弹性伸缩通过 ALTER RESOURCE TENANT 动态调整，无需重启。

生产踩坑案例：
某 SaaS 厂商在 OB 集群上部署了 50+ 租户，部分中小租户活跃度低但占用大量内存（MemTable 长时间不释放）。排查发现 MemTable 的 freeze_trigger_percentage 全局设置对中小租户不友好。解决：为不同规格的租户设置不同的 freeze_trigger_percentage，大租户 30%，小租户 60%。

0x02-01 ClickHouse 整体架构是怎样的？Shard/Replica 如何组织？

标准答案：
ClickHouse 采用 Shared-Nothing 的 MPP 架构，典型部署包含以下角色：

  1. Coordinator 节点：接收客户端查询，解析 SQL，生成分布式执行计划，协调各节点执行。CH 中没有独立的 Coordinator 进程——任意节点都可以作为 Coordinator。
  2. Shard（分片）：数据水平切分单元。一张分布式表的数据按分片键分布到多个 Shard 上。
  3. Replica（副本）：每个 Shard 可以有多副本保证高可用。通过 ReplicatedMergeTree 引擎实现跨节点数据复制（基于 ZooKeeper/ClickHouse Keeper 的日志复制）。
  4. ClickHouse Keeper：替代 ZooKeeper 的一致性服务，用于分布式 DDL 协调和 ReplicatedMergeTree 的元数据同步。

分布式表：通过 Distributed 表引擎实现，本身不存数据，作为”视图”将查询分发到所有分片后聚合结果。

深入思考：

  • 对比延伸：CH 没有独立的 Coordinator/Scheduler 进程（不像 Doris 有 FE，Trino 有 Coordinator），任何节点都可以是查询起点。这简化了架构但增加了查询调度的不确定性。
  • Keeper vs ZooKeeper：CH 从 21.8 开始内置 ClickHouse Keeper，使用 Raft 替代 ZooKeeper，运维负担大幅降低。但 Keeper 的稳定性在超大规模集群（50+ 节点）上仍不如 ZK 久经考验。

生产踩坑案例：
某 30 节点集群使用 ZooKeeper 管理 ReplicatedMergeTree 元数据，ZK 节点处理约 5000 TPS 的写入请求（来自 CH 的复制日志），频繁触发 ZK 的 Sync 超时导致 CH 节点间复制中断。解决：迁移到 ClickHouse Keeper 并使用独立磁盘（NVMe SSD），KV 请求延迟从 5ms 降到 0.3ms。

0x02-02 MergeTree 表引擎家族详解？各引擎的适用场景？

标准答案：
MergeTree 是 CH 的核心引擎，所有变体都基于 MergeTree：

1. MergeTree（基础引擎）

  • 数据按 ORDER BY 排序，按 PARTITION BY 分区
  • 支持稀疏主索引（主键隔 N 行建一个索引标记）
  • 适用：原始事件日志、时序数据

2. ReplacingMergeTree

  • 合并时按 ORDER BY 去重，保留最后一条（可配版本列）
  • 适用：实时更新的维度表、变更数据捕获（CDC）
  • 注意：去重只在合并时发生，查询时需加 FINAL 保证拿到最新版本

3. SummingMergeTree

  • 合并时按 ORDER BY 累加数值列，保留一条汇总行
  • 适用：预聚合指标（PV/UV/金额），对实时性不敏感的场景
  • 注意：非确定性合并——相同分片键的行不一定都被合并到一条

4. AggregatingMergeTree

  • 合并时用 AggregateFunction 类型存储聚合中间状态（如 uniqState/avgState）
  • 写入用 -State 函数，查询用 -Merge 函数
  • 适用：复杂预聚合（Distinct Count/分位数/自定义聚合），物化视图底层引擎

5. VersionedCollapsingMergeTree

  • 通过 sign 列（1/-1）和 version 实现”折叠”——相同 version 的 +1 和 -1 行互相抵消
  • 适用：金融数据变更流、平衡表，需要精确的实时合并
  • 查询时用 sum(sign) > 0 过滤有效行

6. CollapsingMergeTree（旧版）

  • 同 VersionedCollapsingMergeTree 但不支持 version，只支持 sign
  • 不推荐新项目使用

深入思考：

  • 选型决策：绝大部分场景用 MergeTree（只追加）即可。需要去重用 ReplacingMergeTree，需要预聚合看 SummingMergeTree 够不够用，不够上 AggregatingMergeTree。需要精确版本控制用 VersionedCollapsingMergeTree。
  • FINAL 的性能代价：SELECT ... FINAL 会在查询时做去重/合并，单线程执行，扫描量巨大。大表频繁加 FINAL 不现实，应配合物化视图 + AggregatingMergeTree 预聚合。

生产踩坑案例：
某用户使用 ReplacingMergeTree 做实时数据去重，高峰 10 万+ TPS 写入，FINAL 查询耗时从 10ms 升到 5s。排查发现存量数据 parts 数太多（3000+），FINAL 需要合并所有 parts。解决：设置 merge_with_ttl_timeout 主动触发合并，同时用物化视图预聚合替代实时 FINAL。

0x02-03 ClickHouse 分区、分片、物化视图

标准答案：

分区（Partition）：

  • 按 PARTITION BY 表达式将数据分为不同目录（partition_id），查询可做分区裁剪
  • 典型：PARTITION BY toYYYYMM(event_date) — 按年月分区
  • 分区数建议 100-1000 个，过多（>1000）导致 PART 数量爆炸

分片（Sharding）：

  • 通过 Distributed 表 + 分片键实现数据水平拆分
  • 分片键一般用 rand() 随机分布，或按业务 ID 保证关联数据在同一分片
  • 注意：CH 不支持自动 rebalancing——增加新分片后老数据不会自动迁移

物化视图（Materialized View）：

  • CH 的物化视图是”触发器”模型：数据写入源表时触发生成新数据写入物化视图表
  • 不是传统意义上的”查询结果快照”
  • 典型架构：源表（MergeTree）+ 聚合表（AggregatingMergeTree）+ 物化视图桥接

深入思考：

  • 物化视图的陷阱：数据是增量写入的——如果源表有历史数据，物化视图不会自动回填。需要先 INSERT INTO events_agg SELECT ... 回填历史，再建物化视图。
  • 分区过多隐患：每个 INSERT 生成一个新 part。如果分区数 3000，每秒 1000 次插入，part 数会爆炸，合并线程跟不上。解决方案：减少分区数 + 使用 UNIQUE 约束合并前合并。

生产踩坑案例：
某日志平台每天 10TB 数据入 CH，按小时分区（toYYYYMMDDHH），每分区每天有 24 个。6 个月后分区数超过 4300，parts 总数 100 万+，合并线程 CPU 耗尽，查询延迟从 100ms 升到 30s。解决：改为按天分区 + 每个分区内使用 TTL 自动清理过期的 parts，同时调大 max_bytes_to_merge_at_max_space_in_pool 加速合并。

0x02-04 ClickHouse 分片策略与数据分布最佳实践

标准答案：
ClickHouse 的分片通过 Distributed 表引擎实现，核心概念包括：

建表方式：集群中的每个分片（Shard）节点创建本地表（Local Table），然后创建分布式表作为统一查询入口。

分片键选择：

  1. rand() — 随机分布，数据均匀但无法利用 Join 本地性
  2. hash(user_id) — 按用户 Hash 分布，同用户数据在同一分片
  3. intHash64(partition_key) — 结合分区键的 Hash 分布

跨分片查询：Distributed 表发送查询到所有分片 → 各分片并行执行 → Coordinator 节点汇总结果。对于 GROUP BY，各分片先做本地聚合，Coordinator 再做二次聚合。

数据重分布难题：

  • CH 不支持内置的自动数据重平衡
  • 增加分片后，新数据写入新分片，但老数据不会自动迁移
  • 解决：使用 ALTER TABLE ... MOVE PARTITION 手动搬迁，或重建表使用 INSERT INTO ... SELECT

深入思考：

  • 分片数量规划：分片数 = 集群计算能力目标 / 单节点算力。但要注意——分片太多（>20）会导致 Coordinator 节点上的二次聚合成为瓶颈。
  • 过少分片问题：如果只有 2 个分片，其中一个故障时系统只剩 50% 计算能力，且 Replica 恢复期间大量数据需要追赶。

生产踩坑案例：
某用户集群从 5 个分片扩容到 10 个分片（节点数翻倍），发现老数据仍然集中在原来的 5 个分片上，半数新节点无历史数据。大范围时间查询时，5 个老节点 CPU 满载而 5 个新节点空闲。解决：使用 ALTER TABLE events_local MOVE PARTITION 将老分区逐步搬迁到新节点，配合业务低峰期分批次执行。

0x02-05 ClickHouse 性能调优：数据排序、稀疏索引、预聚合

标准答案：

1. 数据排序（ORDER BY）

  • ORDER BY 决定数据的物理存储顺序，也是稀疏索引的构建基础
  • 最佳实践：高基数列在前，过滤频繁的列在前
  • 示例：ORDER BY (event_date, city, user_id) — 日期过滤先裁剪大范围

2. 稀疏主索引

  • 主键不是唯一约束，而是”每隔 N 行记一个索引标记”
  • 默认 index_granularity = 8192（每 8192 行一个标记）
  • 索引大小只有行数的 1/8192，完全可加载到内存
  • 查询时先查稀疏索引定位可能的 granules，再逐一扫描

3. 预聚合

  • 物化视图 + AggregatingMergeTree 是最常用的预聚合手段
  • 按常见查询维度（日期/城市/页面）预计算指标
  • 典型场景：实时报表从扫描百亿行变为扫描预聚合表百万行

4. 其他调优参数：

参数 作用
max_threads 单查询并行度（默认 CPU 核数，大查询适当降低）
max_memory_usage 单查询内存限制
merge_tree.min_rows_for_compact_part 控制 part 合并策略
use_uncompressed_cache 未压缩数据缓存（列扫描场景关闭）

深入思考：

  • ORDER BY 顺序至关重要：如果 ORDER BY 把高基数列放前面（如 user_id），稀疏索引会非常”稀释”，分区裁剪效果差。正确做法：过滤频率最高、基数适中的列在前。
  • 稀疏索引 vs B+Tree：CH 的索引粒度是 8192 行，MySQL B+Tree 索引粒度是一行。CH 牺牲了点查能力换取了极致的分析扫描性能。

生产踩坑案例：
某广告分析系统，ORDER BY 设为 (ad_id, event_time)，ad_id 是 UUID，基数千万级。查询按 event_time 过滤时无法有效裁剪，全表扫描。解决：ORDER BY 改为 (event_date, ad_id, event_time)，跳数索引加 event_time。查询速度提升 20 倍。

0x02-06 ClickHouse JOIN 优化策略

标准答案：
ClickHouse 的 JOIN 曾是其最大短板（右表全量加载到内存建 Hash Table），但 24.x+ 版本已经有显著改进。

JOIN 算法演进：

  1. Hash Join（默认）：右表全量加载到内存建 Hash Table。适用：右表小（<内存）。
  2. Parallel Hash Join（CH 24.12+ 默认）：多线程并行建 Hash Table，大右表性能提升 3-5x。
  3. Grace Hash Join（CH 24.x+）：右表超出内存时分片到磁盘，避免 OOM。
  4. Full Sorting Merge Join：双表按 Join Key 排序后归并。适用于大表 Join 且 Join Key 有序。
  5. Direct Join：右表使用 Dictionary/Join 表引擎时，点查式 Join，避免全量加载。

Global Join：

  • 分布式场景中，默认 Join 只在本地节点上执行（右表取本地数据）
  • 使用 GLOBAL JOIN 使右表广播到所有节点，保证数据完整

优化策略：

  1. 右表越小越好：大表左、小表右
  2. 用字典替代维度表 JOIN：使用 dictGet() 函数比 JOIN 快 10-100 倍
  3. 过滤条件下推：WHERE 条件在 JOIN 前执行
  4. 合理使用 ANY JOIN：右表只需要任意匹配一行时，比默认 JOIN 快

深入思考：

  • CH JOIN vs 传统 OLAP：Doris/StarRocks 使用 CBO（代价优化器）支持 10 张以上的复杂 Join，CH 的 Join 能力仍有差距。这是 CH 被批评最多的地方。
  • 生产建议：如果业务需要 5+ 表复杂 Join，慎重选择 CH；优先考虑预 Join（物化视图将 Join 结果提前算好）或使用数据湖方式（Trino/Iceberg）。

生产踩坑案例：
某 BI 看板需要将事实表（50 亿行）与维度表（500 万行）Join，每次查询都 OOM。排查发现 CH 默认将右表（维度表）加载到内存，500 万行维度表加上 Hash 结构占用 2GB 内存，多个并发查询叠加导致 OOM。解决：使用 join_algorithm = 'partial_merge' 将右表排序后分片加载，同时把维度表加载到 Dictionary，查询改为 dictGet 免去 JOIN。

0x02-07 ClickHouse 数据类型与表结构设计最佳实践

标准答案：
ClickHouse 的数据类型设计注重压缩效率和计算性能，常见的最佳实践：

数值类型选择：

  • 能用 UInt8/16/32 不用 Int64——每列根据字段真实取值范围选择最小类型
  • DateTime（秒级）vs DateTime64（毫秒/微秒级）——精确到秒用 DateTime（4 字节），需要毫秒用 DateTime64(3)
  • Decimal 用于金额——Decimal(P, S) 精度由 P 决定，避免 Float 精度问题

字符串类型：

  • FixedString(N) 定长字符串：适用于固定长度编码如 MD5（32 字节）、IP 地址
  • String 变长：压缩率高，适合长文本
  • LowCardinality(String)：字符串取值有限（如状态、国家代码），使用字典编码，查询和存储兼顾

特殊类型：

  • UUID — 16 字节，写入效率低，考虑用 FixedString(36) 替代
  • IPv4 / IPv6 — 比 String 更紧凑的存储格式
  • Nested — 嵌套数据结构，适合数组/子事件
  • AggregateFunction — 存储聚合函数中间状态（仅用于 AggregatingMergeTree）

Nullable 的代价：

  • Nullable(Int32) 比 Int32 多一条标记数组（存储是否为 NULL），增加 1 倍存储 + 额外 CPU
  • 最佳实践：用默认值替代 NULL（如中位数填充、-1 表示未知、空字符串）
  • 只有在业务语义必须区别”0”和”无数据”时才用 NULL

深入思考：

  • LowCardinality 的原理：写入时对值做字典编码（Global Dictionary），存储字典 ID（Int32），减少重复字符串。适合 1000 万以内不同值。超过时字典膨胀，性能反而下降。
  • Nullable 对索引的影响：稀疏索引在标记位置跳过 NULL 值的效率低，因为标记位置是固定间隔，不因 NULL 而跳过。使用 isNotNull 过滤并不能有效减少扫描量。

生产踩坑案例：
某日志平台将 HTTP 状态码定义为 Nullable(UInt16)，状态码取自登录页面的随机用户名以替代默认值。数据量 5 亿行/天，NULL 比例约 30%。存储占用比非 NULL 版本多 40%。排查发现 Nullable 额外标记列占用了大量空间。解决：将默认值改为 0 （代表未知状态码），省去 Nullable 开销，压缩后存储降低 30%。

0x02-08 ClickHouse 的 TTL 与数据生命周期管理

标准答案：
ClickHouse 支持列级别和表级别的 TTL（Time-To-Live），用于数据生命周期管理。

表级 TTL：

  • 过期数据在后台合并时删除
  • 支持 TO DISK 'cold_volume' 将冷数据迁移到廉价存储

列级 TTL：

  • 不同列可以设置不同过期时间
  • 原始数据 30 天后清空值（置零），聚合结果保留一年

TTL 执行机制：

  • TTL 删除/迁移由后台合并线程触发
  • 非强制：如果合并被阻塞，TTL 数据不会立即清理
  • 性能优化：设置 ttl_only_drop_parts = 1 让整个 part 全部过期时直接删除文件（更快）

深入思考：

  • TTL 不立即执行：如果表 TTL 是 90 天，可能 95 天后数据才实际被删除。因为 TTL 依赖合并线程，而合并可能在等待或者被跳过。需要手动触发 OPTIMIZE TABLE events FINAL。
  • 列级 TTL 的代价：列级 TTL 在合并时需要重建整个 part，即使是删除一列的数据。大表列级 TTL 的合并代价与删除整行相当。

生产踩坑案例：
某监控系统设置 TTL 7 天后删除，但 15 天后磁盘满告警。发现 TTL 过期数据未被清理——因为该表 merge_with_ttl_timeout 设置为 0（禁用 TTL 合并），且 ttl_only_drop_parts 未开启。解决：启用 ttl_only_drop_parts = 1，并设置定期的 OPTIMIZE TABLE ... FINAL 作业强制触发 TTL 清理。

0x02-09 ClickHouse 查询性能诊断：system.query_log 与 Profile

标准答案：
ClickHouse 内置强大的自诊断体系，80% 的性能问题可通过 system 表定位。

核心诊断表：

  1. system.query_log：记录每条查询的执行信息（类型、SQL、查询耗时、读取行数、读取字节数、内存使用、Profile Events）
  2. system.query_thread_log：每个线程的执行细节
  3. system.parts：表的 part 信息（行数、大小、活跃状态）
  4. system.merges：正在执行的合并操作
  5. system.mutations：正在执行的突变操作
  6. system.metrics / system.events：集群级指标和累计事件

典型诊断查询：

Profile Events：

  • 通过 system.query_log 的 ProfileEvents 字段查看内部事件计数
  • 关键事件：MergeTreeDataReadRows、MergeTreeDataReadUncompressedBytes、SelectedPart、BloomFilterFalsePositive
  • 可以精确判断索引是否被有效利用，Bloom Filter 是否误判过多

深入思考：

  • 生产经验：query_log 本身就是 MergeTree 表，在高峰期写入查询日志也会消耗 IO。建议保留 7-30 天，设置 TTL 自动清理。
  • 关键判断：SelectedParts 和 read_rows 比例可以判断索引裁剪效果——如果 SelectedParts 很多但 read_rows 很大，说明索引选择后扫描量仍然较高，需要考虑优化 ORDER BY。

生产踩坑案例：

0x03-01 Apache Doris 整体架构？FE 与 BE 的职责？

标准答案：
Doris 采用 MPP 架构，分为 Frontend（FE）和 Backend（BE）两层：

Frontend（FE）— 无状态计算协调层：

  • 负责 SQL 解析、元数据管理、查询计划生成与调度
  • 通过 Raft 选举（多 FE）保证高可用
  • 采用 MySQL 协议兼容客户端

Backend（BE）— 有状态存储层：

  • 负责数据存储、本地查询执行、聚合计算
  • BE 之间通过 Brpc（百度自研 RPC 框架）通信
  • 支持列式存储、向量化执行引擎
  • 数据以 Tablet（数据分片）为单位分布，每个 Tablet 多副本

数据流：FE 接收查询 → 生成分布式执行计划 → 下发任务到 BE → BE 并行执行 → FE 聚合结果返回客户端

深入思考：

  • 对比延伸：Doris 的 FE 类似 TiDB 的 PD+TiDB Server 合体，但 FE 将元数据（存储在 BDBJE）和查询优化结合在同一进程中。BE 类似 TiKV+TiFlash 的列存版本。
  • 存储计算一体 vs 分离：Doris 默认存储计算一体（BE 负责存储和执行），但 3.0+ 开始支持存算分离（独立 Compute Group）。

0x03-02 Doris 的三种存储模型以及主键模型

标准答案：
Doris 提供三种数据模型，在建表时指定：

1. Duplicate Key Model（明细模型）

  • 最通用的模型，数据按 Key 列排序但不做任何合并/去重
  • 适用：原始日志、事件数据、不需要聚合的场景
  • Key 列用于排序和前缀索引加速

2. Aggregate Key Model（聚合模型）

  • 相同 Key 的行按聚合函数合并（SUM/REPLACE/MAX/MIN/HLL_UNION/BITMAP_UNION）
  • 适用：预聚合报表、维度指标、PV/UV 统计
  • 注意：Key 列顺序影响聚合语义——REPLACE 类型的列会用最后写入的值

3. Unique Key Model（唯一键模型）

  • 相同 Key 只保留最后一条（类似 UPSERT）
  • Doris 3.0 前通过 Merge-on-Read 实现（读时合并，读取性能损耗大）
  • Doris 3.0+ 引入”主键模型”（Primary Key Model）——使用写时合并（Delete+Insert），读取性能接近 Duplicate Key

4. Primary Key Model（主键模型，3.0+）

  • 真正的 Upsert 存储，写时直接标记删除旧版本
  • 主键列建立索引，支持高性能点查和实时更新
  • 适用：实时维表、库存流水、CDC 场景

深入思考：

  • 选型决策：APS 看板用 Aggregate Key；CDC 实时同步用 Primary Key；原始日志用 Duplicate Key。Aggregate Key 和 Unique Key（Merge-on-Read）对高频扫表性能有影响。
  • Primary Key 的代价：写时需要同时写主键索引，内存占用高。如果主键列过长（如字符串 UUID），内存中索引可达 GB 级。

0x03-04 Doris/StarRocks 物化视图与 Rollup

标准答案：

Rollup（上卷）— Doris 特有：

  • Rollup 是基于基表的预聚合索引，自动维护聚合结果
  • 查询时 FE 自动匹配最优 Rollup
  • 可理解为”自动物化视图”
  • 示例：基表按 (site_id, date, hour) 聚合，创建 Rollup 按 (date) 预聚合，查询按天统计时自动走 Rollup

物化视图（Materialized View）：

  • Doris 3.0 支持异步物化视图，StarRocks 同时支持同步和异步
  • 同步 MV：写入主表时自动更新 MV，实时性强但写入延迟增加
  • 异步 MV：定时或手动触发刷新，适合大量预计算场景

区别总结：

特性 Rollup 物化视图
维护方式 自动，写时维护 异步/同步
可自定义 仅支持聚合和排序列变化 支持任意 SQL（Join/Filter/子查询）
查询匹配 自动匹配 手动匹配（Base Table）或自动匹配改写
适用场景 简单预聚合 复杂 ETL、跨表 Join

深入思考：

  • Rollup 的局限性：Rollup 只能在基表的排序键上做聚合，灵活性远不如物化视图。Doris 社区已逐步用物化视图替代 Rollup。
  • StarRocks 同步 MV 的代价：同步 MV 在写入时会增加一次写放大（同时写主表和 MV），大列场景写入性能下降明显。

0x03-05 Doris/StarRocks 查询引擎与 MPP 执行

标准答案：
Doris/StarRocks 都基于 MPP（Massively Parallel Processing）架构：

MPP 执行流程：

  1. FE 解析 SQL → 生成逻辑计划 → CBO 生成物理计划
  2. 物理计划由多个 Fragment 组成，每个 Fragment 是一段可并行执行的算子
  3. Fragment 通过 Exchange 算子跨 BE 传输数据（Shuffle/Broadcast）
  4. 各 BE 并行执行 Fragment，最终 FE 汇聚结果

核心优化：

  • 向量化执行：以 Batch（列存向量）为单位处理数据，而非逐行，大幅减少虚函数调用和 CPU 缓存 miss
  • CBO（代价优化器）：基于统计信息（行数/NDV/数据分布）选择最优 Join Order 和执行策略
  • Colocate Join：将 Join 两表按相同分布键和 Bucket 数分布，Join 时避免 Shuffle 直接在本地执行
  • Runtime Filter：在 Join 执行动态生成 Bloom Filter，下推给 probe 端提前过滤，减少 IO

深入思考：

  • Colocate Join 的黄金法则：分布键和 Bucket 数必须完全相同，否则退化为普通 Shuffle Join。建表时需提前规划好分布策略。
  • Runtime Filter 的效果：典型场景下 Reduce 90%+ 的 Probe 端扫描量，是 Doris/SR 在 Join 性能上吊打 ClickHouse 的核心原因之一。

生产踩坑案例：
某 Doris 集群大表 Join 性能差，排查发现两表 COLOCATE 属性定义正确但 Bucket 数不同（10 vs 20），导致 Runtime 走 Shuffle Join。修正 Bucket 数一致后，Join 延迟从 30s 降到 2s。

  • 简单判断：如果 Join >= 3 张表 + 频繁的复杂查询 → Doris/SR；如果单表大吞吐扫描 + 压缩优先 → ClickHouse

0x04 选型决策树

0x04-01 行存 vs 列存 vs HTAP 选型矩阵

工作负载 推荐方案 原因
核心 OLTP（高并发 CRUD） MySQL/PostgreSQL/OceanBase/TiDB 行存，支持事务
分析报表（大表扫描、聚合） ClickHouse/Doris/StarRocks 列存，高效压缩
混合负载 HTAP TiDB (TiFlash) / OceanBase 4.x 行存+列存双引擎
轻量分析（内嵌） DuckDB 进程内列存，免部署
云原生数仓 Snowflake/Redshift 存算分离，弹性扩展
数据湖分析 Trino/Spark + Iceberg/Hudi 开放格式，灵活

HTAP 的真相：

  • “一份数据既跑 TP 又跑 AP” 是理想，现实是 HTAP 需要资源隔离（TiDB Resource Control / OB 租户隔离），否则 AP 大查询必然影响 TP 延迟。
  • 如果 TP 和 AP 负载都很重，物理隔离（两个集群 + 数据同步）仍是更可靠的选择。

0x04-02 MySQL（分库分表）vs TiDB vs OceanBase 选型

维度 MySQL + ShardingSphere/MyCat TiDB OceanBase
分布式事务 XA/Seata（跨库事务弱） Percolator（强一致） Paxos（强一致）
MySQL 兼容 100% ~95%（协议兼容，部分语法差异） MySQL 模式 + Oracle 模式
水平扩展 手动分片 + 运维复杂 自动扩缩容 自动扩缩容
高可用 异步/半同步复制（丢数据风险） Raft（RPO=0） Paxos（RPO=0）
OLTP 性能 极高（单机） 中（分布式开销） 高（TPC-C 世界纪录）
OLAP 能力 无（需额外导出） HTAP（TiFlash） 4.x 列存实验性
运维复杂度 中（sharding 中间件） 中（PD/TiKV/TiFlash 组件多） 中（OBServer 对等）
信创合规 否 是（国产开源） 是（蚂蚁集团）
成本 低 中（3 副本存储） 中（依赖配套生态）

选型建议：

  • 选 MySQL 分库分表：业务体量可控（<100 节点），团队 MySQL 经验丰富，不需要跨库事务
  • 选 TiDB：需要自动水平扩展 + MySQL 生态 + HTAP 能力，适合互联网/电商/SaaS
  • 选 OceanBase：金融核心交易（银行/保险/支付），需要 Oracle 兼容，TPC-C 标杆性能，信创最佳案例

0x04-03 ClickHouse vs Doris vs StarRocks vs Snowflake 选型

场景 推荐 理由
日志分析（写多读少、单表扫描） ClickHouse 极致写入速度 + 压缩率
实时 BI 看板（多表 Join） Doris / StarRocks MPP + CBO + Runtime Filter
CDC 实时数仓（从 Kafka 实时入仓） StarRocks / Doris Primary Key 模型 + 实时 Upsert
云原生数仓（存算分离、按量付费） Snowflake 无需运维，弹性扩展
数据湖查询加速 Doris / StarRocks（Lakehouse） 内置多 Catalog
高并发点查（广告/推荐） StarRocks / Doris 3.0+ 主键索引 + 短路径查询

0x04-04 国产化数据库替代路径（信创）

替代路径：

信创选型核心考量：

  1. 兼容性：SQL 方言、存储过程、触发器的兼容程度
  2. 迁移成本：ETL 工具、CDC 同步、应用改造成本
  3. 生态成熟度：运维工具、监控对接、人才储备
  4. 性能替代性：能否支撑原库的峰值 TPS/QPS
  5. 安全合规：国测认证、EAL4+ 安全、信创目录

实战案例：
某国有大行将 Oracle RAC 迁移至 OceanBase（3 地 5 中心部署），涉及 2000+ 存储过程、500+ 应用系统。迁移周期 18 个月，投入 50+ DBA + 100+ 开发。最终 TPC-C 性能提升 30%，硬件成本降低 60%。

某互联网公司将 MySQL 分库分表（256 个分片）迁移至 TiDB，7 人 DBA 团队缩至 2 人，查询响应 P99 从 200ms 降至 30ms，存储成本降低 40%（无需冗余备份）。
