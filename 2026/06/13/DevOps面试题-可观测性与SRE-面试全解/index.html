0x00 可观测性基础

Q1：什么是可观测性（Observability）？它与传统监控的区别是什么？

标准答案

可观测性来源于控制理论，指通过系统外部输出（日志、指标、链路）推断内部状态的能力。传统监控是”已知故障模式告警”，而可观测性让你能回答从未遇到过的问题。

核心区别：

  • 监控（Monitoring）：已知-已知（Known Knowns）——对已知指标设阈值告警
  • 可观测性（Observability）：未知-未知（Unknown Unknowns）——通过高维数据探索根因

Rudolph 教授在 2024 年的论文中强调：可观测性本质是调试体验（debugging UX），不是工具堆砌。三大支柱只是手段，目标是快速定位问题。

深入思考

可观测性 ≠ 三个 Pillar。新兴趋势包括：

  1. Event（事件）：被 ClickHouse 和 Honeycomb 推崇的”宽事件”模型，将 metrics/logs/traces 统一为带时间戳的结构化事件
  2. Profiling / Continuous Profiling：通过 eBPF 或 phrof 持续采集 CPU/内存/IO 调用栈，定位性能瓶颈
  3. 行为驱动观测：OpenTelemetry 的 SpanEvent + Metric + Log 关联，基于 Context Propagation

生产踩坑案例

某电商团队搭建了完善的 Prometheus + Grafana + ELK + Jaeger，但每次故障仍需人工交叉比对三个系统。原因是 metrics 没有 trace_id 标签，logs 没有 span_id 字段，三者无法关联。真正的可观测性需要打通数据孤岛——通过 OpenTelemetry 的 Context Propagation 将 trace_id 注入 metrics 和 logs。

Q2：可观测性的三大支柱（Three Pillars）分别是什么？各有什么优缺点？

标准答案

支柱 定义 优点 缺点
Metrics（指标） 聚合数值的时间序列数据 存储成本低，查询快，适合长期趋势分析 丢失个体请求细节，无法排查单次故障根因
Logging（日志） 结构化/非结构化的事件记录 细节最丰富，适合搜索和审计 噪声大，存储成本高，难以关联
Tracing（链路追踪） 记录请求在分布式系统中的完整路径 清晰展现服务依赖和延迟分布 采样后可能丢失关键路径，实现复杂

深入思考

Google 的 “四大黄金信号”（Four Golden Signals）不直接等于三大 Pillar：

  • 延迟（Latency）
  • 流量（Traffic）
  • 错误（Errors）
  • 饱和度（Saturation）

这三个是”存储格式”，四个是”业务关注维度”。

新趋势——Wide Events：不再区分 metric/log/trace，统一为带 Tags 的结构化事件。如 Honeycomb 的”高维事件”模型和 ClickHouse 的列式存储方案。

生产踩坑案例

某支付团队使用 Prometheus 只记录 P99 延迟，某次故障中 1% 的用户完全无法支付，但 P99 只从 200ms 升到 500ms，阈值未触发告警。如果同时使用 Tracing + 全量日志，就能发现这部分用户的请求被错误路由。

Q3：Metrics 的类型及其适用场景

标准答案

Prometheus 定义了四种核心 Metric 类型：

  1. Counter（计数器）：只增不减（重启后归零），用于 HTTP 请求总数、错误总数
  2. Gauge（仪表盘）：可增可减，用于内存使用量、CPU 使用率、在线连接数
  3. Histogram（直方图）：对观测值分桶计数，用于延迟分布、请求大小分布。可计算 histogram_quantile
  4. Summary（摘要）：在客户端计算分位数后上报，与 Histogram 的区别是不支持聚合

深入思考

Histogram 和 Summary 的选型陷阱：

  • Histogram：服务端可聚合（跨实例），但分桶预定义，精度受桶边界限制
  • Summary：客户端预计算分位数，不能跨进程聚合，存储开销固定

推荐策略：默认用 Histogram，配合 bucket 定义要关注的延迟区间（如 {0.1, 0.25, 0.5, 1, 2.5, 5, 10}）。对需要精确分位数的场景，使用 Summary 但注意不能全局聚合。

生产踩坑案例

某团队将 Histogram 桶设置为 {0.01, 0.02, 0.05, 0.1, 0.2, 0.5, 1}，导致 99% 的请求落在 0.01 桶内，P99 估算严重不准确。应参考业务实际延迟分布设置桶边界，保证各桶均匀分布数据。

Q5：OpenTelemetry 架构详解

标准答案

OpenTelemetry（OTel）是 CNCF 可观测性标准，统一了 Metrics/Logs/Traces 的采集和导出。

架构组件：

核心概念：

  1. TracerProvider / MeterProvider / LoggerProvider：SDK 入口
  2. Span：一次操作的上下文，包含 TraceID、SpanID、ParentSpanID
  3. Context Propagation：通过 W3C Trace Context 协议（traceparent header）跨服务传递
  4. Collector：Pipeline 架构——Receivers → Processors → Exporters
  5. Exporter：支持 OTLP、Prometheus、Jaeger、Zipkin、Loki 等

深入思考

Collector 部署模式：

  • Agent 模式（Sidecar/DaemonSet）：每个节点部署一个 Collector
  • Gateway 模式：集中式 Collector 集群做数据路由和缓冲
  • 混合模式：Agent 做基础处理，Gateway 做高级处理（如 Tail Sampling）

OTel 的优势在于 Vendor-neutral——更换后端只需改 Exporter 配置，不需要改代码。

生产踩坑案例

某公司直接通过 SDK 将数据发送到 Jaeger，迁移到 Tempo 时需修改所有服务的 SDK 配置。正确做法：内部部署 OTel Collector 作为 Gateway，所有服务只发数据到 Collector，后端切换仅改 Collector 配置。

Q6：W3C Trace Context 与 Baggage

标准答案

W3C Trace Context 定义了标准 HTTP 头用于分布式上下文传播：

  • traceparent：格式 version-trace_id-span_id-trace_flags
    • trace_id：128-bit（32 位 hex），全局唯一标识一次请求
    • span_id：64-bit（16 位 hex），标识当前 Span
    • trace_flags：采样标记（01 采样，00 不采样）
  • tracestate：用于传递厂商特定数据，支持多 Vendor 共存

Baggage：通过 baggage 头传递业务上下文（用户 ID、请求来源等），但存在安全风险——注入恶意 baggage 会放大基数。

深入思考

W3C Trace Context 要求 128-bit 的 Trace ID，但 Jaeger 旧版使用 64-bit。跨系统追踪时务必统一为 128-bit。Baggage 的使用原则：仅传递调试所需的少量键值对，不要在 Baggage 中传递高基数数据（如每次请求不同的 user_id）。

生产踩坑案例

某团队在 Baggage 中传递 user_id，导致经过的每个服务都记录了这个 label，Prometheus 指标基数爆炸（每天新增 100 万+ 时间序列）。教训：Baggage 只传低频标签和环境信息，不要传高基数业务标识。

Prometheus TSDB 三大关键技术：

Q8：PromQL 核心函数与三大 Counter 陷阱

标准答案

核心函数：

函数 作用 注意
rate() 计算每秒平均增长率（推荐） 自动处理 Counter 重置，末尾自动降噪
irate() 计算最后两个采样点的增长率 对毛刺敏感，适合短期暴涨检测
increase() 计算区间总增量 精度受 Counter 重置影响
histogram_quantile() 计算 Histogram 中的分位数 需满足桶递增且 le 覆盖完整范围

Counter 三大陷阱（Counter 三座大山）：

  1. 重置陷阱：Counter 在进程重启时会归零，increase() 未正确处理会导致突降。用 rate() 或 increase() 配合 @ 时间偏移
  2. 慢速 Counter 陷阱：Counter 增长很慢导致 rate 估算为 0。增加 scrape 间隔或使用更长的时间窗口
  3. 内存膨胀陷阱：rate[5m] 的 5m 区间内数据全部缓存。有大量高基数 metric 时内存消耗是 5m/15s × series 数量

深入思考

rate() vs irate() 选型原则：

  • rate()[5m]：平滑趋势，适合告警和长期分析
  • irate()[5m]：对突发敏感，适合 Dashboard 展示实时变化
  • 绝不要用 rate() 和 increase() 做短于三倍 scrape interval 的窗口

Q10：Thanos vs VictoriaMetrics 选型对比

标准答案

维度 Thanos VictoriaMetrics
架构复杂度 多组件（Sidecar/Store/Query/Compact/Ruler） 单体极简，集群模式也简单
存储后端 依赖对象存储（S3/GCS/MinIO） 本地磁盘 + 可选对象存储
查询性能 慢查询较多（大范围查询从对象存储读取） 极快（自定义 TSDB 引擎，十亿级查询毫秒级响应）
数据复制 Sidecar 强制上传到对象存储 -replicationFactor 参数
Downsampling 内置（5m/1h 降采样） 自动降采样（配置 -downsampling）
PromQL 兼容 完全兼容 99% 兼容（少数边界函数差异）
运维成本 高（6+ 组件） 低（1~2 个组件）
社区活跃度 CNCF Graduated 独立开源，社区活跃
集群版 无原生集群（需外部负载均衡） 有集群版（vminsert/vmselect/vmstorage）

选型建议：

  • 已有 Prometheus、需要长期存储和全局视图 → Thanos
  • 追求极致性能、低运维、高性能写入 → VictoriaMetrics
  • 小团队、少资源 → VictoriaMetrics 单节点版
  • 大团队、复杂多集群 → Thanos 或 VictoriaMetrics 集群版

深入思考

Thanos 的 Receive 模式（类似 Remote Write）可以替代传统 Sidecar 模式，适合没有对象存储的场景。VictoriaMetrics 单节点版能处理每秒百万级样本写入，且 vmagent 比 Prometheus 内存节省 5~10 倍。

生产踩坑案例

某公司使用 Thanos + S3 做长期存储，大范围查询（如 30 天 P99 延迟趋势）经常超时 30 秒。原因：Thanos Store 从 S3 读取数据时网络延迟高。迁移到 VictoriaMetrics 后，同类查询耗时从 15 秒降至 0.5 秒。

Q11：Prometheus 高基数问题（High Cardinality）与解决方案

标准答案

高基数：单个 Metric 的 Label 组合（Series 数量）过多，导致 TSDB 内存暴涨、查询变慢、Scrape 超时。

危险来源：

  • 用户 ID、Session ID、容器 ID、Pod IP、Trace ID 作为 Label
  • URL 作为 Label（含动态参数）
  • 无限制的 Baggage 传播到 Metric

解决方案：

方案 方法 效果
Relabel 丢弃 metricRelabelings 丢弃高基数字段 立即减少 Series
聚合降维 Recording Rule 预聚合（去掉高基数标签） 保留数据但不爆炸
标签白名单 只保留需要的标签 可控
观测端高基数据库 使用 VictoriaMetrics/Honeycomb 等支持高基数写入的系统 牺牲成本换取维度
分级存储 Prometheus 存核心指标，Thanos/VictoriaMetrics 存详情 成本平衡

深入思考

如何发现高基数？Prometheus 提供以下手段：

  • TSDB Status 页面：/tsdb-status 展示 Head Series 数量
  • prometheus_tsdb_head_series 指标
  • 使用 count by(__name__) ({__name__=~".+"}) 查出 Series 最多的 Metric

最佳实践：每个 Metric 的 Series 数量建议控制在 10 万以内，总 Series 数不超过 1000 万（单节点 Prometheus 极限）。

生产踩坑案例

某金融科技公司将 user_id 作为 Counter Label 记录每个用户的请求数。一周后 Series 数达到 500 万，Prometheus 内存 40GB+，Scrape 频繁超时。解决方案：将 user_id 标签丢弃，改用 histogram 统计用户请求延迟分布，用户级指标写入 ClickHouse。

Q14：日志存储与索引优化

标准答案

Elasticsearch 索引优化：

  1. ILM（Index Lifecycle Management）：热（Hot）→ 温（Warm）→ 冷（Cold）→ 删（Delete）
    • Hot：SSD，副本数 1，高频写入
    • Warm：HDD，副本数 0
    • Cold：可搜索快照（S3）
  2. Shard 优化：单 Shard 建议 20~50GB，日志场景按天索引
  3. Mapping 优化：
    • 禁用 _source（节省 50% 存储）
    • 使用 keyword 替代 text（不需要全文搜索的字段）
    • 关闭不需要的 norms 和 doc_values
  4. Rollover：基于大小（50GB）或时间（24h）滚动索引

深入思考

ES 日志场景三大坑：

  1. Field 爆炸：日志格式变化导致动态 mapping 产生大量字段。解决方案：设置 index.mapping.total_fields.limit
  2. 磁盘 IOPS 不足：日志写入频繁，WAL 写入成为瓶颈。解决方案：使用 SSD + 调整 translog 配置
  3. Merge 风暴：大量 Segment 合并消耗 IO。解决方案：调整 merge.scheduler.max_thread_count 和 index.merge.policy

生产踩坑案例

某团队每天产生 800 亿条日志，所有字段都开 text 全文索引，ES 集群 50 节点写性能下降，查询响应缓慢。优化方案：只对 message 字段保留 text 类型，其他字段（level, service, trace_id 等）改为 keyword；开启 ILM 策略，7 天后从 Hot 节点移到 Warm 节点。

Q20：SLO 目标设定与错误预算

标准答案

SLO（Service Level Objective）：SLI 的目标阈值，例如 “99.9% 的请求在 200ms 内完成”。

错误预算（Error Budget）：

示例：SLO = 99.9%（30 天），30 天总请求 = 10,000,000

  • 允许的错误数 = 10,000,000 × (1 - 0.999) = 10,000
  • 当前的错误数 = 3,000
  • 剩余错误预算 = 7,000

错误预算策略：

  • 预算充足（> 50%）→ 正常迭代
  • 预算紧张（< 20%）→ 停止新功能，优先稳定性
  • 预算耗尽 → 紧急响应，不可发布新版本

深入思考

SLO 不是越高越好。99.99% 相比 99.9%，成本可能增加 10 倍，收益边际递减。Google 的实践：99.9% 是最常见的 SLO，99.99% 只给核心支付/搜索服务。

SLO 设定方法论：

  1. 先用观测数据确定当前实际表现
  2. 设置比当前略高的目标（+0.1%~0.5%）
  3. 根据业务重要程度调整
  4. 与产品经理对齐目标

生产踩坑案例

某公司对所有微服务统一设定 99.99% SLO，导致每个服务都需要 4 副本 + 多 AZ 部署。三个月后发现成本增长了 3 倍，但用户满意度并没有提升。审计后发现，非核心服务（如用户头像服务）设定 99.9% 就完全够用。教训：SLO 应分级设定。

SLA vs SLO vs SLI 关系：

典型的 SaaS SLA 条款：

  • 99.9% 可用性（月度）：延长时间 10%
  • 99.0%~99.9%：延期 25%
  • < 99.0%：延期 50%+ 现金赔偿

深入思考

  1. 产品级 SLA：对客户的正式承诺
  2. 服务级 SLO：各个微服务的内部目标
  3. 工程级 SLI：具体技术指标

Q24：告警风暴治理

标准答案

告警风暴（Alert Storm）：短时间内大量告警涌入，导致 On-Call 无法处理关键告警。

治理策略：

  1. 告警分组（Alertmanager group_by）：按服务/集群分组，同源告警聚合
  2. 告警抑制（Inhibition）：高级别告警抑制低级别告警
  3. 告警静默（Silence）：已知问题临时静默
  4. 依赖拓扑感知：数据库宕机最多发一个告警，所有上游服务的依赖告警抑制
  5. 容量规划：避免规模化扩缩容导致的误报

深入思考

告警风暴是系统设计问题，不只是配置问题：

根本原因分析：

  • 是否存在级联故障设计缺陷？
  • 告警阈值是否合理？
  • 是否有故障注入演练？

告警治理的”金字塔”模型：

每次 P0 告警应有一个负责人并事后追溯。P2/P3 告警应持续优化直至趋近于 0 或变为 Dashboard 指标。

生产踩坑案例

某电商大促期间，数据库 Proxy 故障导致 500+ 个 POD 实例全部触发 OOMKilled 告警，2 分钟内产生 2000+ 条告警。On-Call SRE 手机持续震动 15 分钟，完全无法识别真实根因。事后优化：

  1. 按 Deployment 分组告警
  2. 数据库故障抑制所有业务服务告警
  3. 引入告警风暴检测规则（5 分钟内告警 > 50 条触发风暴通知而非逐条通知）

Q25：On-Call 最佳实践

标准答案

轮值模式：

  • Follow-the-Sun：全球三地（美国、欧洲、亚太），白班轮值
  • 7×24 轮值：一周轮值，下一周休息（不推荐长期）

  2. Playbook 自动化：常见故障应有自动恢复脚本
  3. 值班备份：至少两人才算 On-Call 可用（Primary + Secondary）
  4. 战后复盘：5 Whys 分析法，避免重复故障

Q26：容灾架构（同城双活 / 异地多活 / RTO / RPO）

标准答案

模式 RTO RPO 成本 复杂度
同城双活（Active-Active） 分钟级 秒级 2x 中
两地三中心 30分钟 分钟级 3x+ 高
异地多活（Multi-Region Active-Active） < 1分钟 0~秒级 5x+ 极高

术语定义：

  • RTO（Recovery Time Objective）：从故障发生到服务恢复的最大可接受时间
  • RPO（Recovery Point Objective）：最大可接受的数据丢失时间

同城双活架构要点：

  • 两个 AZ（可用区），各部署一套完整服务
  • 前端：DNS 加权轮询 / Global Load Balancer
  • 数据层：DB 跨 AZ 同步（MySQL Group Replication / Oracle RAC）
  • 缓存：Redis 跨 AZ 复制
  • 消息队列：Kafka 跨 AZ MirrorMaker

异地多活架构要点：

  • 全球多 Region，每个 Region 独立 deployment
  • 数据分片/分区，按照用户地理位置就近接入
  • 跨 Region 数据异步复制（最终一致性）
  • 域名：GSLB（全局负载均衡）

深入思考

异地多活是系统工程中最难的课题之一，主要挑战：

  1. 数据一致性：跨 Region 异步复制存在数据冲突
  2. 业务逻辑：强一致性事务无法跨 Region
  3. 部署同步：多 Region 发布节奏一致
  4. 测试验证：多活架构验证本身很复杂

大多数公司不需要 3AZ + 异地多活。99.9% 可用性用同城双活即可满足。只有全球服务（Google/Meta/Netflix）才需要真正意义上的异地多活。

生产踩坑案例

某公司设计”两地三中心”架构但只做了基础设施层面的双活——数据库主备、应用层双活。一次光缆中断后，备库数据延迟 30 分钟（异步复制），导致切换后大量用户数据不一致。教训：RPO 取决于数据同步方式，同城双活应使用同步复制或半同步复制。

Q27：混沌工程（Chaos Engineering）

标准答案

混沌工程是对分布式系统的实验性实践，通过注入受控故障来验证系统的弹性。

核心原则（来自 Principles of Chaos Engineering）：

  1. 定义稳定状态（Steady State）
  2. 提出假设（Hypothesis）
  3. 引入故障变量
  4. 对比实验结果与假设
  5. 缩小爆炸半径

主流工具对比：

工具 类型 语言 K8s 支持 故障类型
Chaos Mesh 云原生 Go 原生 Pod 故障/网络延迟/IO 压力/DNS 故障
Litmus 云原生 Go 原生 Pod 故障/网络/Disk/节点故障，Workflow 编排
ChaosBlade 通用 Go 通过 Operator CPU/磁盘/网络/进程/JVM/Docker
Gremlin SaaS - 支持 广泛，商业支持
Chaos Monkey Netflix Java 基础 实例随机终止

Chaos Mesh 故障类型：

  • PodChaos：Pod 故障/删除
  • NetworkChaos：网络延迟/分区/丢包/带宽限制
  • IOChaos：磁盘 IO 延迟/错误
  • StressChaos：CPU/内存压力
  • DNSChaos：DNS 解析错误
  • HTTPChaos：HTTP 响应修改

深入思考

混沌工程成熟度模型：

  1. 阶段 1：非生产环境手动注入（开发/测试环境）
  2. 阶段 2：非生产环境自动化实验（定期执行）
  3. 阶段 3：生产环境只读/低流量实验（小范围）
  4. 阶段 4：生产环境全量实验 + 自动回滚

GameDay 演练：定期（每季度）组织的混沌工程实验日，所有 SRE 团队参与。目标不是制造灾难，而是验证系统韧性。

生产踩坑案例

某团队在 Staging 环境跑 Chaos Mesh 实验发现 Redis 主节点挂了后，从节点提升为主节点时有 5 秒连接中断。业务代码没做重试，导致雪崩。修复：在业务代码中加入 Redis 连接重试 + 熔断机制。这个故障如果不是 Chaos Mesh 主动注入，可能要生产事故才能发现。

Q28：故障注入与自动恢复

标准答案

故障注入（Fault Injection） 是混沌工程的实践手段，有三种注入层次：

  1. 基础设施层：主机宕机、网络中断、磁盘故障
  2. 平台层：Pod 崩溃、配置错误、镜像拉取失败
  3. 应用层：接口超时、数据库慢查询、业务逻辑异常

自动恢复机制：

故障类型 检测手段 自动恢复动作
Pod OOM K8s OOMKilled K8s 自动重启 + 限制内存
节点宕机 Node NotReady 调度到其他节点
数据库死锁 慢查询告警 Kill 死锁事务 + 连接池重连
流量突增 HPA Metrics K8s HPA 自动扩容
异常发布 Error Budget 告警 自动回滚（Canary 发布失败回退）

深入思考

自动恢复不是万能的——需要关注”恢复的副作用”。例如：数据库自动 Kill 死锁语句，但被 Kill 的事务未被补偿，导致数据不一致。自动恢复需要配合幂等性和补偿机制。

建议：自动恢复 + 人工确认（Semi-auto）是更务实的做法。将自动恢复定义为”自动执行恢复脚本，但通知 SRE 确认”。

生产踩坑案例

某微服务配置了 HPA 自动扩容，大促时流量 10 倍增长，HPA 扩容 100 个 Pod。但数据库连接池未预配，100 个 Pod 瞬间建立 1000+ 数据库连接，数据库 OOM。教训：自动扩容必须有配套的资源变更，否则只是”把问题从上游移到下游”。解决方案：使用 KEDA + 数据库连接池配额控制 + 压力测试验证扩容上限。

Q31：KSM（kube-state-metrics）详解

标准答案

Q32：eBPF 可观测性（Cilium / Pixie）

标准答案

eBPF（extended Berkeley Packet Filter） 是一种在内核中运行沙箱化程序的技术，无需修改内核代码即可插入观测点。

Cilium：基于 eBPF 的 CNI 插件和可观测性平台。

核心能力：

  • Hubble：Cilium 的可观测性层，提供 Service Mesh 级别的网络流量可视化和安全策略监控
  • 自动生成请求级延迟、吞吐量、TCP 状态
  • 支持基于 Identity 的安全策略（替代传统 NetworkPolicy）
  • 无需 Sidecar 即可实现 L3-L7 的可观测性

Pixie：K8s 原生 eBPF 可观测性工具，自动采集应用级指标。

核心能力：

  • 自动 Instrumentation：无需修改代码，自动采集 HTTP/gRPC/MySQL/Redis 等协议的调用链
  • 全量跟踪（Full-body Tracing）而非采样
  • 动态 BPF 程序注入
  • 支持实时查看 Pod 的 CPU Profile 和内存分配

深入思考

eBPF 带来的变化：”零代码可观测性”时代。传统 OpenTelemetry 需要应用层手动或自动注入（代码修改），eBPF 可以直接从内核态捕获数据，对应用完全透明。

但 eBPF 不是万能的：

  1. eBPF 不能捕获应用层业务逻辑（如”订单流程”的语义）
  2. 只能观测到系统调用级别（网络、系统调用协议）
  3. 对内核版本有要求（需要 4.19+ 以上内核支持 BTFeBPF）

建议的组合方案：

  • eBPF（Pixie/Cilium）：基础网络层和协议层可观测性
  • OTel SDK：业务层可观测性
  • KSM + cAdvisor：K8s 资源层可观测性

生产踩坑案例

某团队使用 Pixie 自动采集所有 Pod 的 HTTP 调用，打开后发现单节点 eBPF 程序占用 5% CPU，千 Pod 集群产生每天 10TB+ 数据（因为 Pixie 是全量跟踪）。教训：Pixie 适合故障时刻的临时诊断，长期运行需要考虑存储成本和性能开销。折中方案：默认开启采样模式，Dynatrace 模式（在问题发生前后动态开启全量跟踪）。

Q35：Prometheus 高基数故障

背景

某金融科技初创公司，3 个月后 Ops 发现：

  • Prometheus 内存从 4GB 增长到 32GB
  • 每个 Scrape 周期（15s）需要 30 秒才能完成
  • 告警延迟 10 分钟
  • Prometheus 频繁 OOM

user_action_counter 包含 user_id 标签！

  • 10 万用户 → 10 万条 Series
  • 该指标还有 action_type（10 种）、device（3 种）
  • 合计：100,000 × 10 × 3 = 300 万条 Series

处理过程

紧急止血（Hotfix）：

  1. 丢弃 user_id 标签：

长期方案：

  1. Metrics 只保留聚合标签（action_type, device），不保留用户级标签
  2. 用户级数据（高基数）写入 ClickHouse
  3. 添加 Prometheus 内存告警（process_resident_memory_bytes > 15GB）
  4. 实施指标审计流程——每次新增标签需 SRE Review

教训

  • 任何新增 Label 都要评估基数
  • 开发工程师不能随意添加 Prometheus 标签
  • 建立 Cardinality 看板和告警

Q36：Trace 采样导致问题定位失败

背景

某微服务架构（30+ 服务）引入 Jaeger 做分布式追踪。配置了概率采样 1%。

问题：生产环境偶发”下单失败”错误，日志显示 500 Internal Server Error，但后端服务日志无异常，Trace 也无法查到。

排查过程

  1. 查看 Jaeger——找不到该错误请求的 Trace（因为 1% 采样没采到）
  2. 查看日志——找到了 order_id 但无 Trace ID
  3. 尝试复现——无法稳定复现
  4. 持续观察 3 天——错误每天出现 2-3 次

根因

最终通过增加一个特定 API 的全量采样（head sampling + tail sampling 兜底）定位到问题：

  • 下单请求经过 3 个微服务：Order → Payment → Inventory
  • Inventory 服务偶发 Redis 连接超时（1% 的请求）
  • 由于采样 1%，100 个错误请求才采到 1 个

改进方案

  1. 分层采样策略：

    • 所有 ERROR Span 全量采样（tail sampling）
    • 核心 API（createOrder/pay）概率 10%
    • 其他 API 概率 1%
    • 慢 Span（> 1s）全量采样

  2. 日志关联 Trace：

    • 所有日志注入 trace_id 和 span_id
    • 即使未采样，错误日志的 trace_id 也可用于查询 Trace

  3. 代价：Tail Sampling Processor 需要 OTel Collector 维持 Span 缓冲区，内存从 2GB 增加到 8GB

教训

  • 单一概率采样在微服务环境中极度危险
  • 错误路径必须 100% 采样（使用 Tail Sampling 或加标签采样）
  • Trace 和 Log 必须关联，Log 是错误场景的兜底
