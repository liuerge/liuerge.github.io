0x00 LLM 推理部署

0x00-1 请解释 vLLM 的核心架构与 PagedAttention 原理

标准答案

vLLM 是加州大学伯克利分校 LMSYS 团队开源的高吞吐 LLM 推理引擎。其核心创新是 PagedAttention——受操作系统虚拟内存分页机制启发，将 KV Cache 按固定大小的块（Block）管理，而非为每个请求预分配连续显存。

传统方案为每个请求静态分配最大序列长度的 KV Cache 空间（如 2048 token），导致严重的显存碎片——论文实测 60%-80% 的预留 KV 显存被浪费。PagedAttention 将 KV Cache 切分为 16 或 32 token 的 Block，通过 Block Table（类似页表）建立逻辑页到物理块的映射，按需分配、按页换入换出。

深入思考

KV Cache 生命周期管理：每个 Sequence 持有一个 Block Table，解码过程中逐 Block 追加。当 Block 写满时分配新 Block。Sequence 结束时释放所有 Block。vLLM 的 --gpu-memory-utilization 参数控制 GPU 显存中 KV Cache 的比例（默认 0.9），剩余的留给模型权重和中间激活。

连续批处理（Continuous Batching / Iteration-level Batching）：传统静态批处理必须等整个 Batch 所有序列完成 Prefill 再统一 Decode，空闲 GPU 等待慢序列。vLLM 采用迭代级调度——每步 Forward Pass 完成后立即检查：已完成序列退出、新序列加入、未完成序列继续。这使 GPU 利用率从 30-40% 提升到 85%+。

Chunked Prefill：当 Prefill 阶段计算量过大（长 prompt），vLLM 可将 Prefill 拆分为多个 Chunk，穿插 Decode 请求，避免长 prompt 阻塞解码。--max-num-batched-tokens 控制每批次总 token 上限。

投机解码（Speculative Decoding）：用轻量 Draft Model（如小 10 倍）快速预测 K 个候选 token，Target Model 单次 Forward Pass 验证。若全部接受，吞吐量提升 1.5-3x。vLLM 支持 draft_model 配置、EAGLE 方法（从 Target Model hidden states 预测）、Medusa 多头投机解码。

  • vLLM Blog - Anatomy of vLLM: https://vllm.ai/blog/2025-09-05-anatomy-of-vllm

0x00-3 推理优化技术全览：量化、Flash Attention、投机解码

标准答案

量化（Quantization）

类型 精度 显存节省 质量损失 代表方法
FP8 8-bit 浮点 ~50% 几乎无损 原生 FP8 推理（H100）
INT8 8-bit 整型 ~50% 极微 LLM.int8()
INT4 4-bit 整型 ~75% 轻微 AWQ / GPTQ / GGUF
NF4 4-bit 正态浮点 ~75% 轻微 QLoRA 所用
  • AWQ（Activation-aware Weight Quantization）：根据激活值分布寻找 1% 的”显著权重通道”保留 FP16，其余 INT4。质量优于 GPTQ。
  • GPTQ：基于 Hessian 矩阵的逐层量化，后训练量化方案的经典方法。
  • GGUF：llama.cpp/Ollama 使用的格式，支持多种量化级别（q2_k 到 q8_0）。

Flash Attention

核心思想：通过 tiling（分块） 将注意力计算分片到 SRAM 上完成，避免频繁读写 HBM（高带宽显存）。传统 Attention 的 O(N²) 中间矩阵（S = QK^T, P = softmax(S)）写回 HBM 再读取，Flash Attention 将其合并为一次 kernel，IO 复杂度从 O(N²d) 降至 O(N²d / M)（M 为 SRAM 大小）。

  • FlashAttention-2：减少 non-matmul 操作、并行化 sequence 维度，2x 加速。
  • FlashAttention-3：利用 Hopper 架构的 WGMMA 指令和异步拷贝，FP8 支持，H100 上再提升 1.5-2x。

投机解码

三种主流实现：

  1. Draft Model：用小模型（如 7B→70B）预测 K 个 token，Top-1 采样快速生成候选。
  2. EAGLE-2/3：从 Target Model 的 hidden states 接入轻量预测头，不依赖外部 Draft Model，接受率更高。
  3. Medusa：在 Target Model 最后几层添加多个预测头，同时预测多个后续 token。

核心指标：Acceptance Rate（被接受的 draft token 比例）。70B 模型 paired with 7B draft，~80-90% 接受率，实际吞吐 1.5-2.5x 提升。

深入思考

量化与精度之间始终存在 tradeoff。FP8 在 H100/B200 上有原生硬件支持，是 2026 年生产环境最佳折衷。AWQ 在 INT4 级别质量最优，适合带宽受限场景（如单卡部署 70B）。Flash Attention 已近乎成为生产部署的标配，vLLM/SGLang 均默认集成。投机解码对延迟敏感场景（如实时对话）提升有限，但对高吞吐离线批处理场景价值巨大。

  • FlashAttention: https://arxiv.org/abs/2205.14135
  • Speculative Decoding: https://arxiv.org/abs/2211.17192

0x00-4 分布式推理架构设计

标准答案

策略 原理 适用模型 通信开销 典型工具
张量并行（TP） 沿 hidden 维度切分单层 >30B 极高 vLLM TP, Megatron
流水线并行（PP） 按层切分到不同 GPU >70B 中等 DeepSpeed PP
数据并行（DP） 每个 GPU 跑完整模型 <13B 低 vLLM DP
分离式 Prefill/Decode P/D 分在不同 GPU 集群 大规模服务 中等 DistServe, MoonCake

张量并行实例：将 QKV 投影矩阵按列切分到 2 张 GPU，每个 GPU 计算部分注意力头，最后 all-reduce 拼接。通信需要在每层做 all-reduce，是主要的延迟瓶颈。

Prefix Caching（前缀缓存）：vLLM 自动缓存已计算的 prompt 前缀 KV Cache（以 block 粒度）。当新请求的 prompt 前缀与缓存匹配（如 system prompt），直接从缓存加载，跳过重算。缓存命中可节省 30-70% 的 Prefill 计算。SGLang 的 RadixAttention 使用前缀树（Trie）管理，共享率更高。

部署架构建议：

  • 单机多卡 2-4×H100 部署 70B：TP=2 或 TP+PP
  • 跨节点分布式 8×H100 部署 70B：TP=8（单节点内部），PP 跨节点
  • 大规模服务 >100K QPS：分离式 P/D + Multi-Layer Prefix Cache + 模型分片路由

0x00-5 OpenAI 兼容 API 服务设计

生产级 OAI 兼容 API 需要实现的核心端点：

  • POST /v1/chat/completions：聊天补全（messages 格式、stream、temperature、top_p、tools）
  • POST /v1/completions：文本补全
  • POST /v1/embeddings：向量嵌入
  • GET /v1/models：模型列表查询
  • POST /v1/tokenize/detokenize：分词服务

设计要点：

  1. Streaming 实现：通过 SSE（Server-Sent Events），vLLM 的 AsyncStream 逐 token 推送。
  2. Tool Use 支持：需要解析 tool_calls 参数，在生成过程中识别 function call token。

  4. 请求排队：vLLM 内置 Scheduler 处理请求排队及优先级（AsyncLLMEngine）。
  5. 多模型路由：API Gateway 层根据模型名称分发到不同推理后端，支持模型级 autoscaling。

0x01-1 全量微调 vs LoRA vs QLoRA vs DoRA 对比

标准答案

方法 可训练参数 显存需求（7B） 训练速度 质量（相对全量） 适用场景
全量微调 100% 120-160 GB 1x（基准） 100% 有充足算力、需要最大质量
LoRA 0.1-1% 24-48 GB 1.5-2x 95-99% 大部分生产场景的首选
QLoRA 0.1-1% 12-24 GB 2-3x 93-98% 显存受限、RTX 4090 单卡
DoRA 0.1-1% 24-48 GB 1.5-2x 96-99.5% 需要比 LoRA 更好质量
AdaLoRA 自适应 24-48 GB 1.5-2x 95-99% 需要自动秩选择

全量微调：更新所有模型参数，需要足够显存存储完整模型权重（BF16 下 7B ≈ 14GB）、优化器状态（Adam 额外 2×）、梯度（1×），合计约 120GB+。

LoRA（Low-Rank Adaptation）：冻结原始权重，在 Attention 层插入低秩矩阵 W=W₀+BA（B∈R^{d×r}, A∈R^{r×d}）。训练时只更新 BA，推理时合并回 W₀（无额外延迟）。典型 r=8-64。

QLoRA：在 LoRA 基础上将原始权重 NF4 量化，增加 Double Quantization（量化常数的二次量化）和 Paged Optimizer（显存分页），使 7B 模型可在 24GB 单卡上微调。

DoRA（Weight-Decomposed Low-Rank Adaptation）：将预训练权重分解为幅度（Magnitude）和方向（Direction），只对方向分量做 LoRA 更新，保留幅度可学习。此分解更接近全量微调的学习模式，在多项 Benchmarks 上超越 LoRA，且不增加推理延迟（推理时可合并）。

深入思考

LoRA 的秩 r 是关键超参：r=8 对领域适配通常足够，r=32-64 对复杂任务（代码、数学推理）效果更好。秩越大可学习参数越多但趋近全量微调成本。生产经验：先用 r=8 跑一轮验证数据质量，再决定是否增大秩。

合并权重：LoRA 推理时可合并到原始权重（model = model.merge_and_unload()），或保留 Adapter 文件独立加载。合并后推理延迟与原始模型完全一致。生产部署建议合并后重新量化，减少部署复杂度。

  • DoRA: https://arxiv.org/abs/2402.09353

0x01-2 LoRA 原理与秩的选择

标准答案

LoRA 基于低秩假设：大语言模型在特定下游任务上的权重更新量 ΔW 是低秩的。因此可将 ΔW 分解为两个小矩阵的乘积 ΔW = BA（B∈R^{d×r}, A∈R^{r×d}），其中 r << min(d, d)。

在前向传播中：h = W₀x + BAx，仅 B 和 A 需要梯度。初始化时 B=0，A 使用随机高斯分布，确保训练开始时 ΔW=0。

秩 r 的选择：

  • r=1-4：简单任务（情感分类、实体识别），参数极少有正则化效果
  • r=8-16：通用领域适配（客服、文档问答）——生产中最常用
  • r=32-64：复杂任务（代码生成、数学推理、多轮对话）
  • r=128+：逼近全量微调，失去低秩优势

经验法则：r=8 时 LoRA 参数量约为原始模型的 0.1%（7B 约 7M 参数），r=64 时约 0.8%。在 LLaMA 系列上，r=8 即可达到 r=64 约 95% 的质量，性价比最高。

LoRA 的目标模块：通常只对 Attention 层的 Q、K、V、O 投影矩阵注入 LoRA。也可扩展到 FFN 层（gate_proj、down_proj、up_proj），但参数量和计算量随之增长。生产实践：Q+V 或 Q+K+V+O 较常见。

参考链接

  • LoRA 原理详解: https://huggingface.co/blog/peft

0x01-3 PEFT 框架详解

标准答案

HuggingFace PEFT 是目前最主流的参数高效微调框架，提供统一接口：

  • LoRA：LoraConfig(r=8, lora_alpha=16, target_modules=["q_proj","v_proj"])
  • IA³：学习向量 rescale 激活值
  • Prefix Tuning：在每层前添加可学习前缀 token
  • Prompt Tuning：仅在输入层添加可学习 soft prompt
  • AdaLoRA：自适应分配秩预算，SVD 剪枝

使用模式：

lora_alpha 是缩放因子：推理时 LoRA 权重按 alpha/r 缩放。典型配置 alpha=16 时 r=8，scale=2。alpha 越大 LoRA 贡献越强。

0x01-4 SFT vs DPO vs GRPO vs PPO 对比

标准答案

方法 核心原理 需要奖励模型 训练复杂度 数据需求 代表模型
SFT 交叉熵监督学习 ❌ 低 指令-回复对 所有 instruct 模型
PPO RL 策略梯度优化 ✅ 需要独立 RM 高 偏好数据+无标注 ChatGPT (GPT-4)
DPO 偏好分类损失 ❌（隐式 RM） 中 偏好对 (win/lose) Llama-3-Instruct
GRPO 组内相对奖励 ❌（组内对比） 中 无标注+奖励信号 DeepSeek-R1
ORPO 偏好概率对比 ❌ 低 偏好对 部分社区模型

SFT（Supervised Fine-Tuning）：最基本的微调方式，在高质量指令-回复数据上最小化负对数似然（NLL）。模型学会遵循指令格式，但不一定学会偏好对齐。

PPO（Proximal Policy Optimization）：经典 RLHF 三阶段：SFT → 训练奖励模型（RM）→ PPO 优化策略。奖励模型为每个回复打分，PPO 在此信号下优化策略同时约束策略更新幅度（clip 机制）。计算开销大、训练不稳定。

DPO（Direct Preference Optimization）：将 RLHF 映射为分类问题——最大化偏好对中胜出回复的概率，同时降低落选回复的概率。不需显式训练 RM，收敛更稳定。2025-2026 年已成为对齐事实标准。

GRPO（Group Relative Policy Optimization）：DeepSeek-R1 使用的对齐方法。对同一个 prompt 采样多个回复，组内计算相对奖励（减去组均值、除以组标准差），不依赖独立奖励模型。GRPO 的 KL 散度约束直接在组内比较中实现。

深入思考：SFT → DPO 是 2025-2026 年最主流的两阶段对齐路线。SFT 建立基础能力，DPO 做偏好对齐。GRPO 在需要探索的推理场景（数学、代码）表现优于 DPO，计算量介于 DPO 和 PPO 之间。PPO 由于训练复杂度和稳定性问题，在生产中使用率逐年下降。

参考链接

  • DPO: https://arxiv.org/abs/2305.18290
  • GRPO: https://arxiv.org/abs/2402.03300
  • PPO: https://arxiv.org/abs/1707.06347

SFT 数据

  • 格式：{"instruction": "...", "input": "...", "output": "..."} 或 OpenAI 的 messages 格式
  • 关键原则：质量 >> 数量。1000 条高质量 SFT 数据 > 10 万条低质量数据
  • 数据类型：单轮指令、多轮对话、思维链（CoT）推理、工具调用（function calling）

偏好数据（DPO 数据）

  • 格式：{"chosen": "好回复", "rejected": "差回复"}
  • 构建方式：人工标注、AI 对比（LLM-as-Judge）、离线筛选（如基于规则的 rejection sampling）
  • 数据配比：SFT:DPO ≈ 1:1 到 1:3，DPO 数据量一般在 5K-50K 之间

数据配比策略（Data Mixing）

  • 领域配比：生产对话 40% + 领域知识 30% + 通用能力 20% + 安全对齐 10%
  • 去重：MinHash LSH 近重复检测，避免冗余数据稀释学习信号
  • 质量过滤：基于困惑度打分（用参考模型）、规则过滤（长度、特殊字符）

数据量参考：7B 模型 SFT 一般 5K-50K 条，DPO 一般 3K-20K 条。70B 模型可适当增加。

0x02-1 基础 RAG → 高级 RAG → 模块化 RAG 演进

标准答案

基础 RAG（Naive RAG）

  • Pre-Retrieval：Query 重写（HyDE）、Query 分解（Sub-Query）、查询扩展（Multi-Query）

  • Routing：根据 Query 类型路由到不同检索策略（Web Search vs 知识库 vs 数据库）

  • Tool Use：调用外部 API（数据库查询、计算器）辅助检索
  • Memory：多轮对话上下文管理、缓存常见查询结果

Agentic RAG：2025-2026 年趋势，LLM 作为 Agent 自主决定何时检索、用什么工具检索、是否需多步检索。

0x02-5 混合检索：BM25 + Dense + ReRank

标准答案

BM25（稀疏检索）：基于词频-逆文档频率（TF-IDF 增强版）。精确匹配关键词，对专有名词（如产品型号、人名）不可或缺。但语义匹配能力弱。

Dense Retrieval（稠密检索）：Embedding 模型将 Query 和文档映射到语义空间，cosine 相似度检索。语义匹配强但对拼写错误和罕见词敏感。

混合检索策略：

  • 权重融合：RRF（Reciprocal Rank Fusion，按排序位置融合，无参数）、CC（Convex Combination，需调权重）
  • 分步流程：BM25 + Dense 并行检索各 Top-K → 合并去重 → Cross-Encoder ReRank → 取 Top-N

Cross-Encoder ReRank：与 Bi-Encoder（Embedding 模型，query 和文档分别编码）不同，Cross-Encoder 将 query 和文档拼接后输入模型，输出相关性分数。精度高但计算成本大（O(N) 而非 O(1)）。常用模型：BGE-Reranker-v2、Cohere Rerank。

生产建议：Bi-Encoder 做第一轮召回（Top-100），Cross-Encoder 做第二轮重排序（Top-10→5）。延迟分配：召回 ~5ms，重排序 ~30-100ms（取决于模型大小）。

0x02-6 HyDE / Self-RAG / Corrective RAG

标准答案

HyDE（Hypothetical Document Embeddings）：先用 LLM 基于 Query 生成一个”假设性文档”（理想中应该检索到的内容），然后用这个伪文档的 Embedding 做检索。核心思想：在 Embedding 空间中，答题比问问题更接近答案。对于训练数据和 Query 分布差异大的场景效果显著。

Self-RAG：LLM 在生成过程自主决定是否需要检索、检索什么。通过 Reflection Token（检索/批评标记）控制流程。步骤：是否检索 → 检索哪些段落 → 生成 → 批评生成结果 → 修正。优势：减少不必要检索、提高生成质量。

Corrective RAG（CRAG）：检索后增加一个评估模块判断检索结果的 Relevance。如果评估认为不相关，触发：

  • 低分（不相关）：用 Web Search 替代知识库检索
  • 中分（部分相关）：保留检索结果但增加知识图谱查询
  • 高分（相关）：直接使用

对比总结：

方法 核心思路 适用场景
HyDE 用 LLM 生成中介查询 Embedding 分布不匹配
Self-RAG 检索决策内化到生成过程 需要精确控制检索时机
CRAG 检索后增加质量门控 知识库质量不稳定

0x04 LLM 评估与监控

0x04-1 LLM 评估方法论：LLM-as-Judge / 人工 / 自动

标准答案

人工评估（Human Eval）：

  • 黄金标准，成本最高（每条 $0.5-$3），速度慢
  • 适用于：小规模上线前评估、奖励模型训练数据
  • 方式：A/B 盲测（Side-by-Side）、LMSYS Chatbot Arena 风格

LLM-as-Judge：

  • 用一个 LLM（通常是 GPT-4 / Claude）评估另一个 LLM 的输出
  • 单维评分：对准确性/安全性/相关度等逐项打分（1-5 或 1-10）
  • 成对比较：A vs B 哪个更好（更鲁棒，减少位置偏差）
  • G-Eval：使用 Chain-of-Thought 逐步推理打分
  • 问题：位置偏差（favors first/last position）、自增强偏差（Judge 偏爱自己同类模型）、长度偏差（偏爱更长输出）

自动评估（Automatic Metrics）：

  • BLEU / ROUGE：N-gram 重叠（翻译/摘要场景可用，对话场景无效）
  • BERTScore：基于 BERT 嵌入的语义相似度
  • Perplexity：困惑度评估（仅衡量语言建模能力，不衡量有用性）
  • Token 级指标：符合 JSON Schema 比例、tool use 正确率、代码可执行率

最佳实践：三层评估体系

0x05-3 结构化输出：JSON Mode / Structured Output / Tool Use

标准答案

方法 原理 保证程度 适用场景
JSON Mode System prompt 要求输出 JSON 软约束，可能失败 简单结构化

Tool Use / Function Calling 模型原生 API 调用 中（取决于模型） 工具调用场景

Structured Output（约束解码）原理：在采样阶段，将指定的 JSON Schema 编译为有限状态机（FSM），每一步解码时只允许符合 Schema 的 token。SGLang 的 @function API 原生支持。

生产实践：对需要 100% 格式正确率的场景（API 集成、结构化数据提取），强制使用 Structured Output（SGLang/Outlines）。对非关键场景（文本生成中附带结构化信息），JSON Mode 即可。

0x06 成本优化

0x06-1 KV Cache 量化与复用

标准答案

KV Cache 占推理成本的 60-80%（对长序列场景）。优化手段：

KV Cache 量化：

  • FP8 KV Cache：H100 原生支持，精度损失几乎为零，显存减半
  • INT8 KV Cache：更激进的压缩，某些模型上 1-2% 精度损失
  • KIVI（INT4 KV Cache）：按通道和 token 维度非对称量化，精度损失可控

KV Cache 复用：

  • Prefix Caching：vLLM 自动缓存相同前缀的 KV 块（system prompt、few-shot examples），避免重复计算。命中率 30-70%
  • Session 级缓存：多轮对话中缓存已生成的 KV，每轮只计算新增 token。vLLM 的 --enable-prefix-caching 开启
  • Cross-Request 缓存：vLLM 支持跨请求的 KV Cache 共享（相同 prompt 前缀）

参考链接

  • KIVI: https://arxiv.org/abs/2402.02750

0x06-2 Prompt 缓存策略

标准答案

缓存类型 粒度 实现方式 节省 适用场景
精确匹配缓存 完整 prompt 哈希值 + KV Cache 快照 最大 System Prompt 固定
前缀缓存 Prompt 前缀 vLLM RadiAttention/Trie 中 共享 System Prompt
语义缓存 语义相似度 Embedding 聚类 + 近似匹配 中 FAQ / 常见问题
结果缓存 完整回复 Redis/Memcache 最大 幂等查询

语义缓存：对用户 Query 做 Embedding，在缓存库中查找语义相似度 > threshold 的历史 Query，直接复用之前 LLM 的回复。核心权衡：threshold 高则命中率低但准确性高，threshold 低则反之。典型 threshold 0.90-0.95。

生产经验：组合使用三层缓存——精确缓存（L1）→ 前缀缓存（L2, vLLM 内置）→ 语义缓存（L3, 业务层实现），可减少 45-80% 的 LLM API 调用成本。

0x06-3 模型路由

标准答案

核心思想：不是所有查询都需要最强模型。将简单/高频查询路由到小模型，复杂/低频查询路由到大模型。

路由策略：

策略 原理 成本节省 实现复杂度
规则路由 关键词/用户等级/字数阈值 30-50% 低
分类器路由 小分类模型判断查询难度 40-60% 中
Cascade（级联） 小模型先试，信心不足时级联大模型 50-70% 高
LLM-as-Router LLM 判断使用哪个模型 30-50% 中

Cascade 模式流程：

生产经验：Cascade 模式可将推理成本降低 50-70%，同时保持 95%+ 的回复质量（仅 5% 请求走昂贵模型）。需注意延迟：级联路径延迟 = 小模型延迟 + 判断逻辑 + 大模型延迟。

0x06-4 Batch API vs Streaming

标准答案

模式 延迟 吞吐量 成本 适用场景
实时（Streaming） 极低（TTFT < 500ms） 低 高 对话、聊天
同步（非 Streaming） 中（等待完整回复） 中 中 API 调用、简单任务
Batch API 高（分钟级） 极高 低（~50% 折扣） 离线推理、数据分析

Batch API 优势：OpenAI、vLLM 等均支持 Batch 模式，将多个请求打包发送，享受 50% 左右折扣。适合数据标注、批量总结、内容审核等延迟不敏感场景。

Streaming 成本：Streaming 本身不增加 Token 成本，但会增加基础设施成本（长连接、更复杂的请求处理）。

生产建议：

  • 实时交互 → Streaming（用户感知延迟更低）
  • 后台任务/数据处理 → Batch API
  • 混合策略：对话用 Streaming，后台用 Batch

0x07-3 微调模型过拟合 / 灾难性遗忘处理

标准答案

过拟合（Overfitting）：

  • 现象：训练 Loss 持续降低，验证 Loss 升高；回复变得单一、重复
  • 原因：SFT 数据太少、学习率过高、训练轮次过多
  • 方案：
    1. 减少训练轮次（1-3 epoch 足够）
    2. 降低学习率（SFT: 1e-5 到 2e-4, DPO: 1e-6 到 5e-6）
    3. 增加 lora dropout（0.05-0.1）
    4. 数据增强：轻微改写数据增加多样性

灾难性遗忘（Catastrophic Forgetting）：

  • 现象：微调后模型在通用任务上能力下降（如数学、推理、代码）
  • 原因：微调数据太偏领域、缺少通用能力数据
  • 方案：
    1. 数据配比：领域数据 + 20-30% 通用数据（如 OpenHermes 子集）
    2. EWC（Elastic Weight Consolidation）：对重要权重的更新施加正则惩罚
    3. 知识蒸馏：保留部分原始模型的输出分布作为约束
    4. 混合微调：同时训练领域任务和通用任务
    5. LoRA 合并检查：合并后评估通用能力，必要时回退

生产经验：LoRA 本身自带正则化效果，灾难性遗忘比全量微调轻得多。建议在微调后做全面的能力评估——不仅测目标领域，也要测通用能力（MMLU、HumanEval）。

0x07-4 Embedding 维度选择错误导致检索精度下降

标准答案

问题场景：某团队选择 4096 维 Embedding（e5-mistral-7b）做 10 亿级检索，结果召回率低于预期，且检索延迟无法接受（>500ms）。

问题分析：

  1. 维度灾难：高维空间中的距离趋于均匀化，区分度下降
  2. 索引瓶颈：HNSW 在 4096 维上的内存消耗是 1024 维的 4 倍，且查询速度指数下降
  3. 数据量-维度比：向量数量与维度的比例影响检索精度。10 亿级数据配 4096 维，密度过低

解决方案：

  1. 维度压缩：PCA 将 4096 维降至 768-1024 维，保持 95%+ 精度
  2. 模型更换：改用 bge-m3（1024 维）或 OpenAI text-embedding-3-large（可配置 256-1024 维）
  3. 多级检索：先用低维 Embedding（256 维）做粗召回，再用高维做细排
  4. 量化补偿：高维度下使用更激进的 PQ 配置，或改用 SCANN 索引

通用原则：

  • < 100 万向量：任意维度都可接受
  • 100 万 - 1 亿：推荐 768-1024 维

  • 1 亿：推荐 512-768 维，辅助 IVF_PQ/DiskANN

  • 维度选择决策：精度 vs 性能 vs 存储成本的三角权衡
