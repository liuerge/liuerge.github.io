0x00 分布式基础理论

0x00.1 CAP 定理与 PACELC 扩展

标准答案：
CAP 定理由 Eric Brewer 于 2000 年提出，指出分布式系统在**网络分区（Partition）发生时，必须在一致性（Consistency）和可用性（Availability）**之间做出取舍。三者不可兼得。

  • C（Consistency）：所有节点在同一时刻看到相同的数据。写操作完成后，所有后续读操作都必须返回最新值。
  • A（Availability）：每个请求都能获得一个（非错误的）响应，但不保证返回的是最新数据。
  • P（Partition Tolerance）：系统在网络分区（节点间通信中断）时仍能继续运行。

PACELC 由 Daniel Abadi 提出，弥补了 CAP 的不足：在分区时（P），选择可用性（A）或一致性（C）；否则（Else，即无分区时），选择延迟（L）或一致性（C）。

类别 系统示例 分区时选择 正常时选择
PA/EL Cassandra, DynamoDB 可用性 低延迟
PC/EC Spanner, HBase 一致性 一致性
PA/EC MongoDB (部分配置) 可用性 一致性
PC/EL PNUTS 一致性 低延迟

深入思考：

  • 边界情况：CAP 中定义的可用性是非常宽松的——一个 30 天后才返回响应的系统在 CAP 定义下仍然”可用”。PACELC 的贡献正是把延迟问题显式提出。
  • 真实案例：某大型电商在跨 AZ 部署时使用最终一致性的 DynamoDB，双十一期间出现购物车商品数量不一致（A AZ 的写入在 B AZ 读取不到）。原因是网络抖动导致分区，选择了 PA，但 Read Repair 异步执行的窗口期产生了不一致。解决方案：关键路径使用 DynamoDB 的强一致性读取（牺牲部分延迟），非关键路径使用最终一致性。
  • 生产经验：跨 AZ 部署时，哪怕你认为网络不会分区，也要假设分区会发生。跨 AZ 延迟本身就在 1-5ms 级别，PACELC 的 E→L 或 E→C 选择直接影响架构设计。

面试金句：CAP 不是让你三选二，而是告诉你当网络分区时你必须放弃一项。PACELC 则进一步指出，即便没有分区，你也在延迟和一致性之间做选择。关键不是选哪个，而是在系统的每个数据路径上分别做出明确的 trade-off 决策。

参考来源：

  • PACELC Theorem - Daniel Abadi, University of Maryland
  • Beyond CAP: Unveiling the PACELC Theorem
  • CAP Theorem - Eric Brewer, IEEE Computer 2012

0x00.2 一致性模型（详解）

标准答案：
一致性模型定义了分布式系统中数据写入后在什么条件下对读取可见。按强度递减排列：

模型 描述 典型实现 性能特征
线性一致性 (Linearizability) 操作实时有序，就像单机系统。一旦写完成，所有后续读都能看到。 Raft/Paxos 提交后、Spanner 用 TrueTime、单节点数据库 最慢，延迟较高
顺序一致性 (Sequential) 每个进程的操作按程序顺序执行，但不要求实时顺序。 分布式锁服务、Chubby 中等
因果一致性 (Causal) 有因果关系的操作按因果顺序可见，无因果关系的可乱序。 CRDT、DynamoDB 的向量时钟、COPS 较好
最终一致性 (Eventual) 如果没有新写入，最终所有副本会收敛到相同值。 DNS、CDN、DynamoDB (默认)、Cassandra 最高性能
读己之写 (Read-your-writes) 写入者总能读到自己的写入，但其他用户可能看不到。 会话级别的一致性 良好
单调读 (Monotonic Read) 用户不会读到旧版本——时间只进不退。 粘性会话 良好

读写对账表：

读一致性 \ 写一致性 强同步写 异步写
强一致性读 Spanner、ZooKeeper 不可能（读可能看到旧数据）
最终一致性读 通常没问题（但延迟写入窗口内可能读到旧值） Cassandra 默认
单调读 自动满足 需额外保证（粘性路由）

深入思考：

  • 线性一致性的代价：Google Spanner 的 TrueTime 使用 GPS + 原子钟的硬件方案来提供精确的时钟误差边界（commit wait ~7ms）。没有 TrueTime 这种硬件支持，纯软件实现线性一致性需要至少 2 轮 RTT（如 Raft 的 commit 流程）。
  • 因果一致性的计算复杂度：追踪因果关系需要向量时钟（Vector Clock），每个副本维护一个 N 维向量。当节点数 N 很大时（100+），向量时钟的存储和传输开销不可忽略。Dynamo 的实践是限制向量时钟的大小（如最多 10 个节点），超过时裁剪最旧条目——这理论上可能丢失因果关系信息。
  • 生产经验：阿里的 OceanBase 在金融场景使用强同步（多数派提交），而淘宝商品浏览使用最终一致性 + 本地缓存 TTL 1s 来兼得性能和体验。

面试金句：一致性模型不是选择题，而是光谱。从强到弱逐步降级，每条路径都需要明确的业务容忍度声明。面试时能说出「我们支付系统用线性一致性，商品推荐用最终一致性，用户在个人中心用 read-your-writes」，就是真懂的人。

参考来源：

  • Dynamo: Amazon’s Highly Available Key-value Store - SOSP 2007
  • DDIA - Chapter 9: Consistency and Consensus

0x00.3a CRDT（无冲突复制数据类型）

标准答案：
CRDT（Conflict-free Replicated Data Type）是一种不需要冲突解决就能在多节点间自动合并的数据结构。CRDT 保证：只要所有节点最终收到所有更新，它们的状态就数学上保证收敛。

两种类型：

  1. 基于状态的 CRDT（CvRDT）：节点之间同步完整状态。要求合并函数满足交换律、结合律、幂等律（如 Max、Set-union、G-Counter）。
  2. 基于操作的 CRDT（CmRDT）：节点之间同步操作日志。要求操作的因果序和操作本身可交换。

常见 CRDT 类型：

类型 数据结构 合并方式 实用场景
G-Counter (Grow-only Counter) 每个节点维护自己的计数 取每个节点的 max PV 统计
PN-Counter G-Counter 合成（+ 和 - 各一个） 两个 G-Counter 分别合并 点赞（+1/-1）
G-Set (Grow-only Set) 元素集合 取并集 标签、白名单
2P-Set add 和 remove 两个 G-Set add set 并集 + remove set 并集 黑白名单
LWW-Register (时间戳, 值) 保留时间戳最大的值 购物车
OR-Set (Observed-Remove Set) 每个元素带唯一 tag 基于 tag 的添加/移除合并 协作编辑

深入思考：

  • CRDT 在 Riak 中的实践：Riak 的 riak_dt 模块实现了多种 CRDT，用于自动解决多节点写入冲突。当 Dynamo 风格的向量时钟超过限制时，CRDT 可替代应用层冲突解决。
  • CRDT vs OT（Operational Transform）：两者都用于协作文档编辑。OT 需要中心化的服务器做操作转换（Google Docs 使用），CRDT 天然去中心化（如 Yjs、Automerge）。近年来 CRDT 正在逐步取代 OT——Yjs 是目前最流行的协作编辑 CRDT 库，支持 Rich Text。
  • CRDT 的存储膨胀问题：OR-Set 每个元素需要唯一 tag，且移除操作并不真正删除 tag（只标记删除），导致存储持续增长。需要手动做 GC（Garbage Collection）——在确认所有节点都已看到移除操作后，可以安全清理 tag。
  • 生产经验：Notion 的实时协作使用 CRDT（基于 Logoot 算法变体），但数据层最终用 JSON patch 同步到后端数据库。纯 CRDT 存储成本高，前端实时协作 + 后端快照/增量存储是更经济的混合方案。

面试金句：CRDT 的数学基础（交换律+结合律+幂等律）保证了去中心化场景下数据自动收敛。面试中如果能把 CRDT 和 OT 的区别讲清楚，说明你对分布式数据同步有深入的横向视野。记住：Riak 用 CRDT 做冲突解决、Yjs 用 CRDT 做协作编辑、Redis 的 CRDT 用于 Redis Enterprise 的多活。

参考来源：

  • CRDTs: Consistency without concurrency control - Letia et al. 2009
  • A comprehensive study of CRDT - Shapiro et al. 2011
  • Yjs - CRDT for collaborative editing

0x00.3 Quorum NWR + Hinted Handoff + Read Repair

标准答案：
Dynamo 架构使用 Quorum NWR 模型提供灵活的一致性保障：

  • N：数据的副本数
  • W：写操作需要确认的节点数
  • R：读操作需要返回结果的节点数

条件：W + R > N 保证强一致性（读一定能看到最新写入）。

特性：

  1. Sloppy Quorum（宽松仲裁）：当 preference list 中的节点不可用时，协调器继续沿一致性哈希环寻找后续健康节点。这保证可用性，但可能降低 durability（数据写到了非理想节点）。
  2. Hinted Handoff（提示移交）：如果目标节点不可用，数据被写入一个临时替代节点，并在该节点上附加”提示”信息。当原节点恢复后，临时节点将数据转发回去。
  3. Read Repair（读修复）：读取时若有节点返回旧版本数据，协调器会异步将新版本写入该节点。

深入思考：

  • 边界情况：W + R > N 在并发写入场景下仍不能保证线性一致性。原因是：两个写入并发发生时，一个写入只到了部分节点，另一个写入到了另一部分节点，读仲裁时可能基于过时的版本号做出错误判断。解决：引入权威版本（如 Cassandra 的 last write win 依赖时间戳）或使用读修复+读前仲裁（先读最新版本再写）。
  • 源码层面：Cassandra 的 Read Repair 实现在 StorageProxy.java 的 RowIteratorAndMutations 中——当发现 digest 不一致时，发起 full data read 比较所有副本的完整数据，然后生成 delta mutation 发送给过期副本。
  • 生产经验：某大厂使用 Casssandra 时，Read Repair 在高峰期触发大量背景读，导致网络带宽被打满。解决方案：设置 read_repair_chance = 0.1（仅 10% 的读触发修复），配合手动 Anti-Entropy 窗口。

面试金句：NWR 是 Dynamo 优雅的核心。高手会在面试中补充：W=1 表示最高写入可用性但最低 durability；R=N, W=1 表示读强一致但写性能极差。真实生产中 W 和 R 通常取 N/2+1（多数派）。

参考来源：

  • Dynamo: Amazon’s Highly Available Key-value Store
  • Cassandra Architecture - Read Repair

0x00.4 FLP Impossibility 与共识算法

标准答案：
FLP Impossibility（Fischer, Lynch, Patterson, 1985）证明：在一个异步系统中，只要存在哪怕一个进程可能崩溃，就不存在一个确定性算法能保证所有非故障进程达成共识。 核心原因是在异步模型下无法区分”进程挂了”和”进程太慢”。

实际工程如何绕过：

  1. 使用故障检测器（Failure Detector）——虽然不是完美的（在异步系统中不可能完美），但足够好。如 Φ-Accrual Detector。
  2. 使用随机化——如共识协议中的随机超时（Raft 的选举超时 150-300ms 随机化）。
  3. 部分同步模型（Dwork, Lynch, Stockmeyer）——系统大多数时候同步，偶尔异步。这是工程中最常用的假设：NTP 同步的时钟 + TCP 超时机制。

深入思考：

  • 为什么 FLP 重要：它告诉我们共识在理论上需要某种”非确定性”或”部分同步”的假设。任何声称能在纯异步模型下保证安全达成共识的系统，要么是错的，要么做了隐蔽的假设。
  • 工程含义：Raft 的 leader election 使用随机超时，实际上引入了”概率性”保证——大多数情况下很快达成共识，但理论上可能永远选不出 leader（概率趋近于 0）。Paxos 的活锁也是同理。
  • 对比 Paxos vs Raft：两者都依赖部分同步模型。区别在于 Raft 显式地表述了所有假设并分解为可理解的子问题。

面试金句：FLP 是分布式系统的”测不准原理”——它不阻止你做共识，但告诉你要付出什么代价。你必须在同步性、故障模型或确定性三者中放弃一个。

参考来源：

  • Impossibility of Distributed Consensus with One Faulty Process - JACM 1985
  • DDIA - Chapter 9
  • Consensus in the Presence of Partial Synchrony - DLS 1988

0x00.5 分布式系统中的时间与顺序

标准答案：
分布式系统中时间是一个棘手问题——不同机器的物理时钟无法完全同步，需要引入逻辑时钟来建立事件顺序。

三种时钟类型：

类型 定义 限制
物理时钟 (Physical Clock) NTP 同步的墙上时钟 NTP 误差 1-100ms，NTP 本身可能回拨时钟
逻辑时钟 (Lamport Clock) 每个事件递增计数器，附带进程ID 只能捕获 happens-before 关系，不能检测并发
向量时钟 (Vector Clock) 每个进程维护 N 维计数器 存储 O(N)，裁剪大向量可能丢失信息

Lamport Clock：每个进程维护一个计数器。发送消息时 counter += 1，附带在消息中。收到消息时 counter = max(local, received) + 1。可确定两个事件的偏序，但不能判断并发。

TrueTime（Google Spanner）：使用 GPS + 原子钟，返回一个时间区间 [earliest, latest]。TrueTime 保证真实时间一定在这个区间内。Spanner 使用 commit_wait = latest - now + 1ms 确保提交时间晚于写操作的真实发生时间，从而实现外部一致性（external consistency——等同于线性一致性）。

深入思考：

  • NTP 回拨问题：NTP 检测到系统时钟偏差 > 1000s 时会大幅回拨时钟，直接导致 Snowflake ID 重复。生产环境需使用 ntpd -x（渐进调整）代替默认的 ntpd（跳跃调整）。
  • Hybrid Logical Clock（HLC）：谷歌的 Hybrid Logical Clock 结合物理时钟和逻辑时钟——名义上使用物理时间，当物理时钟回拨时用逻辑组件补偿。HLC 的值 l 满足 l ≈ physical_time，且 l 严格单调递增。CockroachDB 使用 HLC。
  • 面试常问：”如何在没有同步时钟的情况下判断事件顺序？”——答案是向量时钟 + 因果检测。如果向量 VA ≤ VB（每个维度的值都 ≤），则 A 是 B 的因果前驱；否则两者并发。

面试金句：时间问题是分布式系统中最微妙的陷阱。能区分 Lamport Clock（偏序）和 Vector Clock（偏序+并发检测）的人，已经超越了 90% 的面试者。 而能说出 TrueTime 的 commit_wait 机制和 HLC 的设计的，就是真懂分布式系统的。

参考来源：

  • Time, Clocks, and the Ordering of Events in a Distributed System - Lamport 1978
  • Spanner: Google’s Globally Distributed Database - TrueTime Section
  • Hybrid Logical Clock - Kulkarni et al. 2014

0x01 高可用与容错

0x01.1 故障转移：主从 / 多活 / 异地多活

标准答案：

模式 架构特点 RPO RTO 适用场景
主从（Active-Standby） 主库写入，从库异步/半同步复制 秒级~分钟级 分钟级 中小规模
多活（Active-Active） 多节点同时读写，冲突处理 RPO≈0 秒级 全球部署
异地多活 跨地域多活，带流量调度 RPO≈0 <30s 必须保证业务连续性的大厂

深入思考：

  • 主从故障转移踩坑：使用异步复制时，主库挂了但未同步的写入会丢失。MySQL 的 semi-sync replication 缓解但非100%保证。实际案例：某金融公司 MySQL 主库宕机，异步复制落后 3s，切换后丢失了 3000 笔订单的支付确认。
  • 多活冲突处理：多活写冲突不可避免。DynamoDB 使用 last-writer-wins（LWW），依赖精确的时间戳。但跨地域时 NTP 误差可达 10-100ms。CouchDB 使用多版本合并（类似 Git 的合并），把冲突留给应用层。
  • 异地多活的全局视角：阿里在双 11 的异地多活方案是”单元化”（Cell-based）：每个单元是一个自包含的部署单元，单元间通过 MQ 异步同步。流量按用户 ID 哈希路由到固定单元，从根源上减少跨单元写入。

面试金句：异地多活的本质不是”多写”，是”分区写入 + 有限同步”。真正的难度在于如何划分数据分片使大多数请求不跨域。

0x01.2 熔断器：Hystrix / Resilience4j / Sentinel

标准答案：
熔断器（Circuit Breaker）是一种快速失败机制，防止依赖故障级联扩散。核心是状态机：CLOSED → OPEN → HALF-OPEN → CLOSED。

Resilience4j 源码级分析（v2.x）：

  • 滑动窗口：两种策略
    • COUNT_BASED：环形缓冲区（Circular Array），固定大小 N，每次新结果覆盖最旧条目。空间复杂度 O(N)。
    • TIME_BASED：环形数组 + 分桶聚合，每个桶聚合 1 秒内所有调用结果。自增更新部分聚合，整体 Snapshot 预聚合（O(1) 获取）。
  • 状态机：CircuitBreakerStateMachine 内部维护 AtomicReference<CircuitBreakerState>。状态切换通过 stateComparator 验证合法后 CAS 切换，避免并发问题。
  • 关键配置：failureRateThreshold、slowCallRateThreshold、slidingWindowSize、minimumNumberOfCalls（窗口未满前不触发熔断）、waitDurationInOpenState、permittedNumberOfCallsInHalfOpenState

Sentinel 对比 Hystrix：

特性 Hystrix Sentinel Resilience4j
隔离方式 线程池 / 信号量 信号量 + 计数器 信号量 / 线程池
熔断策略 错误率 错误率/慢调用比例/异常数 错误率/慢调用率
滑动窗口 滚动时间窗 滑动时间窗（LeapArray） 计数/时间滑动窗
实时监控 Hystrix Dashboard 控制台 + 实时数据 Micrometer/Prometheus
维护状态 维护模式 (2018) 活跃 活跃

深入思考：

  • HALF-OPEN 的陷阱：如果 permittedNumberOfCallsInHalfOpenState 设置过大（如 10），且 10 个请求全部成功，服务可能已被打垮（这些请求仍然通过了熔断器）。AWS 的实际推荐值是 3。
  • 滑动窗口初始填充问题：Resilience4j 的 COUNT_BASED 窗口初始全空，意味着故障率计算只在窗口填满后触发。Issue #1731 曾讨论过：如果 slidingWindowSize=10, failureRateThreshold=20%，启动后连续 10 个请求全失败才触发熔断，而不是 2 个失败。社区建议初始用 SUCCESS 预填充或用 TIME_BASED 窗口。
  • 生产经验：某在线教育公司的 Hystrix 线程池隔离默认线程池大小 10，但高峰期一个依赖的 QPS 到了 200，大量请求被拒绝。根本原因是线程池隔离的线程上下文切换开销 + 池大小估算不准。迁移到 Resilience4j 后改用信号量隔离，减少了 70% 的线程开销。

面试金句：熔断不是万能的——它保护了下游，但增加了上游的调用方重试风暴。熔断 + 指数退避 + 降级三者要配合使用。另外，永远不要忘记监控熔断器状态，如果熔断器开了 20 分钟没人注意到，那就是哑巴告警。

参考来源：

  • Resilience4j CircuitBreaker Docs
  • Hystrix Wiki (维护模式)
  • Sentinel Docs - Alibaba
  • Proposition regarding COUNT_BASED sliding window - Issue #1731

0x01.3 限流：令牌桶 / 漏桶 / 滑动窗口 + 分布式限流

标准答案：
四大限流算法：

算法 原理 优点 缺点
令牌桶 固定速率生成令牌放入桶中，请求消耗令牌 允许突刺（积累的令牌可应对突发），实现简单 桶大小需要合理设置
漏桶 请求以固定速率流出，超出桶容量则丢弃 平滑流量，无突刺 无法应对突发流量
滑动窗口（日志） 记录每个请求的时间戳，统计窗口内请求数 精确，避免临界问题 内存占用 O(窗口请求数)
滑动窗口（计数器） 将时间窗口分为小格，按格统计 空间固定，较精确 存在边界精度损失

分布式限流：Redis Lua 方案：

Sentinel 限流源码：
Sentinel 的核心限流实现是 FlowRuleChecker + LeapArray（滑动窗口数组）。LeapArray 将 1s 分为 2 个 500ms 的格子（窗口数可配置）。每个格子持有一个 MetricBucket 包含 LongAdder 统计（QPS、线程数等）。当新请求到达时，累加当前格子的计数器，并计算过去 1s 内所有格子的总和。LongAdder 保证了高并发下的低竞争（Cell Stripping 技术）。

深入思考：

  • 分布式限流的精度与一致性权衡：使用 Redis 做分布式限流时，最关键的是原子性。Lua 脚本保证了原子性但增加了一次网络往返（约 0.2-2ms）。高并发场景下 Redis 可能成为瓶颈。解法：本地限流 + Redis 同步（如 Guava RateLimiter 本地桶 + 定期与 Redis 同步配额）。
  • Sentinel 的 FlowSlot：Sentinel 的调用链（Slot Chain）中，FlowSlot 执行流控规则。它使用 FlowRuleChecker#canPassCheck 进行校验，内部调用 TrafficShapingController 的不同实现（如 RateLimiterController 基于漏桶、WarmUpRateLimiterController 带预热）。
  • 生产案例：某公司秒杀场景直接上 Sentinel + Redis 分布式限流，结果 Redis QPS 飙到 45w 后 Redis 集群带宽打满。为解决这个问题，他们将限流逻辑改为两级限流：每台机器本地 Guava RateLimiter 1000/s，集群总限流 10000/s 用 Redis。本地拦截大部分请求，Redis 只处理跨机器的配额协调。

面试金句：限流必须考虑精度和性能的平衡。分布式系统中，100% 精确的限流意味着每次请求都要网络往返，这在 sub-ms 延迟要求的场景下不可接受。容忍一定精度损失（如 5-10%）以换取高性能，是完全合理的架构取舍。

0x01.4 降级与隔离

标准答案：

舱壁隔离（Bulkhead Pattern）：将系统资源（线程、连接、内存）分组隔离，防止一个组件故障消耗所有资源导致整体崩溃。

隔离方式 原理 优点 缺点
线程池隔离 每个依赖一个独立线程池 隔离彻底，可独立配置 线程上下文切换开销大，线程数量难规划
信号量隔离 限制并发数量，不切换线程 开销小，适合非阻塞调用 不能超时切断，需配合 TimeLimiter
舱壁分组 按租户/优先级分组隔离 细粒度控制 配置复杂

降级策略：

  • 静默降级（Fail Silent）：返回默认/空值，不抛异常
  • 快速失败（Fail Fast）：直接返回错误
  • 降级数据源：切换到备选服务（如读取二级缓存）
  • Stub 响应：返回模拟数据（如商品信息不可用时返回默认图片和名称）

深入思考：

  • 线程池 vs 信号量：Hystrix 默认每个依赖 10 线程的线程池。假设有 20 个依赖，仅线程池就需要 200 线程，加上应用自身线程，JVM 线程数轻松上千。Java 21 Virtual Threads 可以缓解，但 IO 密集场景下线程切换仍不免费。信号量隔离无此问题，但不能隔离耗时调用。
  • Sentinel 的做法：Sentinel 使用信号量（计数器）而非线程池。DegradeSlot 检查资源是否进入熔断/降级状态。当资源熔断时，引用 DegradeException 快速失败。
  • 生产经验：美团的大促活动中，降级策略分四级：L0（完全不降级）→ L1（降级非核心接口，如推荐、广告）→ L2（降级半核心接口，如评论、评分）→ L3（只保证核心交易链路可用）。降级等级通过配置中心实时推送。

面试金句：降级不是”功能出了问题不要了”，而是主动放弃非核心功能以保全核心功能。好的降级方案需要业务方参与，定义好每个功能的优先级和降级表现。

0x01.5 重试策略与幂等性设计

标准答案：

指数退避 + Jitter：

Jitter 三种策略：

  1. Full Jitter：wait = random(0, base * 2^attempt) —— 最均匀分散
  2. Equal Jitter：wait = base * 2^(attempt-1)/2 + random(0, base * 2^(attempt-1)/2)
  3. Decorrelated Jitter（AWS 推荐）：wait = min(cap, random(base, wait * 3)) —— 自适应

幂等性设计：

方式 原理 适用场景
唯一请求 ID 每次请求带上 UUID，服务端去重（如幂等表） 支付、下单等敏感操作
乐观锁（版本号） 操作条件 where version = expected 库存扣减
Token 机制 先获取 token，再操作，token 一次性 表单防重复提交
状态机 通过业务状态判断是否已执行 订单状态流转
去重表 事务中插入去重键，利用唯一索引 消息消费去重

深入思考：

  • 重试风暴：某公司核心支付服务挂了 10s，所有调用方同时发起重试，流量放大到正常 30 倍，导致整个数据中心入口打满。根本原因是重试没有加 Jitter，所有客户端退避时间一致。解决方案：全链路加 Jitter + 设置重试预算（retry budget，重试流量不超过总流量的 10-20%）。
  • AWS 的实践：AWS SDK 默认使用 Decorrelated Jitter，最大重试次数 3（某些服务 5 次）。每次重试在应用层超时时间要小于上游的等待超时，否则重试链导致请求堆积。
  • 幂等性的挑战：基于唯一请求 ID 的幂等处理需要一个高可用的去重存储。如果去重存储本身挂了怎么办？大厂通常用本地磁盘持久化 + 分布式存储备份两层保证。

面试金句：重试不是免费的——每一次重试都是对下游的一次伤害。好的重试策略 = 指数退避 + Jitter + 重试预算 + 配合熔断器。没有幂等的重试就是灾难。

0x02 缓存架构

0x02.1 多级缓存架构

标准答案：
现代大型系统通常使用三级或四级缓存：

各级缓存特性：

层级 延迟 容量 典型 TTL 一致性要求
CDN 1-50ms 无限（付费） 1h - 7d 弱
Redis 1-5ms 内存有限（GB级） 1min - 1h 中等
Caffeine <0.1ms 内存有限（MB级） 10s - 5min 可容忍不一致
DB 10-100ms 最大 - 强

深入思考：

  • 为什么需要本地缓存：Redis 的 1-5ms 延迟在微服务中已经是极快了，但在 10k QPS 下意味着 50-500ms 的网络开销。本地缓存的 <0.1ms 延迟可以减少 90% 以上的 Redis 请求。
  • 缓存一致性问题：多级缓存最大的难点是一致性。常见方案：本地缓存用很短 TTL（30s-5min），配合 Redis Pub/Sub 广播失效消息。
  • Caffeine 源码：Caffeine 的 W-TinyLFU 策略是 LFU + LRU 的变体，使用 Count-Min Sketch（概率计数器）追踪高频访问项，兼顾频率和新近度。相比 Guava Cache 的 LRU，在扫描场景下命中率提升 30%+。

面试金句：多级缓存的本质是用空间换时间 + 用本地网络换远程网络。每一级的加入都增加了一致性的维护成本，不要为了用缓存而用缓存。

0x02.2 缓存策略与三大问题

标准答案：

缓存策略对比：

策略 读流程 写流程 一致性 适用场景
Cache Aside 先读缓存，miss 则读 DB 后回填 先写 DB，后删缓存 最终一致性（有窗口） 通用选择
Read Through 从缓存读，缓存负责回填 同 Cache Aside 同 Cache Aside 缓存中间件
Write Through 同 Cache Aside 先写缓存，缓存同步写 DB 强(写路径) 写敏感的缓存
Write Behind 同 Cache Aside 只写缓存，异步写 DB 弱 写吞吐量大

三大问题终极解法：

问题 现象 原因 解法
缓存穿透 大量请求 miss 缓存 -> DB 压力巨大 查询不存在的数据 1. 缓存空值（TTL 短） 2. 布隆过滤器 3. 参数校验
缓存击穿 热点 key 失效瞬间，大量请求 DB 一个热点 key 过期 1. 互斥锁（只让一个请求回填） 2. 逻辑过期（永不过期，后台异步刷新） 3. 二级缓存兜底
缓存雪崩 大量 key 同时过期 -> DB 被击穿 多个 key TTL 相同 1. TTL 加随机值（如 1-5min） 2. 本地缓存前置 3. 限流降级

深入思考：

  • 布隆过滤器细节：布隆过滤器的假阳性概率为 (1 - e^(-kn/m))^k（k 哈希函数数，m 位数组大小，n 插入元素数）。推荐配置：m = 10*n, k = 7 时假阳性率约 1%。但布隆过滤器不支持删除，升级版布谷鸟过滤器支持删除和动态扩容。
  • 互斥锁的坑：分布式缓存使用互斥锁时，锁的超时时间必须 > DB 查询时间，否则锁失效时仍有并发请求穿透。但锁超时太长又影响性能。实际做法是：本地锁 + 短超时降级为允许少量穿透。
  • 逻辑过期方案：缓存不设 TTL，但同时存储逻辑过期时间（如 expire_time: timestamp）。读时判断是否过期，过期则加锁更新。优点：不会有空窗期。缺点：数据在更新前短暂不一致。
  • 生产经验：Reddit 的缓存层使用 Memcached，通过一致性哈希分散热点。当单个 key 流量过大（如超级碗直播帖），他们启动热点 key 自动分裂机制：在一个 key 后面加 _1, _2, _3 后缀，请求随机分散，读时遍历所有子 key 取最新。

面试金句：穿透、击穿、雪崩是缓存面试的”三件套”。能区分这三者说明你理解了缓存的常见陷阱，能逐一说出 2-3 种解法说明你有实践经验。最终极的解法永远是本地缓存 + 分布式缓存 + TTL 分散 + 熔断兜底的组合拳。

  • Caffeine Cache - W-TinyLFU
  • Redis 缓存穿透/击穿/雪崩 - Alibaba Cloud Community
  • Caching Challenges in Distributed Systems - ByteByteGo

0x02.3 一致性哈希

标准答案：
一致性哈希（Consistent Hashing）将节点和数据映射到同一个哈希环上，数据沿环顺时针找到第一个节点。

关键特性：

  • 单调性（Monotonicity）：节点增删时只有少量数据需要迁移，而非全部重新哈希
  • 分散性（Spread）：数据均匀分布在各节点
  • 平衡性（Balance）：引入虚拟节点（Virtual Nodes）实现均匀分布

虚拟节点的作用：

  • 每个物理节点在环上有 100-200 个虚拟位置
  • 节点增减时，负载变化平滑（仅迁移 1/N 的数据量）
  • 异构集群可通过调整虚拟节点数量分配不同权重

生产 Rebalance 优化：

  1. Pending Rebalance：不立即移动数据，先标记节点状态为 LEAVING/JOINING，让新请求走新的路由，同时后台缓慢迁移旧数据。
  2. 增量迁移：每次只迁移一小批数据，配合限流，避免带宽打满。
  3. 双写双读：迁移期间新旧节点同时提供读写，等数据一致后再切流。

深入思考：

  • Hotspot 问题：即使虚拟节点很均匀，某些 key（如大 V 的 feed）仍可能导致单节点热点。解法：热点 key 分裂（如 key_1, key_2, ..., key_N 分布在多个节点）。
  • Rebalance 生产案例：某公司 Redis Cluster 扩容时，重新分片导致集群吞吐从 20w QPS 降到 5w，持续 3 小时。原因是没有限制 rebalance 的 slot 迁移速率。优化后限制每秒迁移 500 个 slot，集群影响降到 10% 以内。
  • 一致性哈希 vs Rendezvous Hashing：Rendezvous（最高随机权重哈希）不需要维护环结构，通过计算每个 key 与所有节点的哈希值取最大来决定归属。复杂度 O(N)（N 为节点数），在节点数 <1000 时可行。优点是更均匀。

面试金句：一致性哈希不是银弹。真正的挑战不在算法本身，而在运维层面——如何做平滑的 rebalance、如何处理热点、如何监控分布均匀度。这比算法本身更显功力。

  • Consistent Hashing and Random Trees - Karger et al. 1997
  • Dynamo Paper - Consistent Hashing Section
  • Cassandra Virtual Nodes

0x02.4 CDN 原理

标准答案：
CDN（Content Delivery Network）通过在全球部署边缘节点，将内容缓存到离用户最近的位置。

关键概念：

  • 回源（Origin Pull）：边缘节点缓存 miss 时，向源站请求内容并缓存
  • 预热（Cache Warming）：提前将内容推送到 CDN 节点，避免大量请求同时回源
  • 边缘计算（Edge Computing）：在 CDN 边缘节点执行轻量计算逻辑（如 Cloudflare Workers）

深入思考：

  • 动态加速（DCDN）：动态内容（如 API 响应）不可缓存，但可通过 CDN 路由优化——选择最优路径到源站，而非公网 BGP 路由。Akamai 的 SureRoute 是最早的实现。
  • 回源风暴：CDN 节点大量同时回源可能导致源站被打挂。解法：回源限流（最多每秒回源 N 次）、分布式回源（回源请求经过另一层缓存）、CDN 断路器（回源超时时返回过期缓存并增加 TTL）。
  • 生产案例：某站点在新闻热点爆发时 CDN 大量 key 同时过期，瞬发 200Gbps 回源流量，源站 Nginx 扛不住挂了。事后加了：1) 回源限流 2) 默认 TTL 从 1h 改为 1h + random(0, 30min) 3) 热点内容自动延长 TTL。

面试金句：CDN 不只是图片加速——动态 CDN、边缘计算、回源优化是高级话题。能聊到 CDN 的回源风暴防护和动态加速路由选择，说明你有 CDN 调优的实际经验。

0x03.1 分库分表

标准答案：

分片策略对比：

策略 方式 优点 缺点 典型场景
范围分片 按 ID/时间范围分段 适合范围查询，扩容只需加新段 数据可能倾斜（热点集中在最近段） 时序数据、日志
哈希分片 对分片键取模或哈希 数据分布均匀 范围查询需全部扫描，扩缩容代价大 用户数据、订单
一致性哈希 哈希环 + 虚拟节点 扩缩容影响小，分布均匀 实现复杂，C++复杂查询难 Redis Cluster
地理位置分片 按地域分片 数据本地化，延迟低 跨地域查询困难 多活部署

ShardingSphere 源码分析：

  • SQL 解析：使用 ANTLR 解析器生成 AST。SQLStatementParserEngine 解析 SQL → SQLStatement 对象。
  • 路由计算：ShardingRouter 接收 SQLStatement，提取分片键值，匹配分片策略（ShardingAlgorithm），计算目标数据源 + 表。
  • SQL 改写：SQLRewriter 根据路由结果改写 SQL（表名替换、分片条件注入）。
  • 结果归并：MergeEngine 支持流式归并（StreamMergedResult，不占用全部内存）和内存归并（MemoryMergedResult，需要排序/聚合时）。

深入思考：

  • ShardingSphere-JDBC vs Proxy：JDBC 模式轻量，无网络开销，但强耦合应用（需要引入 jar），适合 Java 应用。Proxy 模式独立部署，支持异构语言但有额外网络开销（1-3ms 延迟），适合跨语言架构或已有 Proxy 层。
  • 分布式主键：ShardingSphere 内置雪花算法和 Leaf-Segment 两种分布式 ID 方案。
  • 分片键选择错误案例：某公司用订单创建时间做范围分片，双 11 当天所有写入集中在最新段，该 MySQL 实例 CPU 飙升到 100%。解决方案：用 buyer_id 哈希分片 + buyer_id 做查询路由，避免单热点。
  • 分库分表的第二维度：当分片后的单表仍超过 5000w 行时，需要二级分区（如先按 buyer_id 哈希分库，再按月份分表）。

雪花算法（Snowflake）：
Twitter 设计，64 位长整型：1bit(符号) + 41bit(时间戳) + 10bit(工作机器ID) + 12bit(序列号)

  • 每个节点每毫秒可生成 4096 个 ID

时钟回拨处理：

策略 做法 优缺点
阻塞等待 回拨后等待时间同步追上 简单，但回拨大时阻塞时间长

0x03.3 分布式事务

标准答案：

方案 原理 一致性 性能 侵入性 适用场景
2PC (XA) 准备 + 提交/回滚 两阶段 强一致性 低（资源锁定） 高（需 XA 驱动） 短事务、跨库强一致
TCC Try + Confirm + Cancel 最终一致 高（无锁） 高（需业务实现） 核心链路、高频写
Saga 子事务链 + 补偿 最终一致 最高 中 长流程、异步事务
Seata AT 自动生成反向 SQL 最终一致 中 低（@GlobalTransactional） 快速接入分布式事务
本地消息表 MQ + 状态表 + 回查 最终一致 高 低 解耦场景
MQ 事务消息 半消息 + 回查 最终一致 高 低 RocketMQ/RabbitMQ 用户

Seata AT 模式原理（重点）：

  1. 一阶段：TM（Transaction Manager）开启全局事务，RM（Resource Manager）执行业务 SQL + 生成回滚日志（before image + after image） 写入 UNDO_LOG 表，完成后上报 TC（Transaction Coordinator）。
  2. 二阶段提交：TC 通知所有 RM 成功，RM 删除 UNDO_LOG。
  3. 二阶段回滚：TC 通知 RM 失败，RM 根据 UNDO_LOG 的 before image 生成反向 SQL 回滚。

选择决策树：

深入思考：

  • TCC 的 Cancal 幂等性：TCC 的 Cancel 方法必须幂等，因为网络超时可导致重复调用。实现方式通常是：status = CANCELLING → CANCELLED 状态机转换，幂等判断。
  • Saga 与 TCC 的本质区别：Saga 的补偿和正向操作是相同粒度的（Service 级别的 rollback），补偿是业务语义的撤销。TCC 的 Try 阶段保留资源（加锁/预留），Cancel 释放资源——更像”资源预留”。Saga 的中间状态对外可见，TCC 隔离性更好。
  • Seata AT 的 BIG 问题 —— 多表聚合：如果一个全局事务跨了多个表，Seata 的 UNDO_LOG 只记录了单表的 before/after image。但业务 SQL 可能涉及多表关联的写入。此时回滚不能保证正确的业务语义（例如 A 表成功、B 表回滚后 A 表的数据在业务上可能是孤儿）。
  • 生产案例：某支付公司使用 Seata AT 模式处理跨服务转账。上线后发现性能问题——UNDO_LOG 在高峰期的 MySQL 磁盘 IO 暴增。原因：每个事务写入 2 条 UNDO_LOG（转入 + 转出），每秒 1000 笔就是 2000 次额外写入。优化：转到 TCC 模式，手动实现资源预留，没有了 UNDO_LOG 开销。

面试金句：分布式事务选型的核心是确认业务是否真的需要强一致性。大量场景其实在业务层面可以用”对账 + 补齐”代替分布式事务。比如资金转出成功后如果转入失败，晚上对账脚本自动补发即可。能用业务补偿解决的问题，不要用技术方案解决。

0x03.4 一致性与高性能的 Trade-off

标准答案：
一致性与高性能是分布式系统的核心矛盾。降低一致性级别可以显著提升性能。

具体 Trade-off 分析：

场景 强一致方案 性能 弱一致方案 性能
读取 读多数派（R + W > N） 2-3x RTT 读本地副本 1 RTT
写入 同步写入所有副本 N x RTT 异步复制 1 RTT
分布式事务 2PC/Seata AT 额外 1-2 RTT Saga/最终一致 0 额外 RTT
缓存更新 写后立即删缓存 + 分布式锁 高 设置 TTL + 异步淘汰 低

工程实践：

  1. 读写分离：强一致读走主库，最终一致读走从库
  2. 事务粒度：用最大努力保证最终一致性，通过定时对账修复
  3. 异步化：写操作只保证存储到本地，异步复制到其他节点

面试金句：如果你能告诉面试官你们系统在哪个路径选择了强一致、哪个路径选择了高性能、以及这样选择的业务理由，说明你真正理解了 trade-off 的艺术。

0x04 一致性协议

0x04.1 Raft

标准答案：
Raft 将共识问题分解为三个子问题：领导者选举、日志复制、安全性。

领导者选举：

  • 节点角色：Leader / Candidate / Follower
  • 选举超时：随机化 150-300ms，防止 Split Vote
  • 当选条件：Candidat 获得多数派投票 + 自己的日志比投票者更新（至少一样新）
  • Pre-Vote：候选人在发起正式选举前，先发送 PreVote RPC，如果发现已有 Leader 在工作则放弃选举。避免网络分区节点恢复时扰乱集群。

日志复制：

成员变更：

  1. Joint Consensus（联合共识）：新旧配置同时运行，在过渡期需要两个多数派同时同意（旧 N 和新 M 的各多数派）
  2. 单步变更（Single-Server Change）：一次只增删一个节点。通过在旧配置中提交新配置来实现。注意：存在 Bug——若同时移除两个节点并发执行会出问题（详见 Raft 开发者邮件列表讨论）。

深入思考：

  • Pre-Vote 细节：Pre-Vote 在 TiKV/etcd 中都有实现。核心是防止网络分区后的节点重新接入时引起的”脑震荡”——假设 5 节点集群，3 节点分区，2 节点分区。2 节点无法选出 leader（需要 3 票），3 节点正常工作。等网络恢复后，2 节点意识到自己的 term 落后，放弃选举。
  • etcd 的优化：etcd v3 使用 ReadIndex + Follower Read（线性一致性读取不经过 Leader），只在 follower 上确认 commit index ≥ read index 后直接响应。减少了 1 次 RTT。
  • 生产案例：某公司 5 节点 etcd 集群版本升级时，逐台重启，第 3 台重启后集群失联。原因：Leader 在第 2 台重启时已经切过一次，第 3 台重启后剩余的 2 个节点无法构成多数派。解决方案：生产环境升级最少保留 3 节点在线，或使用滚动升级工具如 etcdadm。

面试金句：能说出 Pre-Vote、Joint Consensus 和 ReadIndex 三个优化，表明你对 Raft 的理解超出了表层。一定要知道 Raft 的”坑”——比如单步变更的 Bug，比如 3 节点集群升级时必须保活至少 2 台。

0x04.2 Paxos

标准答案：

Basic Paxos：

  • 两阶段：Prepare → Promise | Accept → Accepted
  • 角色：Proposer / Acceptor / Learner
  • 安全性保证：最多只有一个值被选定（Chosen）
  • 活锁问题：多 Proposer 交替提出更高轮次的 Prepare，导致永远不会选出值。实际用 Leader 解决。

Multi-Paxos：

  • 选出一个稳定的 Leader（Proposer），所有提案由 Leader 提出
  • Leader 可跳过 Prepare 阶段（假设自己的提案编号最大），只走 Accept → Accepted
  • 日志复制：每个日志条目是一个独立的 Paxos Instance

Fast Paxos：

  • Client 直接向 Acceptors 发送提案，跳过 Leader
  • 需要至少 2F+1 个 Acceptors（而不是 2F+1 个参与者）来容忍 F 个故障
  • 适合 WAN 场景（减少一次 RTT），但实现复杂

深入思考：

  • Paxos 的多 Leader 问题：Multi-Paxos 本身不假设唯一 Leader，允许并发提案。但并发时安全性不受影响（Paxos 的设计保证如此），只是效率降低（有冲突时需要更高轮次解决）。Raft 明确要求唯一 Leader，保证了效率但增加了 Leader 不可用时的可用性窗口。
  • Paxos vs Raft 的根本差异：Paxos 是一个”单决策”算法家族，Raft 是”状态机复制”算法。Paxos 家族有大量变种（Cheap Paxos、Fast Paxos、Byzantine Paxos、EPaxos 等），Raft 则非常聚焦。
  • EPaxos（Egalitarian Paxos）：允许所有节点并行处理提案，没有单一 Leader 瓶颈，但需要 O(3) 轮 RTT，且依赖”冲突检测”。

面试金句：面试中提到 Paxos 时不要试图完整描述算法（那需要 30 分钟）。而是说出 “Basic Paxos 的两阶段本质 + Multi-Paxos 的 Leader 优化 + 与 Raft 的核心区别”。一句话总结：Paxos 是理论基石，Raft 是工程实现。

0x04.3 Gossip 协议

标准答案：
Gossip（流言协议）是一种去中心化的信息传播协议，灵感来源于流行病传播模型。

三种模式：

  1. Push：有更新的节点主动推送给随机选择的节点 → 适合传播新信息
  2. Pull：节点随机从其他节点拉取更新 → 适合同步状态
  3. Push-Pull：既是 Push 又是 Pull，效率最高

Anti-Entropy（反熵）：节点之间定期比较数据状态，使用 Merkle Tree 快速定位差异数据的范围，只同步差异部分。Cassandra 在节点恢复后使用 Anti-Entropy 做背景修复。

Phi Accrual Failure Detector：

  • 不设固定超时，而是计算连续心跳间隔的正态分布累积概率
  • 累积概率的负对数 = phi（φ）值
  • φ = 8（默认）意味着节点存活的概率 < 0.00000001（约 10^-8）
  • 自适应网络环境：网络抖动时 phi 自然变大，不会误报

深入思考：

  • Gossip 的收敛速度：每个周期，节点随机选择 f 个节点传播。经过 O(logN) 轮后信息可达所有节点（概率 >= 1 - 1/N）。每轮选择 3 个节点时，10 轮可覆盖 1000 节点集群。
  • Gossip 代价：O(N^2) 的消息冗余（每个节点每轮 O(f) 条消息，N 个节点共 O(f*N) 条）。对于 1000 节点、每轮 3 个节点，消息量适中。但超过 5000 节点时需要考虑分组策略（Hierarchical Gossip）。
  • Cassandra 的实现：每 1 秒 Gossip 一次，每次与 1-3 个随机节点交换信息。包含：心跳计数器（单调递增）、节点状态（UP/DOWN）、负载信息、Schema 版本等。Phi 值超过 8 时标记节点疑似故障，超过 20 时标记下线。

面试金句：Gossip 的核心优势是去中心化和自适应。当面试官问”如何做大规模集群的失败检测”时，回答 Gossip + Phi Accrual 比回答”心跳检测”高级两个量级。

  • The Phi Accrual Failure Detector - Hayashibara et al. 2004
  • Cassandra Gossip Docs

0x04.4 Zab / ZooKeeper

标准答案：
Zab（ZooKeeper Atomic Broadcast）是 ZooKeeper 的核心共识协议，分为三个阶段：

  1. Discovery（发现）：新 Leader 收集所有 Follower 的已提交日志，确定 Epoch
  2. Synchronization（同步）：Leader 使所有 Follower 的日志与其一致（truncate 不需要的条目 + 补充缺失的条目）
  3. Broadcast（广播）：正常运行，Leader 以两阶段提交的方式广播提案

Zab vs Raft：

特性 Zab Raft
协议基础 类似 Paxos，但专注有序广播 独立设计，Three-way split
成员变更 通过 Zab 协议本身完成 Joint Consensus / 单步变更
选举约束 必须有最新 ZXID 的节点才能当选 日志必须至少和投票者一样新
提交确认 2PC（准备 + 提交） 1PC（Leader 确认多数派即可提交）
典型应用 ZooKeeper etcd, Consul, TiKV

深入思考：

  • Zab 的 2PC vs Raft 的 1PC：Zab 需要 Leader 先向所有 Follower 发送 Proposal，收集 ACK 后再发送 Commit。两步消息意味着更低吞吐。而 Raft 中 Leader 发送 AppendEntries 后，只要收到多数派成功的回复，即可在本地 commit。这是 Raft 的一个关键优化。
  • ZXID（ZooKeeper Transaction ID）：64 位，高 32 位为 Epoch（Leader 任期），低 32 位为计数器。决定了 Zab 中日志的全序关系，也决定了选主时的约束。
  • 生产经验：ZooKeeper 3.5+ 引入动态重新配置（Dynamic Reconfiguration），不再需要滚动重启所有节点来变更成员。同时引入 Learner 角色——只复制数据但不参与投票，用于只读扩展。

面试金句：Zab 和 Raft 是非常相似的协议，核心区别在日志提交的阶段数和选举的约束方式。实际选型时不要纠结协议差异，而是看生态：用 etcd 做服务发现、用 ZooKeeper 做协调服务、用 Consul 做服务网格——都是好的选择。

短链接本质是转换映射：短链key → 原始URL → 302重定向。

架构图：

核心要点：

  • 哈希方案：用 Base62 编码（0-9a-zA-Z）= 62 个字符，6位= 62^6 ≈ 568 亿组合
  • ID 生成：Snowflake 生成唯一 ID → Base62 编码作为短链 Key
  • DB 设计：shorted_key (PK), original_url, created_at, expire_at
  • 缓存：Redis 缓存短链映射，TTL 30min - 7d 视业务而定
  • 重定向：使用 302（临时重定向）而非 301，以跟踪点击数据

深入思考：

  • 自定义短链：支持用户自定义（如 /my-link），将自定义 Key 映射到保留的 ID 范围
  • 链接过期和回收：过期链接可回收，但加入冷却期（避免指向旧内容的安全风险）
  • 防攻击：限制单 IP 生成频率、链接内容安全校验（防钓鱼）
  • 百万级 QPS：写入用 DB 批量写，读取全部走 Redis（99% QPS 是读）

面试金句：短链接的”短”不是核心问题，”唯一”才是。关键设计决策是ID 生成方案和重定向响应码的选择（302 才支持统计）。

0x05.2 设计秒杀系统

标准答案：
秒杀系统的核心挑战：高并发瞬时流量、库存准确性、防刷单。

架构图：

关键技术：

  1. 前端限流：CDN 缓存静态页，JS 控制按钮置灰、随机延迟请求
  2. 接口限流：Sentinel/Redis Lua 限流，限制总 QPS + 单用户频率
  3. 库存预热：Redis 缓存库存数量，Lua 脚本原子扣减
  4. 队列消峰：MQ/Redis List 缓冲请求，Worker 串行化处理
  5. 限时+限购：每人限购 N 件，使用 Redis 记录购买记录

深入思考：

  • Redis 扣库存的原子性陷阱：如果使用 WATCH + 事务而非 Lua，高并发下 WATCH 频繁冲突，重试过多。必须使用 Lua 脚本保证原子性。
  • 库存超卖防护：Redis 扣减库存后再落库，数据库有乐观锁 UPDATE stock SET count = count - 1 WHERE id = X AND count > 0 作为最终防线。
  • 异步下单的数据一致性：订单创建和库存扣减是分布式操作，可用 RocketMQ 事务消息或本地消息表保证最终一致性。
  • 生产案例：某公司秒杀使用 Redis 单节点（非集群）存库存，秒杀开始后 Redis 直接被打满——所有请求都走了同一个 key 的读写。优化：库存分片到多个 key（如 stock_001, stock_002, ...），请求随机路由。

面试金句：秒杀的本质不是”快”而是”准”——库存不能多卖一件。核心设计思路是层层过滤：前端限流 → 接入限流 → 排队 → 原子扣减 → 异步处理。每一层都拦截掉大部分流量，用最昂贵的资源（DB 写）处理最少的请求。

0x05.3 设计 IM 系统

标准答案：
IM 系统的核心：消息有序、可靠投递、在线感知、多端同步。

架构图：

关键技术：

  1. 连接管理：WebSocket 长连接，一致性哈希路由到特定接入节点

0x05.5 设计搜索引擎

标准答案：
搜索引擎的核心流程：爬取 → 倒排索引 → 查询 → 排序。

架构图：

关键技术：

  1. 倒排索引：Term → Document List 的映射。每个 term 存储 docId + 位置 + 权重
  2. 分词：中文用 IK Analyzer / jieba / HanLP，英文用空格+Stemming
  3. 排序（Ranking）：BM25（Okapi BM25）是最常用的排序算法，公式：∑ IDF * (TF * (k1+1))/(TF + k1*(1-b + b*|d|/avgdl))
  4. 分片（Shard）：ES 将索引分为多个 Shard，分布在集群上，搜索时先在每个 Shard 上查询再合并。

深入思考：

  • 实时索引：增量索引（近实时）和全量索引（批量重建）的组合。ES 使用 Lucene 的 Segment 合并机制——写入到内存 buffer → flush 到磁盘 → merge segment。
  • 拼音搜索 & 纠错：使用拼写检查算法（如 Levenshtein Distance，BK-Tree）进行查询自动纠正。
  • 性能优化：分页使用 Search After 而非 From+Size（避免深度分页的 O(N) 复杂度）。

面试金句：讲到搜索引擎必须说出倒排索引和 BM25 排序。如果能补充”分片 + 合并”的分布式查询过程和”Segment 合并”的写入策略，就覆盖了面试的全部核心。

0x06.1 微服务拆分原则

标准答案：
微服务拆分的最佳实践基于 DDD（Domain-Driven Design）限定上下文（Bounded Context）：

拆分原则：

  1. 按业务能力拆分：订单服务、支付服务、库存服务
  2. 按子域拆分：销售域、物流域、财务域
  3. 按变化频率拆分：高频变化与低频变化分离
  4. 按数据边界拆分：数据访问完全独立的服务

COLA 架构（Clean Object-oriented and Layered Architecture）：

深入思考：

  • 过度拆分：一个 30 人的团队拆了 40 个微服务，每个服务 0.75 人维护，导致跨服务调用的开销超过了服务本身的价值。经验法则：每 5-8 个开发人员对应 1 个服务，或按”一个服务可以独立开发、测试、部署”为标准。
  • 拆分时机：不要一开始就微服务。先做模块化单体（Modular Monolith），当出现以下信号时考虑拆分：1) 团队扩大 -> 部署冲突增多 2) 某模块需要独立扩缩容 3) 某模块技术栈需要更换。

面试金句：微服务不是免费的午餐。DDD 的意义不是让你分得细，而是让你知道边界在哪。没有 DDD 指导的微服务拆分，终将沦为分布式单体（Distributed Monolith）。

0x06.5 Service Mesh 与服务网格

标准答案：
Service Mesh 将微服务间的通信逻辑（服务发现、负载均衡、熔断、重试、可观测性）从应用代码中剥离，下沉到基础设施层（Sidecar 代理）。

架构模型：

主流对比：

特性 Istio Linkerd Consul Connect
Sidecar Envoy (C++) Linkerd2-proxy (Rust) Envoy
控制面 Istiod (Go) 控制器 (Go) Consul Server
协议支持 HTTP/gRPC/TCP HTTP/gRPC/TCP HTTP/gRPC/TCP
mTLS ✅ 自动 ✅ 自动 ✅ 自动
性能开销 5-15% (高) <5% (低) 5-10%
K8s 集成 深度 深度 一般
复杂度 高 低 中

深入思考：

  • 为什么要有 Service Mesh：在服务数 ≤ 20 时，熔断、重试、限流可在框架层（如 Spring Cloud）实现；当服务数 > 200 时，框架的统一升级变得不可能。Service Mesh 通过 Sidecar 实现流量层面的统一治理，故障注入、灰度发布、流量镜像等功能无需修改代码。
  • Envoy 的核心：Envoy 的 xDS API 是服务网格的事实标准。Pilot 通过 ADS（Aggregated Discovery Service）下发 LDS（Listener）、RDS（Route）、CDS（Cluster）、EDS（Endpoint）等配置。Envoy 的线程模型是 1 主线程 + N Worker 线程，使用非阻塞 I/O，单机性能可达 50w QPS。
  • Sidecar 性能开销分析：Istio + Envoy 在 1000 QPS、1KB payload 下基准测试显示 P99 延迟增加约 5-15ms（约 10-20%）。主要开销来自：1) HTTP 头解析和修改 2) mTLS 加密解密 3) 遥测数据上报。使用 Ambient Mesh 模式（无 Sidecar 的节点级代理）可将开销降到 2-5%。
  • 生产经验：某公司 Istio 集群在 500+ 服务上线后，控制面 Pilot 的内存消耗达到 8GB。原因：大量 VirtualService 和 DestinationRule 导致 Envoy 的配置更新风暴。优化：合理使用 namespace 隔离、减少不必要的高级路由规则数量、开启 PILOT_ENABLE_UNIQUE_CLUSTER。
  • Istio Ambient Mesh：2023 年 Istio 引入 Ambient 模式，去除了 Sidecar，使用每个节点一个的 ztunnel 代理 + 可选方式 waypoint proxy。降低了资源消耗，但牺牲了部分 L7 能力的灵活性。

面试金句：Service Mesh 的本质是将分布式系统通信的复杂度从应用层下沉到基础设施层。当面试官问”你需要 Service Mesh 吗”时，最有力的回答是：“当你的团队维护超过 50 个微服务时，框架层面的统一治理已经不可持续，此时 Service Mesh 的收益远大于成本。” 能说出 Envoy xDS 和 Ambient Mesh 的区别，就证明你真正关注过 Service Mesh 的演进。

0x06.6 分布式链路追踪

标准答案：
分布式链路追踪（Distributed Tracing）解决微服务架构中请求在多个服务间流转时的全链路可视化问题。

核心概念：

概念 描述 示例值
Trace 一次请求的完整链路 一个用户下单的完整调用链
Span Trace 中的一个调用单元 Order Service → Payment Service 的一次 RPC
Trace ID 全局唯一标识一次 Trace abc123def456
Span ID 标识一个 Span span0001
Parent Span ID 父级 Span ID，构建调用树 root_span

常用协议：OpenTelemetry （OTel）是 CNCF 的标准化可观测性框架，统一了 OpenTracing 和 OpenCensus。

架构：

深入思考：

  • 采样策略：100% 采样在高流量下不可行（存储成本）。常用策略：1) 头部采样（根据 TraceID 哈希决定是否采样，保证同一 Trace 的所有 Span 一致采样） 2) 尾部采样（根据 Span 特性决定，如慢调用必然采样） 3) 概率采样（如 1% 全量 + 10% 随机）。
  • 上下文传播：HTTP 用 traceparent header（W3C Trace Context 标准），gRPC 用 Metadata，MQ 用消息属性。跨线程时通过 Context Propagation API 传递。
  • 生产实践：某公司使用 Jaeger 集群，每天产生 10TB Span 数据。优化：1) 将正常的 200 响应降低采样率到 0.1% 2) 错误和慢调用 100% 采样 3) 使用头部采样保证 Trace 完整性。

面试金句：链路追踪的关键不在 SDK 接入，而在采样策略和低成本存储。能说出 W3C Trace Context 协议和如何处理高流量下的采样，就是生产级经验。

0x07.0 分布式系统中的反压（Backpressure）机制

标准答案：
反压（Backpressure）是一种流量控制机制，当下游处理速度跟不上上游数据生产速度时，上游感知到压力并主动降低生产速率，防止系统被压垮。

反压的三种实现层级：

层级 实现方式 优点 缺点
网络层 TCP 滑动窗口 / 流量控制 协议原生 不能识别数据语义
框架层 Reactive Streams (RxJava / WebFlux) 应用透明 要求全链路 Reactor 模型
应用层 Kafka Consumer lag 监控 + 动态限流 灵活可控 需要业务代码配合

Flink 的反压机制：
Flink 使用基于信用度的反压：下游 Task 向上游发送”我还剩多少缓冲”（信用度）。当信用度为 0 时，上游停止发送。Flink 1.5+ 默认使用此机制，替代了老版本的 TCP 反压（会导致 Checkpoint barrier 对齐延迟）。

Kafka 消费者的 Backpressure：

  • max.poll.records：单次 poll 最多获取的消息数
  • max.partition.fetch.bytes：每个分区拉取的最大字节数
  • Kafka Consumer 通过控制 poll 频率间接实现反压——处理慢时 poll 间隔自然增大

深入思考：

  • Reactive Streams 的规范：4 个接口——Publisher、Subscriber、Subscription、Processor。核心是 Subscription.request(n) ——下游告诉上游”给我 N 个元素”。
  • 生产经验：某实时推荐系统使用 Flink + Kafka，某次上游 Kafka 流量暴涨（3x 正常），Flink 的反压机制自动降低消费速率，Checkpoint 延迟从 10s 增加到 45s，但系统未崩溃。恢复后 Checkpoint 正常，无数据丢失。这就是反压的价值——优雅降级，不死不丢。
  • 无反压的后果：无限制的流量积压在内存中 → GC 压力 → Full GC → 服务失联 → OOM → 进程挂掉 → 数据丢失。

面试金句：反压是分布式系统的”安全阀”。能说出 Flink 的 Credit-based 反压和 TCP 反压的区别，说明你做过流式处理系统的生产运维。

0x07.1 千万级 QPS 秒杀系统设计全历程

背景：某电商双 11 大促，秒杀峰值 QPS 达 1500 万，涉及 1 万+ SKU。

架构演进：

V1 - 单体闯关（第一年）：

  • 架构：Nginx → Tomcat Cluster → MySQL
  • 结果：QPS 1 万时 MySQL 连接池打满，502 响应
  • 问题分析：没有缓存、没有限流、DB 写热点

V2 - 加缓存+限流（第二年）：

  • 架构：CDN → Nginx + Lua 限流 → Redis Cluster + Local Cache → Tomcat → MySQL
  • 结果：QPS 提升到 10 万，但 Redis 单 key 热点导致 Redis 集群倾斜
  • 问题：库存扣减使用单个 Redis key → 热点

V3 - 库存分片+MQ（第三年）：

  • 架构：接入层限流 + Redis Lua 原子扣减 + 库存分片到 128 个 key + RocketMQ 异步订单
  • 结果：QPS 200 万，库存精度 99.997%
  • 关键优化：库存预热到 Redis，分 128 片，请求随机哈希到某一片

V4 - 单元化+多活（第四年）：

  • 架构：按用户 ID 哈希路由到独立单元（Cell），每个单元独立部署全链路（限流→Redis→MQ→订单）
  • 结果：QPS 1500 万，跨单元请求 < 1%
  • 关键：单元内独立数据库、独立 Redis 集群、独立 MQ 集群

踩坑记录：

  1. 库存超卖：V2 时 Redis 使用 GET + DECR 非原子操作，高并发下超卖 0.3%。修正：全部改为 Lua 原子脚本。
  2. MQ 堆积：V3 时 RocketMQ 集群单 Topic 堆积超过 100 万条，消费者跟不上。修正：拆分多个 Topic + 消费者线程池从 20 加到 200。
  3. 预热失败：V3 时 Redis 预热脚本未做幂等，运行两次导致库存翻倍。修正：预热前先检查 key 是否存在。

面试金句：千万级秒杀的核心思想不是”提高 QPS”，而是降低到达核心系统的 QPS。每一层都过滤掉 90% 的请求，最后一层只处理下单成功的那 0.1%。这个”漏斗”思维是架构师的基本功。

0x07.2 全球多活部署的 Kafka 集群踩坑

背景：某跨国公司在 5 个 AWS Region 部署 Kafka，用于日志收集和数据同步。

架构：

遇到的坑：

  1. 跨 Region 同步延迟（坑 1）：

    • 现象：us-east-1 到 ap-southeast-1 同步延迟经常超过 30s
    • 原因：MirrorMaker 2 使用单线程 consumer → producer，无法利用多分区并行
    • 修正：每个 Topic 分区数对齐，MM2 使用 tasks.max 配置为分区数，实现并行同步

  2. Offset 冲突（坑 2）：

    • 现象：MM2 双向同步时产生无限循环
    • 原因：A Region 的消息复制到 B → B 的 MM2 又复制回 A → 死循环
    • 修正：MM2 的 offset.translation 功能 + 使用 replication.policy 添加 Region 前缀过滤

  3. Broker 跨 Region 连接失败（坑 3）：

    • 现象：分区 Leader 被选在 us-west-2，而该 Region 网络不稳定，导致其他 Region 的 follower 长期 ISR 不足
    • 原因：Kafka 默认 min.insync.replicas=2，但全球集群中跨 Region 的 broker 可能无法相互到达
    • 修正：使用 Rack Awareness，确保每个 partition 的 replica 分布在不同 AZ/Rack，但不要跨 Region。全球复制用 MirrorMaker 而非 Kafka 原生 Replication。

  4. Consumer Group Rebalance 风暴（坑 4）：

    • 现象：一个 Region 网络波动导致 200 个 Consumer 同时 Rebalance，服务漂移 5 分钟
    • 原因：Kafka 2.x 的 Eager Rebalance（全部 Stop → 重新分配）
    • 修正：升级到 Kafka 3.x 启用 Cooperative Rebalancing（增量重新分配，不停机）

面试金句：Kafka 的全球多活难点不在于”怎么搭”，而在于跨 Region 网络的不可靠性假设。永远不要假设跨 Region 的网络延迟和可用性符合预期。使用 MirrorMaker 而非原生跨域复制，是工程的最佳实践。

0x07.3 异地多活数据库同步故障案例

背景：某互联网金融平台使用 MySQL + Canal（binlog 同步）实现异地多活。

架构：

故障全过程：

T0: 正常：3 个机房同步延迟 < 2s

T+5min: 杭州机房网络抖动：

  • Canal 连接杭州主库中断 30s
  • 杭州到上海、深圳的 binlog 同步滞后到 5 分钟

T+10min: DBA 手动切主：

  • 将上海机房提升为写主库
  • 但杭州机房的写入还没完全同步到上海

T+15min: 数据丢失：

  • 杭州 5min 内 327 笔交易没有同步到上海
  • 上海接管后，用户看到数据”回滚”
  • 杭州恢复后试图同步已丢失数据，但与上海的新写入产生主键冲突

T+2h: 数据恢复：

  • 工程师手动逐条比对 binlog，提取丢失的 327 笔交易
  • 用业务字段（订单号、时间）匹配，解决冲突后补录

根本原因分析：

  1. 缺乏数据一致性校验：没有实时检测同步延迟的告警
  2. 切主没有数据完整性校验：切主前没有确认源端数据已全部同步
  3. 缺少幂等写入机制：数据恢复时没有去重，导致重复扣款

事后改进：

  1. 增加 GTID 模式 的 MySQL 复制（基于事务 ID 的精确同步定位）
  2. 同步延迟超过 10s 自动告警
  3. 切主流程：停机 → 确认延迟为 0 → 切主 → 双写验证 → 开放流量
  4. 增加单元化架构（按用户 ID 分片到固定机房，避免多写冲突）

面试金句：异地多活最大的谎言就是”数据最终一致”。当故障发生时，”最终”可能是 3 天，而你需要的是精确的 RPO 和 RTO 指标。所有切主操作都必须有数据完整性校验步骤，否则一次的疏忽就是灾难。

0x07.4 Redis 大 Key 与热 Key 故障案例

背景：某社交平台使用 Redis Cluster 做缓存和计数器。

故障一：大 Key 引发的集群雪崩

现象：Redis 某个分片节点 OOM 挂掉，然后整个集群逐个节点挂掉。

排查：

  1. 发现有一个 Key 存储了某个大 V 的 200 万粉丝 ID 列表（Set 类型），value 大小约 400MB
  2. 这个 Key 上的操作（SMEMBERS, SADD, SREM）全部是 O(N) 复杂度
  3. 当某个请求对该 Key 执行 SMEMBERS 时，Redis 单线程阻塞了 15 秒
  4. 其他请求排队等待，导致该节点 CPU 100%、连接数打满
  5. 集群判断该节点失联，出发 failover，slave 提升为 master
  6. 但新的 master 同样在下次请求时被打挂，如此循环

修复：

  1. 在线：使用 RENAME 将大 Key 重命名为隐藏 Key，让请求走降级逻辑
  2. 永久：将粉丝列表拆分到 1000 个小 Key（fans:v123:1 到 fans:v123:1000），每个 Key 约 400KB
  3. 最佳实践：Redis 单 Key 不要超过 10KB，string 最大不超过 100KB，集合类元素不超过 1 万
  4. 监控：添加 redis-cli --bigkeys 定期扫描 + Grafana 监控 key 大小

故障二：热 Key 打爆单个节点

现象：双 11 直播弹幕区，某直播间 ID 的 key 访问量瞬间 200 万 QPS，单一 Redis 分片（最大 10 万 QPS）被打穿。

排查：

  • Redis Cluster 的 key 归属于某个固定 slot → 所有请求打到一个分片
  • 该分片的 CPU 和带宽被拉满，其他分片空闲

修复：

  1. 本地缓存：直播间热度信息使用 Caffeine 本地缓存，TTL=5s，减少 Redis 请求 90%
  2. 热点 key 分裂：将 room:12345:heat 拆成 room:12345:heat:1 到 room:12345:heat:10，客户端写入时随机写一个，读时遍历求和
  3. 白名单机制：Redis 检测到热点 key 后自动通知客户端本地缓存该 key
  4. 架构层面：Redis Enterprise 支持 Redis on Flash（内存+SSD 混合存储）缓解内存压力

生产经验总结：

问题 检测 预防 治理
大 Key --bigkeys / MEMORY USAGE / 监控 key 大小 Split Key / 限制 Key 大小 RENAME + 拆分
热 Key hotkeys 参数 / 客户端请求统计 本地缓存 / key 分裂 本地缓存 + 自动降级
慢查询 SLOWLOG / 监控 latency 避免 O(N) 操作 拆分复杂操作

面试金句：Redis 面试中，大 Key 和热 Key 是最有价值的踩坑话题。能说出具体的数据（200 万粉丝列表=400MB、15 秒阻塞）和一个完整的排查→修复→预防流程，面试官就知道你是真正上过生产的人。

0x07.5 消息队列幂等消费与数据一致性案例

背景：某支付系统使用 RocketMQ 做异步对账，消息重复消费导致重复扣款。

问题现象：

幂等消费的六种方案：

方案 核心原理 优点 缺点
唯一主键表 每次消费先 insert 唯一键 简单可靠 需要额外 DB 表
业务状态机 根据订单状态判断是否已处理 无额外存储 状态流转需要统一
Redis 幂等 Token SETNX + TTL 高性能 TTL 过期可能重复
分布式锁 消费前加锁，消费后释放 通用 锁开销
去重表（唯一索引） 事务内 insert 业务 ID 事务保证 需要修改业务表
MQ 事务消息（RocketMQ） 半消息 + 业务执行 + Commit 强一致 复杂度高

最佳实践组合：

深入思考：

  • RocketMQ 事务消息：生产者发送”半消息”到 Broker，Broker 不会投递给消费者。生产者执行本地事务，成功后发送 Commit，Broker 才开始投递。如果生产者挂了，Broker 回查生产者的事务状态。这保证了”生产消息”和”本地事务”的原子性。
  • Kafka 的幂等生产者：Kafka 3.0+ 支持 enable.idempotence=true，通过 Producer ID (PID) + Sequence Number 实现生产端的幂等，防止重复写入 Partition。但消费端幂等仍需业务自行保证。
  • 生产经验：某公司使用 RocketMQ 事务消息处理订单系统，上线后发现回查接口的压力过大——每笔事务消息的回查次数平均 3 次。优化：降低回查频率（默认 6 秒 15 次改为 6 秒 6 次）+ 生产者本地事务完成后尽快异步发送 Commit。

面试金句：MQ 幂等消费不是难题，难点在效率和一致性的平衡。唯一主键表最可靠但需要一次 DB 写入，Redis Token 最快但有 TTL 窗口。面试中要说出每种方案的适用场景和极限情况下的失效模式。
