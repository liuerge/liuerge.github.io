数据库面试题：MySQL 面试全解

面向全栈高级/资深研发的 MySQL 深度全解。覆盖 InnoDB 引擎、MVCC、锁机制、SQL 优化、主从高可用、分库分表、分布式数据库对比与生产实战。
每道题包含：标准答案 + 深入思考 + 生产踩坑案例 + 参考链接。

0x00 InnoDB 存储引擎

0x00-01 B+ 树数据结构（高度/扇出/为什么选 B+ 树而非 LSM/B 树/红黑树）

标准答案：

MySQL InnoDB 存储引擎使用 B+ 树作为索引的底层数据结构。B+ 树是 B 树的一种变体，所有数据仅存储在叶子节点，内部节点只存储键值和指针（路由信息）。叶子节点之间通过双向链表连接，形成有序的数据集合。

  • 高度：对于 16KB 的默认页大小，假设主键为 BIGINT（8 字节）加指针（6 字节），一个内部节点可存放约 16KB / 14B ≈ 1170 个键值对。三层 B+ 树即可存储 1170 × 1170 × 16 ≈ 2190 万 条记录。四层可支撑约 256 亿条记录。实际中三层高度即可覆盖绝大多数生产场景，从根节点到叶子节点仅需 3~4 次磁盘 I/O。
  • 扇出（Fanout）：B+ 树的扇出系数远高于二叉搜索树（如红黑树）。16KB 页的扇出约 1000+，二叉搜索树的扇出为 2。在相同数据量下，B+ 树高度为 log_{1170}(N)，红黑树高度为 log_2(N)。对于 10 亿条数据，B+ 树需要 3 层，红黑树需要 30 层——磁盘 I/O 差距悬殊。

为什么 MySQL 选 B+ 树而非其他结构？

数据结构 不选的原因
红黑树 高度随数据量线性增长，每个节点 2 个子节点。千万级数据需 20+ 次 I/O，磁盘寻道时间不可接受。仅适合内存场景（如 TreeMap）。
B 树 内部节点也存数据，同样大小的页能容纳的键值对更少，扇出系数低。范围查询需中序遍历多次回溯，不如 B+ 树叶子链表高效。
LSM-Tree 写放大严重（Level-0 到 Level-n 的 Compaction），读放大明显（需检查多层级）。引擎实现复杂，读写放大不可同时优化。更适合写多读少场景（如 RocksDB、HBase）。
哈希表 不支持范围查询、前缀匹配、排序。仅适合等值查询场景（如 NoSQL 的 key-value 存储）。

深入思考：

B+ 树的真正优势在于磁盘 I/O 局部性和空间利用率的平衡：

  1. 磁盘预读：操作系统按页（4KB）读取磁盘，B+ 树的页大小（16KB）与之匹配，一次 I/O 可获取整个节点。顺序叶子链表使范围扫描变为顺序 I/O。
  2. 缓存效率：内部节点可常驻 Buffer Pool，实际查询仅叶子节点产生磁盘 I/O。热点中间节点被频繁命中，缓存友好。
  3. B+ 树 vs LSM-Tree 的取舍：在 OLTP 场景下，B+ 树的点查和范围查均优于 LSM-Tree。LSM-Tree 在写入吞吐上有优势（顺序写），但读路径需逐层查找 Bloom Filter 和 SSTable。这就是为什么 InnoDB 用 B+ 树做主存储，而 TiKV（TiDB 的存储层）用 RocksDB（LSM-Tree）做分布式 KV 存储——场景不同。

生产踩坑案例：

Uber 的 B+ 树页分裂灾难：某金融系统使用 UUID 作为主键，在亿级数据插入时频繁触发 B+ 树页分裂（随机插入导致节点 50% 利用率时即触发分裂），产生大量碎片和 IO 写入。最终导致 Buffer Pool 脏页刷盘压力过载，查询延迟从 5ms 飙升至 3s。解决方案是将主键改为自增 BIGINT，页分裂频率降低 90%，写入吞吐提升 5 倍。

0x00-02 聚簇索引与二级索引

标准答案：

InnoDB 使用聚簇索引（Clustered Index）作为主键索引。表数据本身就是 B+ 树，叶子节点存储完整的行数据。每个表有且仅有一个聚簇索引。如果表没有定义主键，InnoDB 会选择一个唯一非空索引代替，否则隐式生成一个 6 字节的 ROWID。

二级索引（Secondary Index）的叶子节点存储的是主键值而非行数据地址。这带来了几个关键概念：

概念 说明
回表（Bookmark Lookup） 通过二级索引找到主键后，再通过聚簇索引查找完整行数据。每次回表 ≈ 一次磁盘 I/O。
索引覆盖（Covering Index） 查询所需字段全部包含在二级索引中时，无需回表。Extra 列显示 Using index。
ICP（Index Condition Pushdown） MySQL 5.6+ 将 WHERE 条件中可以用索引列判断的部分下推到存储引擎层。减少回表次数。
MRR（Multi-Range Read） MySQL 5.6+ 将二级索引扫描得到的多个主键排序后再回表，将随机 I/O 转为顺序 I/O。
Index Merge 对同一条查询使用多个索引取交集/并集。Extra 显示 Using intersect/Using union。

深入思考：

  1. 为什么二级索引存主键值而不是行指针？ 避免数据移动时的索引更新。如果使用行指针，当 B+ 树页分裂或行迁移时，所有二级索引都需更新。主键值”逻辑地址”使得二级索引与物理存储解耦。
  2. 聚簇索引的隐含代价：插入主键不连续时（如 UUID），会产生”页分裂风暴”。二级索引的主键字段也会膨胀——二级索引叶子节点存的是主键，主键越大，每个二级索引占用空间越大。
  3. ICP 生效条件：仅适用于 range/ref/eq_ref/ref_or_null 的索引访问方式。Extra 列显示 Using index condition。ICP 对于复合索引的”中间列”过滤特别有效。

生产踩坑案例：

电商订单系统的索引覆盖优化：某电商订单表 1.2 亿行，查询 SELECT order_id, status, amount FROM orders WHERE status = 1 ORDER BY create_time DESC LIMIT 20，status 和 create_time 各有独立索引。MySQL 选择 status 索引 + 回表 + filesort，耗时 3.2s。优化方案：创建复合索引 idx_status_create_time (status, create_time, order_id, amount) 实现索引覆盖，Extra 变为 Using where; Using index，耗时降至 8ms。

0x00-03 Change Buffer / Adaptive Hash Index / Doublewrite Buffer

标准答案：

Change Buffer（变更缓冲）：
InnoDB 用于缓冲对二级索引页的变更操作（INSERT/UPDATE/DELETE）。当目标页不在 Buffer Pool 中时，不立即从磁盘读取该页，而是将变更记录在 Change Buffer 中。后续当该页被读入 Buffer Pool 时，再将缓冲的变更合并（Merge）。Change Buffer 只针对唯一性索引以外的二级索引。

Adaptive Hash Index（自适应哈希索引，AHI）：
InnoDB 根据访问模式自动为 B+ 树索引的热点页面建立哈希索引。当检测到某个索引页被频繁通过等值条件访问时，会在内存中创建哈希索引。仅对等值查询加速，范围查询仍走 B+ 树。完全自管理、不可干预。

Doublewrite Buffer（双写缓冲）：
InnoDB 的数据页完整性保护机制。在写入数据文件前，先将 1MB 的数据（128 个页，每个 16KB）连续写入双写缓冲区的共享表空间，再写入实际数据文件。解决”部分页写入（Partial Page Write）”问题——数据库因断电/崩溃导致一个页只写了一半的损坏。

深入思考：

  1. Change Buffer 的配置陷阱：参数 innodb_change_buffer_max_size（默认 25%）控制 Change Buffer 占用 Buffer Pool 的最大比例。在写密集且二级索引多的场景下可以调大（如 50%），但会挤压 Buffer Pool 的缓存空间，导致读命中率下降。在 SSD 环境下，Change Buffer 的收益下降——因为 SSD 随机读性能远高于 HDD。
  2. AHI 的 Hash 冲突问题：AHI 使用部分主键的哈希值，在高并发对同一锁争用热行时，AHI 的锁竞争反而可能成为瓶颈。在极端场景（如秒杀扣减库存），建议关闭 AHI（innodb_adaptive_hash_index=OFF）。

AHI 运维监控：

AHI 适用场景判断：

  • 适用：查找极度频繁、写入很少的只读场景（如配置中心、字典表），AHI 能带来明显的性能提升。
  • 不适用：高并发写入场景，AHI 的维护开销大于收益。阿里云 RDS 默认关闭 AHI，替换为自研的 PolarDB Buffer 优化。
  • SSD 时代收益递减：SSD 普及后，B+Tree 3 次随机 IO ≈ 0.03ms，AHI 的 O(1) 查找相比 B+Tree 的 O(log n) 收益大幅缩小，而 AHI 的内存开销和写入维护开销不会消失。
  1. Doublewrite Buffer 与文件系统：如果底层文件系统支持原子写（如 ZFS、使用 Fusion-io 直写），可以关闭 Doublewrite Buffer 以节省 2x 写放大。但在常规的 ext4/xfs 上绝对不可以关闭。

生产踩坑案例：

微博推送系统的 Change Buffer 耗尽问题：某社交平台的”关注关系”表有 5 个二级索引，每秒 2 万 INSERT，Change Buffer 设置为 25%。因二级索引页频繁不在 Buffer Pool 中，Change Buffer 快速膨胀至上限，触发批量 Merge。Merge 时大量随机 I/O 导致磁盘利用率 100%，写入延迟从 1ms 飙升至 500ms。最终调整 innodb_change_buffer_max_size=10%，并将在内存热点高的二级索引改为唯一索引（跳出 Change Buffer），问题解决。

0x00-04 行格式（Compact/Redundant/Dynamic/Compressed）

标准答案：

InnoDB 支持四种行格式，分属两个文件格式世代：

行格式 文件格式 特点 适用场景
Redundant Antelope 最早的格式，数据字典兼容 仅 MySQL 5.0 以下兼容需求
Compact Antelope Redundant 的改进版，约减少 20% 存储 MySQL 5.5/5.6 默认格式
Dynamic Barracuda LONG 列完全存储在溢出页，不浪费主数据页空间 MySQL 5.7+ 默认格式，推荐
Compressed Barracuda Dynamic 基础上增加页级别压缩（zlib） 存储成本敏感的只读/低频场景

关键区别：

  • Compact/Redundant 对于 VARCHAR/VARBINARY/BLOB/TEXT 等长字段，在页内存放 768 字节的前缀 + 20 字节指针指向溢出页。这导致即使只存小数据也浪费空间。
  • Dynamic 不在主数据页预分配 768 字节，仅当整行能放入页内时才全量存储，否则整列放入溢出页。大幅提高 B+ 树叶子节点可容纳行数。

深入思考：

  1. Compressed 的代价：页压缩（Page Compression）会在每次页面修改时进行压缩/解压缩，显著增加 CPU 消耗。在 OLTP 高并发下不推荐。但在分析型查询和数据归档场景，压缩比可达 3~5x，显著减少存储成本和 I/O。
  2. MySQL 8.0 的透明页压缩：MySQL 8.0 引入基于文件系统的透明页压缩（Transparent Page Compression），在上层实现，数据写入前通过 Punch Hole 释放空闲空间。与 InnoDB 的 Compressed 行格式无关，且兼容 Dynamic 格式。推荐用透明页压缩代替 Compressed 行格式。
  3. 巨型 TEXT/BLOB 字段：即使使用 Dynamic 格式，频繁访问包含巨量 TEXT/BLOB 的表依然会导致严重的 Buffer Pool 污染和性能抖动。建议将大字段独立成关联表—这是一种经典的”宽表拆窄表”优化。

0x00-05 索引统计信息收集机制与性能影响

标准答案：

MySQL 优化器基于统计信息估算索引的区分度（Cardinality）来选择执行计划。统计信息记录在 mysql.innodb_index_stats 和 mysql.innodb_table_stats 表中。

收集方式：

  1. 自动采样：当 innodb_stats_auto_recalc=ON（默认），DDL 或 DML 修改的行数超过表行数 10% 后，自动触发 ANALYZE TABLE 重算统计信息。
  2. 手动采集：ANALYZE TABLE table_name 强制执行。大表执行较慢（全表扫描采样）。
  3. 随机采样：innodb_stats_persistent_sample_pages 控制采样页数（默认 20），值越大统计越精确，但 ANALYZE 越慢。
  4. 持久化统计信息：MySQL 5.6+ 默认 innodb_stats_persistent=ON，统计信息写入系统表并持久化到磁盘，MySQL 重启后不丢失。

统计信息不准确怎么办？

深入思考：

  1. Cardinality 的算法：InnoDB 使用随机 dive（随机读取若干索引页统计不同键值数），不是精确计数。如果表的某个二级索引的数据分布极不均匀（如 status 列 99% 行=0），随机采样可能大幅低估或高估。
  2. 统计信息过时导致的问题：某表 1 亿行，status 列有索引，其中 99,999,999 行 status=0，1 行 status=1。查询 WHERE status=1 时优化器认为 status 区分度高，走索引回表——实际快速返回。但如果统计信息采样恰恰没采到那 1 行，优化器认为 status=1 也很多（估算返回 1000 万行），选择全表扫描——灾难。
  3. 直方图（Histogram）：MySQL 8.0 引入 ANALYZE TABLE ... UPDATE HISTOGRAM ON column_name WITH N BUCKETS。直方图能精确描述列的数据分布，而不只是简单的区分度。对于分布极不均匀的列（status、city），直方图帮助优化器做出正确选择。
  4. 🔥 生产踩坑：某 C2C 订单系统，大表 orders 按 create_time 分区。一夜过后所有查询都走全表扫描。排查发现是因为 innodb_stats_auto_recalc=ON + 大批量数据迁移（删除大量历史数据），MySQL 自动重算了统计信息——采样时采样到了空页，导致所有索引的 cardinality 估值骤降。修复：重建统计信息 ANALYZE TABLE orders 后恢复正常，后续改为固定时段手动 ANALYZE。

0x01 MVCC 与事务

0x01-01 MVCC 实现原理（Undo Log + Read View + 可见性算法）

标准答案：

InnoDB 的 MVCC（Multi-Version Concurrency Control）是一种乐观并发控制机制，允许读写互相不阻塞。核心组成：

1. Undo Log（回滚日志）版本链

  • 聚簇索引行记录包含三个隐藏字段：DB_TRX_ID（最近修改的事务 ID）、DB_ROLL_PTR（回滚指针，指向 Undo Log 中的上一个版本）、DB_ROW_ID（隐式行 ID）。
  • 每次 UPDATE/DELETE 时，InnoDB 将被修改前的行快照写入 Undo Log，形成一条版本链，按事务修改顺序链接。
  • INSERT 也会生成 Undo Log 但仅在事务回滚时需要，事务提交后可立即清理。

2. Read View（读视图）
Read View 是在事务执行一致性读时创建的快照，包含：

  • m_ids：当前所有活跃事务的事务 ID 列表（未提交的写事务）。
  • min_trx_id：m_ids 中的最小值。
  • max_trx_id：全局下一个将要分配的事务 ID。
  • creator_trx_id：创建该 Read View 的事务 ID。

0x01-02 事务隔离级别（RR/RC 下 MVCC 差异 + 幻读解决 vs 未解决）

标准答案：

MySQL InnoDB 支持 SQL 标准定义的四个隔离级别，默认为 REPEATABLE READ（RR）：

隔离级别 脏读 不可重复读 幻读
READ UNCOMMITTED 可能 可能 可能
READ COMMITTED（RC） - 可能 可能
REPEATABLE READ（RR） - - InnoDB 下解决
SERIALIZABLE - - -

RR 下 MVCC 的差异：

  • Read View 在第一个一致性读时创建并整个事务期间复用。
  • 同一事务中重复执行 SELECT * FROM t WHERE id > 100，由于 Read View 不变，始终看到事务开始时的数据快照——因此 RR 下普通的快照读不存在不可重复读和幻读。

幻读的解决 vs 未解决：

  • 快照读（Snapshot Read）：RR 级别下普通 SELECT 不存在幻读（一直在事务开始的快照中），这是通过 MVCC Read View 解决的。
  • 当前读（Current Read）：SELECT ... FOR UPDATE/UPDATE/DELETE 等在 RR 级别下通过 Next-Key Lock（临键锁） 解决幻读。如果查询条件没有唯一索引，Next-Key Lock 会锁定扫描范围的所有记录和间隙，阻止其他事务插入新行。

因此严格说，MySQL RR 级别通过 MVCC + 间隙锁完全解决了幻读。这是 MySQL 对 SQL 标准的扩展。

深入思考：

  1. 为什么 RC 在 MySQL 中也有广泛实践？ RC 级别下 Read View 每个语句重新创建，锁竞争更少，是 MySQL 的最佳实践级别（不会出现 RR 下间隙锁引起的死锁扩大化）。许多互联网公司（美团、头条等）线上使用 RC 级别配合 Binlog Row 模式。
  2. Gap Lock 与主从一致性问题：RC + Row-Based 模式 下 Gap Lock 被禁用（唯一索引和外键约束除外），不会有 RR 下的间隙锁死锁问题，但 Binlog 需要设置为 ROW 格式以保证一致性。Statement 模式在 RC 下会产生主从不一致。

0x01-03 MVCC 与锁的配合

标准答案：

MVCC 和锁在 InnoDB 中是并行不悖的两个维度，共同支撑事务的 ACID：

功能 机制
一致性读（快照读） MVCC + Undo Log，完全不使用锁
锁定读/修改 行锁 + 间隙锁 + 临键锁
写写冲突 行锁互斥
读写冲突 MVCC 让读不阻塞写，写不阻塞读

配合流程：

  1. 事务 A 执行 UPDATE：获取行上的 X 锁，同时将旧版本写入 Undo Log，更新 DB_TRX_ID。
  2. 事务 B 执行 SELECT（快照读）：MVCC 通过 Read View 判断版本可见性，自动读取 Undo Log 中的旧版本。不加锁，不等待。
  3. 事务 B 执行 SELECT ... FOR UPDATE（当前读）：需要获取行上的 X 锁，如果事务 A 未提交则等待，直到超时或提交。
  4. 在 RR 级别下，SELECT ... FOR UPDATE 不仅加行锁，还加间隙锁防止幻读。

深入思考：

  1. SERIALIZABLE 本质上退化为纯锁：在该级别下，普通 SELECT 也会被升级为 LOCK IN SHARE MODE，MVCC 的快照读优势完全丧失。
  2. 插入意向锁与 MVCC 的交互：插入操作（INSERT）需要插入意向锁，与已有间隙锁不冲突，但与已存在的排他 Next-Key Lock 冲突。MVCC 会帮助 INSERT 前的快照读避免阻塞，但写入行为还是必须等待锁释放。

0x01-04 长事务问题（undo 膨胀/死锁/主从延迟）

标准答案：

长事务是 MySQL 生产环境最常见的性能杀手之一，主要带来三大问题：

1. Undo 膨胀（Undo Tablespace Explosion）

  • 长事务持续运行时，Undo Log 中的旧版本无法被 purge 线程清理（因为有事务还需要看到这些旧版本）。
  • Undo tablespace 持续增长，最终可能导致磁盘爆满。
  • 查看长事务：SELECT * FROM information_schema.INNODB_TRX WHERE TIME_TO_SEC(TIMEDIFF(NOW(), trx_started)) > 60;
  • 定位活跃事务的 Undo 页数：SELECT * FROM performance_schema.data_lock_waits;

2. 死锁概率升高

  • 长事务持有大量锁资源不释放，与其他事务的锁等待构成环状依赖的概率增大。
  • MySQL 会自动检测死锁并使用事务回滚代价最小的策略——回滚锁资源最少的事务。

3. 主从延迟加剧

  • 长事务对应的 Binlog 事件在提交后才写入 Binlog，从库需要拉取并重放。
  • 大事务（如 DELETE FROM big_table）产生巨量 Binlog，从库单线程回放造成长时间延迟。
  • 缓解：binlog_group_commit_sync_delay、并行复制（slave_parallel_workers）。

深入思考：

  1. undo 收缩的最佳实践：MySQL 8.0 支持 ALTER UNDO TABLESPACE tablespace_name SET INACTIVE; 后 truncate 回缩。不要依赖 MySQL 5.7 及之前的自动收缩（经常失败）。
  2. 应用层预防：ORM层（如 MyBatis/MyBatis-Plus、Hibernate）的 @Transactional 若不指定 timeout 会默认无限等待。生产环境强制设置 @Transactional(timeout=30) 可在应用层防止长事务。

生产踩坑案例：

金融账户系统的 Undo 撑爆磁盘：某银行系统因报表导出时开启了一个 2 小时长事务，期间对账户表执行了大量 UPDATE。Undo Tablespace 在 2 小时内从 1GB 膨胀至 600GB，导致数据盘满，MySQL 实例进入”Read Only”模式。回溯发现是 Java 应用中一个 @Transactional 注解包裹了全量导出逻辑，未设置超时。修复：1) 添加 @Transactional(timeout=30)；2) 拆分导出为批处理；3) 使用 information_schema.INNODB_TRX 建立监控告警。

0x02 锁机制

0x02-01 行锁/间隙锁/临键锁/插入意向锁

标准答案：

InnoDB 的锁机制远比表面复杂。按照锁定的粒度维度：

按类型：

锁类型 说明 SQL 示例
Record Lock（行锁） 锁定索引记录本身 SELECT * FROM t WHERE id=1 FOR UPDATE
Gap Lock（间隙锁） 锁定记录之间的间隙，阻止插入 SELECT * FROM t WHERE id>10 AND id<20 FOR UPDATE
Next-Key Lock（临键锁） Record Lock + Gap Lock 的组合，左开右闭区间 InnoDB RR 级别的默认锁策略
Insert Intention Lock（插入意向锁） INSERT 操作的轻量级”排队令牌” 自动在 INSERT 时获取
Intention Lock（意向锁） 表级别的信号机制：IS/IX 表示事务将要执行行级读/写锁 自动管理

Next-Key Lock 的核心规律：

InnoDB RR 级别下，对于当前读（SELECT…FOR UPDATE/UPDATE/DELETE）：

  • 唯一索引等值查询且记录存在：退化为 Record Lock
  • 唯一索引等值查询且记录不存在：退化为 Gap Lock
  • 非唯一索引等值查询：Next-Key Lock（多重间隙）
  • 范围查询：整个扫描范围加 Next-Key Lock

深入思考：

  1. 间隙锁仅在 RR 级别启用，RC 级别下间隙锁会被禁用（唯一索引和外键约束除外）。这就是为什么很多互联网公司选择 RC 级别——减少锁冲突和死锁。
  2. 插入意向锁（Insert Intention Lock）不是锁等待的终点。多个事务可以在同一个 Gap 上持有插入意向锁，只要它们插入的位置不存在冲突。只有当其他事务持有该 Gap 的 Gap Lock 或 Next-Key Lock 时，插入意向锁才会等待。这是一种轻量级的”排队”机制。
  3. 加锁原理的 SQL 实验：通过 SELECT * FROM performance_schema.data_locks;（MySQL 8.0）可直观查看当前所有锁信息，包括 LOCK_TYPE、LOCK_MODE、LOCK_DATA 等字段。强烈建议面试时提到这点以展示你的实战深度。

0x02-03 DDL Online（Online DDL/gh-ost/PT-OSC）

标准答案：

在生产环境中对大表执行 DDL（ALTER TABLE）的三种方案：

1. MySQL 8.0 Online DDL（原生）

  • INSTANT：仅修改元数据，不涉及数据复制（如 ALTER TABLE ... ALTER SET DEFAULT）。MySQL 8.0.12+ 支持 ADD COLUMN 的 INSTANT 模式。
  • INPLACE：不复制表数据，在引擎层直接重建索引或修改。允许并发 DML。例如 CREATE INDEX、ADD COLUMN（非 INSTANT 时）。
  • COPY TABLE：创建临时表并复制数据，阻塞 DML。几乎所有 DDL 在 5.6 之前都走此路径。

ALTER TABLE t ALGORITHM=INPLACE, LOCK=NONE, ADD INDEX idx_name(col);

2. gh-ost（GitHub Online Schema Migration）

  • GitHub 出品，免触发器。通过 Binlog 同步的方式追踪变更，读取主库/从库的 Binlog 流来捕获 DML 变更。
  • 创建影子表 → 逐片拷贝数据 → Binlog 追踪增量写入 → 切换表名（原子性 RENAME）。
  • 优点：对主库负载极低（读 Binlog 时从库可选）、暂停/恢复灵活。
  • 缺点：需要 Binlog Row 格式、需要额外存储空间。

3. pt-online-schema-change（Percona Toolkit，PT-OSC）

  • Percona 出品，基于触发器。在原表上创建三个触发器（INSERT/UPDATE/DELETE）捕获增量变更。
  • 创建影子表 → 拷贝数据 + 触发器同步增量 → 原子切换。
  • 优点：不依赖 Binlog Row 模式。
  • 缺点：触发器增加主库负载、不支持有触发器的主从场景、大表触发器影响写入性能。

深入思考：

  1. 为什么需要 Online DDL？ 5 亿行表直接 ALTER TABLE 会产生 MDL 写锁，阻塞所有查询。在千万级 QPS 的业务上，一个 SHOW TABLES 被阻塞都可能引发边缘雪崩。
  2. MySQL 8.0 原生 DDL 的问题：PRIMARY KEY 变更仍不可用 INPLACE。MODIFY COLUMN 更改数据类型在 8.0 中仍为 COPY 模式。Online DDL 在执行过程中若遇到 MDL 等待超时（lock_wait_timeout），会回滚整个操作。

生产踩坑案例：

阿里某 5 亿行大表 ALTER TABLE 完崩 15 分钟：某电商订单中心使用 MySQL 5.7，对 5 亿行大表执行 ALTER TABLE order ADD INDEX idx_create_time(create_time)。未使用 Online DDL，直接执行导致 MDL 写锁获取后阻塞所有读写查询 150秒，30 个应用节点连接池爆满，服务不可用 15 分钟。复盘：使用 gh-ost 增加索引只需极短暂的切换锁（毫秒级）。

# 场景 示例 原因
1 列上使用函数 WHERE DATE(create_time) = '2024-01-01' 索引存的是原始值，计算后不匹配
2 隐式类型转换 WHERE phone = 13800138000（phone 为 VARCHAR） 字符串转数字比较，索引失效

  3. OR 的替代方案：使用 UNION ALL 分别走两个索引，或使用 IN 替代。MySQL 5.0+ 的 Index Merge 可以在某些 OR 场景下利用多个索引取并集，但性能通常不如单索引。

标准答案：

大分页优化

LIMIT 1000000, 20 极其低效，因为 MySQL 需要扫描并丢弃前 100 万行。

方案 1：游标分页（Cursor-based Pagination / Seek Pagination）

要求：排序字段唯一且有索引。适用于前后翻页（非跳页）的场景。

方案 2：子查询延迟关联

JOIN 优化

基本原则：

  1. 小表驱动大表：MySQL 的 Nested-Loop Join 会对外表扫描的每一行去探测内表。将小的结果集作为驱动表，减少循环次数。
  2. Join 字段必须有索引：内表的 Join 字段必须加索引，否则内表每次循环都做全表扫描。
  3. 避免超过 3 表 Join：多表 Join 的优化空间极其有限，超过 3 表复杂的关联通常需要改写 SQL 或在应用层完成。

Block Nested-Loop Join（BNL） vs Hash Join：

  • MySQL 5.7 及之前：JOIN 无索引时使用 BNL，将驱动表的行缓冲在 Join Buffer 中批量探测内表。
  • MySQL 8.0.18+：引入 Hash Join，对等值 JOIN 在无索引时自动使用 Hash Join 代替 BNL。性能提升 2-10 倍。

ORDER BY 优化

  1. 使用索引消除 filesort：ORDER BY create_time 且 create_time 上有一个作为前缀的索引，MySQL 可以直接利用 B+ 树已经排好序的特性。
  2. 混合排序方向需一致：索引列默认 ASC，ORDER BY a ASC, b ASC 可走索引排序；ORDER BY a ASC, b DESC 需要 filesort（MySQL 8.0 引入了”降序索引”解决此问题）。
  3. 排序缓冲区调优：sort_buffer_size 如果太小，排序需要使用磁盘临时文件（SHOW SESSION STATUS LIKE 'Sort_merge_passes' 看是否有 disk sort）。

深入思考：

  1. 游标分页的局限：不支持随机跳页（用户说”跳到第 50 页”就无法用游标了）。且如果 WHERE 条件中的过滤字段变化（状态筛选），需要复合索引支持才能继续使用游标方式。
  2. Hash Join vs Index Nested-Loop Join：Hash Join 对于大表 JOIN 无索引时的性能远好于 BNL，属于”分批读入、内存建哈希表、批量探测”的思路。但在索引完备的情况下，Index Nested-Loop Join 仍是更快的选择（每次只有点查）。

生产踩坑案例：

字节跳动某信息流系统的 ORDER BY LIMIT 大翻页：埋点表 10 亿行，前端支持跳页到 1000 页（每页 50 条）。ORDER BY create_time LIMIT 49500, 50 耗时 12s。优化：后端限制最大分页深度（max_page_depth=100），超过 100 页不再提供跳页，改用游标 + 模糊搜索。对极少数的强需求用户，提供导出功能离线处理。

0x04-00 Redo Log 和 Binlog 的区别与协作机制（两阶段提交）

标准答案：

维度 Redo Log Binlog
层次 InnoDB 引擎层 MySQL Server 层
作用 crash-safe 保证事务持久性 主从复制、数据恢复
记录内容 物理日志：页级别的修改（页号+偏移+新数据） 逻辑日志：SQL 语句或行变更
写入时机 事务执行中不断写入（顺序写 IB_logfile） 事务提交时写入
清理策略 循环写入，checkpoint 覆盖旧日志 追加写入，全量保留
格式 二进制（redo record） STATEMENT / ROW / MIXED

两阶段提交（2PC）：保证 redo log 和 binlog 的一致性。

  • Prepare：Redo Log 写入并标记为 prepare 状态，此时事务未提交但在 crash recovery 中可恢复。
  • 写入 Binlog：写入成功后，保证主从可同步。
  • Commit：Redo Log 标记为 commit 状态，事务正式提交。

崩溃恢复逻辑：

  • Scan redo log 找到所有 prepare 但未 commit 的事务
  • 检查对应 binlog 是否写入：binlog 已写 → 提交；binlog 未写 → 回滚
  • 这样就保证了 redo log 和 binlog 的最终一致

深入思考：

  1. 为什么 Binlog Position 不一致会导致数据丢失？ 如果先写 binlog 后写 redo log，写完 binlog 后崩溃——主库恢复后无此事务（redolog 未提交），但从库已经收到 binlog。主从不一致。先写 redo log 后写 binlog，写完 redo log 后崩溃——主库恢复后有事务（redolog prepare 接受），但 binlog 没有导致从库丢失数据。2PC 解决了这个问题：只有在 binlog 写入后才提交 redo log。
  2. 组提交（Group Commit）：MySQL 5.6+ 多个事务的 prepare 阶段可以合并为一次 fsync，显著提升写入吞吐。binlog_group_commit_sync_delay（等待延迟）和 binlog_group_commit_sync_no_delay_count（等待数量）可以调优组提交。
  3. Redo Log 大小配置：innodb_log_file_size 和 innodb_log_files_in_group 共同决定 redo log 总大小。太小导致频繁 checkpoint（写入脏页到磁盘，IO 尖刺），太大导致崩溃恢复时间长。MySQL 8.0.30+ 引入 innodb_redo_log_capacity（默认 100MB），动态调整无需重启。
  4. 🔥 生产踩坑：某金融系统 innodb_log_file_size=48M（默认值），TPS 高峰时每秒产生 200MB redo log，磁盘写入带宽打满（FIO 显示 99% 写延时 > 100ms）。调大到 4GB（innodb_log_file_size=4G）后 checkpoint 频率从每秒数次降到几分钟一次，TPS 提升 3 倍。

0x04-01 异步/半同步/组复制

标准答案：

MySQL 复制按同步程度分为三个层次：

1. 异步复制（Async Replication）

  • MySQL 默认模式。主库将 Binlog 写入本地文件后立即返回客户端成功，不等待从库确认。
  • 优势：对主库性能影响最小。
  • 风险：主库崩溃时，已提交的数据可能尚未同步到从库，数据丢失窗口存在。

2. 半同步复制（Semi-Sync Replication）

  • MySQL 5.5+（通过插件实现），MySQL 5.7+ 原生支持。主库提交事务时需要等待至少一个从库确认收到并写入 Relay Log。
  • 当半同步超时（rpl_semi_sync_master_timeout）则降级为异步复制。
  • 优势：减少数据丢失风险（RPO 降低至一个从库确认时间）。
  • 代价：主库写入延迟 ≈ 网络 RTT 到从库。跨机房部署时延迟明显。

3. 组复制（Group Replication / MGR）

  • MySQL 5.7.17+ 引入。基于 Paxos 分布式共识协议的强一致性复制。
  • Multi-Primary 模式：所有节点可写。
  • Single-Primary 模式：只有一个主节点写入，其他为从。
  • 优势：强一致性、自动化成员管理、故障自动切换。
  • 代价：写入延迟至少需要多数派（N/2+1）确认，网络延迟敏感；只支持 InnoDB；复杂 DDL 限制多；高频大事务场景下性能不及异步复制。

深入思考：

  1. 半同步的 AFTER_SYNC vs AFTER_COMMIT（MySQL 5.7+）：AFTER_SYNC 模式在 Binlog 写入磁盘后即等待从库确认，确认后再提交。这种模式下如果主库在确认后崩溃，数据已写入 Binlog，不会丢失。推荐使用。
  2. MGR 的适用场景：MGR 不适合跨城市部署（50ms+ 延迟下性能急剧下降），也不适合大量并发写入的 Pay-as-you-grow 场景。MGR 的踢出机制（节点慢于阈值时被驱逐）容易导致集群抖动。

0x04-02 GTID 原理与故障恢复

标准答案：

**GTID（Global Transaction Identifier）**是 MySQL 5.6.5+ 引入的全局事务标识符。

格式： UUID:sequence_number

  • UUID：服务器的全局唯一标识（server_uuid）。
  • sequence_number：事务在该服务器上分配的递增序号。

每个在服务器上提交的事务都有一个唯一的 GTID，整个复制拓扑中全局唯一。

GTID 的核心原理：

  1. 主库为每个事务生成 GTID，并写入 Binlog。
  2. 从库重放事务时记录 Executed_Gtid_Set，不会重放已执行的事务（天然的自动去重）。
  3. 主从切换时，新主库收集所有从库的 Executed_Gtid_Set，找到差额（Gap）事务，从老主库拉取缺失的事务。

故障恢复步骤（GTID 模式）：

深入思考：

  1. GTID 与 Binlog 格式：GTID 模式下必须使用 ROW 格式的 Binlog（binlog_format=ROW）。原因：GTID 的原子性依赖 Binlog 格式，statement 模式下的非确定 SQL 会导致主从数据不一致，打破 GTID 的 “同一个事务全局唯一执行一次” 的语义。
  2. GTID 跳过一个事务：在 GTID 模式下 SET GLOBAL sql_slave_skip_counter 不可用。需通过注入空事务跳过：SET GTID_NEXT='uuid:N'; BEGIN; COMMIT; SET GTID_NEXT='AUTOMATIC';。

0x04-03 主从延迟分析与缓解

标准答案：

衡量主从延迟的核心指标：Seconds_Behind_Master（SHOW SLAVE STATUS 输出）。

主从延迟的六大根源：

原因 说明 解决方案
单线程 SQL 线程瓶颈 从库 SQL_Thread 单线程重放 Binlog 跟不上主库 开启并行复制
大事务 单条 DELETE/UPDATE 影响百万行 拆分为小批次事务

并行复制（MySQL 5.7/8.0）：

WriteSet 并行复制（MySQL 8.0.1+ 最强方案）：
MySQL 8.0 在 binlog_transaction_dependency_tracking 参数中引入 WriteSet 机制。通过提取事务修改的行的行标识（WriteSet），如果两个事务修改不同的行，则在从库可并行回放，不再受数据库级别限制。写入密集场景下并行度提升 10 倍以上。

深入思考：

  1. 大事务拆解实战：DELETE FROM t WHERE create_time < '2023-01-01' 影响 1000 万行。拆为 DELETE FROM t WHERE id BETWEEN 1 AND 10000 AND create_time < '2023-01-01' LIMIT 1000 循环执行，每 1000 行 COMMIT 一次。既避免了主库 Binlog 暴增，也避免从库因大事务重放而显著延迟。
  2. 延迟监控的缺陷：Seconds_Behind_Master 是基于时间戳的推算，在以下场景中不准确：网络抖动时、MTS（多线程并行复制）下、慢查询导致的伪延迟。推荐配合 SHOW SLAVE STATUS 中的 Retrieved_Gtid_Set vs Executed_Gtid_Set 的差值做精确判断。

0x04-05 MySQL 8.0 的 Binlog 压缩与 Doublewrite 优化

标准答案：

Binlog 压缩（MySQL 8.0.20+）：

  • 支持 binlog_transaction_compression=ON，对事务级的 binlog 内容进行 zstd 压缩。
  • 压缩比通常 3:1-5:1，写入量大幅减少，对 IO 瓶颈的从库提升明显。
  • 可配合 binlog_row_image=MINIMAL（只记录被修改的列）进一步压缩。
  • 主从之间：当主库启用压缩时，从库可以自动解压，无需额外配置。

Doublewrite Buffer 优化：

  • Doublewrite 解决部分写失效（Partial Write）——MySQL 页 16KB，磁盘扇区 512 字节/4KB，写一半时断电导致页损坏。
  • MySQL 8.0.20+ 引入 innodb_use_native_aio 和新的 doublewrite 实现，支持 innodb_doublewrite_pages 和 innodb_doublewrite_batch_size 调优。
  • MySQL 8.0.31+ 支持 per-table doublewrite（innodb_doublewrite_dir），将 doublewrite 文件放到不同磁盘（如 NVMe），减少写入竞争。
  • MySQL 9.0 新增 Doublewrite Buffer 加密支持，对静态数据加密场景不再妥协。

深入思考：

  1. Binlog 压缩的收益评估：事务密集型系统通常 binlog 写入是最大 IO 压力源。某支付系统启用 binlog 压缩后，binlog 写入带宽减少 70%，从库延迟从 8s 降到 1s。
  2. Doublewrite 是否可以关闭？ 可以，innodb_doublewrite=OFF。但不建议，除非使用 ZFS/ZFS Datasets 或 Fusion-IO 闪存卡等保证原子写。关闭 doublewrite 可以提升写入性能约 15-20%，但承受页损坏风险。
  3. Doublewrite 与 Redo Log 的关系：Redo Log 记录的是逻辑修改（页的变更），如果页本身损坏，redo log 写入的是损坏页——这是 doublewrite 解决的问题。Doublewrite 保证写入页的原子性，Redo Log 保证事务的持久性。

0x05-01 水平分片策略（范围/哈希/时间/一致性哈希）

标准答案：

水平分片（Horizontal Sharding）是将同一张表的数据按一定规则分布到多个数据库实例中。

四种核心分片策略：

策略 描述 优点 缺点 适用场景
范围分片 按 ID 范围分片（1-1 千万在 shard0，1 千万-2 千万在 shard1…） 扩展简单、范围查询天然支持、分片键可读 数据倾斜（尾部热点）、扩容需迁移 订单表按时间分片
哈希分片 shard = hash(shard_key) % N 数据均匀分布、避免热点 扩容需重新计算（”模 N”变成”模 2N”），范围查询跨多分片 用户表、高并发写入
时间分片 按年/月/日分片 符合业务冷热分离，冷数据可压缩归档 非时间维度查询需广播 日志表、埋点数据
一致性哈希 shard = binary_search(hash(shard_key), ring) 扩容仅影响相邻节点（K/N 的键需迁移），平滑扩缩容 实现复杂、虚拟节点管理开销 动态扩缩容场景

深入思考：

  1. Range + Hash 混合策略：大厂典型实践是两层分片。第一层按时间范围（月/季度）分区，第二层在时间分区内按用户 ID 哈希。兼顾时间维度的查询剪枝和数据均匀分布。
  2. 一致性哈希的”虚拟节点”：为了避免节点分布不均导致的倾斜，一致性哈希会在环上为每个物理节点分配 150~200 个虚拟节点（虚拟节点越多，负载越均匀）。实际工程中一致性哈希的存储和管理开销较高，时序场景下不如 range 分片直观。
  3. 分片的度数（Fanout）：分片数不是越大越好。每个分片有自己的连接池、内存、后台线程。经验值：每个 MySQL 实例 8~16 个分片。分片过多会增大连接数开销和跨分片协调复杂度。

0x05-02 跨分片查询/全局表/广播表/ER 分片

标准答案：

跨分片查询的三大难题：

  1. 跨分片 Join：分属不同分片的数据无法在单库层面 Join，需要在中间件层做”合并+关联”。
  2. 跨分片聚合：COUNT/SUM/AVG 需要在每个分片执行后汇总（AVG 必须拆解为 SUM/COUNT 两步）。
  3. 跨分片排序分页：ORDER BY create_time LIMIT 0, 10 需要在每个分片取前 10 条，在中间件层合并排序后取前 10 条——随着页码增大，每个分片拉取的量线性增长。

三种表策略：

策略 描述 示例 优点 缺点
全局表 不拆分，每张表在每个分片有完整副本 配置表、字典表 Join 可直接在分片本地完成 维护一致性、更新需广播
广播表 类似全局表，但由中间件自动管理 小数据量的基础数据表 对应用透明 数据一致性确保复杂
ER 分片 将有关联关系的表按同一分片键分片 订单表和订单明细表按 order_id 分片 分片内 Join 本地执行，避免跨分片 设计需提前规划分片键

深入思考：

0x06-01 TiDB（Raft + LSM-tree + HTAP）

标准答案：

TiDB 是由 PingCAP 开发的开源分布式 NewSQL 数据库，兼容 MySQL 协议。

核心架构：

TiDB 的关键技术：

  1. Raft 一致性：TiKV 使用 Raft 共识协议进行数据复制。每个 Region（默认 96MB 数据范围）是一个 Raft Group，多副本（默认 3 副本）。Raft Leader 负责写入，Followers 只读。
  2. LSM-Tree（RocksDB）：TiKV 使用 RocksDB（Facebook 优化的 LSM-Tree 引擎）作为底层存储。写入先在 MemTable 中缓冲，定期刷到 SSTable。读路径需逐层查找 Bloom Filter 过滤不存在的键。
  3. HTAP 能力：TiFlash 以 Raft Learner 角色的方式从 TiKV 异步复制数据，并转换为列存格式。分析型查询自动路由到 TiFlash，事务型查询走 TiKV。同一份数据，两份引擎，用户无感。
  4. 自动分片（Region Auto-Split）：Region 达到 144MB（默认 soft limit）时自动拆分。数据倾斜时会自动调度 Region 到负载低的 TiKV 节点。

TiDB 的 MySQL 兼容性：

  • 兼容 95%+ 的 MySQL 语法
  • 不支持：存储过程、自定义函数、触发器、外键（5.6 之前不支持）、全文索引

深入思考：

  1. TiDB 的写放大问题：由于 LSM-Tree 的本质特征，TiKV 的写放大系数在 520 倍之间（取决于 Level 数量和 Compaction 策略）。而 InnoDB 的写放大在 25 倍（Doublewrite Buffer + Page Flush）。这意味着 TiDB 对 SSD 寿命的消耗是 MySQL 的 2~4 倍，大企业部署时需考虑 TBW（Total Bytes Written）。
  2. TiDB 的 TSO（Timestamps Oracle）：PD 以单点方式全局分配递增时间戳，保证分布式 MVCC 的总序。如果 PD 挂掉（即使部署 3 节点 Raft 副本），新写入阻塞到 PD 选举完毕。但也导致 TiDB 的写入延迟依赖 PD 的网络 RTT。
  3. 大事务限制：TiDB 大事务限制为 100MB（单事务总变更量），超过则报错。这是 LSM-Tree 引擎的内在限制（超大事务会导致大量 PUT 操作在 Compaction 中产生严重延迟）。

0x07-02 MySQL OOM（内存溢出）排查与解决

标准答案：

MySQL OOM 的常见原因和排查方法：

主要原因（排名分先后）：

  1. Buffer Pool 过大：innodb_buffer_pool_size 设置超过物理内存 70%，加上其他组件后 OOM。
  2. 连接数过高：每个连接占用 thread_stack（默认 256KB）+ sort_buffer_size + join_buffer_size 等。max_connections=1000 + 每个连接 2MB = 2GB 仅连接内存。
  3. 临时表过大：tmp_table_size 和 max_heap_table_size 设置过大，复杂查询在内存中建立大临时表。
  4. Prepared Statement 内存泄漏：max_prepared_stmt_count 过高，大量 prepare 未 deallocate。
  5. 大事务长事务：活跃事务的 undo log 持续增长，innodb_max_undo_log_size 不足以限制。

排查工具：

事前预防：

  • CGroup/memory cgroup：通过 systemd/cgroup 限制 mysqld 内存上限（MemoryMax=8G）。
  • connection memory pool：performance_schema 开启后监控连接级内存。
  • 配置最佳实践：
    • innodb_buffer_pool_size ≤ 物理内存 60%（留 40% 给 OS、连接、排序、临时表）
    • max_connections ≤ 200（结合 ProxySQL 连接池）
    • tmp_table_size = max_heap_table_size = 128MB（避免大临时表吃内存）
    • innodb_max_undo_log_size = 4GB（限制 undo 日志大小）

OOM 后的恢复：

  • 如果 OOM kill 了 mysqld，log 中会看到 Out of memory: Killed process (mysqld) → 调优后重启。
  • 如果 mysqld 还在但 SWAP 已用满 → SHOW PROCESSLIST 找到吃内存的查询 → KILL。

深入思考：

  1. Buffer Pool 是”可回收”内存吗？ Buffer Pool 可以 LRU 淘汰，但 SQL 查询过的 page 会留在 buffer pool 中直到被逐出。这意味着 buffer pool 的”冷数据”可以被其他进程复用——但 MySQL 不主动释放给 OS。innodb_buffer_pool_size=512GB + 物理 768GB，当其他进程申请内存时，OS 可能会 OOM kill MySQL。
  2. Performance Schema 内存监控开销：开启 performance_schema 本身消耗约 500MB-2GB 内存（取决于 performance_schema_consumer_global_instrumentation 配置）。8.0.23+ 支持动态调整。
  3. TCMalloc vs glibc malloc：MySQL 8.0 使用 glibc malloc，在内存碎片严重场景下表现不好。TCMalloc（Google）可显著减少内存碎片。Percona Server 默认使用 jemalloc。
  4. 🔥 生产踩坑：某数据库服务器（64GB 内存），innodb_buffer_pool_size=48GB（75%），max_connections=500。运维凌晨执行了一个 SELECT * FROM logs WHERE DATE > subdate(NOW(), 7)（logs 表 8 亿行），结果把 48GB buffer pool 的 hot data 全部刷掉，并把 server 内存打满，触发 OOM Killer 杀掉 mysqld。

0x07-03 MySQL 数据库备份与恢复策略（Xtrabackup / mysqldump / Clone Plugin）

标准答案：

备份方案对比：

方案 备份类型 大小 速度 适用场景
mysqldump 逻辑备份（SQL） 大（3-5× 数据量） 慢 小库（<50GB），迁移，单表恢复
Xtrabackup 物理备份（InnoDB 文件） ≈ 数据量 快 生产大库，全量+增量
Clone Plugin（MySQL 8.0.17+） 物理克隆（远程 clone） 最小（压缩传输） 极快 同架构复制节点搭建

Xtrabackup 核心流程：

  1. xtrabackup --backup：拷贝 InnoDB 数据文件，同时追踪 redo log 的 LSN。
  2. xtrabackup --prepare：应用 redo log 到数据文件（前滚 + 回滚未提交事务），使备份处于一致状态。
  3. xbstream -x 解压备份文件。
  4. copy-back 或直接修改 datadir 还原。

Clone Plugin 流程：

自动完成：数据传输 → 数据文件复制 → redo apply → 实例启动。完全自动化，适合 MySQL 8.0.17+。

深入思考：

  1. 增量备份的原理：Xtrabackup 的增量备份基于 LSN（Log Sequence Number）。第一次全量备份记录 checkpoint LSN，增量备份只拷贝 LSN 之后修改过的 page。注意：增量备份 restore 时需要先恢复全量备份，再依次应用每个增量备份——每次应用 increment 都要 redo。
  2. mysqldump 的 –single-transaction：对 InnoDB 表用 --single-transaction 可以在不加锁的情况下获取一致性快照（利用 MVCC）。但对 MyISAM 表会自动加全局读锁。
  3. 恢复时间目标（RTO）：Xtrabackup 恢复 500GB 数据约需 2-4 小时（包括拷贝+prepare），Clone Plugin 约 1-2 小时。要 RTO < 30 分钟必须做 MySQL Group Replication 或多副本架构，故障时直接切换不用恢复。
  4. 🔥 生产踩坑：某金融公司用 mysqldump 做 2TB 数据库的全量备份，用时 11 小时，期间 --single-transaction 保证一致性，但生成的 SQL 文件约 600GB。恢复时执行 mysql < dump.sql 跑了 24 小时！原因是 INSERT 语句的单条大小太大（net_buffer_length 默认 1MB），加上 binlog 写入放大。改为 Xtrabackup 后备份 40 分钟，恢复 1 小时。

0x07-05 大表 DDL 踩坑（5 亿行表改字段）

标准答案：

真实案例：某 5 亿行订单表将 status 字段从 INT(11) 改为 VARCHAR(10)。

坑点一：直接 ALTER TABLE 引发的 MDL 写锁洪灾

  • 执行 ALTER TABLE orders MODIFY status VARCHAR(10);
  • MySQL 5.7 自动判断该操作无法 INPLACE（改变数据类型需要 COPY），进入 COPY TABLE 模式
  • COPY TABLE 需要 MDL 写锁，而 MDL 写锁需要等待所有之前的 MDL 读锁释放
  • 正在执行的长时间查询（如报表统计 SUM）持有 MDL 读锁不释放
  • ALTER 排队等待时，后续的所有查询（读 + 写）都在等待 MDL
  • 最终：SELECT 也全部阻塞，整个实例的请求排队堆积
  • 数据库不到 2 分钟就连接数满

坑点二：Ghost Table 占满磁盘

  • 使用 gh-ost 执行：先创建影子表，将原表的 5 亿行通过 INSERT…SELECT 边读取边写入
  • 影子表的大小与原表接近（5 亿行 × 行大小），额外需要约 50GB 存储空间
  • 如果未评估磁盘余量，影子表在拷贝过程中磁盘写满，gh-ost 进程直接崩溃
  • 影子表不会自动清理，需要手动 DROP TABLE _orders_gho

坑点三：Binlog 爆炸

  • 即使使用 gh-ost，Binlog 格式为 ROW 的情况下，每一行变更（INSERT + DELETE）都写入 Binlog
  • 5 亿行 INSERT + 5 亿行 DELETE = 10 亿条 Binlog 事件
  • 主从从库同步这 10 亿个 Binlog 事件需要数小时甚至一天
  • 从库延迟飙升、磁盘爆满（Binlog 没设置 expire_logs_days）

总结出的 DDL 最佳实践清单：

  1. 评估先行：INFORMATION_SCHEMA.TABLES 查表大小、行数、碎片率。
  2. 从库先行：先在从库上执行 DDL 验证兼容性和耗时，确认无问题后再在主库执行。
  3. 使用 Online 工具：5 亿行以上优先使用 gh-ost，其次 PT-OSC，最后 ALTER TABLE。
  4. 关注磁盘空间：确保有足够空间放影子表（原表 1.5 倍大小）。
  5. 设置超时保护：lock_wait_timeout = 5（MDL 等待超时），防止 DDL 阻塞一切查询。
  6. 流量窗口选择：业务低峰期（凌晨 2-5 点）执行。预留回滚时间。
  7. 监控关键指标：数据库 QPS/TPS、主从延迟、磁盘空间、锁等待。
  8. 分片分批：如果表本身有分片，可以逐个分片执行 DDL，降低风险。

深入思考：

  1. MySQL 8.0 INSTANT DDL：MySQL 8.0.12+ 对 ADD COLUMN 支持 INSTANT 算法（仅改元数据，毫秒级完成），但 MODIFY COLUMN 改变数据类型仍然需要 COPY 模式。对 5 亿行表，添加新列几乎是瞬时的，但修改已有列还是坑。
  2. 夜间 DDL 的”隐蔽风险”：即使凌晨执行，凌晨的批处理任务（数据备份、ETL）可能因 DDL 阻塞而失败，这些任务失败后第二天才会发现——发现时已经是一个”被阻塞的 DDL + 失败的数据任务”的死锁。

0x07-06 连接池爆满救援全过程

标准答案：

真实案例：某电商平台大促日连接池爆满，全站瘫痪 8 分钟。

事件时间线：

根本原因分析：

  1. 某查询 SQL 因数据量暴增 + 索引缺失，执行时间从 10ms 升至 5s
  2. 慢查询长时间占用连接不释放
  3. 新请求来了创建新连接，但旧连接不释放
  4. 连接池被占满，即使原来与 MySQL 已建立的连接也因等待慢查询而”假死”

救援三步走（按现场情况优先级）：

第一步：即刻止血（0-30秒）

第二步：定位与隔离（30秒-3分钟）

第三步：恢复与加固（3分钟-60分钟）

预防措施（事后复盘必做）：

  1. 慢查询自动 KILL：通过脚本监控 INFORMATION_SCHEMA.PROCESSLIST，自动 KILL 超过阈值的慢查询。
  2. 连接池治理：每个微服务实例按业务重要性分配连接池大小，不要所有服务共用同一个 MySQL 用户上限。总连接数 = Σ(服务实例数 × 每个实例的连接池大小) < max_connections × 0.7。
  3. 熔断降级：应用层加入数据库访问熔断（如 Sentinel/Hystrix），当错误率达到阈值（如 50%）时自动降级——先保证缓存的读可用，不重要的写允许失败返回友好提示。
  4. ProxySQL 队列保护：在 ProxySQL 中设置 mysql-max_connections，对超过阈值的请求排队，避免 MySQL 直接拒绝连接。

深入思考：

  1. max_connections 不是越大越好：max_connections=2000 不是好事。每个连接在 InnoDB 中占一个线程，线程过多导致操作系统上下文切换开销暴增。正常建议 max_connections = 200~500，配合连接池使用。超过 500 的连接数往往说明应用层连接池配置有问题。
  2. 连接池泄漏排查：如果连接数持续上升但不下降，很大概率是应用层有一个 Connection 被获取后没有 close()——也就是连接泄漏。常见于使用了 @Transactional 但未正确配置 rollbackFor 的场景：异常未触发回滚时连接未能正常归还连接池。
