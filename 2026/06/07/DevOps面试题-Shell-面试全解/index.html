7. 如何编写带 trap 信号处理的 Shell 脚本？

标准答案：

trap 用于捕获信号并执行清理逻辑，是生产脚本的标配：

深入思考：

  • 信号差异：EXIT 在所有退出路径触发（正常 exit、信号终止、set -e 检测到错误）。INT（Ctrl+C）、TERM（kill 默认信号）需要单独 trap
  • 锁竞争：使用文件描述符（exec 200>）配合 flock 比传统的 [ -f lock ] && exit 更可靠——前者在进程崩溃时内核自动释放锁，后者会留下孤儿锁文件
  • 生产经验：trap 中慎用 exit，嵌套 exit 可能导致 trap 再次触发。用 kill -9 $$ 会跳过 trap，用于最后的保底强制退出

参考来源：Bash Manual, 生产运维实践

8. Shell 如何实现并行任务处理？（xargs -P / GNU parallel / wait 管理）

标准答案：

三种并行方案，按复杂度递增：

深入思考：

  • xargs -P 陷阱：xargs -P 不保证输出顺序——两个并行进程的 stdout 可能交错。用 --output (GNU parallel) 或重定向到独立文件解决
  • 错误处理：GNU parallel 的 --halt soon,fail=1 可在首个任务失败后停止新任务。wait 方案需手动收集每个后台 pid 的退出状态
  • 生产经验：GNU parallel 支持 --load（CPU 负载阈值）和 --memfree（内存阈值），避免打爆生产服务器。如 parallel -j 0 --load 80% 是自适应并发

参考来源：GNU parallel manual, Bash Cookbook

9. 生产环境 JSON 处理实战（jq 高级用法）

标准答案：

jq 是生产脚本处理 JSON 的事实标准，远胜 grep/sed/python -c：

深入思考：

  • jq vs python -c：jq 是领域特定语言，处理 JSON 比 Python 一行流更简洁，且启动开销远小于 Python（~2ms vs ~50ms）。但复杂逻辑（如网络调用、数据库查询）仍需 Python
  • 大文件处理：jq 是流式处理（streaming parser），可以处理 GB 级 JSON。用 jq -n 'inputs' file 而非 jq '.' file 可以边读边处理
  • 生产经验：jq -r 输出 raw string（去引号），适合管道传递。jq -c 输出紧凑 JSON（一行一条），适合 while read 逐行处理。jq -e 设置退出码（匹配不到时非零）

参考来源：jq manual, 生产运维实践

10. 如何编写进程守护（Daemon Wrapper）脚本？

标准答案：

守护脚本的核心：监控目标进程存活，异常退出时自动重启，同时防止重启风暴：

深入思考：

  • 重启风暴防护：指数退避（1s → 2s → 4s → 8s… → 60s）配合最大重试次数，防止 bug 导致无限重启循环。StackStorm/Supervisor 等成熟方案也采用类似策略
  • 信号转发：守护脚本收到 SIGTERM 时应转发给子进程（trap 'kill -TERM $CHILD_PID' TERM），否则 kill 守护脚本时子进程变成孤儿
  • vs systemd：systemd 的 Restart=on-failure + RestartSec=5s + StartLimitBurst=5 提供了原生的进程守护能力，生产环境应优先用 systemd 而非自定义守护脚本

参考来源：Unix Daemon 编程实践, systemd.service(5)

9. set -euo pipefail 的隐藏陷阱？SIGPIPE (141) 为什么会让脚本意外退出？

难度：⭐⭐⭐⭐（专家）
问题：某运维团队在 CI/CD 脚本中加上了 set -euo pipefail 来保证严格模式，结果发现脚本在正常处理大量文件时意外退出，排查发现是 exit code 141（SIGPIPE）导致的。为什么 set -o pipefail + grep -q 的组合会出问题？如何修复？

核心原理：

  • set -o pipefail：管道中任何命令失败（非 0 退出码），管道整体返回该失败码
  • set -e（errexit）：任何命令返回非 0 时立即退出脚本
  • SIGPIPE（信号 13，退出码 141 = 128+13）：当管道的读取端提前关闭，写入端仍在写时，操作系统发送 SIGPIPE 信号
  • 经典场景：big_generator | head -n 5 → head 读取 5 行后关闭，big_generator 继续写时收到 SIGPIPE → 退出码 141

生产实战：

🔥 踩坑案例：某日志采集脚本 pipefail+SIGPIPE 导致数据丢失

  • 场景：Shell 脚本每日凌晨从 100+ 台服务器采集日志，核心 pipeline：
  • 症状：脚本执行到一半退出，/tmp/errors.txt 为空或内容不全。排查日志发现 exit code 141
  • 排查过程：
    1. 手动模拟：seq 1 100000 | grep "5" | head -3 → 立即退出且 echo $? = 141
    2. 检查 set -o pipefail 的语义：管道退出码以最后一个非 0 命令为准
    3. head 读取足够行后关闭管道 → grep 收到 SIGPIPE → set -e 看到 141 立即退出
  • 根因：
    • set -o pipefail：管道的退出码是 grep（141）而非 head（0）
    • set -e：看到非 0 退出码立即终止整个脚本
    • 问题本质：grep -q + pipefail 是已知的「Bash 陷阱 #1」

哪个 exit code 可能来自 SIGPIPE？

对比延伸：

场景 无 pipefail 有 pipefail
`cmd head`（head 提前关闭） ✅ head 退出码 0

面试金句：

「set -euo pipefail 是 Shell 脚本的安全带，但 SIGPIPE（141）是安全带上的一个易撕口——grep | head 这类管道中读取端提前关闭时，写入端收到 SIGPIPE 退出码 141，在 pipefail 作用下导致整个脚本意外终止。正确做法是用 || [[ $? -eq 141 ]] 显式放行 SIGPIPE。」

核心原理：

  • trap 命令：在接收到指定信号时执行清理函数
  • 关键信号：
    • EXIT：脚本正常/异常退出时触发（exit、set -e、脚本末尾）
    • ERR：命令返回非 0 时触发（需配合 set -e 和 set -E 继承到函数）
    • INT/TERM：Ctrl+C / kill 发送的信号
  • 陷阱：trap 'cleanup' EXIT 比 trap 'cleanup' INT TERM 更可靠——EXIT 在任何退出路径上都触发

11. set -e 在哪些场景下「不生效」？Shell 脚本错误处理的边界情况？

难度：⭐⭐⭐⭐（专家）
问题：某面试者自信地说「我在脚本开头写了 set -e，所以所有错误都能被捕获」。但生产环境中发现管道命令、逻辑判断、子 shell 等场景下 set -e 悄悄「失效」。set -e 有哪些著名的边界情况？

核心原理：

  • set -e（errexit）在以下场景中不生效（按 POSIX 规范 + Bash 行为）：
    1. 管道中非最后一个命令：false | echo ok → set -e 不会退出（除非 pipefail）
    2. 逻辑运算符的短路部分：false && true → false 不会触发退出（这是条件判断）
    3. if / while / until 的条件部分：if false; then → expected，不退出
    4. 函数/子 shell 中用 ! 取反：! false → 预期退出码 0，不退出
    5. grep -q 无匹配：set -e; grep -q pattern file → 退出码 1，set -e 会退出—但注意这是故意的，不是失效
    6. 后续命令是 || true：cmd_fails || true → 不退出（true 覆盖了错误）
    7. 函数调用中的 return：函数内 return 1 → 可能被外部 if/|| 吞掉
  • 真正易踩的失效场景是上述 #1（无 pipefail 时）和 #6（期望 fail 但被掩盖）

生产实战：

🔥 踩坑案例：某自动化部署脚本 set -e 失效导致生产事故

  • 场景：部署脚本，核心逻辑：
  • 预期：健康检查失败时，set -e 使脚本退出，然后 trap ERR 触发回滚
  • 实际：健康检查返回 503，curl -f 返回 22，但 grep -q 的退出码 1 被 pipefail 吞掉（没有 pipefail），脚本继续执行，打印”健康检查通过”并继续部署到坏的版本！
  • 根因：
    1. 没有 set -o pipefail，管道返回的是 grep 退出码而非 curl 退出码
    2. curl -f 失败时，grep 读取空输入 → grep -q 退出码 1
    3. set -e 看到 1 → 退出，但！这里的退出是因为 grep 无匹配，不是因为 curl 失败
    4. 再加上 trap ERR 没有 set -E（errtrace），函数内 ERR 不触发顶层 trap
