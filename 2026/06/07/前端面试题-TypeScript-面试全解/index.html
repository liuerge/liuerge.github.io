0x00 类型系统

1. any、unknown、never 的区别？

  • any：关闭类型检查，会污染后续推导。适合快速原型或渐进迁移
  • unknown：安全的”任意类型”。必须先类型收窄才能操作
  • never：表示永远不会发生的类型（抛出异常、死循环、穷尽性检查）

2. type 和 interface 怎么选？

维度 type interface
扩展 & 交叉 extends
声明合并 ❌ ✅（同名自动合并）
联合类型 ✅ type A = B | C ❌
元组 ✅ type T = [string, number] ❌（可用但丑）
映射类型 ✅ ❌

选型原则：描述对象形状用 interface，需要联合/交叉/映射用 type。

0x05 TypeScript 5.5+ 新特性（2024-2025 热点）

9. TS 5.5 Inferred Type Predicates 解决了什么问题？有什么限制？

TS 5.5（2024年6月发布）最受瞩目的特性——类型谓语自动推断（由社区贡献者 Dan Vanderkam 实现）。

标准答案：

在 TypeScript 5.5 之前，filter/find 等数组方法无法自动收窄类型：

核心原理：TS 5.5 会用控制流分析（Control Flow Analysis）推断函数的返回值是否为 type predicate。对于箭头函数 x => typeof x === 'string'，TS 会自动推断出返回类型为 x is string 而非 boolean。

注意限制：

  1. “当且仅当”语义：类型谓语要求 true 返回 → 参数是 T 类型；false 返回 → 参数不是 T 类型。这意味着”truthiness” 检查（如 x => !!x 或 x => x）不会触发推断，因为它们不满足”false 分支排除 T”的条件
  2. 显式注解优先：如果手动写了 : x is string，TS 不会检查一致性——你可以写 : x is number 而实际逻辑返回 typeof x === 'string'（等同于类型断言，不安全）
  3. Breaking Change：filter(x => x !== null) 现在返回 number[] 而非 (number | null)[]，可能影响依赖旧类型的代码

参考：TypeScript 5.5 Release Notes

10. TS 5.5 isolatedDeclarations 是什么？为什么需要它？

标准答案：

--isolatedDeclarations 是 TS 5.5 引入的新编译选项，要求所有导出（exports）必须显式标注类型注解，以便声明文件（.d.ts）可以完全独立于源文件生成。

为什么需要？

  • 并行构建：isolatedDeclarations 使得 transpileDeclaration API 可以在不进行完整类型检查的情况下生成 .d.ts 文件——每个文件独立处理，可以充分并行
  • 构建速度：对于大型单体仓库（monorepo），声明文件生成通常占总构建时间的 30-50%。isolatedDeclarations + --noCheck 可以将这部分的耗时降低 10 倍以上
  • 更清晰的 API 契约：强制所有导出显式标注类型，提升代码可读性

参考：TypeScript 5.5 Isolated Declarations

深入思考：

  • 适用场景：isolatedDeclarations 对库作者和 monorepo 维护者价值最大——可以显著加速 CI 构建。对于应用项目价值较小（应用代码通常不需要对外暴露声明文件）
  • 工具链兼容：tsc --noCheck --isolatedDeclarations 时，TypeScript 仅做语法转换生成 .d.ts，完全不进行类型检查。这比 tsc --noEmit 快一个数量级。在 CI 中可以拆分为：快速构建阶段（--noCheck 产出 .d.ts）+ 独立类型检查阶段（tsc --noEmit），两者并行运行

11. TS 5.6/5.7 有哪些值得关注的新特性？

标准答案：

TS 5.6（2024年9月）：

  1. Iterator Helper Methods：支持 ECMAScript 的 Iterator Helpers 提案（.map(), .filter(), .take(), .drop(), .reduce() 等方法直接作用于 Iterator 对象）

  1. --noCheck 公开化：此选项在 TS 5.5 内部使用（用于 transpileModule/transpileDeclaration），5.6 起作为 CLI 选项公开，允许构建工具跳过类型检查

  2. --build 支持中间错误：--build 模式现在允许项目引用链中有错误时仍继续构建其他项目（之前会全部停止）

  3. Strict Builtin Iterator Checks：更严格的内置迭代器类型检查

TS 5.7（2024年11月）：

  1. Computed Property Names 改进：计算属性名（[symbolMethodName]）现在会生成索引签名，而非完全忽略
  1. TypedArray 泛型化：Uint8Array、Int32Array 等现在接受泛型参数指定底层 ArrayBuffer 类型
  1. Path Mapping 改进：支持 package.json 中的相对路径重映射

面试金句：TS 5.5-5.7 的演进方向很明确——“Make TypeScript faster and stricter”。inferred type predicates 提升类型精度，isolatedDeclarations 加速构建，noCheck 让大型项目 CI 飞起来。

参考：TypeScript 5.7 Release Notes

12. TypeScript Decorators（Stage 3）与实验性装饰器有何区别？

标准答案：

TypeScript 同时支持两套装饰器系统，这导致了长期的混乱。关键区别：

维度 实验性装饰器（TS 5.0 之前默认） Stage 3 装饰器（TC39 标准，TS 5.0+）
标准 非标准，TypeScript 自行实现 TC39 Stage 3（2022年），接近标准化
启用 experimentalDecorators: true 默认启用（TS 5.0+）
参数装饰器 ✅ 支持 ❌ 不支持（从提案中移除）
元数据 emitDecoratorMetadata: true ❌ 无此功能（需等待 Reflection API）
类装饰器语法 @sealed class C {} @sealed class C {}（语法相同但语义不同）
类装饰器行为 替换构造函数 通过返回新类或修改原类
与 NestJS/Angular 兼容 ✅ ❌（这些框架依赖实验性装饰器）

关键区别：Stage 3 装饰器的 context 参数提供了丰富的元信息（context.kind、context.name、context.metadata），这是实验性装饰器没有的。

选型建议：

  • 使用 NestJS/Angular → 必须用实验性装饰器（experimentalDecorators: true）
  • 新建纯 TS 项目 → 优先 Stage 3 标准装饰器（未来兼容性更好）
  • 类库/框架开发 → 评估社区生态，目前多数框架仍依赖实验性装饰器

参考：TC39 Decorators Proposal

13. 条件类型（Conditional Types）深度实战：infer、分布式条件类型与递归类型推导

标准答案：

条件类型是 TypeScript 类型体操的核心武器。它的语法类似三目运算符：T extends U ? X : Y——如果 T 可赋值给 U，则类型为 X，否则为 Y。

深入思考：

  • 条件类型执行顺序：TypeScript 在求值条件类型时，会先延迟求值直到所有泛型参数确定——这意味着 type Foo<T> = T extends string ? true : false 在 T 不确定时不会报错，只是保持未求值状态
  • infer 的限制：infer 只能在条件类型的 extends 子句中使用，不能单独使用。而且 infer 在同一位置只能出现一次——你不能写 T extends [infer A, infer A] 来约束两个位置类型相同（需要辅助技巧）
  • 分布式条件的陷阱：type IsString<T> = T extends string ? true : false 作用于 string | number 时结果是 boolean（即 true | false），而不是 false！可以通过 [T] extends [string] 禁用分发
  • 递归深度限制：TypeScript 的递归深度默认为 50 层（可通过 --maxNodeModuleJsDepth 调整，但不推荐依赖），过深的递归条件类型会导致编译错误（Type instantiation is excessively deep）

参考：TypeScript Conditional Types

14. TS 5.8 erasableSyntaxOnly 与直接执行 TypeScript

标准答案：

TS 5.8（2025 年 3 月）引入 --erasableSyntaxOnly 选项，禁止那些 Node.js 无法直接剥离的 TypeScript 特有运行时语法：

同时推出的 --rewriteRelativeImportExtensions 自动将 .ts → .js（以及 .tsx→.jsx，.mts→.mjs，.cts→.cjs），让 .ts 文件可以直接被 Node.js（22+/23+）原生执行：

技术背景：

Node.js 22 引入 --experimental-strip-types（23 稳定化），可以直接执行 TypeScript——但只支持类型擦除（type stripping），不能处理 enum、命名空间、参数属性这些需要语法转换的构造。erasableSyntaxOnly 恰好保证你的代码在直接执行时不会崩溃。

深入思考：

  • 迁移路径：对于已有大量 enum 使用的项目，erasableSyntaxOnly 会全面报错。迁移策略是先用 const enum（如果只是编译期使用）或改为联合类型字面量
  • 性能收益：TS 7（规划中）将用 Go 重写运行时，预计构建加速 10×。erasableSyntaxOnly 是这个规划的前置条件——只有消除不可擦除语法，才能实现零成本类型擦除
  • 面试加分：能说出 TS 未来的演进方向——“类型擦除式编译”（type-stripping compilation），说明你对 TS 生态趋势有全局认知

参考：TypeScript 5.8 Release Notes

15. tsconfig 继承与多环境配置策略

标准答案：

大型项目通常需要多个 tsconfig 文件，通过 extends 链式继承：

paths 与 types 配置：

triple-slash directives 的适用场景：

  • path：显式引用其它 .d.ts 文件（仅在不支持 include 的旧项目中用）
  • types：引用 @types/* 包（当 types 字段为空数组时，用此方式手动拉入）
  • lib：追加标准库定义（少数场景需要）

深入思考：

  • paths 的陷阱：paths 只是类型检查时的映射，运行时模块解析不受影响。需要 webpack/vite 的 resolve.alias 或 tsconfig-paths 包在运行时生效
  • types 管理：types: ["node"] 会让 TS 只加载 @types/node，忽略其它 @types/*。这在 monorepo 中有用——避免子项目意外依赖根级别的类型定义。也常用 types: [] 完全关闭全局类型自动加载
  • extends 深度：官方建议不超过 3 层继承，否则调试配置问题极其困难

选型策略（生产推荐）：

性能数据参考（大型 monorepo，5000+ TS 文件）：

  • tsc：约 120 秒（全量）
  • tsc --noEmit：约 45 秒
  • swc 转译：约 1.5 秒
  • esbuild 打包：约 2 秒

深入思考：

  • CI 策略：推荐将类型检查与构建分离。类型检查在 PR 阶段并行执行，构建阶段用 --noCheck 跳过类型检查，可节省 50-70% 的 CI 时间
  • noCheck 的代价：跳过类型检查意味着某些类型错误会延迟到运行时才发现。需要在”构建速度”和”类型安全”间做 trade-off。推荐 PR CI 做完整 tsc --noEmit，发布 CI 用 swc + tsc --emitDeclarationOnly --noCheck
  • Bun 的特殊性：Bun 不仅是个构建工具，还是运行时。它原生支持 TypeScript 执行，不需要转译步骤。但 Bun 的底层 API 和 Node.js 不完全兼容，迁移成本需要考虑

  • 生产价值：模板字面量类型的最大价值在于——让「URL 路径」、「事件名」、「CSS 类名」等字符串参数在编译期就得到类型检查，将运行时错误提前到 IDE 阶段
