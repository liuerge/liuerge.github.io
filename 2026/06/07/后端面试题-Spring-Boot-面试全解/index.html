0x00 自动配置原理

1. @SpringBootApplication 注解到底做了什么？

拆解：

关键在于 @EnableAutoConfiguration：

  1. 通过 @Import(AutoConfigurationImportSelector.class) 导入选择器
  2. 选择器读取 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports（Spring Boot 3.x）或 spring.factories（2.x）
  3. 加载全部候选配置类（约 150+）
  4. 通过 @ConditionalOnClass、@ConditionalOnMissingBean 等条件注解过滤
  5. 最终只加载符合当前环境的配置

深入思考：

  • 边界情况：第三方 Starter 的 AutoConfiguration.imports 可能定义了你不想要的 Bean。排除方式：@SpringBootApplication(exclude = {XXXAutoConfiguration.class}) 或在 application.properties 中 spring.autoconfigure.exclude=...
  • 源码层面：Spring Boot 3.x 换用 .imports 文件替代 spring.factories 是为了减少类加载开销——spring.factories 需要加载所有文件中的所有行，.imports 只加载对应文件

参考：Spring Boot 自动配置文档

4. @ConfigurationProperties vs @Value 怎么选？

维度 @ConfigurationProperties @Value
绑定方式 按前缀批量绑定（类型安全） 逐个注入
松散绑定 支持（my-prop → myProp） 不支持
校验 支持 JSR303 @Validated 不支持
复杂类型 支持 List/Map/嵌套对象 仅支持简单类型
适用场景 一组相关配置 单个独立配置

深入思考：

  • 生产经验：@ConfigurationProperties 需要启用 @EnableConfigurationProperties 或注册为 Bean，否则不生效。最常见踩坑：写了 @ConfigurationProperties 但忘了加 @Component 或 @EnableConfigurationProperties

0x03 Spring Boot 3.x 迁移要点

6. Spring Boot 2.x → 3.x 有哪些 Breaking Changes？

变更 2.x 3.x
Java 基线 Java 8/11 Java 17+
Jakarta EE javax.* jakarta.*
自动配置注册 spring.factories AutoConfiguration.imports
可观测性 无统一 API Micrometer Observation API

  • 大量第三方库的 javax.servlet 依赖需要替换为 jakarta.servlet
  • 如果有自定义 spring.factories，必须迁移到 .imports 文件
  • spring-security 的配置方式有较大变化（lambda DSL）

0x08 基础面试问答（补充）

以下题目整理自 Tom哥《Spring Boot 那点破事》系列，补充一些基础但常考的面试题。

8. Spring Boot 常用注解有哪些？

标准答案：

注解 说明
@SpringBootApplication 组合注解，定义在 main 方法入口类，包含 @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan
@EnableAutoConfiguration 开启自动配置，通过 META-INF/spring.factories 或 .imports 加载候选配置类
@Value 从 application.properties/yml 注入单个配置值
@ConfigurationProperties(prefix="xxx") 按前缀批量注入配置到 POJO，类型安全
@RestController 组合 @Controller + @ResponseBody，用于 API 接口
@SpringBootTest 测试注解，提供 Spring 容器上下文

出处：Tom哥《Spring Boot 那点破事》

9. Spring Boot 配置加载顺序是怎样的？

标准答案：

优先级从高到低：

  1. 命令行参数（最高优先级）：java -jar app.jar --server.port=8081
  2. JNDI 属性：java:comp/env
  3. 系统环境变量：SERVER_PORT
  4. application-{profile}.properties/yml：多环境配置文件
  5. application.properties/yml：主配置文件
  6. @PropertySource 注解指定的配置
  7. Spring 默认配置：spring-boot-autoconfigure 中的默认值

补充说明：YAML 文件比 properties 文件晚加载，所以 properties 中的配置会覆盖 yaml 中的同名配置。Spring Boot 2.x 起推荐统一使用一种格式。

出处：Tom哥《Spring Boot 那点破事》

10. Spring Boot 的运行方式有哪些？

标准答案：

  1. 直接执行 main 方法：在 IDE 中运行 @SpringBootApplication 类的 main 方法
  2. Maven/Gradle 插件运行：mvn spring-boot:run 或 gradle bootRun
  3. 打包后运行：mvn package → java -jar target/app.jar
  4. 部署到外置容器：打成 war 包部署到 Tomcat/Jetty（较少用，Spring Boot 提倡内嵌容器）

深入思考：生产环境应使用 java -jar 配合 Systemd/Supervisor 做进程管理，不要用 mvn spring-boot:run。在 K8s 中配合 PreStop 钩子和 SIGTERM 信号实现优雅下线。

出处：Tom哥《Spring Boot 那点破事》

11. bootstrap.properties 和 application.properties 有何区别？

标准答案：

维度 bootstrap.properties application.properties
加载顺序 先加载（父 ApplicationContext） 后加载（子 ApplicationContext）
用途 引导阶段配置，如配置中心地址、加密密钥 常规应用配置
覆盖规则 bootstrap 中的属性不能被 application 覆盖 可以被 profile 和环境变量覆盖
常用场景 Spring Cloud Config / Nacos 配置中心的连接信息 业务相关的常规配置

注意：Spring Boot 2.4+ 引入了新的配置加载方式，可以通过 spring.config.import 替代 bootstrap 的功能。Spring Cloud 2020.0+ 也支持这种方式。

出处：Tom哥《Spring Boot 那点破事》

14. Spring Boot 如何实现定时任务？

标准答案：

两种方式：

方式一：@Scheduled 注解（推荐，Spring 原生支持）

需要在启动类添加 @EnableScheduling 开启定时任务支持。

方式二：Quartz（适合复杂调度需求）

适用于需要持久化、动态调整 Cron 表达式、分布式场景。

深入思考：

  • 单机场景用 @Scheduled 足够，注意方法不能有入参、不能有返回值
  • 集群场景必须加分布式锁（如 Redis 锁）防止多节点重复执行
  • @Scheduled 默认使用单线程执行器，多个定时任务会互相阻塞，可通过 @Async 或自定义 TaskScheduler 改为线程池

出处：Tom哥《Spring Boot 那点破事》

19. Spring Boot 的安全防护方法有哪些？

标准答案：

  1. HTTPS：生产环境强制 HTTPS，配置 SSL 证书
  2. CSRF 保护：Spring Security 默认开启 CSRF 防护
  3. XSS 防护：使用内容安全策略（CSP）头
  4. 依赖安全：定期使用 Snyk 或 OWASP Dependency-Check 扫描依赖漏洞
  5. 版本升级：保持 Spring Boot 及所有依赖的最新稳定版本

  7. 输入校验：对用户输入做白名单校验，防止 SQL 注入和命令注入

20. Virtual Threads（虚拟线程）与 Spring Boot 3.2+ 整合

背景：JDK 21 正式发布 Virtual Threads（JEP 444），Spring Boot 3.2 起通过 spring.threads.virtual.enabled=true 一键开启。

配置方式：

开启后以下组件自动使用虚拟线程：

  • Tomcat：protocol-handler 每个请求一个虚拟线程（替代平台线程池）
  • @Async：异步任务运行在虚拟线程
  • @Scheduled：定时任务使用虚拟线程
  • RabbitMQ/Kafka 监听器：消息消费在虚拟线程

Tomcat 线程模型变化：

🔥 生产踩坑：虚拟线程下的三大陷阱

案例 1：synchronized 导致虚拟线程 pinned（某电商支付服务）

  • 场景：Spring Boot 3.2 + JDK 21 虚拟线程，支付服务 50 台集群
  • 症状：开启虚拟线程后吞吐量下降 40%，线程 dump 显示大量虚拟线程处于 CARRIER 状态
  • 排查：
  • 根因：支付模块在 synchronized 块内调用了远程服务——虚拟线程被 pinned 到 carrier 线程，3s 的 RPC 调用占用了 carrier 线程
  • 修复：将 synchronized 替换为 ReentrantLock
  • 修复效果：吞吐量恢复，P99 延迟从 800ms 降至 200ms

案例 2：ThreadLocal 内存泄漏（某内容推荐服务）

  • 场景：使用 ThreadLocal 存储请求追踪 ID，开启虚拟线程后出现 OOM
  • 根因：虚拟线程池不回收线程，ThreadLocal 值在虚拟线程生命周期内一直存在。百万级请求创建百万级虚拟线程、百万个 ThreadLocal 实例
  • 修复：
  • 最佳实践：
    • 虚拟线程场景下完全禁止 ThreadLocal（JDK 官方也建议迁移到 ScopedValue）
    • 如果必须用，在 finally 块中显式 remove()

案例 3：池化资源（连接池/线程池）与虚拟线程不兼容（某社交平台）

  • 场景：数据库连接池 HikariCP，每个请求获取连接后等待数据库响应
  • 症状：数据库连接池被占满，新的请求等待连接超时
  • 根因：传统模式下 200 个平台线程共用 50 个连接，现在 10 万个虚拟线程同时竞争 50 个连接——连接池被击穿
  • 修复：调大连接池 + 缩短连接持有时间
  • 最佳实践：虚拟线程场景下连接池需要调大 2-3 倍，并设置更短的超时时间

Virtual Threads 兼容性审计清单：

  • synchronized 块/方法 → 替换为 ReentrantLock / StampedLock
  • ThreadLocal 使用 → 替换为 ScopedValue（JDK 21+）
  • 池化资源（连接池、线程池）→ 调大 2-3 倍
  • ThreadPoolTaskExecutor → 移除自定义线程池（虚拟线程用默认即可）
  • 第三方库兼容性 → 检查依赖是否支持虚拟线程（Netty/Undertow 默认支持）
  • 监控指标 → 检查虚拟线程活跃数、队列深度、pinned 次数

面试金句：

“虚拟线程不是银弹。我在 Spring Boot 3.2 生产中遇到三个核心坑：synchronized 导致 pinned（吞吐下降 40%）、ThreadLocal 内存泄漏（百万实例 OOM）、连接池被击穿（10 万虚拟线程争 50 个连接）。迁移顺序：先审计所有 synchronized + ThreadLocal，再调大连接池，最后一键开启 spring.threads.virtual.enabled=true。”

参考：Spring Boot 3.2 Virtual Threads | JEP 444

21. Spring Boot 3.x 从 2.x 迁移全指南

核心变更清单：

变更项 Spring Boot 2.x Spring Boot 3.x 迁移影响
Java 基线 JDK 8/11/17 JDK 17+ 必须升级 JDK
Jakarta EE javax.* jakarta.* 包名全局替换
Hibernate 5.x 6.x 大量 API 变更
Spring Security 5.x 6.x 配置变化大

迁移红牌禁令：

  • ❌ 不要一次性升级所有服务 — 分批灰度，先升级无状态边缘服务
  • ❌ 不要忽略 -Xlint:all 编译警告 — 模块化系统对跨模块反射敏感
  • ❌ 不要跳过 JDK 过渡版本 — JDK 8 → 21 直接升级风险极高
  • ✅ 先升级到 JDK 11 → 17 过渡，验证兼容性后再升级到 JDK 21
  • ✅ 使用 sdkman 管理多版本 JDK，配置切换脚本

必须监控的指标与告警阈值：

指标 PromQL 告警阈值 说明
HTTP 错误率 rate(http_server_requests_seconds_count{status=~"5.."}[1m]) > 1% 5xx 比例
P99 延迟 histogram_quantile(0.99, ...) > 1000ms 用户体验红线
活跃线程数 jvm_threads_live_threads > 80% 池大小 线程耗尽预警

数据库连接池 hikaricp_connections_active > 80% maximum-pool-size 连接池告急
健康检查 probe_success{group="readiness"} < 1 服务不健康

🔥 生产案例：GC 频繁导致 Actuator 响应超时引发的告警风暴（某金融交易系统）

  • 场景：Spring Boot 2.7 + G1 GC，8 节点集群
  • 症状：Prometheus 告警 "Target Down" 每 2 小时报告一次，持续 30-60 秒
  • 排查：抓取 JMX 数据发现 GC 暂停期间 JVM 线程全部 STW，包括 HTTP 请求处理线程。Actuator /actuator/health 端点 30s 无响应，Prometheus 标记为 Down
  • 根因：G1 Mixed GC 阶段回收大对象（Humongous Allocation），单次暂停 800ms-1.2s，导致 Actuator 超时
  • 修复：
  • 最佳实践：Actuator 健康检查设置 time-to-live 缓存，避免 Prometheus 每次拉取都触发类加载和 GC

面试金句：

“生产监控不是配好依赖就完了。我在金融交易系统中踩过 ‘Prometheus Target Down’ 告警风暴——G1 GC 暂停 800ms 导致 Actuator 超时。解决方案是在 Actuator 健康端点加 5s 缓存，把不必要触发的 GC 隔离开。Micrometer + OTEL 很好用，但每个指标的告警阈值要按业务场景调。”
