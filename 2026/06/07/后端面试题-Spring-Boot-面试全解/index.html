后端面试题：Spring Boot 面试全解

Spring Boot 高级面试全解，覆盖自动配置原理、Starter 机制、Actuator 监控、性能调优和常见生产踩坑。面向高级研发，侧重源码理解和架构设计。

0x00 Spring Framework 核心

Spring Framework 核心内容，涵盖 IoC 容器源码、DI 注解详解、AOP 原理、事务管理等。这是 Spring Boot 的底层基础，理解这些有助于深入掌握 Spring Boot 的自动配置和运行机制。

0x00 IoC 容器

0x00-01. BeanFactory 和 ApplicationContext 有什么区别？

标准答案：
BeanFactory 是 Spring IoC 容器的顶层接口，定义了 IoC 容器的基本功能（获取 Bean、判断是否单例等）。ApplicationContext 是它的子接口，增加了企业级功能：

特性 BeanFactory ApplicationContext
延迟加载 默认懒加载（getBean 时创建） 默认预加载（容器启动时创建所有单例 Bean）
国际化 不支持 支持（MessageSource）
事件发布 不支持 支持（ApplicationEventPublisher）
AOP 整合 需手动配置 自动整合
BeanPostProcessor 注册 需手动注册 自动注册
环境抽象 不支持 支持（EnvironmentCapable）

深入思考：

  • 源码层面：DefaultListableBeanFactory 是核心实现，getBean() 方法经过 AbstractBeanFactory.doGetBean() → 检查缓存（单例池）→ 不存在则创建 → 递归处理依赖 → 返回实例。ApplicationContext 的 refresh() 方法内部调用 obtainFreshBeanFactory() 创建 BeanFactory，然后在 finishBeanFactoryInitialization() 中预实例化所有单例。
  • 实际架构：Spring Boot 内部用的就是 AnnotationConfigApplicationContext（继承自 ApplicationContext），但底层操作的 BeanFactory 仍然是 DefaultListableBeanFactory。BeanFactory 是”骨架”，ApplicationContext 是”完整肉身”。
  • 性能差异：大量 Bean（>5000）启动时，ApplicationContext 的预加载策略会导致启动时间显著增加。可用 @Lazy 或设置 spring.main.lazy-initialization=true 改为懒加载，但可能影响首次请求响应时间。

🔥 生产踩坑：某微服务整合了 300+ 个 Starter，启动耗时 90 秒。排查发现大量 Bean 在启动时被预加载，但其实很多只在特定场景使用。解决：对非核心组件（如某些 Report 类、定时任务客户端）加 @Lazy，启动时间降至 25 秒。注意：@Lazy 使用时如果存在循环依赖，可能引发 BeanCurrentlyInCreationException。

0x00-03. @Configuration 的源码原理是什么？和 @Component 有什么区别？

标准答案：
@Configuration 标注的类会被 CGLIB 增强，生成代理子类。Spring 通过 ConfigurationClassPostProcessor（BeanFactoryPostProcessor）处理 @Configuration 类，使其内部的 @Bean 方法变成单例管理——同一个 @Bean 方法被多次调用，每次返回的都是同一个实例（因为被 CGLIB 代理拦截了）。

@Component 不会产生 CGLIB 代理，只是将类注册为普通 Bean。

区别核心：

特性 @Configuration @Component
CGLIB 代理 是（full 模式） 否（lite 模式）
@Bean 方法调用 单例拦截，返回容器中的同一个实例 每次调用都 new 一个新对象
内部 @Bean 方法间引用 Bean 间引用被代理拦截 Bean 间引用直接 new 对象，不是容器中的 Bean
性能 启动稍慢（CGLIB 生成子类） 启动快，无代理开销
典型用途 声明配置类 声明普通组件

深入思考：

  • 源码层面：ConfigurationClassPostProcessor.postProcessBeanFactory() → 遍历所有候选类 → 检查是否有 @Configuration → 使用 ClassPathBeanDefinitionScanner 扫描 → 对 @Configuration 类创建 EnhancedConfigurationBeanDefinition，设置 CONFIGURATION_CLASS_ATTRIBUTE 为 full（对应 @Configuration）或 lite（对应 @Component/@Bean 组合）。
  • CGLIB 增强细节：Spring 使用 ConfigurationClassEnhancer 生成代理子类。增强后调用 @Bean 方法时先查 BeanFactory 缓存，如果存在直接返回，不存在才调用父类方法创建。这就是为什么 full 模式下 @Bean 方法可以无压力互相调用。
  • lite 模式陷阱：如果 @Component 内声明了 @Bean 方法，方法间调用如 dataSource() → transactionManager(dataSource()) 会导致每次 new 一个 DataSource，事务管理器连接的 DataSource 和注入到其它 Bean 的 DataSource 不是同一个对象。
  • Spring Boot 3.x 变化：Spring 6 / Boot 3 引入了 proxyBeanMethods 属性——@Configuration(proxyBeanMethods=false) 可以关闭 CGLIB 代理，适用于不需要 @Bean 方法间调用的场景，启动速度更快（减少了 CGLIB 生成开销）。

🔥 生产踩坑：某项目中在 @Configuration 类上用了 @Component 注解而非 @Configuration，导致数据库中插入了重复数据。原因是用 Spring Data JPA 时，多个地方调用了同一个 @Bean 方法，每次 new 了一个新 Repository Bean 出来，连接不同的 DataSource 实例写入。排查两天，最终对比 Bean 的 hashCode 发现不是同一个对象。修复：改为 @Configuration 即可。

0x01 DI 与注解

0x01-01. @Autowired、@Resource、@Inject 有什么区别？

标准答案：

维度 @Autowired @Resource @Inject
所属 Spring 原生 JSR-250 标准 JSR-330 标准
默认匹配方式 byType（按类型） byName（按名称） byType（按类型）
支持 @Qualifier 是 是（通过 name 属性） 是（通过 @Named）
required 属性 支持（required=false） 不支持 不支持（可用 Optional）
支持 @Primary 是 是 是
构造器注入 支持（推荐方式） 不支持 支持
推荐度 ⭐⭐⭐ ⭐⭐ ⭐⭐⭐

源码层面：

  • @Autowired：由 AutowiredAnnotationBeanPostProcessor 处理。先 byType 匹配，如果找到多个候选，再通过 @Qualifier/byName/@Primary 降级筛选。若仍无法唯一确定则抛 NoUniqueBeanDefinitionException。
  • @Resource：由 CommonAnnotationBeanPostProcessor 处理。默认按 name 查找，name 可以指定（如 @Resource(name="xxx")），如果未指定 name 则以字段名作为 name 查找；按 name 找不到时回退到 byType。
  • @Inject：由 AutowiredAnnotationBeanPostProcessor 同一套机制处理（Spring 5.x 起内置支持）。行为与 @Autowired 几乎一致，区别在于没有 required 属性。

深入思考：

  • 字段注入 vs 构造器注入：字段注入使 Bean 在构造完成后才能注入，导致与 final 关键字不兼容、无法实现不可变对象、测试时需要反射。Spring 官方推荐构造器注入（不仅是最佳实践，更是防止未来循环依赖的手段——构造器注入在初始化时就会暴露循环依赖，不会等到运行时才报错）。Spring 4.3+ 如果类只有一个构造器，可以省略 @Autowired。
  • @Resource 的 byName 回退陷阱：当 bean name 和字段名不一致时，@Resource 先按字段名找 name，找不到再按类型匹配。如果同类型有多个 Bean，可能注入到错误的对象。
  • @Inject + @Named 组合：等价于 @Autowired + @Qualifier，但 @Inject 是 Jakarta EE 标准，在不同 DI 容器（如 Guice）间可移植——如果团队有跨容器需求，优先用 @Inject。

🔥 生产踩坑：老项目从 XML 配置迁移到注解时，某 Service 注入两个同类型 DataSource Bean。原来用 @Resource(name=”primaryDataSource”) 工作的很好，有人改成 @Autowired 后应用启动失败——@Autowired 默认 byType，两个 DataSource 导致冲突。修复：加 @Qualifier(“primaryDataSource”) 或保留 @Resource 不变。关键教训：不要用自动化重构工具批量替换 @Resource → @Autowired 而不仔细检查多候选场景。

  • 为什么需要三级缓存，两级不够？：两级（singletonObjects + earlySingletonObjects）理论上可以解决 Setter 注入的循环依赖。但三级缓存引入 ObjectFactory 是为了延迟 AOP 代理的创建时机——如果 Bean 被 AOP 代理，需要提前创建代理对象暴露给依赖方。三级缓存在 getEarlyBeanReference() 中调用 SmartInstantiationAwareBeanPostProcessor.getEarlyBeanReference() 来实现。如果只有两级，每次都要在实例化后立即判断是否需代理，性能浪费。
  • 什么情况下循环依赖无法解决？：① 构造器注入（constructor injection），实例化时就需要依赖，此时三级缓存还没放数据；② prototype 作用域的 Bean——Spring 不缓存 prototype，每次都是新实例；③ @Async 注解——异步会在后置处理器中额外生成代理，与循环依赖机制冲突；④ 开启了 allowCircularReferences = false。
  • Spring Boot 2.6+ 默认禁止循环依赖：spring.main.allow-circular-references=false（新项目默认）。如果你的项目有大量循环依赖，建议重构而不是依赖三级缓存——循环依赖本质是设计缺陷。

🔥 生产踩坑：某团队写了一个非常复杂的 Service 层，A → B → C → A 的三方循环依赖。Spring Boot 2.6 升级后启动报错。由于业务逻辑深度耦合，重构代价极大。临时方案：设置 spring.main.allow-circular-references=true。长期方案：抽取出公共依赖层，用 Event/Listener 解耦 A → C 的调用关系。教训：循环依赖是设计警钟，不是容器的 Bug。

0x01-03. @Lazy、@Scope、@Profile 的作用和原理是什么？

标准答案：

@Lazy：延迟初始化。单例 Bean 默认在容器启动时创建，加上 @Lazy 后推迟到第一次 getBean() 时才创建。可用于 @Configuration、@Bean、@Component 上。

@Scope：定义 Bean 的作用域。Spring 支持的作用域：

作用域 说明 使用场景
singleton 默认，容器中只有一个实例 无状态 Service、DAO
prototype 每次 getBean() 创建新实例 有状态的组件、非线程安全的对象
request 单次 HTTP 请求共享一个实例 Web 应用中的请求上下文数据
session 单次 HTTP 会话共享一个实例 用户登录信息
application ServletContext 级别共享 全局配置缓存
websocket WebSocket 会话范围 WebSocket 应用

@Profile：根据运行环境条件加载 Bean。

深入思考：

  • @Lazy 与 @Bean 的关系：@Bean 默认是 Eager（急加载），加上 @Lazy 后延迟加载。但注意：如果其他非懒加载的 Bean 依赖了这个懒加载 Bean，依赖注入时会立即创建它，@Lazy 失效。
  • @Lazy 解决循环依赖：构造器注入的循环依赖可以用 @Lazy 打断——在其中一个依赖上 @Lazy，Spring 会生成一个代理对象注入，真正的创建延迟到实际调用时。
  • @Scope 的代理模式：request/session 等 Web 作用域 Bean 注入到 singleton Bean 时，需要设置 proxyMode（如 @Scope(value="request", proxyMode=ScopedProxyMode.TARGET_CLASS)），否则 singleton Bean 启动时就会尝试注入 HTTP 请求上下文（此时请求不存在）。Spring 注入一个代理对象，实际调用时每次委托给当前请求的上下文。
  • @Profile 的底层：基于 Condition 接口实现——ProfileCondition 检查 Environment.getActiveProfiles() 中是否匹配。Spring Boot 通过 spring.profiles.active 激活配置，也支持 spring.profiles.include 无条件包含某些 profile。
  • Spring Boot 的 profile 层叠优先级：application-default.yml（默认）→ application-dev.yml（profile 激活时）→ application.yml 中定义的值被 application-dev.yml 覆盖。同优先级下最后加载的生效（spring.config.additional-location 可以改变加载顺序）。

🔥 生产踩坑：某团队在 @Configuration 上用了 @Lazy，期望整个配置类延迟加载。但是另一个 @Component 通过 @Import 导入了这个配置类中的 Bean，导致 @Lazy 完全失效。原因是 @Import 导入的 Bean 会在容器启动时就处理。解决：在具体 @Bean 方法上加 @Lazy 而不是类级别。另一个踩坑：@Profile(“!prod”) 用于测试隔离，但有人拼写错误写成了 @Profile(“!pro”)（少了个 d），导致生产环境也加载了测试 Bean，关闭了安全校验——排查了整整一天。

0x01-04. 什么是 Setter 注入 vs 构造器注入？为什么 Spring 官方推荐构造器注入？

标准答案：

维度 构造器注入 Setter 注入 字段注入
写法 public class A { private final B b; public A(B b) { this.b = b; } } public class A { private B b; @Autowired public void setB(B b) { this.b = b; } } @Autowired private B b;
不可变性 ✅ 支持 final ❌ 不支持 final ❌ 不支持 final
循环依赖检测 ✅ 启动时就检测到 ❌ 运行时才知道 ❌ 运行时才知道
测试便利性 ✅ 直接 new 传参 ✅ setter 设置 mock ❌ 需要用反射/Spring 容器
必要依赖说明 ✅ 显式声明必需依赖 ❌ 可选依赖的意味 ❌ 隐式依赖
Spring 推荐 ✅ 最推荐 ⚠️ 可选依赖时使用 ❌ 不推荐（IDE 也会警告）

深入思考：

  • Spring 官方在 Spring 4.x 起就推荐构造器注入，关键原因是：构造器注入确保 Bean 在创建时所有依赖都已就位，不存在 NullPointerException 隐患；final 字段保证了不可变性和线程安全；更容易测试（不需要启动 Spring 容器）。
  • Spring 4.3+ 的隐式构造器注入：如果一个类只有一个构造器，可以省略 @Autowired，Spring 会自动选择这个构造器注入依赖。这让代码更简洁。
  • 循环依赖与构造器注入：构造器注入的循环依赖无法用三级缓存解决——因为实例化阶段就需要依赖，此时三级缓存还未放置 Bean 的工厂。解决方案：用 @Lazy 打断循环（但会引入代理），或重构代码消除循环。
  • Lombok 配合：@RequiredArgsConstructor 为 final 字段生成构造器，配合 Spring 的隐式构造器注入非常优雅。

🔥 生产踩坑：有团队统一使用了字段注入，后来添加了新的依赖关系产生循环依赖，应用在测试环境跑了一周才在特定路径下出现 StackOverflowError。如果用构造器注入，这个问题在启动时就能发现。后来加了一条 Sonar 规则：禁止字段注入，强制构造器注入。

0x02-01. JDK 动态代理和 CGLIB 代理有什么区别？Spring 如何选择？

标准答案：

维度 JDK 动态代理 CGLIB 代理
原理 基于接口生成代理类（Proxy.newProxyInstance） 基于继承，生成目标类的子类
要求 目标类必须实现至少一个接口 目标类不能是 final 的，方法不能是 final
代理对象 实现目标类的所有接口 继承目标类
性能（创建） 较快（字节码生成简单） 较慢（需生成子类字节码）
性能（调用） 较慢（通过 InvocationHandler 反射调用） 较快（直接调用父类方法，无需反射，但 Spring 5+ 两者差距缩小）
目标类要求 需有接口 需非 final 类 + 非 final 方法

Spring 选择策略：

  1. 如果目标类实现了接口 → JDK 动态代理
  2. 如果目标类未实现接口 → CGLIB 代理
  3. 如果显式配置 proxy-target-class=true → 强制 CGLIB
  4. Spring Boot 2.0+ 默认 spring.aop.proxy-target-class=true（强制 CGLIB，即使有接口）

深入思考：

  • 源码层面：DefaultAopProxyFactory.createAopProxy(AdvisedSupport config) 判断：如果 config.isOptimize() 或 config.isProxyTargetClass() 或目标类没有接口 → 用 CglibAopProxy；否则用 JdkDynamicAopProxy。
  • Spring Boot 为何默认 CGLIB：因为几乎所有 Service 都加了 @Transactional，而 @Transactional 本身不需要接口。如果默认 JDK 代理，未实现接口的类就必须手动配置 proxy-target-class=true。Spring Boot 为了”开箱即用”索性默认 CGLIB。
  • 性能 benchmark：JDK 动态代理调用耗时 ≈ 35ns，CGLIB 耗时 ≈ 25ns（Spring 5.3 + JDK 17）。CGLIB 在方法调用链特别深时有优势，但在大多数业务场景下可以忽略不计。
  • 自调用问题：无论是 JDK 还是 CGLIB，同一个类内部的方法调用（this.methodB()）都不会触发 AOP——因为代理对象拦截的是外部调用，内部 this 引用的是原始对象。

🔥 生产踩坑：某团队升级 Spring Boot 2.x 后，发现部分 Bean 注入失败，错误信息 @Autowired failed: no qualifying bean。排查发现某个 Service 类没有任何接口，但在另一个模块中被注入时用了接口类型声明。Spring Boot 2.0 默认开启了 CGLIB，注入时按实际类名匹配不成功。解决：要么加接口，要么统一用按类型注入。另一个踩坑：CGLIB 代理无法代理 final 方法，导致某些使用 final 的 @Transactional 方法事务失效，且没有任何警告日志。排查极其困难。

0x02-02. @Transactional 失效的常见场景有哪些？请列出至少 5 种

标准答案：
@Transactional 基于 Spring AOP 实现，本质是通过代理对象拦截方法调用。以下场景会导致事务失效：

1. 自调用（Self-Invocation）

2. 私有方法

CGLIB 是基于继承的，私有方法不可见无法被覆盖；JDK 代理基于接口，私有方法不在接口中。

3. 异常类型不匹配

4. 异常被捕获

5. 不同 Propagation 类型的影响

6. 数据库引擎不支持事务
MySQL MyISAM 引擎不支持事务，即使在 Service 上加了 @Transactional 也不生效。InnoDB 才支持。

7. 多数据源没有正确配置 TransactionManager
多个 DataSource 时，@Transactional 默认用主事务管理器，如果方法操作的数据库是另一个 DataSource，事务不会在该数据源上开启。

8. final 方法

深入思考：

  • 自调用的根本原因：Spring AOP 是代理模式，只有在调用代理对象的方法时才触发拦截逻辑。内部方法调用 this.save() 中的 this 指向原始 Bean 对象，不是代理对象。解决方案：① 注入自身代理（@Autowired private UserService self; 然后调用 self.save()）；② 拆到另一个 Service 类（推荐）；③ 用 AopContext.currentProxy()（需配置 exposeProxy）。
  • Spring 5.x+ 事务异常回滚策略：默认只回滚 RuntimeException 和 Error，不回滚 checked exception。可以通过 rollbackFor 属性指定：@Transactional(rollbackFor = Exception.class)。
  • 源码验证：TransactionInterceptor.invoke() → 调用 TransactionAspectSupport.invokeWithinTransaction() → 判断 txAttr.rollbackOn(exception) → 默认委托给 DefaultTransactionAttribute.rollbackOn() → 返回 ex instanceof RuntimeException || ex instanceof Error。

🔥 生产踩坑：一个支付回调接口中，方法内调用 this.updateOrderStatus()（@Transactional），外部方法包裹 try-catch 捕获了所有异常返回成功给支付网关。导致：内部事务异常被捕获不回滚，但支付回调以为成功了。后果：订单状态不一致，用户未扣款但订单显示已支付。排查结果：自调用 + 异常捕获双重踩坑。修复：拆出事务方法到独立 Service 类 + 不吞异常。

标准答案：
ProxyFactoryBean 是 Spring 1.x 时代的 AOP 实现方式，需要 XML 配置指定 Target、拦截器和接口。AspectJ 是更现代的切面表达方式。

维度 ProxyFactoryBean AspectJ 注解

切入点表达 名称匹配简单 Pointcut 表达式语言强大

深入思考：

  • AspectJ 编译时织入 vs 运行时织入：Spring AOP 默认是运行时织入（通过代理），AspectJ 支持编译时织入（ajc compiler）和类加载时织入（Load-Time Weaving, LTW）。编译时织入不需要代理，性能最强，也不需要接口。但配置复杂（需修改 Maven/Gradle 构建插件）。
  • 什么时候需要 AspectJ 编译时织入？：当需要切面应用到 final 类/方法、需要内部方法调用触发切面（自调用问题）、或对性能有极致要求（零反射开销）时。
  • @EnableAspectJAutoProxy 的 proxyTargetClass 和 exposeProxy 属性。

🔥 生产踩坑：一个高并发系统使用 Spring AOP 记录方法耗时，发现性能监控本身带来了 15% 的性能损耗——代理 + 反射调用在高频方法下很可观。最终对核心热点方法（每秒调用 10 万+）切换到 AspectJ 编译时织入。但迁移过程中发现构建流程改造成本极大，最终选择只对最热门的 5 个方法做了手动字节码增强（Javassist），保持其他方法用 Spring AOP。

0x03 事务管理

0x03-01. 请详细解释 @Transactional 的传播行为（REQUIRES_NEW vs NESTED 的陷阱）

标准答案：
Spring 定义了 7 种事务传播行为：

传播行为 说明
REQUIRED（默认） 支持当前事务，不存在则新建
SUPPORTS 支持当前事务，不存在则不开启事务
MANDATORY 必须已有事务，否则抛异常
REQUIRES_NEW 挂起当前事务，新启一个独立事务
NOT_SUPPORTED 挂起当前事务，非事务执行
NEVER 非事务执行，有事务则抛异常
NESTED 嵌套事务，如果有事务则在 Savepoint 内执行

REQUIRES_NEW 详解：

结果：T2 回滚（child 插入回滚），T1 不受影响（parent 插入不受影响，继续执行至完成并提交）。但 T1 中调用 child 之后的代码会继续执行，因为异常从 child 抛到 T1，是否回滚取决于 T1 的异常处理。

陷阱：如果 child 抛异常且 T1 不 catch，T1 也会回滚。这不是 REQUIRES_NEW 的失效，而是异常传播到 T1 的后果。

NESTED 详解：

结果：child 回滚到 Savepoint（不影响 T1 已执行的 insertParent），T1 继续执行。关键区别：REQUIRES_NEW 是挂起 T1 开启独立物理事务（两阶段提交开销大），NESTED 是使用同一个数据库连接的 Savepoint（无独立事务开销，但 JDBC Savepoint 只在支持嵌套事务的数据库上有效）。

深入思考：

  • JDBC 驱动限制：REQUIRES_NEW 要求底层支持事务挂起/恢复（大部分连接池都支持），而 NESTED 依赖 JDBC 3.0 的 Savepoint 机制。MySQL InnoDB 支持 Savepoint，但 Oracle 的 JDBC 驱动可能行为不一致。
  • NESTED 的”不是真正嵌套事务”：NESTED 回滚到 Savepoint 后，T1 仍然可以继续提交，但子事务的修改被回滚。但 NESTED 不支持两个事务独立提交——如果 T1 最终回滚，子事务的修改（即使没抛异常）也会一起回滚。
  • 事务挂起开销：REQUIRES_NEW 每次切换要 suspend/resume 事务上下文（TransactionSynchronization 列表的保存和恢复），频繁使用会导致性能下降。建议只在真正需要独立事务的场景使用 REQUIRES_NEW。
  • 自调用与传播行为：同一个类中 parent → child（REQUIRES_NEW）不会触发传播行为切换——因为自调用不走代理。

🔥 生产踩坑：一个批量订单处理任务中，外层事务逐条处理订单，每条调用 REQUIRES_NEW 子事务。开发预期是某条失败不影响其他订单。上线后发现数据库连接很快被耗尽（每条订单都需要一个新连接，旧事务被挂起但连接不释放）。排查发现：REQUIRES_NEW 在新事务开启时，原连接不能归还到连接池。如果连接池最大 20 个，同时处理 10 条订单就可能耗尽连接。解决方案：要么用 NESTED（同一个连接，Savepoint 回滚），要么改用编程式事务逐条控制。另一个大坑：有人把 NESTED 理解为”两个独立事务可以部分提交”，测试时发现子事务回滚后父事务提交，但子事务的修改还在——实际上 NESTED 回滚到 Savepoint，子事务插入的数据确实不存在了，但父事务其他修改仍在。这和独立事务的概念不同。

0x03-02. 声明式事务 vs 编程式事务，如何选择？

标准答案：

维度 声明式事务（@Transactional） 编程式事务（TransactionTemplate）
实现方式 AOP 拦截 + 声明注解 手动 TransactionTemplate.execute()
代码量 少（一行注解） 多（需要模板代码）
细粒度控制 粗（方法级别） 细（代码块级别，可以部分回滚）

异常处理 自动回滚策略 精确控制回滚时机
调试难度 低（声明式清晰） 中（代码分散）
可测试性 需代理 好（直接传 TransactionTemplate mock）
多事务组合 单一方法只能一种传播行为 一个方法内可以开多个独立事务

选择策略：

  • 80% 场景用声明式事务：简单 CRUD、单个操作、不需要精确回滚控制
  • 20% 场景用编程式事务：① 一个方法内需要多个独立事务（如批量处理各条独立回滚）；② 需要根据运行时条件决定回滚；③ 事务代码块中混合了缓存操作/消息发送等非事务资源；④ 精细控制事务超时/只读属性

  • 声明式事务的 AOP 原理：@Transactional 由 TransactionInterceptor 处理，运行时判断是否有事务注解，然后调用 TransactionAspectSupport.invokeWithinTransaction()。该方法创建 TransactionInfo（绑定当前线程的事务上下文），执行目标方法，然后 commit/rollback。
  • 编程式事务的内部逻辑：TransactionTemplate.execute() 内部调用 TransactionCallback → 成功则 commitTransactionAfterReturning() → 异常触发 completeTransactionAfterThrowing() → 判断 rollbackOn 决定是否回滚。
  • TransactionTemplate 的隔离级别：可设置 txTemplate.setIsolationLevel(TransactionDefinition.ISOLATION_REPEATABLE_READ)，可每次调用动态修改。

🔥 生产踩坑：一个数据同步服务，每次同步 1 万条记录。用声明式事务在方法级别加 @Transactional，一条失败全部回滚，不但浪费了之前处理成功的记录，还因长事务锁表导致线上查询堵塞。解决：改为编程式事务，每条记录独立事务提交。性能：从”全成功 60 秒 / 一条失败 0 秒”变为”1 万条 65 秒，每条独立回滚”。教训：批量操作慎用声明式事务。

深入思考：

  • MySQL 的 REPEATABLE_READ 如何避免幻读？：InnoDB 通过 Next-Key Lock（记录锁 + 间隙锁的组合）锁定索引记录之间的间隙，防止其他事务插入新记录。同时 MVCC 的 Read View 机制确保同一事务内的多次查询看到的是同一个快照。所以 MySQL 的 REPEATABLE_READ 实际上达到了 SERIALIZABLE 的”无幻读”效果，但并发性能好得多。

  • 分布式事务的隔离级别：XA 分布式事务、Seata AT 模式等分布式事务框架的隔离级别与数据库级别不同，它们有自己的隔离机制（如 Seata 的写隔离 + 读隔离通过全局锁实现）。
  • isolation 在 @Transactional 中的配置：Spring 枚举 Isolation.DEFAULT（使用数据库默认）、READ_COMMITTED、REPEATABLE_READ、SERIALIZABLE。注意：Isolation.DEFAULT 表示使用数据源的默认级别，不是 Spring 的默认。

🔥 生产踩坑：某金融系统中对账户余额做 UPDATE 操作，用了默认的 REPEATABLE_READ 隔离级别。高并发下发生死锁（两个事务互相等待对方释放间隙锁）。通过 SHOW ENGINE INNODB STATUS 分析死锁日志，发现 REPEATABLE_READ 的 Next-Key Lock 导致非主键索引上的间隙锁冲突。解决方案：① 降低隔离级别到 READ_COMMITTED（大多数金融场景够用，因为用了乐观锁 version）；② 确保 UPDATE 走唯一索引，减少间隙锁范围。③ 加 @Transactional(isolation = Isolation.READ_COMMITTED) 替换默认。”降级”不可怕，”死锁”才可怕。

0x03-04. 什么是事务超时和只读事务？@Transactional(readOnly=true) 真的只是读吗？

标准答案：
事务超时（timeout）：指定事务允许执行的最大时间（秒），超时则自动回滚。

只读事务（readOnly）：提示数据库当前事务只进行读取操作，可以进行一些优化。

readOnly=true 的实际作用：

  1. MySQL（InnoDB）：readOnly=true 会设置 SET TRANSACTION READ ONLY，允许数据库跳过写操作相关的锁（如 undo log 记录），减少内部开销。但不会阻止你执行 INSERT/UPDATE/DELETE（数据库层面不会报错，只是性能上”不优化写路径”）。
  2. SQL Server：会设置 SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED 或 SNAPSHOT 类似物。
  3. Oracle：实际上会禁止写操作——如果尝试写入会抛异常。
  4. JPA/Hibernate：readOnly=true 还会禁用脏检查（flush mode 设为 MANUAL），对未修改的实体不会执行 SELECT FOR UPDATE。
  5. Spring Data JPA + readOnly=true：Spring 会设置 FlushMode.MANUAL，延迟加载的懒属性在事务外访问会抛 LazyInitializationException（因为 Session 关闭了）。

深入思考：

  • readOnly 的真正价值：不是”阻止写操作”，而是向底层基础设施传达”这是一个读操作”的信号，让数据库/ORM 层去做优化。真正阻止写操作需要数据库权限层面限制。
  • 性能优化：在 MySQL 中 readOnly 事务可以减少 trx_sys 的锁竞争，因为只读事务不需要分配事务 ID（trx_id），直到第一次写操作才会分配。在高并发读场景下（每秒数万次读），readOnly=true 可以降低 5-15% 的事务元数据开销。
  • @Transactional(readOnly=true) 放在 Service 类上：如果类上标了 @Transactional(readOnly=true)，其内部有写操作的方法需要单独覆盖：@Transactional(readOnly = false)。

🔥 生产踩坑：某报表查询接口加了 @Transactional(readOnly=true)，使用了懒加载的关联实体集合。返回 JSON 时 Jackson 序列化触发了懒加载，但因为事务已提交，Session 已关闭 → LazyInitializationException。解决：要么用 JOIN FETCH 提前加载，要么配置 spring.jpa.open-in-view=true（不推荐——会导致事务时间过长）。另一个踩坑：测试环境 readOnly=true 的方法居然成功插入了数据（MySQL 不禁止），但某天为 Oracle 做兼容性测试时炸了。教训：readOnly 的语义在不同数据库间不可移植。

  • RabbitMQ/Kafka 监听器：消息消费在虚拟线程

Tomcat 线程模型变化：

🔥 生产踩坑：虚拟线程下的三大陷阱

案例 1：synchronized 导致虚拟线程 pinned（某电商支付服务）

  • 场景：Spring Boot 3.2 + JDK 21 虚拟线程，支付服务 50 台集群
  • 症状：开启虚拟线程后吞吐量下降 40%，线程 dump 显示大量虚拟线程处于 CARRIER 状态
  • 排查：
  • 根因：支付模块在 synchronized 块内调用了远程服务——虚拟线程被 pinned 到 carrier 线程，3s 的 RPC 调用占用了 carrier 线程
  • 修复：将 synchronized 替换为 ReentrantLock
  • 修复效果：吞吐量恢复，P99 延迟从 800ms 降至 200ms

案例 2：ThreadLocal 内存泄漏（某内容推荐服务）

  • 场景：使用 ThreadLocal 存储请求追踪 ID，开启虚拟线程后出现 OOM
  • 根因：虚拟线程池不回收线程，ThreadLocal 值在虚拟线程生命周期内一直存在。百万级请求创建百万级虚拟线程、百万个 ThreadLocal 实例
  • 修复：
  • 最佳实践：
    • 虚拟线程场景下完全禁止 ThreadLocal（JDK 官方也建议迁移到 ScopedValue）
    • 如果必须用，在 finally 块中显式 remove()

案例 3：池化资源（连接池/线程池）与虚拟线程不兼容（某社交平台）

  • 场景：数据库连接池 HikariCP，每个请求获取连接后等待数据库响应
  • 症状：数据库连接池被占满，新的请求等待连接超时
  • 根因：传统模式下 200 个平台线程共用 50 个连接，现在 10 万个虚拟线程同时竞争 50 个连接——连接池被击穿
  • 修复：调大连接池 + 缩短连接持有时间
  • 最佳实践：虚拟线程场景下连接池需要调大 2-3 倍，并设置更短的超时时间

Virtual Threads 兼容性审计清单：

  • synchronized 块/方法 → 替换为 ReentrantLock / StampedLock
  • ThreadLocal 使用 → 替换为 ScopedValue（JDK 21+）
  • 池化资源（连接池、线程池）→ 调大 2-3 倍
  • ThreadPoolTaskExecutor → 移除自定义线程池（虚拟线程用默认即可）
  • 第三方库兼容性 → 检查依赖是否支持虚拟线程（Netty/Undertow 默认支持）
  • 监控指标 → 检查虚拟线程活跃数、队列深度、pinned 次数

完整优雅关闭方案（K8s 环境）：

最佳实践要点：

  1. -XX:MaxRAMPercentage=75.0 — 给 JVM 分配容器内存的 75%，剩余 25% 留给操作系统和监控 Agent
  2. -XX:+UseZGC — JDK 21+ 的 ZGC 默认是分代模式，延迟 < 1ms，比 G1 更适合容器环境
  3. resources.requests == resources.limits — 在 Pod 级别设置相等的 request 和 limit（Guaranteed QoS），防止 OOM Kill 导致的抖动
  4. requests.memory 不应小于 256Mi——Spring Boot + JVM 本身需要约 150-200MB 基础内存
  5. 启用 ExitOnOutOfMemoryError——OOM 时快速失败，让 K8s 重启 Pod，而非停留在半死状态

    1. 升级基础镜像到 JDK 17+（JDK 17 已完整支持 Cgroup v2）
    2. 显式指定 -XX:MaxRAMPercentage=75.0 作为兜底
    3. 在 K8s Pod 的 annotation 中确认 cgroup 版本：
    1. 验证方式：进入 Pod 执行 cat /sys/fs/cgroup/memory.max 确认值与 limit 一致

方案 实现方式 优缺点 适用场景

生产实践：

  • 内部服务用 Header 版本化，外部 API 用 URI 版本化
  • SB4 内置方案适合全新微服务；存量系统应保持现有方案
  • 版本化不等于无限兼容——建议最多维护 3 个活跃版本，N-2 规则
