后端面试题：Spring Cloud Alibaba 核心原理与生产避坑（2026版）

发表于 2026-06-07 分类于 后端

本文整理了 Spring Cloud Alibaba 生态中高频出现的核心原理面试题、两个线上经典事故的排查复盘，以及一个网关+配置中心联动场景的实战设计。适合中高级 Java/微服务工程师准备面试。

0x00 面试题

1. Nacos 同时支持 AP 和 CP 模式，什么场景选哪个？和 Eureka / Consul 的对比？

Nacos 是目前国内微服务体系的事实标准注册中心。它与 Eureka 最大的区别在于 Nacos 支持 临时实例（AP） 和 持久实例（CP） 两种模式，可以在一致性与可用性之间按需切换。

AP 模式（临时实例） —— 默认行为，对应服务发现场景：

  • 实例通过心跳续约（默认 5s），15s 无心跳则剔除
  • 节点间采用 Distro 协议异步同步，强调可用性
  • 类似 Eureka 的自保护模式，网络分区时优先保证能注册和发现

CP 模式（持久实例） —— 对应配置管理/DNS 场景：

  • 使用简化的 Raft 协议，写操作必须经过 Leader
  • 网络分区时少数派拒绝写，保证数据一致性
  • 类似 Consul 的强一致性模式

一句记住：Nacos = Eureka（服务发现）+ Config（配置中心）的合体，但 Nacos 2.x 后通信协议从 HTTP 升级为 gRPC，大幅降低了心跳压力，单机可支撑 10w+ 实例。

参考：Nacos 官方架构文档 / 掘金《Nacos 2.x 架构演进》

2. Sentinel 的滑动窗口限流是怎么工作的？和 Hystrix 有什么本质区别？

Sentinel 的 QPS 限流基于滑动窗口计数器实现，核心数据结构为 LeapArray：

  • 将 1 秒时间窗口切分为 N 个格子（默认 2 格，每格 500ms）
  • 请求到来时计算当前窗口内所有格子的累计通过数
  • 超过阈值则触发限流（快速失败 / Warm Up / 排队等待三种效果）

与 Hystrix 的本质区别：

维度 Sentinel Hystrix
线程模型 信号量隔离（无额外线程开销） 线程池隔离（额外线程开销大）
规则推送 控制台实时下发，支持动态规则 静态配置，修改需重启
熔断策略 慢调用比例 / 异常比例 / 异常数 三种 仅异常比例
扩展性 SPI 插件机制，支持规则动态数据源 接口实现，灵活性较低
维护状态 活跃维护（阿里开源） 停止维护，进入维护模式

一句话：Sentinel 是”规则驱动的流量整形工具”，Hystrix 是”资源隔离的熔断工具”。2026 年选型优先 Sentinel，Hystrix 仅遗留系统还在跑。

参考：Sentinel Wiki / 牛客网《微服务治理面试高频考点》

3. Seata 分布式事务 AT 模式和 TCC 模式怎么选？各自有什么坑？

Seata 提供四种分布式事务模式，最常用的是 AT 和 TCC：

AT 模式（自动补偿） —— 适合无外部依赖的纯数据库操作：

  • 一阶段：拦截 SQL，解析出前后镜像（undo_log），把业务 SQL + undo_log 在同一本地事务提交
  • 二阶段：成功则异步删 undo_log；失败则根据 undo_log 反向生成回滚 SQL
  • 核心前提：必须用支持 ACID 的关系型数据库，且 SQL 能被 Seata 正确解析

TCC 模式（手动补偿） —— 适合有外部 RPC/缓存/MQ 的复杂场景：

  • Try：资源预留和检查
  • Confirm：真正执行业务，幂等
  • Cancel：释放预留资源，幂等
  • 需要业务方实现三个接口，代码侵入性强

选型决策：

  • 纯数据库操作 → AT（零侵入，前提是 SQL 可解析）
  • 涉及 Redis / MQ / 第三方 RPC → TCC（必须业务自己写补偿）
  • 长事务（分钟级） → SAGA 或最终一致性消息（避免长时间锁表）

参考：Seata 官方文档 / 掘金《Seata AT 模式踩坑记》

现象：服务重启后，消费方连续 10 秒内大量调用报错 java.net.ConnectException: Connection refused。

Bug 2：Seata AT 模式下全局锁等待超时，事务被无限回滚

现象：某下单接口偶发超时，日志显示 LockConflictException: get global lock fail，然后不断重试直到超时回滚，但下一个请求又遇到同样的锁冲突。

原因：两个全局事务分别操作 order 表和 inventory 表，但执行顺序相反——事务A先锁 order 后锁 inventory，事务B先锁 inventory 后锁 order，形成死锁。Seata 的全局锁等待超时（默认 10s）后回滚，但如果请求量大，回滚后立即有新请求进来再次死锁。

0x03 小结

Spring Cloud Alibaba 面试的核心不是背题，而是理解三个”为什么”：

  1. Nacos 为什么选 AP 做服务发现？ — 可用性优先，服务列表的短暂不一致通常不致命
  2. Sentinel 为什么用信号量而不用线程池？ — 降低资源开销，适合高 QPS 网关场景
  3. Seata AT 为什么要解析 SQL？ — 只有拿到前后镜像才能自动生成回滚 SQL，这是它”零侵入”的代价

掌握这些”为什么”，比记住配置参数重要得多。面试官要的不是运维手册，是你能在设计选型时说出 trade-off。
