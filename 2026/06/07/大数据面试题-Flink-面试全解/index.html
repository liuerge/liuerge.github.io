大数据面试题：Flink 面试全解

发表于 2026-06-07 分类于 大数据

Flink 高级面试全解，覆盖 Checkpoint 与状态后端、Window 窗口机制、反压原理、Exactly-Once 语义和水位线。面向高级研发。

0x00 Checkpoint 与状态

1. Flink Checkpoint 的 Chandy-Lamport 算法原理？

Flink 的 Checkpoint 基于 Chandy-Lamport 分布式快照 的变种（异步屏障快照）：

流程：

  1. JobManager 向所有 Source 注入 Checkpoint Barrier
  2. Barrier 随数据流向下游传播，将流分为”Barrier 之前的记录”和”之后的记录”
  3. 算子收到所有上游分区的 Barrier 后，触发状态快照（异步写入状态后端）
  4. 所有算子完成快照后，JobManager 确认 Checkpoint 完成

对齐（Alignment）：

深入思考：

  • 源码层面：CheckpointBarrierHandler 负责 Barrier 对齐逻辑。CheckpointedInputGate 在非对齐模式下直接转发数据而不缓存
  • 生产经验：Flink 1.15+ 默认启用非对齐 Checkpoint，大状态场景 Checkpoint 时间平均缩短 50%+

参考：Flink Checkpoints

2. Flink 三种状态后端有什么区别？

状态后端 存储位置 适用场景 限制
HashMapStateBackend JVM Heap 小状态、低延迟 受 JVM 堆大小限制
EmbeddedRocksDBStateBackend 本地磁盘（RocksDB） 大状态（TB 级） 每次读写涉及序列化
FsStateBackend 已废弃 — —

选型决策：

  • 每个算子状态 < 几百 MB → HashMapStateBackend（纯内存，零序列化）
  • 状态 > GB 级别 → RocksDB（增量 Checkpoint 是杀手锏）

Session Window 按活动间隙切分：EventTimeSessionWindows.withGap(Time.minutes(30))

4. Watermark 是怎么生成的？如何处理迟到数据？

三层迟到处理：

  1. Watermark 容错（forBoundedOutOfOrderness）— 乱序容忍
  2. allowedLateness — 窗口触发后仍可追加
  3. sideOutputLateData — 最终兜底，不会丢

0x03 反压 (Backpressure)

5. Flink 如何检测和处理反压？

检测原理（Flink 1.13+）：

反压传播：下游 BufferPool 满 → 上游无法写入 → 上游 BufferPool 满 → 继续向上传播

定位：

  • Flink Web UI → 反压指标标签：OK(绿) / LOW(黄) / HIGH(红)
  • HIGH 表示该算子的输出 Buffer 使用率 > 50%

常见原因和修复：

原因 修复
Sink 写入慢（如 MySQL） 增加 Sink 并行度、批量写入
数据倾斜 keyBy 前加随机前缀打散
Checkpoint 对齐 启用非对齐 Checkpoint
外部服务慢 异步 IO (AsyncDataStream)
