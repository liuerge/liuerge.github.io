为什么 Hooks 不能放在条件语句里？
因为 Hooks 靠调用顺序索引。如果某次渲染跳过一个 Hook，所有后续 Hook 的索引都会错位，导致状态混乱。

4. useMemo、useCallback、React.memo 什么时候用？

不滥用原则：只在确实有性能问题时才用。useMemo 本身有开销（比较依赖数组），对简单计算反而更慢。

深入思考：

  • 传给子组件的回调函数 → 必须用 useCallback，否则子组件的 React.memo 形同虚设（每次父组件渲染都会生成新函数引用）

0x03 状态管理

5. Context 和状态管理库（Zustand/Jotai）怎么选？

方案 适用场景 缺点
useState 组件内部状态 跨组件共享需 prop drilling
Context 主题/语言/认证等全局低频变更 任何 Context 值变化，所有消费者重渲染
Zustand 中大型应用全局状态 需要理解 selector 机制
Jotai 原子化状态，细粒度更新 原子多了难管理

Context 优化：拆分 Context 或使用 useMemo 包裹 value，避免不必要的重渲染。

0x06 React 19 新特性（2024-2025 热点）

8. React 19 Actions 是什么？useActionState / useFormStatus / useOptimistic 分别怎么用？

标准答案：

React 19（2024年12月正式发布）引入了 Actions 概念——一种简化数据突变（mutation）和表单处理的范式，核心是三个新 Hook：

1. useActionState — 管理异步 Action 的状态

2. useFormStatus — 在子组件中读取父表单的状态

3. useOptimistic — 乐观更新

Actions 的核心能力：

  • 自动管理 pending 状态（无需手动 setIsPending）
  • 与 Error Boundary 集成（Action 抛出异常自动触发 Error Boundary）
  • <form action={fn}> 直接支持（<form>, <input>, <button>）

参考：React v19 Blog

面试金句：Actions 将 React 从 “UI 库”变成了 “UI + 数据流” 的框架——你不再需要手动编排 loading/error/success 三态，React 19 帮你做完了。

9. React Server Components (RSC) 和 Server Actions 的协作模式是什么？

标准答案：

React Server Components（RSC，React 19 中稳定化）和服务端 Actions 构成了 React 全栈开发的核心范式：

Server Components：组件在服务端运行，可以直接访问数据库/文件系统，无需 API 层

Server Actions：在服务端执行的函数，客户端可以直接调用

RSC + Server Actions 协作模式：

关键规则：

  • 'use server' — 标记文件/函数在服务端执行
  • 'use client' — 标记组件在客户端执行（需要交互）
  • Server Component（默认）不能使用 useState/useEffect/onClick 等客户端 Hook

面试金句：RSC 消除了传统的 “前端 → API → 后端” 三层架构——你现在可以在一个组件文件里完成从数据库到 UI 的全过程。代价是服务端成本上升（需要 Node.js 运行时支撑渲染）。

参考：React Server Components

10. React Compiler（React Forget）解决了什么问题？原理是什么？

标准答案：

React Compiler（原名 React Forget，2024年开源，2025年逐步成熟）是一个构建时编译器，自动为组件和 Hook 添加 useMemo/useCallback/React.memo 级别的优化——开发者不再需要手动写这些优化。

解决的问题：

工作原理：

  1. 编译器分析组件内部的数据流 —— 识别哪些值是”衍生”的（如 filteredItems）、哪些函数”捕获了外部变量”
  2. 对衍生值自动生成 useMemo 缓存
  3. 对传递给子组件的函数自动生成 useCallback 稳定引用
  4. 对纯展示子组件自动添加 React.memo

React Compiler 的局限：

  • 不能违反 Rules of React（Hooks 调用顺序必须一致、不能在条件中调用 Hooks）
  • 不能优化真正的”昂贵计算”（如 10000 项的排序）——这类性能瓶颈需要算法层面优化
  • 目前为 Beta 状态，推荐先在部分页面验证后逐步推广

面试金句：React Compiler 要做的是 “编译器懂得你需要什么优化，比你手动写得更准确”。它不会让你写出更快的代码，但会让你写出更正确、更简洁的代码——而”正确的不必要重渲染消除”通常已经足够快。

参考：React Compiler 官方文档

12. React 19 中 ref 作为 prop 和 forwardRef 废弃意味着什么？

标准答案：

React 19 最受开发者欢迎的简化之一：函数组件可以直接接收 ref prop，不再需要 forwardRef。

这意味着什么？

  1. forwardRef 被废弃（不会删除，但不推荐使用）
  2. TypeScript 类型更简单：不再需要区分 Props 和 Ref 类型参数
  3. 组合更自然：HOC 不再需要用 forwardRef 透传 ref
  4. 与 Server Components 的兼容性：forwardRef 无法用于 Server Component，而 ref as prop 可以

深入思考：

  • 迁移注意：React 19 中 forwardRef 仍然可用但会在 DevTools 中显示弃用警告。社区库（如 react-hook-form、Radix UI）正在逐步迁移
  • 类型推断改进：React 19 配合 TypeScript 5.5+ 可以更好地推断 ref 类型，不再需要手动标注 React.Ref<HTMLInputElement>
  • ref 回调函数：ref 作为 prop 同样支持回调函数形式（ref={(el) => ...}），与 createRef/useRef 无行为差异

参考：React v19 - ref as a prop
