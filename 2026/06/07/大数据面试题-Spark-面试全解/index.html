大数据面试题：Spark 面试全解

发表于 2026-06-07 更新于 2026-06-08 分类于 大数据

Spark 高级面试全解，覆盖 Shuffle 原理、内存管理、Spark SQL 优化和流处理。面向高级研发。

0x00 核心架构

1. Spark 的 Job、Stage、Task 是怎么划分的？

  • Job：一个 Action 操作触发一个 Job
  • Stage：遇到 Shuffle 依赖（宽依赖）时切分。Stage 内全是窄依赖（pipeline 执行）
  • Task：Stage 内每个分区对应一个 Task。Task 数 = Stage 最后一个 RDD 的分区数

深入思考：

  • 生产经验：Shuffle 是 Spark 的最大瓶颈。减少 Shuffle 次数是最有效的优化手段——能用 reduceByKey 不用 groupByKey

参考：Spark 调度文档

2. Spark Shuffle 的三种实现及演进？

版本 实现 特点
Spark 1.x HashShuffle 每个 Task 为每个下游分区创建文件，M×N 个文件。不实用
Spark 1.2+ SortShuffle 每个 Task 写一个文件 + 索引。文件数降为 M
Spark 2.0+ Tungsten-Sort 堆外内存 + 指针排序，避免 GC。默认启用

0x02 内存管理

3. Spark 统一内存模型是怎样的？

深入思考：

  • 生产经验：OOM 排查第一步——看是 Execution Memory 还是 User Memory 溢出。(1) Execution OOM → 增加分区数、减少单分区数据量；(2) Storage OOM → 用 MEMORY_AND_DISK_SER 溢出到磁盘；(3) User Memory OOM → 减少 UDF 中创建的大对象

0x03 Spark SQL 优化

4. DataFrame 比 RDD 为什么快？

  1. Catalyst 优化器：逻辑计划 → 优化 → 物理计划 → 代码生成
  2. Tungsten 执行引擎：堆外内存 + 直接操作二进制数据
  3. 列式存储：只读需要的列，减少 IO 和内存
  4. Whole-Stage Code Generation：把多个算子编译成一个函数，减少虚函数调用

5. 数据倾斜怎么解决？

发现倾斜：某个 Task 处理时间远超其他（Spark UI 可直观看到）

分场景解决：

场景 方案
GroupBy 倾斜 两阶段聚合（加盐去盐）
Join 倾斜 广播小表（Broadcast Join）
大表 Join 大表倾斜 倾斜 key 单独处理

0x04 流处理

6. Structured Streaming 的 Exactly-Once 怎么保证？

Structured Streaming 通过 WAL（Write-Ahead Log）+ 幂等 Sink 保证端到端 Exactly-Once：

  1. Source：Kafka offset 存储在 Checkpoint
  2. State：状态通过 Checkpoint 持久化到 HDFS/S3
  3. Sink：foreachBatch 幂等写（如 upsert 到 MySQL）

0x06 内存管理进阶

7. Spark 内存管理模型

难度：⭐⭐
问题：Spark 3.x 的统一内存管理是如何分配内存的？

解析：
Spark 将 JVM 堆内内存分为三大区域（spark.memory.fraction=0.6）：

  • Storage Memory：spark.memory.fraction * spark.memory.storageFraction（默认 0.6 * 0.5 = 0.3）
    • 用途：缓存 RDD、Broadcast 变量
    • 可被 Execution 抢占
  • Execution Memory：同上，与 Storage 共享剩余 50%
    • 用途：shuffle、join、sort、aggregation 的中间缓冲区
    • 可抢占 Storage 的空间
  • Reserved Memory：300MB，保护 JVM OOM

动态占用：

  • Execution 可占用 Storage 的闲置空间，反之亦然
  • 如果 Storage 需要空间而 Execution 正在使用，不会被强制回收（避免性能退化）

0x07 流处理对比与演进

8. Spark Streaming 与 Structured Streaming

难度：⭐⭐⭐
问题：Spark Streaming（DStream）和 Structured Streaming 的区别？Structured Streaming 的”端到端 Exactly-Once”是如何实现的？

解析：
区别：

维度 Spark Streaming Structured Streaming
语义模型 DStream（离散化流） 无界 DataFrame
时间语义 基于批处理间隔 Event Time 原生支持
一致性 At-Least-Once Exactly-Once（提交式 Sink）
延迟 秒级（mini-batch） 毫秒级（Continuous Processing）
Watermark 需要手动实现 内置 Watermark 支持
API DStream API DataFrame/SQL API

Exactly-Once 实现：

  1. Source：Kafka 偏移量记录在 checkpoint
  2. 处理：幂等计算 + 预写日志（WAL）
  3. Sink：幂等写入（如 Kafka 的 idempotent producer）或事务写入（如 FileSink 的 _SUCCESS 文件）
  4. 失败恢复：从 checkpoint 恢复偏移量重新处理，通过 WAL + 幂等 Sink 保证结果不重复

10. Spark Structured Streaming 3（流式物化视图与异步状态管理）——Spark 4.x 流处理深度剖析

难度：⭐⭐⭐
问题：Spark 4.x 引入了全新的 Structured Streaming 3 流处理引擎。相比 Spark 3.x 的 Structured Streaming，Structured Streaming 3 在流式物化视图（Streaming Materialized View）、异步状态管理（Async State Management）和流式 Delta Join 方面有哪些重要改进？Spark 4.x 为何将 RocksDB 作为流处理状态存储的默认后端？

解析：

Structured Streaming 3 架构变革：

Structured Streaming 3 在 Spark 4.x 中进行了深度重构，核心变化如下：

特性 Spark 3.x Structured Streaming Spark 4.x Structured Streaming 3
状态后端 HashMapStateBackend（默认） RocksDBStateBackend（默认）
状态管理 同步状态访问（算子内串行） 异步状态访问（多线程并发）
物化视图 ❌ 不支持 ✅ 流式物化视图（持续更新）
Delta Join 普通 Stateful Join 流式 Delta Join（增量关联）
背压机制 Rate Controller 控制摄入 自适应 Backlog 反压
Exactly-Once WAL + 幂等 Sink WAL + 状态快照 + 幂等 Sink
Watermark 固定配置 可动态调整

流式物化视图（Streaming Materialized View）：

这是 Spark 4.x 最重磅的流处理特性。传统流处理中的聚合结果需要写入外部存储（如 Redis/ES），再供查询引擎读取，存在数据延迟和口径不一致问题。流式物化视图直接让 Spark 作为”实时物化视图引擎”，持续更新物化结果。

异步状态管理（Async State Management）：

Spark 3.x 中状态访问是同步的——每个微批处理批次中，状态读写都在单线程中串行执行，随着状态量增长（TB 级），状态访问成为瓶颈。Spark 4.x 引入了异步状态管理：

流式 Delta Join：

传统流式 Join 需要将完整的历史数据存储在状态中，随着时间推移状态无限膨胀。Streaming Delta Join 只存储需要参与 Join 的增量数据：

RocksDB 作为默认状态后端：

面试金句：

“Spark 4.x Structured Streaming 3 最核心的三个改进是：流式物化视图让 Spark 本身成为实时物化引擎，无需外部存储中转；异步状态管理消除了 TB 级状态下的访问瓶颈；RocksDB 作为默认状态后端使状态量不再受 JVM 堆限制。这三个改进让 Spark Streaming 在流批一体和实时数仓场景下的竞争力大幅提升，直接对标 Flink 的状态管理能力。流式物化视图尤其值得关注——它模糊了流处理和 OLAP 的边界，是 Lakehouse 架构中实时服务层的关键组成部分。”
