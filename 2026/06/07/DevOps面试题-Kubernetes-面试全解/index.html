0x00 核心概念

1. Pod 的生命周期和探针？

三种探针：

探针 作用 失败后果
livenessProbe 容器是否存活 重启容器
readinessProbe 容器是否就绪 从 Service 摘除
startupProbe 容器是否启动完成 阻塞 liveness/readiness

深入思考：

  • 生产经验：readinessProbe 失败不重启容器，只是摘流量。这让你可以在不重启的情况下做限流降级

类型 集群外可访问 实现

LoadBalancer ✅ 外部 LB IP 云厂商 LB → NodePort

iptables 模式流程：

5. Resource 的 requests 和 limits 怎么设？

QoS 等级：

  • Guaranteed：requests=limits（内存 CPU 都要），最高优先级
  • Burstable：requests < limits，中等优先级
  • BestEffort：没设 requests/limits，最低优先级，Node 压力大时先被杀

深入思考：

  • 生产经验：CPU limits 不要设太低，否则即使 Node 有空闲 CPU，容器也会被 throttle。CPU requests 影响调度，CPU limits 影响使用。内存 limits 是硬限制，超过就 OOM Kill

0x04 基础概念与架构

9. Kubernetes 是什么？和 Docker 是什么关系？

Kubernetes 是什么：
Kubernetes（K8s）是 Google 开源的容器集群管理系统，基于 Docker 技术为容器化应用提供部署运行、资源调度、服务发现和动态伸缩等功能。

Kubernetes 与 Docker 的关系：

工具 职责
Docker 容器的生命周期管理（构建、运行、停止），将应用打包为容器
Kubernetes 编排和管理多个主机上的容器，负责调度、扩缩容、服务发现

Docker 是「集装箱」，Kubernetes 是「港口调度系统」。

Kubernetes 的优势：

  1. 容器编排 — 自动化部署和管理
  2. 轻量级 — 基于容器技术
  3. 开源 — 广泛的社区和生态
  4. 弹性伸缩 — 根据负载自动扩缩容
  5. 负载均衡 — 内建服务发现和负载均衡

Kubernetes 的不足：

  1. 安装和配置相对复杂
  2. 管理服务相对繁琐
  3. 资源开销比其他容器管理方案高
  4. 对于简单应用可能过于复杂

出处：《Kubernetes 那点破事》— Tom哥

10. etcd 的特点和应用场景？

etcd 特点（CoreOS 开源，基于 Go 语言）：

  • 简单：REST 风格 HTTP+JSON API
  • 安全：支持 HTTPS 访问
  • 快速：支持并发 1k/s 写操作
  • 可靠：基于 Raft 一致性算法，支持分布式

主要应用场景：

场景 说明
服务发现 分布式集群中进程和服务如何找到对方并建立连接
消息发布与订阅 配置共享中心，数据提供者发布消息，订阅者实时接收
负载均衡 分布式架构信息存储支持负载均衡，每个节点都可处理请求
分布式通知与协调 利用 Watcher 机制，实现异步通知和实时处理
分布式锁 Raft 算法保证数据强一致性，可保持独占或控制时序
集群监控与 Leader 竞选 通过 etcd 实现简单且实时性强的监控

  • Minikube — 在本地轻松运行单节点 K8s 集群
  • Kubectl — 命令行工具，控制集群（创建/删除/更新资源、查看应用）
  • Kubelet — 每个节点上的代理服务，使 Worker 与 Master 通信

12. Kubernetes 集群架构是怎样的？

核心概念：

概念 说明
Master 集群管理节点，运行 API Server、Controller Manager、Scheduler，提供资源数据访问入口
Node (Worker) 运行 Pod 的服务节点，运行 Docker、kubelet、kube-proxy
Pod 最小调度单元，一个或多个相关容器的组合，共享网络命名空间和存储
Label Key/Value 键值对，附加到资源对象上，通过 Label Selector 查询筛选
ReplicationController 保证指定数量的 Pod 副本运行，是弹性伸缩和滚动升级的核心
Deployment RC 的升级版，可随时获知部署进度，支持滚动更新和回滚
HPA 横向自动扩容，根据 CPU 等指标自动调整 Pod 副本数
Service Pod 的逻辑集合和访问策略，统一入口 + 负载均衡
Volume Pod 中可被多个容器访问的共享目录
Namespace 多租户资源隔离，逻辑上分组项目/用户

出处：《Kubernetes 那点破事》— Tom哥

13. Kubernetes 集群有哪些核心组件？

组件 功能
API Server 集群入口，封装核心对象增删改查，RESTful API，各模块通信枢纽
Scheduler 为新 Pod 选择最优 Node，负责资源调度
Controller Manager 执行各种控制器，保证集群期望状态

— Node Controller 管理 Node，定期检查健康状态
— Namespace Controller 管理 Namespace，定期清理无效资源
— Service Controller 管理 Service，提供负载和服务代理
— Endpoints Controller 管理 Endpoints，关联 Service 和 Pod
— PV Controller 管理 PV/PVC 绑定和回收
— Daemon Set Controller 管理 DaemonSet，保证指定 Node 运行 Daemon Pod
— Deployment Controller 管理 Deployment，保证运行指定数量 Pod
— Job Controller 管理 Job，创建一次性任务 Pod
— Pod Autoscaler Controller 自动伸缩 Pod，定时获取监控数据执行伸缩

14. Replication Controller 和 Replica Set 的区别？

维度 Replication Controller (RC) Replica Set (RS)
选择器 基于权限（相等）的选择器 基于集合的选择器（支持 In、NotIn、Exists）
表达能力 弱 强
当前状态 已被 Deployment 替代 Deployment 内部使用 RS

RC 机制：RC 提交到集群后，Controller Manager 持续巡检目标 Pod，确保实例数量等于期望值。多余则停止，不足则创建。

出处：《Kubernetes 那点破事》— Tom哥

15. kube-proxy 原理：iptables vs IPVS？

kube-proxy 作用：运行在所有 Node 上，监听 Service 和 Endpoint 变化，创建路由规则提供服务 IP 和负载均衡功能。

iptables 模式（默认模式，K8s 1.2+）：

  • 通过 API Server Watch 接口跟踪 Service/Endpoint 变更
  • 更新 iptables 规则
  • 客户端请求通过 iptables NAT 机制直接路由到目标 Pod

IPVS 模式（GA 稳定版，K8s 1.11+）：

  • 使用 IPVS 专门做高性能负载均衡
  • 使用 ipset（带索引数据结构）替代线性 iptables 规则链
  • 规则多时依然高效
对比维度 iptables IPVS
定位 防火墙 高性能负载均衡
数据结构 线性规则链 Hash 表
扩展性 规则多时性能下降 几乎无限规模扩张
负载算法 仅随机 rr/lc/sh/dh 等（最小负载、加权等）
健康检查 不支持 支持健康检查和连接重试

出处：《Kubernetes 那点破事》— Tom哥

16. 创建和删除一个 Pod 的完整流程？

创建流程：

  1. 客户端提交 Pod 配置到 kube-apiserver
  2. apiserver 通知 controller-manager 创建资源对象
  3. controller-manager 将配置存入 etcd
  4. scheduler 检测到 Pod 信息，先过滤不符合条件的 Node，再优选打分，最后将 Pod 绑定到目标 Node
  5. 目标 Node 的 kubelet 根据资源配置单创建并运行 Pod

删除流程（优雅退出，默认 30s）：

  1. Pod 从 Service 的 Endpoint 列表中移除
  2. 如有 preStop 钩子，在 Pod 内部调用
  3. 进程收到 TERM 信号（kill -14）
  4. 超过优雅退出时间后，发送 SIGKILL（kill -9）

出处：《Kubernetes 那点破事》— Tom哥

17. Pod 重启策略有哪些？

策略 行为 适用控制器
Always（默认） 容器失效自动重启 RC、DaemonSet（必须）
OnFailure 退出码非 0 时重启 Job
Never 从不重启 Job

注意：Pod 重启策略应用于 Pod 内所有容器，仅在 Pod 所处 Node 上由 kubelet 判断执行。

出处：《Kubernetes 那点破事》— Tom哥

18. 什么是初始化容器（Init Container）？

Init Container 的运行方式与应用容器不同：

  • 必须先于应用容器执行完成
  • 多个 Init Container 按顺序逐个运行
  • 前一个成功运行后才能运行后一个
  • 所有 Init Container 成功后，Kubernetes 才初始化 Pod 并启动应用容器

适用场景：等待数据库就绪、初始化数据、权限设置等前置任务。

出处：《Kubernetes 那点破事》— Tom哥

19. Deployment 升级策略有哪些？

策略 说明
Recreate（重建） 先杀掉所有正在运行的 Pod，再创建新 Pod。有停机时间
RollingUpdate（滚动更新）（默认） 逐个更新 Pod，支持设置 maxUnavailable（最大不可用数）和 maxSurge（最大额外数）

RollingUpdate 参数：

  • maxUnavailable：更新过程中最多有多少个 Pod 不可用（默认 25%）
  • maxSurge：更新过程中最多可以超出期望副本数的 Pod 数量（默认 25%）

0x05 扩展机制与运维

24. Kubernetes Scheduler 的调度算法？

Scheduler 使用两步算法将 Pod 绑定到最优 Node：

① 预选（Predicates） — 过滤不满足条件的 Node

  • 输入：所有 Node
  • 输出：满足条件的 Node 列表
  • 过滤条件：资源不足（CPU/内存）、Node 标签不匹配、污点不兼容等

② 优选（Priorities） — 打分排序

  • 输入：预选后的 Node 列表
  • 输出：得分最高的 Node
  • 打分指标：资源富余度、负载均衡、亲和性规则等

出处：《Kubernetes 那点破事》— Tom哥

25. Kubernetes 如何保证集群安全性？

多维度安全机制：

维度 措施
基础设施 容器与宿主机隔离
权限最小化 各组件只执行授权行为，限制单个组件权限
用户权限 区分普通用户和管理员角色
API Server 认证 使用 HTTPS/Token 识别客户端身份
API Server 授权 建议使用 RBAC（基于角色的访问控制）
敏感数据 使用 Secret 机制保护敏感信息
准入机制 认证 → 授权 → 准入控制（Admission Control）

出处：《Kubernetes 那点破事》— Tom哥

26. Kubernetes 数据持久化方式？PV 和 PVC 是什么？

常见持久化方式：

类型 说明 生命周期
EmptyDir Pod 内容器共享目录，Pod 删除时数据清除 与 Pod 同生命周期
HostPath 挂载宿主机目录/文件 节点级别
PersistentVolume (PV) 底层网络存储的抽象资源（如 NFS、GFS） 独立于 Pod
PersistentVolumeClaim (PVC) 用户对存储资源的「申请」 与 Pod 绑定

PV 生命周期阶段：

  1. Available — 可用，未绑定 PVC
  2. Bound — 已绑定 PVC
  3. Released — PVC 删除，资源释放但未回收
  4. Failed — 自动回收失败

出处：《Kubernetes 那点破事》— Tom哥

27. Worker 节点如何加入集群？

  1. 在 Node 上安装 Docker、kubelet、kube-proxy
  2. 配置 kubelet 和 kube-proxy 启动参数，指定 Master URL
  3. 启动服务，kubelet 自动注册到集群
  4. Master 接受注册后自动纳入调度范围

出处：《Kubernetes 那点破事》— Tom哥

28. 什么是 Kubernetes Metric Service？如何用 EFK 管理日志？

Metrics Server（K8s 1.10+ 默认）：

  • 提供核心指标（Core Metrics），包括 Node 和 Pod 的 CPU/内存使用
  • 自定义指标（Custom Metrics）由 Prometheus 等组件完成

EFK 日志管理（Elasticsearch + Fluentd + Kibana）：

组件 职责
Elasticsearch 存储日志，提供查询接口
Fluentd 在每台 Node 上以 DaemonSet 方式运行，收集容器日志
Kibana Web GUI，浏览和搜索日志

日志采集原理：
Fluentd 挂载 /var/lib/docker/containers 和 /var/log 目录，在 /var/log/pods/ 下创建目录区分不同容器日志输出。

出处：《Kubernetes 那点破事》— Tom哥

29. 如何进行优雅的节点关机维护？

使用 kubectl drain <node-name> 将该节点上的 Pod 进行安全驱逐，然后再关机维护。

注意：直接关机可能导致 Pod 非正常终止，造成数据丢失或服务中断。

出处：《Kubernetes 那点破事》— Tom哥

30. Helm 是什么？有什么优势？

Helm 是 Kubernetes 的软件包管理工具，类似 Ubuntu 的 apt、CentOS 的 yum、Python 的 pip。

每个包称为一个 Chart（目录/压缩包，格式 name-version.tgz）。

Helm 优势：

  1. 统一管理、配置和更新分散的 K8s 应用资源文件
  2. 分发和复用应用模板
  3. 将应用的一系列资源当作一个软件包管理
  4. 使用者无需编写复杂部署文件，可查找、安装、升级、回滚、卸载应用

出处：《Kubernetes 那点破事》— Tom哥

0x06 Kubernetes 1.32 新特性与深度排障

31. K8s 1.32 的 DRA（Dynamic Resource Allocation）重构了什么？

标准答案：

K8s 1.32 彻底重构了 DRA（从 K8s 1.26 引入的原始实现），移除旧版 API，用 Structured Parameter 模型替代。核心变化：从”用户态兜底调度”变成”调度器可感知的声明式模型”。

深入思考：

  • 迁移影响：如果你在用旧版 DRA（Dynamic Resource Allocation with control plane controller），升级到 1.32 后需迁移到新 Structured Parameter 模型。旧 YAML 不兼容
  • Cluster Autoscaler 联动：新 DRA 的最大收益是 CA 可以读取 ResourceClaim 的 Structured Parameters，在扩容时自动选择有 GPU/NPU 等特殊硬件的 Node
  • 生产经验：如果集群单纯用 nvidia.com/gpu 这种传统 device plugin 方式管理 GPU，不受 DRA 重构影响。DRA 适用于需要更复杂设备分配逻辑的场景（如多 GPU 卡间拓扑感知）

参考来源：KEP-4381, Kubernetes 1.32 Release Notes

34. StatefulSet PVC 自动清理（KEP-1847）解决了什么运维痛点？

标准答案：

K8s 1.32 中 StatefulSet PVC 自动清理（Auto-Delete PVC）升级到 Stable。当 StatefulSet 缩容或 Pod 被删除时，关联的 PVC 可以根据策略自动清理：

策略 行为
whenScaled: Delete 缩容时自动删除多余 PVC
whenScaled: Retain（默认） 缩容时保留 PVC（手动处理）
whenDeleted: Delete 删除 StatefulSet 时级联删除 PVC
whenDeleted: Retain 删除 StatefulSet 时保留 PVC

深入思考：

  • 运维痛点：旧版 StatefulSet 缩容后，PVC 仍然存在（孤儿 PVC），需手动 kubectl delete pvc。生产环境 Kafka/ES 集群常有几十个孤儿 PVC
  • 安全权衡：whenScaled: Delete 是”高危”选项——数据随 PVC 一起删除！仅适用于数据可重建的场景（如 Kafka，数据有副本）
  • 生产经验：数据库类 StatefulSet（MySQL/ZK）建议 whenScaled: Retain + whenDeleted: Retain，由运维手动确认后再清理。流处理类（Kafka/Flink）可用 whenScaled: Delete

参考来源：KEP-1847, Kubernetes 1.32 Release Notes

35. K8s 1.32 的 Memory Manager GA 和 CPU Manager 严格预留有什么变化？

① Memory Manager GA：

Memory Manager 在 NUMA 拓扑感知的内存分配上正式稳定，保证 Guaranteed Pod 的内存分配在同一 NUMA 节点：

② CPU Manager 严格预留（Alpha）：

开启后，独占 CPU cores 的 Pod 保证不与其他容器共享物理核，避免 L1/L2 cache 竞争：

场景 传统 static policy strict-cpu-reservation
Guaranteed Pod 请求 2 CPUs 可能与其他 Pod 共享物理核 独占 2 个完整物理核
Burstable Pod 请求 0.5 CPU 正常调度 若使用独占核会被拒绝

  • NUMA 亲和性：Memory Manager 对延迟敏感应用（DPDK、数据库、AI 推理）至关重要。跨 NUMA 内存访问延迟高出 30-50%

23. Pod Priority 与 Preemption 在生产中的坑？如何设计优先级体系？

难度：⭐⭐⭐⭐（专家）
问题：某金融公司集群资源紧张时，业务 Pod 被 Kubernetes 调度器「无声」驱逐，导致线上服务中断。排查发现是 PriorityClass 配置不当导致高优 Pod 抢占低优 Pod 引发级联重启。Pod Priority 和 Preemption 的机制是什么？如何避免踩坑？

核心原理：

  • PriorityClass：定义 Pod 的优先级（1-10亿），数值越大优先级越高
  • Preemption（抢占）：当高优先级 Pod 无法调度时，调度器会驱逐（evict）低优先级 Pod 来腾出资源
  • 调度队列中高优先级 Pod 始终排在前面
  • 默认 PriorityClass：system-cluster-critical（2000000000）和 system-node-critical（2000001000）

生产实战：

🔥 踩坑案例：某金融公司监控 Pod 被抢占导致雪崩

  • 场景：3 节点集群（32C/128GB），运行 80+ Pod（核心交易 + 日志采集 + 监控 + 后台批处理）
  • 症状：某次促销流量突增，调度器自动创建了一组高优交易 Pod。随后：
    1. Prometheus + Grafana 监控 Pod 被抢占驱逐
    2. 告警系统沉默（无法采集指标）
    3. 新交易 Pod 的依赖（Redis/MySQL 连接池）未准备好，交易超时
    4. 用户重试导致流量进一步增加 → 更多 Pod 被抢占 → 连锁反应
  • 排查过程：
    1. kubectl get events --sort-by='.lastTimestamp' → 看到大量 Preempted 事件
    2. kubectl describe pod <monitoring-pod> → 显示 Reason: Preempted
    3. 检查 PriorityClass 发现所有非系统 Pod 使用同一个默认优先级（1000）
  • 根因：
    • 未区分「业务 Pod」和「基础设施 Pod」的优先级
    • 监控 Pod 被抢占后，无法感知集群故障（监控盲区 = 雪崩催化剂）
  • 修复：设计三级优先级体系：
  • 修复效果：监控 Pod 优先级高于批处理 Pod，不再被抢占；核心交易服务获取最高优先级

优先级体系设计原则：

源码分析：

  • 核心调度逻辑：pkg/scheduler/framework/preemption/preemption.go → Preempt()
  • 关键算法：NodesWherePreemptionMightHelp() → 筛选可被抢占的节点
  • selectVictimsOnNode() → 在每个候选节点上选择可以被驱逐的 Pod（驱逐优先级最低的，且满足新 Pod 资源需求的 Pod 集合）

面试金句：

「Pod Preemption 是一把双刃剑——高优 Pod 挤占低优 Pod 时，如果监控/日志这类基础设施 Pod 恰好是”最容易被驱逐”的，你会先失明再死亡。三级优先级体系（核心 → 基础设施 → 批处理）是生产环境的底线要求。」

24. HPA 为什么在流量突增时「来不及扩容」？如何优化响应速度？

难度：⭐⭐⭐⭐（专家）
问题：某电商公司大促期间流量突增 10 倍，HPA 却在 3-5 分钟后才开始扩容，导致大量 503。为什么 HPA 响应这么慢？如何让它「更灵敏」？

核心原理：

  • HPA 的扩容延迟主要来自 3 个环节：
    1. 指标采集间隔：Metrics Server 默认 15s 采集一次
    2. 控制循环间隔：HPA controller 默认 15s 评估一次
    3. 扩容冷却窗口：--horizontal-pod-autoscaler-downscale-stabilization（默认 300s）
    4. Pod 启动时间：拉镜像 + 启动进程 + readiness 就绪（30s-3min）
  • 总延迟 = 指标采集(15s) + 评估周期(15s) + 扩容生效(15s) + Pod 就绪(30s~3min) ≈ 1-4 分钟

生产实战：

🔥 踩坑案例：某电商大促 503 事故

  • 场景：6 节点集群（16C/64GB），订单服务 10 个 Pod，HPA 基于 CPU（target=60%）
  • 症状：流量在 30 秒内从 2000 QPS 飙升至 18000 QPS，大量请求返回 503。HPA 日志显示 3 分钟后才触发扩容到 30 个 Pod，但此时大量请求已超时
  • 排查过程：
    1. kubectl describe hpa orders-hpa → 看到 Conditions: AbleToScale=True 在 3 分钟前
    2. 查看 HPA Events → Scaling up 触发后，Pod 启动又用了 90s（大镜像拉取慢）
    3. 检查 readiness probe → initialDelaySeconds: 60 导致就绪更慢
  • 根因：
    1. HPA 默认冷却窗口 + 指标采集间隔合计响应约 45s
    2. Pod 启动速度慢：镜像 1.2GB，readiness 探针等待 60s
    3. 没有配置 HPA 的 behavior.scaleUp 自定义策略
  • 修复方案：

HPA 响应速度优化 Checklist：

优化项 默认值 推荐值 收益
--horizontal-pod-autoscaler-sync-period 15s 10s 控制循环加快 33%
Metrics Server 采集间隔 15s 10s 指标更新加快 33%
scaleUp.stabilizationWindowSeconds 隐含 3min 0 立即响应
scaleUp.percent 100% 100% 翻倍扩容
Pod 镜像大小 - <500MB 启动更快
readinessProbe.initialDelaySeconds 0-60s 5-10s 更早进入服务

面试金句：

「HPA 不是实时的——从流量暴增到新 Pod 开始处理请求，至少经历 4 个时间窗口：指标采集(15s) + 控制循环(15s) + Pod 创建(30s) + 就绪等待(30s+) = 至少 90s。解决之道：HPA behavior.scaleUp 设 stabilizationWindowSeconds: 0 + 翻倍扩容策略 + 小镜像 + 快就绪探针 + 预留缓冲 Pod。」

31. Kubernetes Gateway API 与 Ingress 对比——生产迁移指南

难度：⭐⭐⭐⭐（专家）

问题：Kubernetes Gateway API 正在取代 Ingress，其核心优势和迁移路径是什么？

核心原理：

① 单资源瓶颈：一个 Ingress 规则包揽全部路由，无法按团队拆分
② 注解地狱：canary/rewrite/header 操作全靠供应商注解，不跨控制器可移植
③ 跨命名空间无能：Ingress 只能引用同命名空间的 Service
④ 协议受限：仅 HTTP/HTTPS，不支持 TCP/UDP/gRPC

↓ 迁移后 Gateway API：

Ingress vs Gateway API 对比：

维度 Ingress v1 Gateway API
关注点 第 7 层 HTTP 路由 第 4/7 层 + 服务网格集成
实现模型 单一路由规则 Gateway → Listener → Route 三层模型
跨命名空间 ❌ 需 annotation ✅ 原生支持
Header 修改 ❌ 需 annotation ✅ HTTPRouteFilter
灰度发布 ❌ 复杂 ✅ RequestMirror / 权重路由
TLS 配置 annotation 中指定 独立 Secret 资源
后端协议 仅 HTTP/HTTPS HTTP, HTTP/2, gRPC, TCP, TLS
成熟度 稳定（GA v1） Beta（v1.0 2024）

迁移路径：

  1. 先用 Gateway API 替换简单的 L7 路由
  2. 逐步迁移高级功能（灰度、Header 改写、gRPC）
  3. 最后启用 L4 TCP 路由替代 LoadBalancer Service

面试金句：

2. Sidecar-less Service Mesh（Istio Ambient Mesh）

Sidecar 模式的痛点：

Ambient Mesh 架构（Istio 1.22+ GA）：

ztunnel（Zero-Trust Tunnel）：

  • 用 Rust 编写，每个节点一个 DaemonSet
  • 处理 L3/L4：mTLS 认证、加密隧道（HBONE）、L4 鉴权、遥测
  • 不解析 HTTP 头，不处理 L7
  • 资源开销极低：约 10-30MB 内存，< 0.1 CPU core

Waypoint Proxy：

  • 标准 Envoy 代理，按 namespace 部署（不是每个 Pod）
  • 只对有 L7 需求的 namespace 启用（灰度/重试/断路器/header 路由）
  • 与应用 Pod 生命周期解耦，独立升级扩缩

Sidecar vs Ambient 选型决策：

维度 Sidecar 模式 Ambient 模式
资源开销 高（每 Pod 额外 100MB+） 低（节点级共享）
延迟 +5-15ms（串行 Envoy） +0-2ms（内核级透传）
应用侵入 Pod 必须重启注入 零侵入，无需重启
L7 功能 全部 Pod 可用 按 namespace 启用
升级影响 所有 Pod 需 rollout 重启 仅升级 ztunnel/waypoint
安全性 更强隔离 更依赖节点安全
适用场景 强隔离/金融/合规 大规模/成本敏感/增量上网格

深入思考：

  • 混合模式可行：同一集群可以同时运行 sidecar Pod 和 ambient Pod，Istio 自动兼容
  • 性能数据：Tetrate 实测，Ambient 模式在 1000 Pod 集群中相比 Sidecar 减少 60% 资源消耗，P99 延迟降低 40%
  • HBONE 隧道：使用 HTTP CONNECT 封装，天然支持 SNI，可通过标准 L7 LB 负载均衡
  • 迁移路径：先启用 ztunnel (L4) → 观察无问题 → 按 namespace 开 waypoint (L7)，逐步取代 sidecar
  • 当前 GA 状态：Istio 1.22+ Ambient 模式已 GA，CNCF 生态持续跟进

参考来源：Istio Ambient 官方文档, InfoQ Istio Ambient Mesh 深度解析, Tetrate Ambient vs Sidecar 性能评测

3. eBPF 与 Cilium

eBPF 基础知识：

eBPF（extended Berkeley Packet Filter）允许将沙箱程序安全地注入 Linux 内核，无需修改内核源码或加载内核模块。核心能力：

能力 说明 传统方案对比
网络包处理 在最早的时刻处理数据包 iptables 遍历 O(n) 规则链
系统调用拦截 监控所有进程的系统调用 Auditd 无法结构性分析
性能追踪 零开销采样和追踪 Perf 嵌入点有限
安全策略 内核级强制执行 AppArmor/SELinux 配置复杂

Cilium 网络（替代 kube-proxy）：

Cilium 是 CNCF 毕业项目，用 eBPF 重构了整个 Kubernetes 网络栈。

Cilium NetworkPolicy（基于身份，而非 IP）：

对比 NetworkPolicy vs CiliumNetworkPolicy：

特性 K8s NetworkPolicy CiliumNetworkPolicy
范围 L3/L4 L3/L4/L7 (HTTP/gRPC/Kafka)
身份 IP/标签 标签 + CIDR + DNS + FQDN
L7 规则 ❌ ✅ 方法/路径/header 匹配
DNS 安全 ❌ ✅ DNS 规则
集群内外 仅集群内 支持外部服务 (toFQDN)

Hubble 可观测性：

Hubble CLI 示例

Hubble 提供：

  • Service Map：可视化服务间依赖关系
  • L7 流详情：每个 HTTP 请求的延迟、状态码
  • DNS 解析追踪：查看哪些 Pod 访问了哪些域名
  • 策略审计：哪些 NetworkPolicy 命中了流量

Cilium Tetragon — eBPF 安全：

Tetragon 是 Cilium 子项目，实现内核级运行时安全：

Tetragon 关键能力：

  • 进程执行监控（谁启动了哪个进程）
  • 文件访问审计（敏感文件读取/写入）
  • 网络连接监控（出站连接检测）
  • 容器逃逸检测（内核级实时拦截）
  • TOCTOU 攻击防护（内核级同步检测，不依赖 hook）

深入思考：

  • 内核版本依赖：eBPF 大部分特性需要内核 5.10+（建议 5.15+ 或 RHEL 8.6+）
  • Cilium vs Calico：Cilium 在 1000+ 节点集群中 Service 转发性能比 Calico (iptables) 高 3-5 倍；Calico eBPF 模式也在追赶，但生态不如 Cilium
  • Cluster Mesh：Cilium 支持多集群互联，Pod IP 可以直接跨集群通信，无需 VPN/overlay
  • 生产踩坑：替换 kube-proxy 时需确保 kubelet --node-ip 配置正确；eBPF 程序在旧内核上可能失败
  • 迁移建议：新集群直接 Cilium + kube-proxy 替换；存量集群先用 Cilium + kube-proxy 共存模式验证，再切 kube-proxy-free

深入思考：

  • VPA vs HPA 协同：VPA 设 request，HPA 设 replica。两者不建议同时作用于同一个 Deployment（VPA 的调整会干扰 HPA 的稳定窗口）
  • KRR 局限：基于 Prometheus 历史数据，对新部署或突发流量不准确。建议先运行观察 7 天，结合业务流量峰值人工微调

  • FinOps 即文化：工具只能解决 30% 问题，剩下 70% 靠组织 — 让每个开发看到自己代码的”云账单”

深入思考：

  • 推理与训练不同：训练追求吞吐（throughput），推理追求延迟（TTFT + TPOT）。vLLM 的 PagedAttention + 连续批处理是当前最优方案
  • GPU 弹性扩缩挑战：GPU 节点池扩容通常 3-10 分钟（云服务商冷启动+驱动加载），需要提前规划 buffer 节点
  • 推理成本公式：单次推理成本 = GPU 小时单价 × (TTFT + TPOT × 输出长度) / 3600，量化的 AWQ/FP8 可直接降低成本 40-60%

  • KEDA GPU Scaler：基于 DCGM 指标（GPU 利用率、显存使用率）自动扩缩推理 Pod，避免 CPU-based HPA 的不准确性
  • 模型版本管理：结合 OCI 镜像存储 HuggingFace 模型，用 Helm Chart 管理不同模型版本的部署参数
