Python 高级面试全解，覆盖 GIL 机制、协程与异步编程、内存管理、装饰器高级用法和性能优化。面向高级研发，侧重底层原理。

0x00 GIL 与并发

1. GIL 是什么？为什么 CPython 不删除它？

GIL（Global Interpreter Lock） 是 CPython 解释器级别的互斥锁，确保同一时刻只有一个线程执行 Python 字节码。

为什么不去掉 GIL？

  • GIL 让 CPython 的单线程性能极好（无需细粒度锁）
  • C 扩展库大量依赖 GIL 保证线程安全，去掉会破坏生态
  • Python 3.13 引入了实验性的 --disable-gil 模式，走了 PEP 703 路线（偏向锁 + 延迟引用计数）

绕过 GIL 的方式：

  • 多进程：multiprocessing，每个进程有独立 GIL
  • C 扩展：在 C 代码中手动释放 GIL（Py_BEGIN_ALLOW_THREADS）
  • 异步 IO：asyncio 单线程协程，不涉及 GIL 竞争

深入思考：

  • 边界情况：IO 密集型多线程不受 GIL 影响——线程在 socket.recv() 阻塞时会释放 GIL，其他线程可以继续执行

2. 协程 (coroutine) 和线程的区别？async/await 底层怎么工作？

维度 协程 线程
调度 用户态协作式（自己让出） OS 抢占式
切换成本 函数调用级别（微秒） 上下文切换（毫秒）
并发数 数万 数百
共享状态 无需锁（单线程） 需要锁
阻塞 await 不阻塞事件循环 系统调用阻塞 OS 线程

async/await 底层：

事件循环（Event Loop）是一个 while True，每次循环从就绪队列取出协程执行，遇到 await 就挂起当前协程、切换到下一个。

深入思考：

  • 源码层面：async def 编译后是一个 coroutine function，调用返回 coroutine object。await 等价于 yield from，底层通过 __await__ 协议实现
  • 生产经验：不要在协程里调用同步阻塞函数（如 time.sleep(1)）——换成 await asyncio.sleep(1)。否则整个事件循环被阻塞

0x01 内存管理

3. Python 的垃圾回收机制是怎样的？

三层回收：

  1. 引用计数（主）：每个对象维护 ob_refcnt，为 0 时立即释放
  1. 标记清除（辅）：解决循环引用
  1. 分代回收：新对象更容易变成垃圾

深入思考：

  • 边界情况：__del__ 方法会影响 GC——有 __del__ 的循环引用对象在 Python 3.4+ 仍会被回收，但顺序不确定
  • 生产经验：大内存应用用 gc.disable() + 手动 gc.collect() 可避免 GC 的 STW 暂停

4. Python 的 __slots__ 是什么？什么时候用？

使用场景：创建大量小对象（如数据类的数百万实例），__slots__ 可节省 50%+ 内存。

注意：__slots__ 不继承——子类需要重新声明，且和多重继承有兼容性问题。

0x05 生产环境踩坑与调试

7. Python 3.13 无 GIL（free-threaded）模式——生产就绪了吗？

背景：PEP 703 在 Python 3.13 以 --disable-gil 编译选项引入实验性无 GIL 模式，Python 3.14 进一步改进。目标：让多线程计算密集型 Python 代码真正并行。

工作机制（偏向锁 + 延迟引用计数）：

核心变化：

机制 有 GIL 无 GIL
引用计数 线程安全（GIL 保护） 延迟引用计数 + 偏向锁（PEP 703）
对象分配 无需额外锁 每线程本地缓存 + 原子操作
容器线程安全 天然安全 需要显式锁（list/dict 不再线程安全）
C 扩展兼容性 全部兼容 需要标记 Py_MOD_GIL_NOT_USED
单线程性能 基准 下降约 5-10%（偏向锁开销）

生产实战：

踩坑案例 1：C 扩展兼容性事故（某 AI 推理服务）

  • 场景：某 AI 公司 Python 3.13 无 GIL 模式，用 NumPy 1.26 + PyTorch 2.3
  • 症状：import torch 后解释器段错误（Segfault），崩溃无堆栈
  • 排查：gdb python 定位到 NumPy C 扩展尝试获取 GIL（PyGILState_Ensure()），但无 GIL 模式该函数内部断言失败
  • 根因：C 扩展未标记 Py_MOD_GIL_NOT_USED，旧版 ABI 不兼容
  • 修复：升级 NumPy 2.0+（支持 free-threaded），PyTorch 需用 nightly 版
  • 修复效果：导入正常，4 线程推理吞吐从 1.0x（有 GIL）提升到 3.2x

踩坑案例 2：list/dict 竞态条件（某爬虫平台）

  • 场景：多线程爬虫，用 threading.Thread 并发写入共享 dict
  • 症状：偶发 KeyError 访问明明已存在的 key，Dict 长度偶尔丢失
  • 排查：打印 dict 大小发现不一致 len(d) 在不同线程返回不同值
  • 根因：无 GIL 模式下 dict 的 __setitem__ 不再原子，len() 读取到中间状态
  • 修复：对共享 dict 加 threading.Lock 保护，或改用 queue.Queue
  • 最佳实践：无 GIL 模式下所有可变容器需要加锁

调优指南：

场景 推荐模式 原因
CPU 密集型多线程 无 GIL 真正并行，4 线程提升 3-4x
单线程 IO 密集型 有 GIL 无额外偏向锁开销
C 扩展密集（NumPy/Pandas） 有 GIL C 扩展兼容性最好
混合负载（Web 服务） 有 GIL + 多进程 进程隔离更成熟

面试金句：

“Python 3.13 无 GIL 模式让多线程计算有了真正并行能力，但生产落地需审慎评估 C 扩展兼容性和容器线程安全。我在 AI 推理服务中实测 4 线程吞吐提升 3.2 倍，但历经了 NumPy/PyTorch 的段错误排查。预计 Python 3.15+ 才是无 GIL 的生产就绪版本。”

参考：PEP 703 | Python 3.13 Free-threaded FAQ

内存泄漏排查清单：

上线 Checklist（Python Web 服务）：

  • asyncio 调试模式在生产关闭（PYTHONASYNCIODEBUG=0，性能开销约 10%）
