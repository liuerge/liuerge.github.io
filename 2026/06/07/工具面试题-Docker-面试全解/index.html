2. Docker 镜像分层原理和写时复制？

  • 每一层是一个文件系统的增量变更
  • 拉取镜像时，已存在的层不重复下载（不同镜像可共享层）
  • 写操作使用 COW：修改只读层的文件时，先复制到容器层再修改

优化策略：把变化频率高的指令放下面（如 COPY 源码），变化频率低的放上面（如 apt-get install）

0x05 基础概念与原理

7. 什么是 Docker？

答案：
Docker 是一个容器化平台，将应用及其所有依赖环境打包成一个标准化的容器，保证应用在不同的平台（开发、测试、生产）上可以无缝运行。

Docker 容器将一个软件包装在一个完整的文件系统中，包含运行所需的一切：代码、运行时、系统工具、系统库等，可以安装在服务器上的任何东西。

出处：《Docker 那点破事》— Tom哥

8. Docker 与虚拟机的区别？

维度 Docker 容器 虚拟机
内核 共享宿主机内核 每个 VM 有独立内核
启动时间 毫秒级 分钟级
资源损耗 仅进程级隔离，几乎无损耗 Hypervisor 层占用大量资源
隔离性 通过 namespace/cgroup 实现进程隔离 完全虚拟化，安全性更高
密度 同等资源可运行数十倍于 VM 的容器 数量有限

深入思考：

  • 安全权衡：Docker 与宿主机共享内核，容器逃逸风险比 VM 高。生产环境建议：敏感业务用 VM，微服务/无状态应用用容器
  • 性能：Docker 网络和 IO 接近原生，VM 因虚拟化层有 5-15% 性能损耗

出处：《Docker 那点破事》— Tom哥

9. Docker 三大核心组件是什么？

组件 说明
镜像（Image） 创建容器的基础，类似 VM 快照。一个只读模板，包含运行应用所需的所有内容（代码、运行时、库、环境变量、配置文件）。实际上是一个压缩包，包含完整操作系统文件
容器（Container） 从镜像创建的运行实例。可启动、停止、删除。每个容器相互隔离，可看作一个简易版 Linux 环境（root 用户、文件空间、网络空间）
仓库（Registry） 存放镜像的服务器。每个仓库含多个镜像，每个镜像有不同 tag。公开仓库（Docker Hub）和私有仓库

出处：《Docker 那点破事》— Tom哥

10. Docker 后台的标准运行过程？

当执行 docker run 创建容器时，后台执行流程：

  1. 检查本地是否存在指定镜像，不存在则从仓库下载
  2. 利用镜像创建并启动一个容器
  3. 分配文件系统给容器，在只读镜像层外挂载一层可读写层
  4. 从宿主机网桥接口桥接一个虚拟接口到容器
  5. 分配 IP 地址给容器
  6. 执行用户指定的应用程序，执行完毕后容器终止

出处：《Docker 那点破事》— Tom哥

11. 镜像本地存放在哪里？容器退出后数据会丢失吗？

镜像存储位置：
Docker 本地资源存放在 /var/lib/docker/ 目录下：

  • container/ — 存放容器信息
  • image/ — 存放镜像信息
  • overlay2/ — 存放具体的镜像底层文件

容器数据：
容器退出后处于 Exited 状态，通过 docker ps -a 可以查看。数据不会丢失，可通过 docker start 重新启动。只有执行 docker rm 删除容器才会清除数据。

15. 可以在一个容器中运行多个应用进程吗？

一般不推荐。如果需要，可以通过额外的进程管理机制实现，例如使用 supervisord 来管理多个进程。

最佳实践：一个容器只运行一个进程/服务，便于管理、监控和扩缩容。

18. Docker 应用流程是怎样的？

  1. Dockerfile — 定义应用的构建方式（源代码）
  2. docker build — 构建镜像（编译产物）
  3. Registry — 发布/共享镜像（类似 Git 仓库）
  4. docker run — 从镜像启动容器运行

出处：《Docker 那点破事》— Tom哥

19. 如何创建与环境无关的容器系统？

三种关键技术：

  1. 存储卷（Volumes） — 将数据持久化到宿主机，与容器生命周期解耦
  2. 环境变量注入（Environment Variable Injection） — 通过 -e 或 --env-file 注入配置
  3. 只读文件系统（Read-only） — 容器根文件系统设为只读，强制数据写入 Volume

深入思考：

  • 生产经验：--mount=type=cache 是 CI/CD 构建提速的关键。Go 的 /go/pkg/mod、npm 的 ~/.npm、pip 的 ~/.cache/pip 都适合缓存。CI 环境需确保 Docker daemon 支持 BuildKit（DOCKER_BUILDKIT=1）

参考来源：Docker BuildKit Documentation, Docker Engine 26+ Release Notes

22. Docker Compose Watch 是什么？解决了什么痛点？

标准答案：

Docker Compose Watch（Docker 26+，Compose v2.22+）实现了开发时的文件热同步，无需重建镜像即可将本地代码变更实时注入容器：

action 行为 适用场景
sync 文件变更时直接拷贝到容器 源代码热更新（不重启）
rebuild 文件变更时重建镜像并重启 依赖变更（package.json）
sync+restart sync 后自动重启容器 配置文件变更

深入思考：

  • 对比 bind mount：bind mount 也能实现文件同步，但 Compose Watch 更精细——可以根据文件类型选择 sync/rebuild/sync+restart，且不会因宿主机和容器文件系统差异导致权限问题
  • 性能：sync 使用 docker cp 机制实现增量同步，比 bind mount 在 macOS/Windows 文件系统性能更好（避免 osxfs 性能损耗）
  • 生产经验：node_modules 等大型依赖目录应排除在 sync 之外（通过 .dockerignore），避免同步大量依赖文件拖慢开发体验

参考来源：Docker Compose Watch Documentation

深入思考：

  • 存量兼容：开启 containerd image store 后，旧的 overlay2 镜像不会自动迁移。需 docker save 导出再 docker load 导入，或重新 docker pull

25. Docker 容器中 JVM OOM 的根因是什么？如何配置 JVM 内存限制？

难度：⭐⭐⭐⭐（专家）
问题：某金融公司 Java 微服务容器化后，JVM 频繁被 OOM Kill，即使设置了 -Xmx 也无法解决。为什么 JVM 在容器中会超限使用内存？

核心原理：

  • JVM 在 JDK 8u131 之前完全不知道 cgroup 的存在，-Xmx 限制的是 Heap，而非整个进程内存
  • JVM 进程 = Heap + Metaspace + Thread Stack + Code Cache + Direct Buffer + Native Memory + JNI
  • 这三项最容易超出预期：
    • Direct Buffer：-XX:MaxDirectMemorySize 默认等于 -Xmx
    • Metaspace：动态加载类时持续膨胀（无上限）
    • Thread Stack：每个线程默认 1MB（-Xss），200 个线程 = 200MB

生产实战：

🔥 踩坑案例：某互金公司电商订单服务（2024）

  • 场景：8 个 Java 微服务，每个容器 memory limit = 4GB，-Xmx=3GB
  • 症状：每天凌晨 02:00-04:00 随机 OOM Kill 3-5 次，docker inspect 显示 OOMKilled: true
  • 排查过程：
    1. 先用 docker stats 发现容器 RSS 实际使用 3.6-3.9GB（远超 -Xmx 3GB）
    2. 看 /sys/fs/cgroup/memory/memory.stat → total_rss 3.8GB，total_cache 0.3GB
    3. 用 jcmd <pid> VM.native_memory summary 发现 Direct Buffer 占用 600MB+NMT overhead 200MB
    4. 检查 JVM 参数发现未设置 -XX:MaxDirectMemorySize 和 -XX:MaxMetaspaceSize
  • 根因：-Xmx=3GB 只限制了 Heap，但 Direct Buffer（默认 = -Xmx = 3GB）+ Metaspace（无上限）+ Thread Stack（64 线程×1MB）使总内存远超容器 4GB 限制
  • 修复方案：
  • 修复效果：容器 RSS 稳定在 3.2-3.5GB，OOM 降为 0

参数调优指南：

内存项 占容器 limit 比例 备注
Heap（-Xmx / MaxRAM%） 50-65% 核心业务内存
Metaspace ~256MB 框架+业务 Class
Thread Stack 线程数 × 512k-1MB 虚拟线程可大幅降低
Direct Buffer 512MB-1GB 网络 IO 密集型
Code Cache 64-256MB JIT 编译
预留余量 10-20% 避免 OOM

源码分析：

  • 核心类：hotspot/src/share/vm/runtime/os.cpp → os::Linux::container_support()（JDK 8u131+）
  • JDK 10+：-XX:+UseContainerSupport 默认启用，读取 /sys/fs/cgroup/memory/memory.limit_in_bytes
  • JDK 17+：-XX:+UseContainerSupport + -XX:MaxRAMPercentage 是最优方案
  • 关键路径：os::Linux::physical_memory() → OSContainer::memory_limit_in_bytes() → cgroup 文件读取

对比延伸：

维度 JDK 8u131- JDK 11 JDK 17+
容器感知 ❌ 需手动配置 ✅ 默认启用 ✅ 默认启用
推荐参数 -XX:MaxRAMFraction=2 -XX:MaxRAMPercentage=70 -XX:MaxRAMPercentage=60
DirectBuffer 手动限制 手动限制 手动限制
虚拟线程 ❌ ❌ ✅ 节省 Stack 内存

面试金句：

「-Xmx 限制的是 Heap 而非进程内存——JVM 总内存 = Heap + Metaspace + DirectBuffer + Thread Stack + CodeCache + Native，容器的 memory limit 必须为所有这些项的总和预留 20-30% 余量。JDK 17+ 用 -XX:MaxRAMPercentage=60 是最安全的方案。」

26. Docker 日志驱动不配置会有什么后果？如何配置生产级日志轮转？

难度：⭐⭐⭐（高级）
问题：某团队上线 Docker 微服务后，运行 7 天后宿主机磁盘写满，所有容器异常重启。排查发现是 Docker 日志文件占用 80GB+ 空间。为什么默认日志配置会导致磁盘满？

核心原理：

  • Docker 默认日志驱动是 json-file，每个容器的 stdout/stderr 写到一个 JSON 文件中
  • 默认无任何日志轮转限制 — 只要容器一直在写，日志文件无限增长
  • 日志文件位置：/var/lib/docker/containers/<CONTAINER_ID>/<CONTAINER_ID>-json.log

生产实战：

🔥 踩坑案例：某 SaaS 公司 API 网关日志撑爆磁盘

  • 场景：6 台 Docker 宿主机，每台运行 15-20 个容器，含一个 Nginx 网关容器（日产生 3GB 日志）
  • 症状：运行 6 天后，某宿主机 /var/lib/docker/ 占用 92%，容器陆续因「设备无空间」报错崩溃
  • 排查过程：
    1. du -sh /var/lib/docker/containers/*/*-json.log → 发现 Nginx 容器的日志文件 47GB
    2. docker info --format '{{.LogDrv}}' → 确认使用 json-file 驱动
    3. /etc/docker/daemon.json → 发现未配置任何 log-opts

27. 挂载 /var/run/docker.sock 有什么安全隐患？如何替代？

难度：⭐⭐⭐⭐（专家）
问题：很多 CICD 工具和监控容器需要挂载 docker.sock 来管理其他容器，但安全团队警告这是「打开了通往主机的后门」。docker.sock 被挂载后到底有多危险？有哪些替代方案？

核心原理：

  • Docker daemon 通过 /var/run/docker.sock Unix Socket 暴露完整 API（无认证）
  • 挂载 docker.sock 的容器内相当于有 Docker 的 root 权限，可以执行任意 docker 命令
  • docker run -v /var/run/docker.sock:/var/run/docker.sock ... 是常见的错误实践

生产实战：

🔥 踩坑案例：某 CI 工具 docker.sock 容器逃逸事故

  • 场景：Jenkins Agent 容器挂载 docker.sock 来构建和推送镜像
  • 症状：某次 Jenkins 构建任务被注入恶意代码，攻击者通过 docker.sock 在宿主机上启动了一个挂载 / 的容器，读取了宿主机上所有容器的环境变量（含数据库密码、API Key）
  • 攻击链：
  • 根因：docker.sock 提供的是 Docker API 的 root 权限，等价于在宿主机上运行 docker 命令
  • 修复：
    1. 立即吊销泄露的凭证
    2. 用 DinD（Docker in Docker）替代 docker.sock
    3. 配置 Docker daemon 的 TLS 认证（双向证书）
    4. 最小权限原则：CI 用 kaniko/buildkit 替代 Docker in Docker

28. Docker Compose depends_on 的坑是什么？如何实现可靠的启动顺序？

难度：⭐⭐⭐（高级）
问题：某团队用 Docker Compose 编排服务，depends_on: - db 后应用还是报「数据库连接被拒绝」。为什么 depends_on 不够？如何确保服务就绪后才启动依赖方？

核心原理：

  • depends_on 只控制容器启动顺序（先启动 db），但不等待 db 的就绪状态
  • 容器启动 ≠ 服务就绪——PostgreSQL 启动可能需要 3-15 秒才能接受连接

生产实战：

🔥 踩坑案例：某初创公司后端服务启动失败

  • 场景：Docker Compose 编排 3 个服务（app → db + redis）
  • 症状：docker compose up -d 后 app 容器启动又崩溃，日志报 dial tcp 127.0.0.1:5432: connection refused
  • 排查：
    1. docker compose logs app → 确认是 db 未就绪，Spring Boot 启动太快
    2. 检查 compose.yaml → depends_on: - db 已配置
    3. docker compose logs db → db 容器日志显示仍在 recovery 阶段
  • 根因：depends_on 只保证 db 容器的进程启动了，不保证 PostgreSQL 的 TCP 端口可连接
  • 修复：使用 condition: service_healthy：

方案对比：

方案 可靠性 复杂度 说明
depends_on 无 condition ❌ 低 低 只保证启动顺序
depends_on + service_healthy ✅ 高 中 等待就绪
restart: always 自愈 ✅ 中 低 崩溃后重试
init container 脚本 ✅ 高 高 循环检测端口

面试金句：

「depends_on 在 Compose v2 中的应用层启动保证机制——depends_on 只管容器进程启动，不管服务就绪。生产环境必须用 depends_on: condition: service_healthy 配合 healthcheck，让每个服务自报健康后才启动依赖方。」

🔥 踩坑案例：Rootless Docker + NFS 卷挂载权限问题（某 AI 公司）

  • 场景：机器学习平台，所有 Model Serving 容器迁移到 Rootless
  • 症状：容器启动后无法写入挂载的 NFS 目录，报 Permission denied
  • 排查：
    1. NFS 服务器的共享目录配置了 root_squash（默认）
    2. Rootless 模式下，容器的 root 被映射为 host 上的普通用户（UID 100000+）
    3. NFS 看到的 UID 是 100000+，但共享目录的 owner 是 1000
  • 根因：NFS root_squash 只对 UID 0 有效。Rootless 模式下容器 root 的映射 UID 不在 NFS 权限范围内
  • 修复：
    1. 在 NFS exports 中加 no_all_squash 和 anonuid=1001
    2. 或者容器内使用与 NFS 匹配的 UID 运行
  • 替代方案：使用 CSI 驱动 + 本地 SSD 而非 NFS

Rootless 迁移红牌禁令：

❌ 不要做 ✅ 应该做
Rootless 直接用于生产未测试 在 staging 运行 1 周验证网络/存储兼容性
依赖特权端口（< 1024） 使用 docker run --privileged 仅对特定容器开启
使用 overlay2 + xfs d_type 检查 /sys/kernel/security 路径权限
忽略 kernel.unprivileged_userns_clone 内核参数 确保 sysctl kernel.unprivileged_userns_clone=1

面试金句：

“Docker Rootless 是安全的基线配置——容器逃逸后只能获得非 root 权限。但最大的坑是文件系统：NFS、FUSE 和 overlay2 在 Rootless 下都有已知问题。先用 dockerd-rootless-setuptool.sh 验证兼容性再上线。”
