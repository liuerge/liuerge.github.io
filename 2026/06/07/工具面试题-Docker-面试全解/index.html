0x00 Dockerfile 最佳实践

1. 如何写一个高质量的 Dockerfile？

深入思考：

2. Docker 镜像分层原理和写时复制？

  • 每一层是一个文件系统的增量变更
  • 拉取镜像时，已存在的层不重复下载（不同镜像可共享层）
  • 写操作使用 COW：修改只读层的文件时，先复制到容器层再修改

优化策略：把变化频率高的指令放下面（如 COPY 源码），变化频率低的放上面（如 apt-get install）

0x02 网络与存储

3. Docker 的五种网络模式？

模式 命令 特点
bridge（默认） --network bridge 独立网络栈，通过 docker0 网桥通信
host --network host 直接使用宿主机网络，性能最高
none --network none 无网络，完全隔离
overlay --network overlay 跨主机通信（Swarm/K8s）
container --network container:xxx 共享另一个容器的网络栈

4. Docker Volume 和 Bind Mount 的区别？

维度 Volume Bind Mount
管理 Docker 管理（/var/lib/docker/volumes/） 宿主机路径
权限 Docker 控制 宿主机文件权限
跨平台 ✅ 可移植 ❌ 依赖宿主机路径
适用 持久化数据、DB 开发时挂载代码

0x05 基础概念与原理

7. 什么是 Docker？

答案：
Docker 是一个容器化平台，将应用及其所有依赖环境打包成一个标准化的容器，保证应用在不同的平台（开发、测试、生产）上可以无缝运行。

Docker 容器将一个软件包装在一个完整的文件系统中，包含运行所需的一切：代码、运行时、系统工具、系统库等，可以安装在服务器上的任何东西。

出处：《Docker 那点破事》— Tom哥

8. Docker 与虚拟机的区别？

维度 Docker 容器 虚拟机
内核 共享宿主机内核 每个 VM 有独立内核
启动时间 毫秒级 分钟级
资源损耗 仅进程级隔离，几乎无损耗 Hypervisor 层占用大量资源
隔离性 通过 namespace/cgroup 实现进程隔离 完全虚拟化，安全性更高
密度 同等资源可运行数十倍于 VM 的容器 数量有限

深入思考：

  • 安全权衡：Docker 与宿主机共享内核，容器逃逸风险比 VM 高。生产环境建议：敏感业务用 VM，微服务/无状态应用用容器
  • 性能：Docker 网络和 IO 接近原生，VM 因虚拟化层有 5-15% 性能损耗

出处：《Docker 那点破事》— Tom哥

9. Docker 三大核心组件是什么？

组件 说明
镜像（Image） 创建容器的基础，类似 VM 快照。一个只读模板，包含运行应用所需的所有内容（代码、运行时、库、环境变量、配置文件）。实际上是一个压缩包，包含完整操作系统文件
容器（Container） 从镜像创建的运行实例。可启动、停止、删除。每个容器相互隔离，可看作一个简易版 Linux 环境（root 用户、文件空间、网络空间）
仓库（Registry） 存放镜像的服务器。每个仓库含多个镜像，每个镜像有不同 tag。公开仓库（Docker Hub）和私有仓库

出处：《Docker 那点破事》— Tom哥

10. Docker 后台的标准运行过程？

当执行 docker run 创建容器时，后台执行流程：

  1. 检查本地是否存在指定镜像，不存在则从仓库下载
  2. 利用镜像创建并启动一个容器
  3. 分配文件系统给容器，在只读镜像层外挂载一层可读写层
  4. 从宿主机网桥接口桥接一个虚拟接口到容器
  5. 分配 IP 地址给容器
  6. 执行用户指定的应用程序，执行完毕后容器终止

出处：《Docker 那点破事》— Tom哥

11. 镜像本地存放在哪里？容器退出后数据会丢失吗？

镜像存储位置：
Docker 本地资源存放在 /var/lib/docker/ 目录下：

  • container/ — 存放容器信息
  • image/ — 存放镜像信息
  • overlay2/ — 存放具体的镜像底层文件

容器数据：
容器退出后处于 Exited 状态，通过 docker ps -a 可以查看。数据不会丢失，可通过 docker start 重新启动。只有执行 docker rm 删除容器才会清除数据。

出处：《Docker 那点破事》— Tom哥

14. 如何临时退出容器终端而不终止？

按 Ctrl+P 然后按 Ctrl+Q。

注意：按 Ctrl+C 会向容器内应用进程发送 SIGINT 信号，导致容器终止。

15. 可以在一个容器中运行多个应用进程吗？

一般不推荐。如果需要，可以通过额外的进程管理机制实现，例如使用 supervisord 来管理多个进程。

最佳实践：一个容器只运行一个进程/服务，便于管理、监控和扩缩容。

出处：《Docker 那点破事》— Tom哥

16. Dockerfile 中 COPY 和 ADD 的区别？

指令 说明 推荐
COPY 仅将本地文件拷贝到容器中，行为简单直观 ✅ 默认推荐
ADD 除了 COPY 的功能，还支持本地归档自动解压（如 ADD rootfs.tar.xz /）和远程 URL 下载 仅在需要自动解压时使用

最佳实践：能用 COPY 就用 COPY，ADD 的额外特性可能带来意外行为。

18. Docker 应用流程是怎样的？

  1. Dockerfile — 定义应用的构建方式（源代码）
  2. docker build — 构建镜像（编译产物）
  3. Registry — 发布/共享镜像（类似 Git 仓库）
  4. docker run — 从镜像启动容器运行

出处：《Docker 那点破事》— Tom哥

19. 如何创建与环境无关的容器系统？

三种关键技术：

  1. 存储卷（Volumes） — 将数据持久化到宿主机，与容器生命周期解耦
  2. 环境变量注入（Environment Variable Injection） — 通过 -e 或 --env-file 注入配置
  3. 只读文件系统（Read-only） — 容器根文件系统设为只读，强制数据写入 Volume

出处：《Docker 那点破事》— Tom哥

20. DevOps 有哪些优势？

  1. 持续交付能力 — 自动化构建、测试、部署流水线
  2. 快速修复问题 — 容器化应用可以快速回滚和更新
  3. 更快的解决问题的速度 — 统一环境降低了「在我机器上是好的」问题

0x06 Docker 26+ 新特性与生产进阶

21. BuildKit 有哪些高级特性？（--mount=type=cache / --mount=type=secret / Here-Docs）

标准答案：

Docker 26+（BuildKit 已成默认 Builder）引入了多项 Dockerfile 高级指令，大幅提升构建效率：

深入思考：

  • 生产经验：--mount=type=cache 是 CI/CD 构建提速的关键。Go 的 /go/pkg/mod、npm 的 ~/.npm、pip 的 ~/.cache/pip 都适合缓存。CI 环境需确保 Docker daemon 支持 BuildKit（DOCKER_BUILDKIT=1）
  • 安全：--mount=type=secret 在最终镜像中不保留密钥文件（构建完成后自动清除），比 COPY token.txt / && RUN ... && rm token.txt 安全（后者会在镜像层历史中残留）
  • 边界情况：Here-Docs 中的变量默认不展开（字面传递），需要使用 RUN <<'EOF' 禁止展开或 RUN <<EOF 允许展开

参考来源：Docker BuildKit Documentation, Docker Engine 26+ Release Notes

22. Docker Compose Watch 是什么？解决了什么痛点？

标准答案：

Docker Compose Watch（Docker 26+，Compose v2.22+）实现了开发时的文件热同步，无需重建镜像即可将本地代码变更实时注入容器：

action 行为 适用场景
sync 文件变更时直接拷贝到容器 源代码热更新（不重启）
rebuild 文件变更时重建镜像并重启 依赖变更（package.json）
sync+restart sync 后自动重启容器 配置文件变更

深入思考：

  • 对比 bind mount：bind mount 也能实现文件同步，但 Compose Watch 更精细——可以根据文件类型选择 sync/rebuild/sync+restart，且不会因宿主机和容器文件系统差异导致权限问题
  • 性能：sync 使用 docker cp 机制实现增量同步，比 bind mount 在 macOS/Windows 文件系统性能更好（避免 osxfs 性能损耗）
  • 生产经验：node_modules 等大型依赖目录应排除在 sync 之外（通过 .dockerignore），避免同步大量依赖文件拖慢开发体验

参考来源：Docker Compose Watch Documentation

23. Docker 的新镜像存储后端（containerd image store）有什么变化？

标准答案：

Docker 26+ 引入了 containerd image store（替代传统 graphdriver），将镜像管理完全委托给 containerd，消除 Docker daemon 与 containerd 之间的镜像重复存储：

深入思考：

  • 存量兼容：开启 containerd image store 后，旧的 overlay2 镜像不会自动迁移。需 docker save 导出再 docker load 导入，或重新 docker pull
  • 性能收益：docker pull 速度提升（不再在 Docker daemon 和 containerd 之间拷贝层数据），docker run 启动更快。镜像共用 containerd snapshot，K8s 可以直接复用
  • 注意事项：部分老旧的 Docker 插件不支持 containerd image store。开启前建议在测试环境验证

参考来源：Docker Engine 26+ Release Notes, Moby Project

24. Docker Bake 是什么？如何组织多镜像构建？

标准答案：

Docker Bake（Docker 28+ GA）是声明式的多镜像构建编排工具，用 YAML/HCL/JSON 定义构建矩阵，替代复杂的 shell 脚本：

深入思考：

  • vs docker-compose build：Compose build 是为运行定义的构建，Bake 是为分发定义的构建（多平台、多 tag、矩阵构建）。Bake 支持 --push 直接推送
  • CI/CD 集成：Bake 文件可以版本管理，CI 中只需要 docker bake --push 一条命令完成所有镜像构建和推送
  • 性能：Bake 自动并行构建不相关的目标，最大化利用 BuildKit 的并发能力

参考来源：Docker Buildx Bake Documentation, Docker Engine 28 Release Notes
