2. Docker 镜像分层原理和写时复制？

  • 每一层是一个文件系统的增量变更
  • 拉取镜像时，已存在的层不重复下载（不同镜像可共享层）
  • 写操作使用 COW：修改只读层的文件时，先复制到容器层再修改

优化策略：把变化频率高的指令放下面（如 COPY 源码），变化频率低的放上面（如 apt-get install）

模式 命令 特点

0x05 基础概念与原理

7. 什么是 Docker？

答案：
Docker 是一个容器化平台，将应用及其所有依赖环境打包成一个标准化的容器，保证应用在不同的平台（开发、测试、生产）上可以无缝运行。

Docker 容器将一个软件包装在一个完整的文件系统中，包含运行所需的一切：代码、运行时、系统工具、系统库等，可以安装在服务器上的任何东西。

出处：《Docker 那点破事》— Tom哥

8. Docker 与虚拟机的区别？

维度 Docker 容器 虚拟机
内核 共享宿主机内核 每个 VM 有独立内核
启动时间 毫秒级 分钟级
资源损耗 仅进程级隔离，几乎无损耗 Hypervisor 层占用大量资源
隔离性 通过 namespace/cgroup 实现进程隔离 完全虚拟化，安全性更高
密度 同等资源可运行数十倍于 VM 的容器 数量有限

深入思考：

  • 安全权衡：Docker 与宿主机共享内核，容器逃逸风险比 VM 高。生产环境建议：敏感业务用 VM，微服务/无状态应用用容器
  • 性能：Docker 网络和 IO 接近原生，VM 因虚拟化层有 5-15% 性能损耗

出处：《Docker 那点破事》— Tom哥

9. Docker 三大核心组件是什么？

组件 说明
镜像（Image） 创建容器的基础，类似 VM 快照。一个只读模板，包含运行应用所需的所有内容（代码、运行时、库、环境变量、配置文件）。实际上是一个压缩包，包含完整操作系统文件
容器（Container） 从镜像创建的运行实例。可启动、停止、删除。每个容器相互隔离，可看作一个简易版 Linux 环境（root 用户、文件空间、网络空间）
仓库（Registry） 存放镜像的服务器。每个仓库含多个镜像，每个镜像有不同 tag。公开仓库（Docker Hub）和私有仓库

出处：《Docker 那点破事》— Tom哥

10. Docker 后台的标准运行过程？

当执行 docker run 创建容器时，后台执行流程：

  1. 检查本地是否存在指定镜像，不存在则从仓库下载
  2. 利用镜像创建并启动一个容器
  3. 分配文件系统给容器，在只读镜像层外挂载一层可读写层
  4. 从宿主机网桥接口桥接一个虚拟接口到容器
  5. 分配 IP 地址给容器
  6. 执行用户指定的应用程序，执行完毕后容器终止

出处：《Docker 那点破事》— Tom哥

11. 镜像本地存放在哪里？容器退出后数据会丢失吗？

镜像存储位置：
Docker 本地资源存放在 /var/lib/docker/ 目录下：

  • container/ — 存放容器信息
  • image/ — 存放镜像信息
  • overlay2/ — 存放具体的镜像底层文件

容器数据：
容器退出后处于 Exited 状态，通过 docker ps -a 可以查看。数据不会丢失，可通过 docker start 重新启动。只有执行 docker rm 删除容器才会清除数据。

15. 可以在一个容器中运行多个应用进程吗？

一般不推荐。如果需要，可以通过额外的进程管理机制实现，例如使用 supervisord 来管理多个进程。

最佳实践：一个容器只运行一个进程/服务，便于管理、监控和扩缩容。

18. Docker 应用流程是怎样的？

  1. Dockerfile — 定义应用的构建方式（源代码）
  2. docker build — 构建镜像（编译产物）
  3. Registry — 发布/共享镜像（类似 Git 仓库）
  4. docker run — 从镜像启动容器运行

19. 如何创建与环境无关的容器系统？

三种关键技术：

  1. 存储卷（Volumes） — 将数据持久化到宿主机，与容器生命周期解耦
  2. 环境变量注入（Environment Variable Injection） — 通过 -e 或 --env-file 注入配置
  3. 只读文件系统（Read-only） — 容器根文件系统设为只读，强制数据写入 Volume

深入思考：

22. Docker Compose Watch 是什么？解决了什么痛点？

标准答案：

Docker Compose Watch（Docker 26+，Compose v2.22+）实现了开发时的文件热同步，无需重建镜像即可将本地代码变更实时注入容器：

action 行为 适用场景
sync 文件变更时直接拷贝到容器 源代码热更新（不重启）
rebuild 文件变更时重建镜像并重启 依赖变更（package.json）
sync+restart sync 后自动重启容器 配置文件变更

深入思考：

  • 对比 bind mount：bind mount 也能实现文件同步，但 Compose Watch 更精细——可以根据文件类型选择 sync/rebuild/sync+restart，且不会因宿主机和容器文件系统差异导致权限问题
  • 性能：sync 使用 docker cp 机制实现增量同步，比 bind mount 在 macOS/Windows 文件系统性能更好（避免 osxfs 性能损耗）
  • 生产经验：node_modules 等大型依赖目录应排除在 sync 之外（通过 .dockerignore），避免同步大量依赖文件拖慢开发体验

参考来源：Docker Compose Watch Documentation

深入思考：

0x10 BuildKit 深度

26. BuildKit 架构：LLB 中间表示与前端-后端分离？

标准答案：

BuildKit 是 Docker 的默认构建引擎（Docker Engine v23.0+），其核心架构采用前端-后端分离设计，中间以 LLB（Low-Level Build） 作为桥梁：

  • LLB：BuildKit 的中间表示，定义为一个 Protocol Buffer，本质是一个有向无环图（DAG）。每个节点代表一个操作（SourceOp/ExecOp/FileOp/MergeOp/DiffOp），边定义依赖关系
  • 前端（Frontend）：将人类可读的构建描述（Dockerfile、Nix、HLB、Bass）转换为 LLB DAG。官方前端是 docker/dockerfile:1，作为容器运行在 BuildKit 守护进程中
  • 后端（Backend）：接收 LLB DAG，使用 runc 或 containerd 执行具体操作，最终产出 OCI 镜像

LLB 核心操作类型：

操作 描述 Dockerfile 对应
SourceOp 加载源文件/镜像 FROM / COPY context
ExecOp 执行命令 RUN
FileOp 文件操作 ADD, COPY, WORKDIR
MergeOp 合并多个输入为一个层 COPY –link
DiffOp 计算两个输入的差异 不直接暴露（内部使用）

深入思考：

  • 为什么 LLB 是 DAG 而不是线性流水线：DAG 允许 BuildKit 自动发现无依赖的阶段并并行执行。例如独立的多阶段构建可以同时运行，极大缩短构建时间
  • 前端容器化有何好处：前端运行在容器中，意味着可以在不更新 Docker Engine 的情况下独立升级 Dockerfile 语法解析器。syntax=docker/dockerfile:1.7 可以锁定特定前端版本
  • 自定义前端：社区已有 Nix（nix2container）、Rust（CargoWharf）、Bazel（rules_oci）等替代前端，突破 Dockerfile 的限制

参考来源：BuildKit README (github.com/moby/buildkit), Depot.dev BuildKit in Depth

27. 多阶段构建优化与 BuildKit 缓存挂载实战？

标准答案：

多阶段构建优化三个核心策略：

① 依赖缓存分层

② 合并 RUN 指令但保持可读性

③ COPY –link 避免破坏缓存

COPY --link 使用 MergeOp 将新层作为独立快照合并，即使上游层变化，这一层也不会被重建。

BuildKit 缓存挂载（–mount=type=cache）实战：

缓存挂载 vs 隐式缓存：

  • 隐式缓存：BuildKit 自动缓存 RUN 指令的整个层，但如果指令内容变化则整个缓存失效
  • 显式缓存挂载：--mount=type=cache 将指定目录持久化到 BuildKit 缓存存储中，即使 RUN 指令内容变化，已缓存的依赖包仍然有效

深入思考：

  • sharing 参数：sharing=shared（所有并发构建共享）、sharing=private（每个构建私有挂载）、sharing=locked（同一时间只有一个构建使用）。CI/CD 并行构建时建议 locked 避免竞争
  • 缓存 ID：可通过 id=cache-id 为不同缓存命名，避免冲突
  • 生产经验：缓存挂载的关键是选对挂载点——包管理器的缓存目录（/var/cache/apt、~/.cache/pip、/go/pkg/mod、/root/.m2）是最佳选择。不要缓存编译中间产物，它们可能因源码变化而过时

参考来源：Docker BuildKit Cache Mounts Documentation, SparkFabrik BuildKit Deep Dive

0x11 OCI 镜像规范

29. OCI Image Spec 的核心组成（Manifest/Index/Layers/Config）？

标准答案：

OCI Image Spec（opencontainers/image-spec）定义了容器镜像的标准格式，核心由四个组件构成，构成一个 DAG 引用结构：

① Image Manifest

  • 描述单个平台镜像的组成：一个 config + 有序的 layers 列表
  • layers 中的 digest 是压缩后的 blob 摘要（与 config 中的 diff_id 不同）

② Image Index（Manifest List）

  • 多架构镜像的入口点，引用不同平台的 manifest
  • 客户端拉取时根据 runtime.GOARCH 选择对应 manifest

③ Image Config

  • 定义容器的运行时配置（环境变量、入口命令、工作目录）
  • rootfs.diff_ids 是未压缩层的 SHA256 摘要，用于校验层内容完整性
  • Image ID = SHA256(config JSON) — 唯一标识一个单平台镜像

④ Filesystem Layers

  • 每个层是一个 tar 归档（可 gzip 压缩），包含文件增、删、改的变更集
  • 删除文件使用 whiteout 文件（.wh.<filename>）
  • 运行时使用 OverlayFS/union mount 将多层叠加为统一视图

深入思考：

  • Content-Addressable Storage（CAS）：所有 blob 通过 digest（SHA256）寻址，相同的层在不同镜像间只存储一次。这也是镜像拉取只下载缺失层的原因
  • Manifest vs Config 的关系：Manifest 是分布式的索引（引用压缩 blob），Config 是运行时的定义（引用未压缩 diff）。两者的 digest 角度不同
  • OCI vs Docker Schema：Docker 早期使用 Schema 1（V1）格式，已被 OCI 标准取代。现代工具都兼容 OCI Spec

参考来源：OCI Image Spec (github.com/opencontainers/image-spec), iximiuz Labs - Container Image From Scratch

32. Rootless 模式原理？

标准答案：

Rootless 模式允许 Docker 守护进程和容器在非 root 用户下运行，无需任何 SETUID 二进制或文件 capabilities（除了 newuidmap/newgidmap）。

工作原理：

核心技术：

  1. 用户命名空间（User Namespace）：宿主机上的非 root 用户（如 uid 1001）在用户命名空间内映射为 root（uid 0），容器进程认为自己是 root，但宿主机上没有任何特权
  2. newuidmap/newgidmap：仅在需要多 UID 映射时使用，从 /etc/subuid 和 /etc/subgid 读取子 UID 范围
  3. fuse-overlayfs：Rootless 模式下无法使用内核 OverlayFS（需要 root），使用 FUSE 实现的 overlayfs 替代
  4. Slirp4netns：用户态网络栈，为 rootless 容器提供网络连接

安装与验证：

局限性：

  • 需要 slirp4netns 用户态网络，性能略低于 bridge 模式
  • 部分存储驱动不可用（overlay2 不可用，需 fallback 到 fuse-overlayfs）
  • 不能绑定 1024 以下特权端口（可通过 setcap 或在宿主机上端口转发解决）
  • 不支持 --privileged 模式

深入思考：

  • Rootless vs userns-remap：userns-remap 是 Docker 的早期方案（dockerd 仍以 root 运行，但将容器 uid 映射到非 root 范围），Rootless 则连 dockerd 本身都是非 root 运行。Rootless 安全性更高但功能受限更多
  • 攻击面收益：即使容器内逃逸成功，攻击者在宿主机上只有一个普通用户权限，无法获得 root，大幅降低破坏范围
  • 生产经验：Rootless 在 Docker Desktop 上默认启用（macOS/Windows），但在 Linux 生产环境中使用仍较少——Kubernetes 节点通常直接用 rootful Docker + userns-remap 作为折中

参考来源：Docker Rootless Mode Documentation

33. Seccomp/AppArmor/Capabilities 安全配置？

标准答案：

三种安全机制形成深度防御的三层体系：

① Linux Capabilities

Docker 默认移除 44 个 capabilities 中的大部分，仅保留 CHOWN、DAC_OVERRIDE、FSETID、FOWNER、MKNOD、NET_RAW、SETGID、SETUID、SETFCAP、SETPCAP、NET_BIND_SERVICE、SYS_CHROOT、KILL、AUDIT_WRITE。

  • Seccomp 的生效时机：Seccomp 策略在系统调用进入内核时拦截，对性能影响极小（纳秒级）。但白名单模式如果遗漏必要的 syscall，容器会静默崩溃——调试非常困难，建议先用 SCMP_ACT_LOG 模式记录

36. 生产环境 Docker Compose 最佳实践？

标准答案：

核心最佳实践清单：

  1. 端口绑定：绑定 127.0.0.1，用反向代理（Nginx/Traefik）对外暴露
  2. 资源限制：始终设置 deploy.resources.limits 防止容器撑爆宿主机
  3. 安全基线：no-new-privileges:true + cap_drop: ALL + cap_add: NET_BIND_SERVICE + read_only: true
  4. 日志管理：设置 max-size 和 max-file，防止无限制写日志耗尽磁盘
  5. Secrets 管理：敏感信息通过 secrets 注入，不写死在 environment 或 Dockerfile 中
  6. 健康检查：所有关键服务配置 healthcheck
  7. YAML 锚点：使用 &/<<:* 避免重复配置

深入思考：

  • Docker Compose vs Kubernetes：Docker Compose 不是 K8s 的替代品。生产环境建议用 Compose + Docker Swarm（轻量多节点）或用 Compose 做单机开发/预发布，K8s 做生产
  • 生产陷阱：Compose 的 restart: always 在系统重启后不会自动拉起——需要使用 restart: unless-stopped（Docker daemon 重启时自动恢复）或 systemd 管理
  • CI/CD 集成：在 CI 中可以使用 docker compose run --rm test 运行临时测试容器，测试完成后自动清理

38. runc / containerd / crun 对比？

标准答案：

各层职责：

组件 角色 描述
containerd 容器管理平台 管理容器生命周期、镜像拉取/存储、网络接口、卷管理。CNCF 毕业项目
containerd-shim 容器守护代理 每个容器对应一个 shim 进程，隔离 runc 退出对容器的影响
runc OCI 运行时 OCI Runtime Spec 参考实现，使用 cgroups/namespaces 创建容器。Go 编写
crun OCI 运行时 C 语言实现的 OCI 运行时，性能显著优于 runc

runc vs crun 性能对比：

指标 runc (Go) crun (C) 差距
启动时间（冷） ~100-200ms ~20-40ms crun 快 5-10x
内存占用 ~10-15MB ~500KB crun 节省 95%+
Fork 容器开销 ~150ms ~15ms crun 快 10x
源码语言 Go C —

选择建议：

  • 通用场景 → runc（兼容性最好，Docker/K8s 默认）
  • 大规模/Serverless → crun（低延迟启动，如 OpenFaaS、Knative）
  • 边缘/IoT → crun（低内存占用，适合资源受限设备）
  • 安全敏感 → runc（更成熟，生态更广）
  • cgroup v2 环境 → runc 1.1+ 或 crun 均支持

深入思考：

  • containerd-shim 的必要性：如果没有 shim，runc 退出后容器会一起退出。shim 充当”代理守护”，即使 runc 退出，容器进程仍作为 shim 的子进程继续运行
  • Kubernetes CRI 集成：Kubernetes 通过 CRI 接口与 containerd 交互（cri-o 是另一个 CRI 实现）。containerd 内部再调用 runc/crun
  • Kata Containers / gVisor：替代 runc 的安全容器运行时——Kata 使用轻量虚拟机隔离，gVisor 使用用户态内核拦截系统调用。它们都实现了 OCI Runtime Spec，因此可以替换 runc
  • 生产经验：从 runc 切换到 crun 只需配置 containerd 的 runtime 配置项，无需重启容器。建议先在非关键节点做灰度切换

参考来源：containerd Architecture, crun GitHub README, OCI Runtime Spec

40. 容器网络（CNI 插件）？

标准答案：

CNI（Container Network Interface） 是 CNCF 项目，定义了配置容器网络接口的标准。containerd 和 Kubernetes 使用 CNI 而不是 Docker 自带的 libnetwork。

CNI 架构：

常见 CNI 插件对比：

插件 模式 性能 特性
Bridge（基础） Linux bridge 高 最基本的容器间通信
Flannel VXLAN / host-gw 中 简单易用，覆盖网络
Calico BGP / IPIP 高 原生 BGP 路由，NetworkPolicy
Cilium eBPF 极高 eBPF 数据面，透明加密，可观测性
Weave 自定义 UDP 中 加密通信，跨主机组播
Macvlan 子接口 极高 直接分配物理网络 IP，无 NAT

Docker 默认网络（bridge）的工作原理：

CNI 与 Docker libnetwork 的差异：

  • Docker/libnetwork：自创 API，绑定 Docker Engine
  • CNI：标准 API，通过 JSON 配置 + 二进制插件执行，与容器运行时解耦
  • Kubernetes 使用 CNI；Docker Swarm 使用 libnetwork

深入思考：

  • Cilium + eBPF 为何成为趋势：eBPF 允许在网络栈中执行用户程序，无需修改内核代码或加载内核模块。Cilium 利用 eBPF 实现了比 iptables 更高效的网络策略（O(1) 匹配 vs 链式遍历），支持透明加密（WireGuard），并提供 Hubble 可观测性
  • Calico BGP 模式：Calico 不使用隧道封装（VXLAN/IPIP），而是通过 BGP 协议将容器 IP 路由通告到物理网络。性能接近裸机但需要支持 BGP 的网络基础设施
  • 网络策略性能：在大量策略下，iptables 规则链遍历成为瓶颈（超过 1000 条规则延迟显著）。Cilium eBPF 的策略匹配是 O(1)，性能不随策略数量下降
  • 生产选择：小集群（<50 节点）→ Flannel（简单可靠）；中大型（50-500 节点）→ Calico（BGP 路由 + NetworkPolicy）；大型 + 安全敏感 → Cilium（eBPF + 加密 + 可观测）

41. 存储驱动 overlay2 原理？

标准答案：

OverlayFS 核心概念：

关键机制：

  • lowerdir：镜像的只读层，多个 lower 层堆叠，最多 128 层
  • upperdir：容器的读写层，容器中修改的文件写入此层
  • merged：lowerdir + upperdir 的统一视图
  • workdir：OverlayFS 内部使用，用于原子操作

读写操作流程：

操作 流程
读取已有文件 从 upperdir 查找 → 未找到则从最上层 lowerdir 向下遍历
首次写入文件 Copy-up：将 lowerdir 中整个文件复制到 upperdir，再修改
删除文件 在 upperdir 创建 whiteout 文件（c 0 0 字符设备）
重命名目录 仅在同一层中允许，跨层返回 EXDEV 错误

overlay2 磁盘结构：

页面缓存共享：多个容器使用同一镜像层时，内核共享同一页面缓存（Page Cache）。这意味着从镜像层读取文件的 RAM 消耗不随容器数量线性增长，这是 Docker 能实现高密度的关键。

深入思考：

  • copy-up 的性能影响：首次写入大文件时（如日志文件、数据库写入），整个文件从 lowerdir 复制到 upperdir。对于 1GB 文件，首次写入延迟可达数百毫秒。解决方案：写入密集的数据放 volume（绕开存储驱动）
  • overlay vs overlay2：overlay（旧版）最多只支持 2 个 lower 层，overlay2 支持 128 层。后者也是自 Docker 18.06 起推荐使用的版本
  • containerd overlayfs snapshotter：Docker Engine 29+ 默认使用 containerd 镜像存储，其 snapshotter 本质与 overlay2 相同但实现不同。迁移后目录结构变为 /var/lib/docker/containerd/daemon/io.containerd.snapshotter.v1.overlayfs/
  • 磁盘使用监控：docker system df -v 可以查看每个镜像/容器占用的存储空间。注意镜像层是共享的，计入 Shared Size

参考来源：Docker overlay2 Storage Driver Documentation, Linux Kernel OverlayFS Documentation
