后端面试题：Java 高级面试全解 — 高并发、JVM、设计模式

发表于 2026-06-07 更新于 2026-06-08 分类于 后端

Java 高级面试全解，覆盖多线程与锁机制、JVM 内存与 GC 调优、经典设计模式实战。面向高级研发，侧重源码理解与架构设计。

0x00 多线程与锁

1. synchronized 锁升级过程是怎样的？

JDK 6 后 synchronized 引入了锁升级机制，性能大幅提升：

  • 偏向锁：第一个线程获得锁时，在对象头记录线程 ID，后续该线程再次进入无需 CAS
  • 轻量级锁：第二个线程竞争时，偏向锁撤销。线程通过 CAS 在对象头设置 Lock Record 指针。失败则自旋等待（自适应自旋）
  • 重量级锁：自旋超过一定次数（默认 10 次或 CPU 核数/2）仍未获取，膨胀为重量级锁。线程进入 BLOCKED 状态，由 OS 调度

深入思考：

  • 边界情况：偏向锁在竞争激烈的场景下反而是累赘——每次撤销偏向锁需要到达 SafePoint（STW）。高并发场景建议 -XX:-UseBiasedLocking 关闭偏向锁
  • 源码层面：锁状态存储在 Mark Word 的最低 3 位：001(无锁) → 101(偏向) → 00(轻量级) → 10(重量级)

生产实战：

  • 踩坑案例：某金融风控服务（JDK 8，4C8G × 6 节点）每天 10:00 和 14:00 准时出现 200-500ms 的 STW 抖动，GC 日志显示 SafePoint 时间异常偏高。最终通过 -XX:+PrintSafePointStatistics 定位到偏向锁撤销是罪魁祸首——风控规则引擎每秒创建大量短期线程竞争同一个锁对象，每次偏向锁撤销都需要全局 SafePoint。修复方案：-XX:-UseBiasedLocking 关闭偏向锁后，SafePoint 暂停时间从平均 150ms 降至 5ms 以下，P999 延迟从 800ms 降至 120ms。
  • 运维检查清单：
    检查项 命令/参数 说明
    查看偏向锁是否启用 -XX:+PrintFlagsFinal | grep BiasedLocking JDK 15+ 默认关闭
    SafePoint 统计 -XX:+PrintSafePointStatistics 生产谨慎开启（有性能开销）
    锁撤销次数监控 JMX LockInfo.getOwner() 趋势 撤销次数突增需关注
    JDK 版本检查 JDK 8u191+ 偏向锁延迟开启 低版本有 Bug 导致异常 STW
  • 面试金句（升级版）：

    “偏向锁是 JDK 6 为低竞争场景做的优化，在高并发服务中它是性能杀手——每次撤销都需要全局 SafePoint。我在金融风控服务中实测，关闭偏向锁后 SafePoint 暂停从 150ms 降到 5ms。JDK 15+ 已经默认关闭偏向锁，团队升级到 JDK 17+ 时建议直接 -XX:-UseBiasedLocking。”

参考：OpenJDK Synchronization | JDK-8186043 BiasedLocking 默认关闭

2. AQS (AbstractQueuedSynchronizer) 是怎么实现 ReentrantLock 的？

AQS 是 JUC 的核心基石：

ReentrantLock 公平锁实现：

非公平锁：去掉 hasQueuedPredecessors() 检查，新线程直接 CAS 抢锁，插队成功就获取

深入思考：

  • 性能差异：非公平锁吞吐量更高（减少线程切换），但可能导致队列尾部线程饥饿。默认 ReentrantLock() 是非公平的
  • 与 synchronized 对比：AQS 支持超时获取 (tryLock(timeout))、可中断获取 (lockInterruptibly)、条件变量 (Condition)——这些都是 synchronized 不具备的

生产实战：

  • 公平锁 vs 非公平锁选型依据：不是简单的”非公平更快”。具体决策参考下表：

    场景 推荐策略 原因
    分布式锁客户端（ZooKeeper/etcd 封装层） 公平锁 客户端间需公平竞争，防止某个客户端永远抢不到锁
    本地资源池（连接池、内存缓存） 非公平锁 吞吐优先，短暂的 unfairness 可接受
    定时任务调度（同一任务多节点争抢） 公平锁 避免某个节点因饥饿导致任务超时未被调度
    高吞吐消息处理（Kafka 消费线程） 非公平锁 + tryLock(timeout) 吞吐第一，超时未获取直接跳过

  • Condition 的坑：signal() 之后并非立即执行——只是将等待队列中的节点转移到同步队列，被 signal 的线程和其他正在 acquire 的线程一起竞争。踩坑案例：某消息推送服务使用 Condition.await() 做批量消费，每次 signalAll() 后只有一个线程抢到锁，其余被 signal 的线程回到等待队列——本该并行处理的消息变成了串行。修复方案：改用 signal() 逐个唤醒，或者使用 Semaphore 代替 Condition。

  • AQS 队列头节点丢失的踩坑：ReentrantLock 的 lockInterruptibly() 在中断时会将当前节点从等待队列移除，但如果节点恰好是 head，会导致头指针悬空，极端情况下触发 NPE。JDK 9 已修复（JDK-8154414），低版本用户要注意。

  • 监控指标：

    指标 获取方式 告警阈值
    队列等待线程数 ReentrantLock.getQueueLength() 持续 > 10 需关注
    Condition 等待线程数 getWaitQueueLength(condition) 持续增长需排查
    持有锁时间 通过 AOP 埋点统计 P99 > 100ms 告警

  • 面试金句（升级版）：

    “ReentrantLock 在生产中的选型不是简单的’非公平更快’。我维护过一个分布式任务调度系统，使用公平锁确保了每个 Worker 节点都能分配到任务，避免了非公平锁下的任务饥饿。Condition 的 signal 只是节点转移，不是立即执行——这个语义差异在实现批量消费时踩过大坑。”

3. CountDownLatch、CyclicBarrier、Semaphore 的区别？

0x02 线程池

4. ThreadPoolExecutor 的 7 个参数及工作流程？

工作流程：

四种拒绝策略：

  • AbortPolicy（默认）：抛 RejectedExecutionException
  • CallerRunsPolicy：由提交任务的线程自己执行（天然限流）
  • DiscardPolicy：静默丢弃
  • DiscardOldestPolicy：丢弃队列最老的任务，重试提交

深入思考：

  • 生产经验：不要用 Executors.newFixedThreadPool() 和 Executors.newCachedThreadPool()——前者无界队列可能 OOM，后者无限线程创建。必须用 new ThreadPoolExecutor() 显式指定参数
  • 大小公式：CPU 密集型 coreSize = CPU核数 + 1；IO 密集型 coreSize = CPU核数 × 2

生产实战：

  • 生产级参数制定方法：不要迷信”CPU密集=cores+1, IO密集=cores×2”这个简单公式。真实场景中线程池承载的通常是混合负载，需要基于 实际压测 制定参数：

  • 线程池预热：prestartAllCoreThreads() 在服务启动时立即创建所有核心线程，避免第一个请求到达时因线程创建产生延迟。踩坑案例：某网关服务上线后前 30 秒的请求延迟高达 2s，GC 日志无异常——JFR 显示是线程创建耗时。加上 prestartAllCoreThreads() 后首请求延迟降至 50ms。类似的，prestartCoreThread() 逐个预热（适用于需要阶梯式初始化的场景）。

  • 线程池监控（线上必设）：

    指标 获取方式 告警阈值 说明
    活跃线程数 executor.getActiveCount() > corePoolSize × 0.8 线程资源吃紧
    队列积压 executor.getQueue().size() > 队列容量 × 0.7 处理速度跟不上入队速度
    拒绝次数 封装 Handler 计数器 > 0 系统过载，需扩容或限流
    最大池大小 executor.getLargestPoolSize() > corePoolSize 曾触发创建临时线程
    完成任务数 executor.getCompletedTaskCount() 速率突降 可能线程被阻塞或死锁

  • 面试金句（升级版）：

    “线程池参数没有银弹公式——我在八个生产服务中总结出的规律是：压测定基线、监控保稳定、预热防抖动。prestartAllCoreThreads() 被我列为所有线上服务的启动标配，它能消除上线后前 30 秒的’冷启动’延迟。队列必须是有界的——newFixedThreadPool 的无界队列是 OOM 的温床。拒绝策略我倾向 CallerRunsPolicy，因为它不仅降级，还天然限流——提交线程变慢，迫使上游背压。”

5. 如何优雅关闭线程池？

关键区别：shutdown() 不中断正在执行的线程，shutdownNow() 调用 Thread.interrupt() 中断。被 shutdownNow() 中断的线程如果吞掉 InterruptedException 则不会停止。

生产实战：

  • 关闭顺序的坑：如果服务中多个线程池存在依赖关系（例如：生产者线程池 → 队列 → 消费者线程池），必须先关闭生产者再关闭消费者。某支付结算服务曾因先 shutdown 消费者线程池，导致生产者仍在提交任务到阻塞队列，消费者已关闭——任务堆积最终触发拒绝策略，产生大量支付结果未处理的脏数据。

  • 线程池泄露排查：服务下线后发现 JVM 进程未退出，jstack 看到线程池的 worker 线程仍在阻塞等待队列任务。原因是 shutdown() 后队列中有未处理任务，但 awaitTermination 超时时间设得太短，代码直接走 catch 分支忽略了。修复方案：超时后调用 shutdownNow() 清空队列，然后 List<Runnable> remaining = executor.shutdownNow() 记录未执行任务日志，由人工确认补偿。

  • Java 19+ 的改进：ExecutorService 在 JDK 19 引入了 close() 方法（实现 AutoCloseable），配合 try-with-resources 可以自动关闭：

  • 面试金句（升级版）：

    “优雅关闭线程池本质是’先断粮、再收队、后清理’。shutdown() 断新任务，awaitTermination 等现有任务收尾，超时后用 shutdownNow() 强制收回——但收回的任务必须记录日志，确保可以人工补偿。我遇到过因为关闭顺序反了导致支付数据丢失的 P0 事故，从此所有线程池关闭逻辑都加了任务追踪。”

0x03 JVM 内存模型

6. JVM 运行时数据区有哪些？各自存什么？

  • 堆：new 出来的对象和数组。GC 的主要战场，分年轻代(Eden+S0+S1)和老年代
  • 方法区（元空间）：类的结构信息（字段、方法、常量池）、JIT 编译后的代码缓存。JDK 8+ 用本地内存实现，不再有 -XX:MaxPermSize
  • 虚拟机栈：每个方法执行时创建栈帧，存局部变量表、操作数栈、方法返回地址。StackOverflowError 和它的关系最大
  • 本地方法栈：native 方法使用的栈
  • 程序计数器：当前线程执行的字节码行号

0x04 GC 调优

7. G1 GC 的核心原理是什么？什么时候选 G1？

G1 将堆划分为大小相等的 Region（默认 2048 个，每个 1-32MB）：

核心流程（Mixed GC）：

  1. 初始标记（STW，随 Young GC 一起）：标记 GC Roots 直接可达对象
  2. 并发标记：三色标记法，与用户线程并发
  3. 最终标记（STW，短）：处理并发期间的变化（SATB 快照）
  4. 筛选回收（STW）：选垃圾最多的 Region 回收（Garbage-First 的由来）

选型标准：

  • 堆 > 4GB → G1（CMS 在大堆下碎片化严重）
  • 延迟要求 < 200ms → G1（可设置 -XX:MaxGCPauseMillis）
  • JDK 9+ → G1 是默认 GC
  • 堆 < 4GB 且对吞吐量要求高 → Parallel GC

生产实战：

  • Region 大小对大对象的致命影响：G1 的 Region 大小决定了”大对象（Humongous）”的判定阈值——超过 Region 大小 50% 的对象直接分配到 Humongous Region。Humongous Region 的 GC 效率极差：Mixed GC 中需要扫描整个大对象区域；Full GC 时大对象区域可能成为碎片化源头，导致晋升失败（Promotion Failed）。

    • 踩坑案例：某广告推荐服务（JDK 11，G1，堆 8GB）每天下午高峰出现 3-5 次 Full GC，每次 STW 长达 5-8 秒。排查发现：推荐模型更新的 byte[] 缓存平均大小为 3.2MB，而 G1 默认 Region 大小按堆推算约为 4MB（8GB / 2048）。3.2MB > 4MB × 50%（2MB），触发了 Humongous 分配。高峰期 100+ 个大对象同时分配，Humongous Region 快速耗尽，触发 Full GC。修复方案：将 -XX:G1HeapRegionSize 从默认的 4MB 增大到 8MB，使大对象判定阈值升至 4MB，3.2MB 的缓存不再被视为大对象，Full GC 消失。
    堆大小 默认 Region 数 默认 Region 大小 Humongous 阈值 建议
    4GB 2048 2MB 1MB 有 1-2MB 对象时调大到 4MB
    8GB 2048 4MB 2MB 有 2-4MB 对象时调大到 8MB
    16GB 2048 8MB 4MB 有 4-8MB 对象时调大到 16MB
    32GB 2048 16MB 8MB 注意大 Region 可能增加暂停时间

  • G1 Mixed GC 触发频率调优：-XX:G1MixedGCLiveThresholdPercent=85（默认 85%）控制 Mixed GC 只回收存活对象占比低于 85% 的 Region。如果线上 Mixed GC 回收效果差（回收前后堆占用变化不大），可以降低此阈值到 65%。

  • 面试金句（升级版）：

    “G1 的 Region 大小不是随便选的——它直接决定什么对象算’大对象’。我遇到过广告推荐服务因为缓存对象刚好超过 Region 50% 阈值，导致频繁触发 Humongous Region 分配和 Full GC。把 Region 从 4MB 调到 8MB 后 Full GC 完全消失。调 G1 的第一步不是改暂停时间，而是确认 Region 大小和你的对象大小匹配。”

8. OOM 排查流程？

常见 OOM 原因：

  • 堆溢出：ThreadLocal 没 remove、集合类无限增长、缓存没上限
  • 元空间溢出：动态生成大量类（CGLIB 代理、Groovy 脚本）
  • 直接内存溢出：NIO 的 DirectByteBuffer 没释放（-XX:MaxDirectMemorySize）

生产实战：

  • 踩坑案例：ThreadLocal 泄漏导致 Full GC 风暴：某风控规则引擎服务（JDK 8，4C8G × 12 节点）上线一周后频繁触发 Full GC（每 10 分钟一次），每次 STW 3-5 秒。jmap -histo:live 显示 char[] 和 HashMap$Node 对象数量异常。MAT 分析 Dominator Tree 发现：ThreadLocalMap$Entry 占用了约 60% 的堆空间，每个 Entry 的 value 都是一个包含 Map<String, Object> 的风控上下文对象。根因：规则引擎在处理每个请求时 ThreadLocal.set(contextMap) 存储上下文，但请求结束后未在 finally 中 remove()。Tomcat 的请求处理线程是复用的——线程不销毁，ThreadLocalMap 持续增长，OOM 只是时间问题。修复方案：在请求过滤器的 finally 块中 tl.remove()，同时增加 Arthas 监控 ognl '#threadLocals=@ThreadLocal@threadLocalMap' 实时检查泄漏。

  • OOM 排查三板斧：

    步骤 工具 关键操作
    第一板斧：快速定位 jmap -histo:live <pid> 查看存活对象类型分布，找到异常增长的对象
    第二板斧：获取快照 jmap -dump:live,format=b,file=/tmp/heap.hprof <pid> 注意 live 参数会触发 Full GC（生产慎用）
    第三板斧：深入分析 MAT / JProfiler Dominator Tree → Leak Suspects → Path to GC Roots

  • 面试金句（升级版）：

    “OOM 排查最重要的是’让证据完整’——-XX:+HeapDumpOnOutOfMemoryError 是所有线上服务的必配参数。我处理过最典型的 OOM 是 ThreadLocal 泄漏：规则引擎服务上线一周后 Full GC 爆发，MAT 一查 ThreadLocalMap 占堆 60%。根因就是 request 处理完没 remove。从此所有 ThreadLocal 使用都加 try-finally 模板——这是团队的铁律。”

11. ConcurrentHashMap 1.7 vs 1.8 底层原理对比

难度：⭐⭐⭐
问题：ConcurrentHashMap 在 JDK 1.7 和 1.8 中的底层实现有什么区别？为什么 JDK 1.8 要重构？

解析：

  • JDK 1.7：采用 Segment（分段锁）+ HashEntry 数组。默认 16 个 Segment，每个 Segment 继承 ReentrantLock，锁粒度较粗。get 操作无锁（volatile 保证可见性），put 操作对单个 Segment 加锁。

  • 优势：锁粒度更细，并发度从 16 提升到数组长度；synchronized 在 JDK 1.8 有偏向锁/轻量级锁优化，性能不低于 ReentrantLock；空间占用更小（省略 Segment 对象）。

问题：Java 8 Stream 的惰性求值是如何实现的？并行流（parallelStream）底层使用什么线程池？有哪些常见的陷阱？

解析：

  • 惰性求值：Stream 分为中间操作（filter、map）和终端操作（collect、forEach）。中间操作只构建操作流水线（Sink 链），不实际计算；终端操作触发完整计算。

  • 边界情况：findFirst() 在并行流中比 findAny() 慢得多——findFirst() 需要保持 encounter order，拆分和合并时强制有序，破坏了并行优势。数据量小于 10,000 元素时，并行流通常比串行慢。limit(n) 在并行流中可能消耗比 n 更多的元素（拆分后再合并取前 n）。

14. ThreadLocal 内存泄漏原理

难度：⭐⭐
问题：ThreadLocal 为什么会发生内存泄漏？如何避免？

解析：
ThreadLocal 的内存泄漏源于其内部存储结构的引用设计：

  • ThreadLocalMap 的 Entry 继承 WeakReference<ThreadLocal>，key（ThreadLocal 对象）是弱引用，value 是强引用。
  • 当 ThreadLocal 对象不再被外部强引用后，GC 会回收 key（变为 null），但 value 仍有强引用链：Thread → ThreadLocalMap → Entry → value。
  • 线程池场景最为致命——核心线程长期存活，线程的 ThreadLocalMap 永远不会被回收，key=null 的 Entry 持续堆积，value 无法释放，最终 OOM。

解决方案：

  1. 每次使用后显式调用 TL.remove()
  2. Try-finally 确保：
  1. JDK 内部做了防御：get()/set()/remove() 时会遍历 Entry 数组，清理 key=null 的过期 Entry（expungeStaleEntry() 方法），但不保证一定会调用到。

深入思考：

  • 边界情况：使用 InheritableThreadLocal 时，子线程会复制父线程的 ThreadLocalMap，如果父线程反复创建子线程（如线程池提交新任务），会导致大量复制和内存泄漏。JDK 21 的 ScopedValue 是更好的替代方案。
  • 源码层面：ThreadLocalMap 的 Entry 数组初始容量 16，负载因子 2/3。expungeStaleEntry() 不仅清理当前 stale slot，还会遍历后续 Entry 重新哈希（rehash），这是 ThreadLocalMap 不使用链表而用开放地址法的原因。
  • 对比延伸：ThreadLocal 的弱引用设计与 Netty 的 FastThreadLocal 形成对比——Netty 用数组替代 Map，直接通过 index 访问，空间换时间。ScopedValue（JDK 21, JEP 429）比 ThreadLocal 更轻量：不可变、不泄漏、不限制虚拟线程使用。
  • 生产经验：Spring 的 RequestContextHolder、TransactionSynchronizationManager 底层都依赖 ThreadLocal，使用完后框架会主动清理。但如果自定义 ThreadLocal 存储用户信息、TraceId、SQL 连接等资源，必须确保在请求结束或线程归还池前 remove()。Arthas 的 ognl 命令行可以实时查看 ThreadLocalMap 内容来排查泄漏。

面试金句：

“ThreadLocal 内存泄漏的根源是’弱引用的 key 和强引用的 value 生命周期不同步’。key 被 GC 回收后，value 仍然活在 Thread 的引用链中。线程池放大了这个问题——线程不销毁，泄漏持续累积。解决方案简单但关键：try{set()}finally{remove()}，这是使用 ThreadLocal 的铁律。”

15. Java 虚拟线程（Virtual Threads）原理与大数据应用

难度：⭐⭐⭐
问题：JDK 21 正式发布的虚拟线程（Virtual Threads / Project Loom）是什么？与传统平台线程相比的优势是什么？在大数据 IO 密集型场景下如何应用？

解析：

虚拟线程（JEP 444，JDK 21 GA）是 Java 的轻量级用户态线程，由 JVM 调度而非 OS 内核调度。一个平台线程（Platform Thread）对应一个 OS 线程，而数百万虚拟线程可以复用在少量平台线程（Carrier Thread）上。

核心原理：

  1. 虚拟线程由 ForkJoinPool（默认 commonPool）中的 carrier 线程执行
  2. 遇到阻塞操作（网络 IO、文件 IO、BlockingQueue.take()）时，虚拟线程自动 yield，carrier 被释放去执行其他虚拟线程
  3. 阻塞恢复后，虚拟线程重新挂载到某个 carrier 上继续执行
  4. 整个过程对业务代码完全透明——写同步阻塞代码即可享受异步性能

平台线程 vs 虚拟线程对比：

调度者 OS 内核 JVM（ForkJoinPool）
数量上限 几千（受限于栈内存 1MB+） 百万级（栈仅几百字节）
创建开销 高（系统调用 + 1MB 栈预分配） 极低（堆内分配，微秒级）
上下文切换 微秒级（内核态） 纳秒级（用户态）
适用场景 CPU 密集型 IO 密集型（最佳）
阻塞行为 阻塞 OS 线程 仅阻塞虚拟线程本身
调试友好性 栈深度清晰 栈深度清晰（不丢失）

注意事项（Pitfalls）：

  1. CPU 密集任务不适合——虚拟线程不减少 CPU 计算量，且多了调度开销
  2. synchronized 钉住（Pinning）：synchronized 块内阻塞会钉住 carrier 线程，应改用 ReentrantLock
  3. 线程局部变量（ThreadLocal）：虚拟线程数量极大，ThreadLocal 可能导致严重内存泄漏
  4. 池化虚拟线程无意义：虚拟线程创建/销毁轻量，不应池化（Executors.newCachedThreadPool() 或 newVirtualThreadPerTaskExecutor()）

  • 生产经验：Kafka 消费者场景中，每条消息一个虚拟线程是可行的（数百万条也无压力），但要小心每条消息处理时可能创建新的数据库连接——连接池大小也需要同步扩大。Flink 用户自定义函数中使用虚拟线程时，需注意 Flink 的 Checkpoint 屏障与虚拟线程的交互（阻塞时是否会影响 Checkpoint 对齐）。

“虚拟线程将 Java 的并发模型从’以线程为中心’转向’以任务为中心’——开发者写同步阻塞代码就能获得异步框架的性能。大数据场景中，Kafka 消费者、HTTP/RPC 调用、数据库写入等 IO 密集环节是最大受益者。但需要警惕：synchronized 钉住和 ThreadLocal 滥用是虚拟线程最容易踩的两个坑。Flink 社区已开始探索在虚拟线程上运行用户自定义函数，以简化 AsyncIO 编程模型。”

解析：

Java 新特性概述：

深入思考：

  • 边界情况：Pattern Matching Switch 中如果 Sealed class 在后续版本增加了新子类，所有 switch 表达式会编译失败（Exhaustiveness check），强迫开发者处理新类型——这是特性而非缺陷。Record 无法继承其他类（extends 已被隐式继承的 java.lang.Record 占用），在需要基类行为复用时不适用，应使用抽象类替代。
  • 源码层面：Record 的编译器糖由 javac 在解析阶段生成：编译器自动为 Record 生成 canEqual() 方法（确保 equals 的对称性），以及 writeReplace() 方法支持序列化。Pattern Matching Switch 的编译产物在 JDK 21 中改用了 invokedynamic + TypeSwitch 匿名类，比 JDK 17 的 tableswitch/lookupswitch 更高效。
  • 对比延伸：Java 17/21 的代数数据类型（ADT）与 Haskell 的 data type、Scala 的 case class + sealed trait、Kotlin 的 sealed class 本质相同。但 Java 的 Record + Pattern Matching 更加”浅显”——没有 Monad 等复杂概念，对团队入门友好。
  • 生产经验：升级到 JDK 21 后，Flink/Spark 中定义事件类型应优先使用 Record 替代 @Data。但需注意：Flink 的 Kryo 序列化对 Record 支持不完美（Record 的序列化走 Java 原生序列化），建议在 Flink ExecutionConfig 中启用 Flink 自带的 PojoSerializer 或者使用 Avro。Spark 3.5+ 的 Dataset 对 Record 的 Encoder 支持良好。

17. 反射的原理是什么？如何提升反射性能？

反射是 Java 语言的重要特性，它允许程序在运行时获取任意类的内部信息并操作其属性和方法。

核心原理：

  • 反射 API 的核心是 Class 对象，JVM 在类加载阶段为每个类在堆中生成对应的 Class 对象
  • 通过 Class.forName()、类名.class、对象.getClass() 三种方式获取 Class 对象
  • 获取 Class 对象后，通过 Field、Method、Constructor 等反射类访问类的所有成员（包括私有成员）

提升反射性能的方案：

  1. setAccessible(true) 关闭 Java 安全检查
  2. 缓存 Method/Field 对象，避免重复获取
  3. 使用 ReflectASM、MethodHandle 等字节码生成方式
  4. 对于高频调用场景，优先使用接口或抽象类替代反射

深入思考：

  • 边界情况：JDK 9+ 模块化系统对反射有严格限制，跨模块反射默认被禁止，需要 --add-opens JVM 参数放行。setAccessible(true) 在模块化系统下可能抛出 InaccessibleObjectException。
  • 源码层面：Class.forName() 默认执行类的静态初始化块，而 ClassLoader.loadClass() 不会触发初始化，两者有本质区别。
  • 生产经验：Spring 框架大量使用反射实现 IoC/DI，但 Spring 会缓存反射元数据（ReflectionUtils），避免重复反射的性能开销。Jackson 序列化库优先使用 Getter/Setter 方法反射，只有在无法获取时才使用字段反射。

参考：technology-talk — Tom哥，Apache-2.0

18. Java 序列化与 serialVersionUID 的作用？

序列化与反序列化：

  • 序列化将 Java 对象转换为字节流，便于网络传输或持久化
  • 反序列化将字节流恢复为 Java 对象
  • 实现 Serializable 接口即可开启序列化能力

serialVersionUID 的作用：

  • 用于标识类的版本兼容性
  • 反序列化时 JVM 比较字节流中的 serialVersionUID 与当前类的 serialVersionUID，不一致则抛出 InvalidClassException
  • 若未显式声明，JVM 会根据类结构自动生成一个 UID，任意字段变更都会导致 UID 变化

最佳实践：

  • 所有 Serializable 类必须显式声明 serialVersionUID
  • 使用 private static final long serialVersionUID = 1L; 固定版本号
  • 仅在无法兼容旧版本时修改 serialVersionUID
  • transient 关键字修饰的字段不会被序列化

19. Java 的五种 IO 模型是什么？

IO 模型 描述 特点
同步阻塞 IO (BIO) 数据未就绪时线程阻塞等待 简单但线程利用率低，每个连接一个线程
同步非阻塞 IO (NIO) 数据未就绪时立即返回，应用线程轮询 避免线程阻塞，但轮询消耗 CPU
IO 多路复用 select/epoll 监听多个 fd，就绪时通知 一个线程管理大量连接，Netty 核心模型

异步 IO (AIO) 发起调用后立即返回，内核完成数据拷贝后回调 真正的异步，JDK 7 引入，但实际应用较少

深入思考：

  • 对比延伸：Netty 基于 NIO（多路复用）而非 AIO，因为 Linux 的 AIO 实现不够成熟，epoll 配合 Reactor 模式已在生产中得到充分验证。Windows 的 IOCP 是真正的异步 IO。
  • 生产经验：Java NIO 的 Selector 存在空轮询 Bug（JDK 中著名的 epoll bug），Netty 通过自旋计数器 + rebuilt Selector 的方式规避。

21. ReadWriteLock 读写锁的原理是什么？

ReentrantReadWriteLock 实现读写分离，内部维护两个锁：

  • 读锁（共享锁）：多个线程可以同时持有
  • 写锁（独占锁）：只有一个线程可以持有

锁互斥规则：

当前持有 请求读锁 请求写锁
读锁 ✅ 允许 ❌ 阻塞
写锁 ❌ 阻塞 ❌ 阻塞
无锁 ✅ 允许 ✅ 允许

实现要点：

  • 读锁和写锁共享同一个 AQS 的 state 状态位，高 16 位记录读锁数量，低 16 位记录写锁重入次数
  • 写锁获取时需等待所有读锁释放（防止脏读）
  • 锁降级：写锁可以降级为读锁（先获取写锁，再获取读锁，再释放写锁），保证数据可见性

24. 类加载机制与双亲委派模型？

类加载过程：加载 → 验证 → 准备 → 解析 → 初始化

  • 加载：通过类的全限定名获取二进制字节流，将静态存储结构转为方法区的运行时数据结构，在堆中生成 Class 对象
  • 验证：确保字节流符合 JVM 规范，不危害 JVM 安全
  • 准备：为静态变量分配内存并设置零值
  • 解析：将符号引用转为直接引用
  • 初始化：执行类构造器 <clinit>() 方法

双亲委派模型：

每个类加载器收到类加载请求时，优先委托给父加载器，父加载器无法加载时再由子加载器尝试。

为什么需要双亲委派？

  • 保证 Java 核心类库的安全——java.lang.Object 永远由启动类加载器加载，不会出现多个版本
  • JVM 中类的唯一性由加载它的类加载器决定，双亲委派确保核心类全局唯一

打破双亲委派：

  • 继承 ClassLoader，重写 loadClass() 方法
  • 典型场景：Tomcat 为每个 Web 应用使用独立类加载器，实现应用隔离
  • SPI 机制（ServiceLoader）使用线程上下文类加载器，打破了双亲委派

参考：technology-talk — Tom哥，Apache-2.0

25. 对象的栈上分配与逃逸分析？

栈上分配：对于生命周期仅在线程内部的对象，在栈帧中分配内存，随方法返回自动销毁，无需 GC 介入。

逃逸分析：JVM 判断对象是否会被外部访问的优化技术：

  • 方法逃逸：对象被传递给其他方法
  • 线程逃逸：对象被其他线程访问
  • 未逃逸：对象仅在方法内部使用

逃逸分析带来的优化：

  1. 栈上分配：未逃逸对象分配在栈上，减少 GC 压力
  2. 标量替换：将对象拆解为多个基本类型变量，直接在栈上分配
  3. 锁消除：未逃逸对象的锁被消除，避免获取无用锁

27. 安全点、安全区域与 OopMap 的作用？

OopMap：记录栈帧中哪些位置保存了对象引用，用于 GC Roots 枚举。JVM 在安全点位置更新 OopMap，避免全栈扫描，帮助 HotSpot 实现准确式 GC。

安全点（Safepoint）：程序执行中的特殊位置（方法调用、循环跳转、异常抛出处），GC 时线程必须到达安全点才能暂停。HotSpot 采用主动式中断：设置标志位，线程轮询检查。

安全区域（Safe Region）：解决 SLEEP/BLOCKED 状态的线程无法主动到达安全点的问题。线程进入安全区域时标记自身状态，GC 时忽略该线程，从安全区域恢复执行时检查 GC 是否完成。

深入思考：

  • 生产经验：偏向锁撤销需要到达安全点，这是高并发场景下关闭偏向锁的原因之一。-XX:+PrintSafePointStatistics 可以输出安全点统计，辅助排查 STW 异常问题。
