0x00 Checkpoint 与状态

1. Flink Checkpoint 的 Chandy-Lamport 算法原理？

Flink 的 Checkpoint 基于 Chandy-Lamport 分布式快照 的变种（异步屏障快照）：

流程：

  1. JobManager 向所有 Source 注入 Checkpoint Barrier
  2. Barrier 随数据流向下游传播，将流分为”Barrier 之前的记录”和”之后的记录”
  3. 算子收到所有上游分区的 Barrier 后，触发状态快照（异步写入状态后端）
  4. 所有算子完成快照后，JobManager 确认 Checkpoint 完成

对齐（Alignment）：

深入思考：

  • 源码层面：CheckpointBarrierHandler 负责 Barrier 对齐逻辑。CheckpointedInputGate 在非对齐模式下直接转发数据而不缓存
  • 生产经验：Flink 1.15+ 默认启用非对齐 Checkpoint，大状态场景 Checkpoint 时间平均缩短 50%+

0x00 GIL 与并发

1. GIL 是什么？为什么 CPython 不删除它？

GIL（Global Interpreter Lock） 是 CPython 解释器级别的互斥锁，确保同一时刻只有一个线程执行 Python 字节码。

为什么不去掉 GIL？

  • GIL 让 CPython 的单线程性能极好（无需细粒度锁）
  • C 扩展库大量依赖 GIL 保证线程安全，去掉会破坏生态
  • Python 3.13 引入了实验性的 --disable-gil 模式，走了 PEP 703 路线（偏向锁 + 延迟引用计数）

绕过 GIL 的方式：

  • 多进程：multiprocessing，每个进程有独立 GIL
  • C 扩展：在 C 代码中手动释放 GIL（Py_BEGIN_ALLOW_THREADS）
  • 异步 IO：asyncio 单线程协程，不涉及 GIL 竞争

深入思考：

  • 边界情况：IO 密集型多线程不受 GIL 影响——线程在 socket.recv() 阻塞时会释放 GIL，其他线程可以继续执行

2. 协程 (coroutine) 和线程的区别？async/await 底层怎么工作？

维度 协程 线程
调度 用户态协作式（自己让出） OS 抢占式
切换成本 函数调用级别（微秒） 上下文切换（毫秒）
并发数 数万 数百
共享状态 无需锁（单线程） 需要锁
阻塞 await 不阻塞事件循环 系统调用阻塞 OS 线程

async/await 底层：

事件循环（Event Loop）是一个 while True，每次循环从就绪队列取出协程执行，遇到 await 就挂起当前协程、切换到下一个。

深入思考：

  • 源码层面：async def 编译后是一个 coroutine function，调用返回 coroutine object。await 等价于 yield from，底层通过 __await__ 协议实现
  • 生产经验：不要在协程里调用同步阻塞函数（如 time.sleep(1)）——换成 await asyncio.sleep(1)。否则整个事件循环被阻塞

0x01 内存管理

3. Python 的垃圾回收机制是怎样的？

三层回收：

  1. 引用计数（主）：每个对象维护 ob_refcnt，为 0 时立即释放
  1. 标记清除（辅）：解决循环引用
  1. 分代回收：新对象更容易变成垃圾

深入思考：

  • 边界情况：__del__ 方法会影响 GC——有 __del__ 的循环引用对象在 Python 3.4+ 仍会被回收，但顺序不确定
  • 生产经验：大内存应用用 gc.disable() + 手动 gc.collect() 可避免 GC 的 STW 暂停

4. Python 的 __slots__ 是什么？什么时候用？

使用场景：创建大量小对象（如数据类的数百万实例），__slots__ 可节省 50%+ 内存。

注意：__slots__ 不继承——子类需要重新声明，且和多重继承有兼容性问题。

后端面试题：Java 高级面试全解 — 高并发、JVM、设计模式

发表于 2026-06-07 分类于 后端

Java 高级面试全解，覆盖多线程与锁机制、JVM 内存与 GC 调优、经典设计模式实战。面向高级研发，侧重源码理解与架构设计。

0x00 多线程与锁

1. synchronized 锁升级过程是怎样的？

JDK 6 后 synchronized 引入了锁升级机制，性能大幅提升：

  • 偏向锁：第一个线程获得锁时，在对象头记录线程 ID，后续该线程再次进入无需 CAS
  • 轻量级锁：第二个线程竞争时，偏向锁撤销。线程通过 CAS 在对象头设置 Lock Record 指针。失败则自旋等待（自适应自旋）
  • 重量级锁：自旋超过一定次数（默认 10 次或 CPU 核数/2）仍未获取，膨胀为重量级锁。线程进入 BLOCKED 状态，由 OS 调度

深入思考：

  • 边界情况：偏向锁在竞争激烈的场景下反而是累赘——每次撤销偏向锁需要到达 SafePoint（STW）。高并发场景建议 -XX:-UseBiasedLocking 关闭偏向锁
  • 源码层面：锁状态存储在 Mark Word 的最低 3 位：001(无锁) → 101(偏向) → 00(轻量级) → 10(重量级)

参考：OpenJDK Synchronization

2. AQS (AbstractQueuedSynchronizer) 是怎么实现 ReentrantLock 的？

AQS 是 JUC 的核心基石：

ReentrantLock 公平锁实现：

非公平锁：去掉 hasQueuedPredecessors() 检查，新线程直接 CAS 抢锁，插队成功就获取

深入思考：

  • 性能差异：非公平锁吞吐量更高（减少线程切换），但可能导致队列尾部线程饥饿。默认 ReentrantLock() 是非公平的
  • 与 synchronized 对比：AQS 支持超时获取 (tryLock(timeout))、可中断获取 (lockInterruptibly)、条件变量 (Condition)——这些都是 synchronized 不具备的

3. CountDownLatch、CyclicBarrier、Semaphore 的区别？

工具 作用 可重用 核心方法
CountDownLatch 等待 N 个线程完成 否 countDown(), await()
CyclicBarrier N 个线程互相等待到齐 是 await()
Semaphore 控制并发访问数 是 acquire(), release()

0x02 线程池

4. ThreadPoolExecutor 的 7 个参数及工作流程？

工作流程：

四种拒绝策略：

  • AbortPolicy（默认）：抛 RejectedExecutionException
  • CallerRunsPolicy：由提交任务的线程自己执行（天然限流）
  • DiscardPolicy：静默丢弃
  • DiscardOldestPolicy：丢弃队列最老的任务，重试提交

深入思考：

  • 生产经验：不要用 Executors.newFixedThreadPool() 和 Executors.newCachedThreadPool()——前者无界队列可能 OOM，后者无限线程创建。必须用 new ThreadPoolExecutor() 显式指定参数
  • 大小公式：CPU 密集型 coreSize = CPU核数 + 1；IO 密集型 coreSize = CPU核数 × 2

5. 如何优雅关闭线程池？

关键区别：shutdown() 不中断正在执行的线程，shutdownNow() 调用 Thread.interrupt() 中断。被 shutdownNow() 中断的线程如果吞掉 InterruptedException 则不会停止。

0x03 JVM 内存模型

6. JVM 运行时数据区有哪些？各自存什么？

  • 堆：new 出来的对象和数组。GC 的主要战场，分年轻代(Eden+S0+S1)和老年代
  • 方法区（元空间）：类的结构信息（字段、方法、常量池）、JIT 编译后的代码缓存。JDK 8+ 用本地内存实现，不再有 -XX:MaxPermSize
  • 虚拟机栈：每个方法执行时创建栈帧，存局部变量表、操作数栈、方法返回地址。StackOverflowError 和它的关系最大
  • 本地方法栈：native 方法使用的栈
  • 程序计数器：当前线程执行的字节码行号

0x04 GC 调优

7. G1 GC 的核心原理是什么？什么时候选 G1？

G1 将堆划分为大小相等的 Region（默认 2048 个，每个 1-32MB）：

核心流程（Mixed GC）：

  1. 初始标记（STW，随 Young GC 一起）：标记 GC Roots 直接可达对象
  2. 并发标记：三色标记法，与用户线程并发
  3. 最终标记（STW，短）：处理并发期间的变化（SATB 快照）
  4. 筛选回收（STW）：选垃圾最多的 Region 回收（Garbage-First 的由来）

选型标准：

  • 堆 > 4GB → G1（CMS 在大堆下碎片化严重）
  • 延迟要求 < 200ms → G1（可设置 -XX:MaxGCPauseMillis）
  • JDK 9+ → G1 是默认 GC
  • 堆 < 4GB 且对吞吐量要求高 → Parallel GC

8. OOM 排查流程？

常见 OOM 原因：

  • 堆溢出：ThreadLocal 没 remove、集合类无限增长、缓存没上限
  • 元空间溢出：动态生成大量类（CGLIB 代理、Groovy 脚本）
  • 直接内存溢出：NIO 的 DirectByteBuffer 没释放（-XX:MaxDirectMemorySize）

0x00 Nacos — 服务发现与配置中心

1. Nacos 的 AP 和 CP 模式分别用于什么场景？切换方式是什么？

Nacos 是目前国内微服务体系的事实标准注册中心，同时支持 AP（临时实例） 和 CP（持久实例）。

AP 模式 — 对应服务发现，默认行为：

  • 实例通过心跳续约（5s），15s 无心跳则自动剔除
  • 节点间通过 Distro 协议异步同步注册信息
  • 类似 Eureka 的自保护：网络分区时优先保证注册和发现可用
  • 适合：无状态微服务，服务实例频繁上下线

CP 模式 — 对应配置管理和 DNS 场景：

  • 使用简化 Raft 协议，写操作必须经过 Leader
  • 网络分区时少数派拒绝写入，保证一致性
  • 适合：配置变更、服务元数据等需要强一致的场景

切换方式：

Nacos 2.x 后通信协议从 HTTP 升级为长连接 gRPC，单机可支撑 10w+ 实例。

参考：Nacos 官方架构文档

深入思考：

  • 边界情况：网络分区时，AP 模式下不同分区的节点各自独立接收注册，分区恢复后通过 Distro 协议做最终一致合并。如果同一服务在两个分区分别注册了不同实例列表，合并策略是「全量保留」而非「冲突裁决」——这可能导致消费者看到已下线的旧实例。生产上需要在客户端加健康检查兜底。

  • 源码层面：Nacos 的一致性协议切换核心在 ConsistencyService 接口，AP 实现为 DistroConsistencyServiceImpl，CP 实现为 RaftConsistencyServiceImpl。临时实例的 ephemeral=true 通过 Instance.isEphemeral() 判断，路由到不同的 ClientManager 处理。

  • 对比延伸：Consul 的 CP 模式基于完整 Raft（非简化版），支持多 DC 的 WAN Gossip。但 Consul 不支持 AP 模式切换——它要么 CP 要么什么都不给。Nacos 的灵活性在此占优，但也意味着使用者必须理解 CAP 才能正确配置。

2. Nacos 配置中心的热更新是如何实现的？

核心机制：

  1. 长轮询（1.x）：客户端发起 HTTP 请求到服务端，服务端持有连接 29.5s，期间配置有变更则立即返回，否则超时后返回 304 让客户端再次发起
  2. gRPC 长连接（2.x）：客户端与服务端建立双向流，服务端主动推送变更通知
  3. Spring 集成：@RefreshScope 或 @NacosValue 注解的 Bean 会在收到变更通知后重新初始化

注意：@RefreshScope 会为每个 Bean 创建代理，有性能开销，高频调用的 Bean 不建议使用。

参考：Nacos Spring Cloud 文档

3. Nacos 和 Eureka 的核心区别是什么？

维度 Nacos Eureka
CAP AP + CP 可切换 AP only
健康检查 TCP/HTTP/MySQL 多种 仅 HTTP 心跳
配置中心 内置 无（需集成 Spring Cloud Config）
通信协议 gRPC 长连接（2.x） HTTP 短连接
实例规模 10w+ / 节点 ~5000 / 节点
控制台 完善的管理 UI 基本的 Dashboard
维护状态 阿里持续活跃维护 Netflix 停止维护

一句话：Nacos = Eureka + Config + 管理面板，是 Eureka 的超集。

参考：Nacos vs Eureka vs Consul

0x01 Sentinel — 流量控制与熔断降级

4. Sentinel 的滑动窗口限流原理是什么？

Sentinel QPS 限流基于 滑动窗口计数器（LeapArray）：

  • 将 1 秒时间窗口切分为 N 个格子（默认 2 格，每格 500ms）
  • 每个格子维护一个 LongAdder 计数器
  • 请求到来时，遍历当前窗口内所有有效格子，累加计数
  • 超过阈值则触发限流

三种限流效果：

  • 快速失败：直接抛 FlowException
  • Warm Up：预热模式，阈值从 count/3 线性增长到 count
  • 排队等待：匀速排队（漏桶算法），请求超过超时时间才失败

参考：Sentinel Flow Control

深入思考：

  • 边界情况：默认 2 格（每格 500ms）的粒度对突发流量不敏感——如果前 500ms 已经用完所有配额，后 500ms 的请求全被拒绝，即使这 1 秒的整体 QPS 远低于阈值。实际应用中应根据流量特征调整 sampleCount。API 网关类建议 10-20 格，让限流更平滑。

  • 源码层面：LeapArray 的核心是 currentWindow() 方法——根据当前时间戳计算窗口下标，如果窗口过期则调用 resetWindow() 清零计数器。这里使用 LongAdder 而非 AtomicLong 是为了减少高并发下的 CAS 竞争。Warm Up 效果背后是 Guava 的 SmoothWarmingUp 算法——令牌桶的变种，冷启动时令牌生成速率从低到高逐渐增加。

  • 生产经验：很多人把 Sentinel 限流阈值设在 QPS=1000 就以为万事大吉。但如果下游单接口 RT=200ms，1000 QPS 意味着 200 个并发线程在等待——线程池瞬间打满。正确做法是 QPS 阈值 ≤ 线程池大小 ÷ 平均 RT，即 count ≤ threads / (rt/1000)。

5. Sentinel 的熔断降级有哪三种策略？各自适用什么场景？

策略 判断依据 适用场景
慢调用比例 慢调用数 / 总调用数 > 阈值 下游服务响应变慢（如 DB 慢查询）
异常比例 异常数 / 总调用数 > 阈值 下游偶发性故障
异常数 异常数绝对值 > 阈值 低流量接口（比例不敏感）

核心逻辑：Sentinel 熔断器是半开-全开-全关三段式，不是 Hystrix 的半开试探模式。熔断时间窗口结束后自动恢复，不需要单独的探测请求。

参考：Sentinel Circuit Breaking

6. Sentinel 和 Hystrix 的本质区别？

维度 Sentinel Hystrix
隔离策略 信号量隔离（无额外线程） 线程池隔离（额外线程开销大）
规则管理 控制台实时推送，动态热更新 静态配置，修改需重启
熔断策略 3 种（慢调用/异常比例/异常数） 1 种（异常比例）
流量整形 预热、匀速排队、冷启动 无
扩展性 SPI 插件机制 接口实现
维护状态 阿里开源活跃维护 Netflix 停止维护

选型建议：2026 年新项目优先 Sentinel。Hystrix 仅遗留系统还在跑。

0x02 Seata — 分布式事务

7. Seata 的四种事务模式分别是什么？如何选型？

模式 侵入性 性能 适用场景
AT 零侵入 中等 纯关系型数据库操作
TCC 高（需实现 Try/Confirm/Cancel） 高 涉及 Redis/MQ/第三方 RPC
SAGA 中（需实现正向+补偿） 高 长事务（分钟级）
XA 零侵入 低 强一致性要求（金融）

选型决策树：

  • 纯 DB 操作 → AT（前提：SQL 可被解析）
  • 涉及外部资源（Redis/MQ/RPC）→ TCC
  • 长事务（分钟级）→ SAGA 或最终一致性消息
  • 金融级强一致 → XA（但很少用）

参考：Seata 官方文档

8. Seata AT 模式的实现原理是什么？为什么需要全局锁？

AT 模式核心是自动补偿，两阶段实现：

一阶段：

  • Seata 拦截业务 SQL，解析出前后镜像存入 undo_log 表
  • 将业务 SQL + undo_log 在同一本地事务中提交
  • 注册分支事务到 TC（Transaction Coordinator）

二阶段：

  • 成功：TC 通知各分支删除 undo_log（异步，无阻塞）
  • 失败：TC 通知各分支根据 undo_log 反向生成回滚 SQL 并执行

全局锁的必要性：
在一阶段提交后、二阶段完成前，其他事务可能修改同一行数据。全局锁保证在全局事务未结束时，其他事务不能写入已被 AT 模式锁定的数据行，防止脏写。

脏写防护：如果全局锁获取失败 → 重试 → 超时则回滚本地事务并上报 TC。

参考：Seata AT Mode

深入思考：

  • 边界情况：AT 模式的前提是数据库支持 ACID 事务且 SQL 能被正确解析。以下场景 AT 会失效：(1) 使用了存储过程/函数——Seata 无法解析内部 SQL；(2) SELECT ... FOR UPDATE 行锁——可能与全局锁冲突；(3) 分库分表后的跨库事务——每个库的 undo_log 独立，回滚时无法保证原子性。遇到这些场景必须切换到 TCC 或 SAGA。

  • 源码层面：SQL 解析的核心是 SQLRecognizerFactory，通过 Druid 的 SQL Parser 解析出表名、WHERE 条件、更新字段。前后镜像的生成在 BaseTransactionalExecutor 中——先 SELECT * FROM table WHERE ... FOR UPDATE（加行锁+拿前镜像），执行业务 SQL，再 SELECT * FROM table WHERE ...（拿后镜像）。正是这个 FOR UPDATE 和全局锁的双重保障防止了脏写。

  • 对比延伸：AT 和 XA 都是两阶段，但本质不同。XA 的一阶段是 prepare（不提交），二阶段是 commit/rollback——数据库连接一直持有，资源锁定时间长。AT 的一阶段直接提交本地事务，只靠后镜像做回滚——连接的持有时间极短。这就是为什么 AT 的性能远好于 XA，代价是需要 undo_log 表和 SQL 解析能力。

10. RocketMQ 的事务消息是怎么实现的？

事务消息用于解决本地事务 + 消息发送的原子性问题。流程如下：

  1. 生产者发送 half 消息（prepare 状态，消费者不可见）
  2. 执行本地事务
  3. 根据本地事务结果：成功 → commit（消息可见），失败 → rollback（消息删除）
  4. Broker 定期回查生产者：如果 half 消息长时间未确认（默认 60s），回调 checkLocalTransaction() 确认状态

核心设计：half 消息不是真实消息，是占位符。只有 commit 后才对消费者可见。

参考：RocketMQ Transaction Message

0x04 Dubbo — RPC 通信

11. Dubbo 的服务调用流程是怎样的？

完整链路（以 Dubbo 3 + Nacos 为例）：

关键步骤：

  1. 代理层：Consumer 通过 @DubboReference 生成动态代理
  2. 路由层：从 Registry（Nacos）获取 Provider 列表，经过路由规则过滤
  3. 负载均衡：从候选列表中选择一个 Provider
  4. 协议层：序列化请求，通过 Netty 发送
  5. Filter 链：Provider 端依次执行 Filter（可插拔，用于监控/限流/鉴权）
  6. 反射调用：最终执行 Provider 实现

参考：Dubbo 官方文档

12. Dubbo 的负载均衡策略有哪些？

策略 原理 适用场景
Random（默认） 加权随机 通用，各节点配置相同
RoundRobin 加权轮询 请求处理时间均匀
LeastActive 最小活跃数（inflight 请求最少） 长连接、处理时间不均
ConsistentHash 一致性哈希（同参数请求总打到同一节点） 有状态服务、缓存亲和
ShortestResponse 最短响应时间（Dubbo 3.2+） 自适应，兼顾延迟和负载

0x04 安装 IK

支持两种形式的安装

1，以plugin方式安装

进入ES的bin路径，执行以下命令：
elasticsearch-plugin install https://github.com/medcl/elasticsearch-analysis-ik/releases/download/v6.1.3/elasticsearch-analysis-ik-6.1.3.zip

执行完毕，重启ES，ik安装完毕

2，手动安装

：注意版本号要求一致
1,在 https://github.com/medcl/elasticsearch-analysis-ik/releases下载这个版本的IK和Es的版本对应关系一致，本文使用es 6.1.3，下载后
2，解压到ES_HOME/plugins/ik 目录下面(直接包含一个conf文件夹和一堆.jar包)
3，重新启动ES
4，如果 看到try load config ….IK相关信息，说明启动完成后和安装IK插件完成。或者访问http://spark01:9200/_cat/plugins 出现node-1 analysis-ik 6.1.3即为完成安装

3，验证

书籍推荐

发表于 2019-08-13 更新于 2026-06-07 分类于 fortest

0x00 概述

初次推荐书籍，这里我看的书籍也不多，目前也在充实自己中，欢迎大家推荐和自荐~

补充，推荐的书籍，各大商城均有销售，价格起伏不定，请适时下手~

以下按照数据中转的流程推荐书籍~

0x01 书单

一、《深入理解Kafka核心设计与实践原理》 Kafka类书籍个人入门推荐5星

厮大著作的书基于Kafka2.0~ 此书基础扎实，适合小白入门，相关慨念颇多，涉及Kafka的架构设计和很多高级应用，读此一本计科了解Kafka的整体架构和理念，建议配合书上搭建源码环境进行学习，效果更佳~

二、《HBase不睡觉书》HBase类书籍个人入门推荐五星

杨曦大佬著作的书，适合小白和有一定基础的工程师入门，书中知识点比较琐碎，但是按照杨大佬的推荐，阅读此书时，需要有选择性的阅读，不适合从头开始看(如果你有基础)。看完确实对Hbase的架构有了一定的认识，还对Hbase的发展有了一定的了解~

三、《大数据之路：阿里巴巴大数据实践》数仓类书籍个人推荐超5星

这本介绍了阿里大数据的最佳实践，适合小白和数仓工程师阅读，书中案例结合阿里遇到的问题进行将结合阐述，在数据模型的设计和相关的平台体系建设都有相应的实践。很多大佬都强烈推荐这一本书,哈哈~
