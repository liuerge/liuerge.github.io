Kafka 高级面试全解，覆盖 ISR 机制、高性能设计、Exactly-Once 和常见生产问题。

0x00 核心机制

1. Kafka 为什么这么快？

四大设计：

  1. 顺序写磁盘：每个 Partition 是顺序 Append 的日志文件，顺序 IO 接近内存速度
  2. 零拷贝（Zero-Copy）：sendfile() 系统调用，数据从 PageCache 直接到网卡，不经过用户态
  3. PageCache：消息先写到 OS 的 PageCache，异步刷盘。读命中 PageCache 时完全不走磁盘
  4. 批量 + 压缩：Producer 批量发送（batch.size/linger.ms），Broker 端支持 gzip/snappy/lz4 压缩

深入思考：

  • 边界情况：linger.ms=0 时每条消息立即发送，吞吐量骤降。acks=0 最快但不保证送达，acks=all 最安全但延迟最高

2. Kafka 的 ISR 机制是什么？与 acks 的关系？

ISR（In-Sync Replicas）：与 Leader 保持同步的 Follower 集合。

replica.lag.time.max.ms（默认 30s）：Follower 超过这个时间没跟上 Leader，被踢出 ISR。

acks 行为 可靠性 延迟
0 不等待任何确认 可能丢 最低
1 Leader 写入即确认 Leader 宕机可能丢 中
all/-1 所有 ISR 写入才确认 最高 最高

深入思考：

  • 生产经验：如果 min.insync.replicas=2 但 ISR 只剩 Leader 一个，acks=all 的写入会一直等待。设 min.insync.replicas=2 时必须 replication.factor >= 2

  • 生产经验：OOM 排查第一步——看是 Execution Memory 还是 User Memory 溢出。(1) Execution OOM → 增加分区数、减少单分区数据量；(2) Storage OOM → 用 MEMORY_AND_DISK_SER 溢出到磁盘；(3) User Memory OOM → 减少 UDF 中创建的大对象

0x03 Spark SQL 优化

4. DataFrame 比 RDD 为什么快？

  1. Catalyst 优化器：逻辑计划 → 优化 → 物理计划 → 代码生成
  2. Tungsten 执行引擎：堆外内存 + 直接操作二进制数据
  3. 列式存储：只读需要的列，减少 IO 和内存
  4. Whole-Stage Code Generation：把多个算子编译成一个函数，减少虚函数调用

5. 数据倾斜怎么解决？

发现倾斜：某个 Task 处理时间远超其他（Spark UI 可直观看到）

分场景解决：

场景 方案
GroupBy 倾斜 两阶段聚合（加盐去盐）
Join 倾斜 广播小表（Broadcast Join）
大表 Join 大表倾斜 倾斜 key 单独处理

0x04 流处理

6. Structured Streaming 的 Exactly-Once 怎么保证？

Structured Streaming 通过 WAL（Write-Ahead Log）+ 幂等 Sink 保证端到端 Exactly-Once：

  1. Source：Kafka offset 存储在 Checkpoint
  2. State：状态通过 Checkpoint 持久化到 HDFS/S3
  3. Sink：foreachBatch 幂等写（如 upsert 到 MySQL）
